# Планування AI-рішення

Перетворити умову задачі на обґрунтований вибір класу моделі, план системи та маршрут через
супутні скіли. Результат — не «назва моделі», а рішення з обґрунтуванням, альтернативами,
ризиками і планом перевірки.

## Довідки

- [references/task-to-model.md](references/task-to-model.md) — процедура вибору класу моделі:
  ознаки задачі → клас (генеративні / CV / дерева / регресія / часові ряди / DL-архітектури),
  типові помилки маршрутизації.
- [references/skill-router.md](references/skill-router.md) — маршрутна карта «сигнал запиту →
  скіл → що читати»: інваріанти завершеності етапів, правила маршрутизації, типові помилки
  маршруту. Спільна для Claude і для ChatGPT-пакета.
- [references/handoff.md](references/handoff.md) — шаблон і правила HANDOFF.md для перенесення
  роботи між сесіями.

## Межі роботи

- Основний фокус плагіна — **генеративні моделі всіх видів** (LLM, VLM, трансформери, енкодери
  та їх поєднання), але клас моделі обирається чесно: якщо задача класична (табличні дані,
  прогноз ряду, проста класифікація) — рекомендувати дерева/регресію/класичні методи і пояснити,
  чому генеративна модель тут гірша (вартість, латентність, детермінізм, валідованість).
- Не вигадувати бенчмарки, ціни й ліміти моделей: каталоги (OpenRouter тощо) змінюються щотижня.
  Навчати процедури вибору; конкретні цифри — перевіряти в актуальному каталозі провайдера.
- Кожну рекомендацію позначати: усталена практика / компроміс із варіантами / експертне судження.

## Початковий контракт

Спочатку встановити або явно позначити як невідоме:

1. **Рішення**, яке підтримує система (не «використати LLM», а «що зміниться для користувача»).
2. **Вхід і вихід**: модальності (текст/зображення/таблиця/ряд/аудіо), формат відповіді,
   допустима недетермінованість.
3. **Обмеження**: латентність, бюджет (грн/токени/GPU), приватність даних, де має працювати
   (локально / хмара / браузер), потреба в поясненні рішень.
4. **Дані**: що є для навчання/донавчання/оцінювання; чи можлива взагалі перевірка якості.
5. **Інтеграції**: зовнішні інструменти (пошук, БД, API), потреба в пам'яті між сесіями.

Не блокувати роботу через невідомі: ставити щонайбільше 2–3 уточнення й лише там, де хибний
здогад знецінює всю рекомендацію; решту фіксувати як припущення з ризиком.

## Вибір класу моделі (крок 1, завжди)

Пройти процедуру з [references/task-to-model.md](references/task-to-model.md) і видати вердикт у
форматі:

1. **Рекомендований клас** + чому саме він (2–4 аргументи, прив'язані до умови задачі).
2. **Найсильніша альтернатива** + чому програла (це обов'язково: рекомендація без переможеної
   альтернативи — не обґрунтування).
3. **Ризики рекомендації** і як їх виміряти до повного впровадження (PoC-метрика).
4. Якщо клас — генеративний: чи потрібен агент (інструменти/цикл рішень) чи досить одного
   виклику моделі; це визначає маршрут далі.

## Оркестрація скілів (крок 2)

| Питання | Скіл |
|---|---|
| Яку саме модель узяти: створювати чи готову, яку з каталогу OpenRouter | `select-genai-models` |
| Написати чи полагодити промпт; чи варта техніка (CoT, few-shot) своєї ціни; бюджет контексту | `engineer-prompt-context` |
| ReAct / RAG / multi-agent / human-in-the-loop; MCP-інструменти; пам'ять; векторні БД | `design-agent-architecture` |
| Де і як запускати: локально, Docker, AWS / GCP / Azure | `deploy-ai-environments` |
| Як міряти якість і оптимізувати токени/час/вартість; експлуатація агента в проді | `evaluate-optimize-models` |
| ЧОМУ модель поводиться так: токенізація, увага, масштабування, сервінг, механіка LoRA | `explain-llm-internals` |
| Робочий приклад на Python + LangChain/LangGraph + Docker + React 19 | `build-ai-examples` |

Порядок за замовчуванням: клас моделі → модель → промпт/контекст → архітектура → оцінювання
(спроєктувати метрики **до** побудови) → розгортання → приклад. Промпт стоїть перед архітектурою
свідомо: дефект промпту, прийнятий за дефект архітектури, породжує агента, який нікому не був
потрібен. Оцінювання проєктується раніше за код: система без
метрики якості не вважається спроєктованою.

Розширена версія цієї таблиці — з інваріантом завершеності кожного етапу, сигналами запиту та
типовими помилками маршруту — у [references/skill-router.md](references/skill-router.md).
Маршрутизувати за **рішенням**, а не за іменником: «зроби RAG» — це запропоноване рішення, і
якщо збій поведінковий (формат, тон, невиконання інструкцій), пошук його не полагодить.

## Handoff між сесіями (обов'язково)

Робота над AI-рішенням майже завжди триває кілька сесій. Правила:

- **На початку сесії**: якщо в проєкті користувача є `HANDOFF.md` — прочитати повністю перед
  будь-якими діями; розбіжності між handoff і фактичним станом коду назвати вголос.
- **Наприкінці сесії** (або на прохання «зроби handoff»): створити/оновити `HANDOFF.md` у корені
  проєкту користувача за шаблоном з [references/handoff.md](references/handoff.md): що це за
  проєкт, поточний перевірений стан, що саме зроблено в цій сесії і чому, відкриті гілки,
  граблі середовища.
- Handoff описує **рішення та їх причини**, а не лише перелік файлів: наступна сесія має
  зрозуміти, чому вибір саме такий, щоб не переграти його наосліп.
- Handoff-файл — власність користувача: не переписувати мовчки його правки.

## Формат результату

1. Контракт задачі (з припущеннями та ризиками).
2. Вердикт про клас моделі (рекомендація / альтернатива / ризики / PoC-перевірка).
3. План системи по етапах із маршрутом через скіли.
4. Перелік рішень, що потребують підтвердження користувача.
5. За потреби — оновлений `HANDOFF.md`.
