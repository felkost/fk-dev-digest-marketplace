# Вибір генеративних моделей

Обрати конкретну модель (або поєднання моделей) під задачу — з обґрунтуванням, а не за звичкою
чи рейтингом. Клас моделі вже має бути визначений через `plan-ai-solution`; тут вирішується
«яку саме і звідки».

## Довідки

- [references/model-landscape.md](references/model-landscape.md) — карта видів: декодерні LLM,
  VLM, енкодери (embedding), encoder-decoder, дифузійні; що кожен вид уміє і де його межа.
- [references/build-vs-use.md](references/build-vs-use.md) — процедура «створювати чи взяти
  готову»: промптинг → few-shot → RAG → fine-tune → навчання з нуля, і коли кожен щабель
  виправданий.
- [references/openrouter.md](references/openrouter.md) — як користуватись каталогом OpenRouter:
  фільтри, порівняння, роутинг, fallback-ланцюжки, підводні камені.

## Обов'язкові правила

- **Не вигадувати цифри.** Контекстні вікна, ціни за токен, рейтинги — змінюються щотижня;
  показувати, *як* перевірити в актуальному каталозі, а не цитувати з пам'яті як факт. Якщо
  цифра потрібна для прикладу — позначати «перевірити в каталозі на момент рішення».
- **Порівнювати на своїх даних, не на чужих бенчмарках.** Публічний лідерборд — фільтр
  кандидатів, а не вердикт: фінальний вибір робиться на власному eval-наборі (див.
  `evaluate-optimize-models`).
- Кожна рекомендація — з переможеною альтернативою і причиною програшу.

## Процедура вибору

1. **Вимоги від задачі**: модальності входу/виходу, потрібна довжина контексту, структурованість
   виходу (JSON/tool calls), латентність, бюджет, приватність (хмарний API чи локальна вага).
2. **Щабель побудови** за [references/build-vs-use.md](references/build-vs-use.md): готова модель
   з промптом → few-shot → RAG → fine-tune → своя модель. Кожен наступний щабель — лише коли
   попередній виміряно недостатній.
3. **Шорт-лист із каталогу** (OpenRouter або провайдер напряму): 2–4 кандидати різних цінових
   класів; великий-розумний, середній-швидкий, дешевий-масовий.
4. **Мікро-eval на своїх даних**: 20–50 репрезентативних прикладів, однакові промпти, порівняти
   якість/латентність/вартість. Це головний крок — без нього вибір є вгадуванням.
5. **Поєднання моделей**, якщо одна не закриває задачу: енкодер для пошуку + LLM для синтезу;
   дешевий роутер + дорогі спеціалісти; маленька модель для чернетки + велика для перевірки;
   VLM для сприйняття + LLM для міркування. Явно назвати, який компонент за що відповідає.

## Типові помилки

- Брати найбільшу модель «про запас» — платити за интелект, який задача не використовує.
- Fine-tune замість RAG, коли проблема у **знаннях** (застарілі/відсутні факти), а не в
  **поведінці** (стиль, формат, домен-специфічна мова).
- Ігнорувати структурований вивід: якщо потрібен строгий JSON — перевіряти підтримку
  structured output / tool calling у кандидата, а не «просити чемно».
- Один провайдер без fallback: продумати деградацію (інша модель, кеш, черга) до інциденту.
