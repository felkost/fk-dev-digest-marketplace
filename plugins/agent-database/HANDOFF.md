## Що це за плагін

Агент `sql-mentor` + **7 скілів** для викладача курсу баз даних. Мова відповідей — українська;
код — мовою оригіналу; frontmatter скілів — англійська (вимога каталогу маркетплейсу).
Аудиторія — **викладач** (повні еталонні розв'язки, критерії оцінювання, типові помилки
студентів). Обов'язковий контракт виводу (7 блоків): середовище/діалект → аналіз → розв'язок →
пояснення → ризики → альтернативи → обмеження.

## Походження і джерело правди

Плагін зібрано **2026-07-19** із проєкту `sql_agent` (окрема тека поза цим репозиторієм), де
агент і скіли розроблялися й тестувалися ітеративно. Під час перенесення змінено рівно три речі:

1. `description` у frontmatter 7 `SKILL.md` і агента перекладено **англійською** (конвенція
   маркетплейсу: frontmatter читає каталог сайту й механізм добору скілів; тіло лишається
   українським).
2. Агент більше не посилається на хардкоджені шляхи `.claude/skills/...` — маршрутизація йде
   через **іменовані скіли** `agent-database:<skill>`, а довідники читаються з теки завантаженого
   скіла. У плагіні, який ставиться в кеш-теку, старі шляхи не резолвилися б.
3. Пакет для ChatGPT перероблено з 12 «плоских» knowledge-файлів на **один zip-архів**
   (`chatgpt/build_gpt_package.ps1` → `dist/agent_database_knowledge.zip`) — так само, як у
   сусідніх плагінах `eda-skills`, `ai-gen`, `agent-ml-interviewer`.

**Джерело правди відтепер — цей плагін.** Подальші поповнення знань (порції матеріалів від
замовника) вносяться сюди, у `skills/**/references/`. Копія в `sql_agent` лишається як історичний
знімок; тримати обидві синхронними ніхто не буде — не редагуй її, щоб не породити розбіжність.

## Ключові домовленості (не порушуй)

- **DDL/приклади проєктування схем/ETL — обов'язково для PostgreSQL + MySQL + ClickHouse**
  (вимога замовника); специфіку ClickHouse (MergeTree, ORDER BY, немає FK) коментувати завжди.
- Технічні приклади, які можливо, **перевіряти через `sqlite3`** (Bash) перед внесенням у знання;
  діалектне — звіряти з документацією/книгами, **не вигадувати**.
- Замовник наповнює знання **ітеративно, порціями** (книги/посилання/задачі) — кожну порцію
  аналізувати і «вбивати» у відповідні `references/`; недоступні джерела позначати.
- **Кураторський режим:** агент *пропонує* (📌 урок / 📒 задача), людина *схвалює*, асистент
  *вносить*. Фактичні помилки виправляються одразу без схвалення — це не нові уроки, а дефекти.
- Секрети в прикладах — з env/секрет-менеджера; нічого не хардкодити.
- **Чужу незакомічену роботу в репозиторії не чіпати.** Під час цієї інтеграції в робочій копії
  паралельно лежали правки `eda-skills` (`tests/*.py`, `imbalance-probes.md`, `plan-eda-dataset/SKILL.md`)
  і локальний `.claude/launch.json` — стейджити файли **поіменно**, а не `git add -A`.
- Після правок довідників — **перезібрати** ChatGPT-архів (`chatgpt/build_gpt_package.ps1`) і
  перевірити, що інструкція лишилась у межах 8000 UTF-8 **байтів**.

## Гейти CI і граблі, на які вже наступали

Перед пушем ганяй **усі чотири** з кореня маркетплейсу — вони перевіряють різні речі, і
проходження одного нічого не каже про інші:

```bash
npm run lint                                       # validate-marketplace.mjs
npx markdownlint-cli2 "**/*.md" "#**/node_modules"
npm run build:catalog                              # dist/index.json + site/public/catalog.json
cd evals && npm run eval:quality                   # gray-matter + метрики якості скілів
```

Три пастки, кожна з яких **пройшла повз локальні перевірки** й спливла лише в CI або пізніше:

1. **Двокрапка з пробілом у нелапкованому `description:` валить `gray-matter`** → `eval:quality` FAIL. У YAML
   двокрапка з пробілом усередині plain-скаляра невалідна. Підступність у тому, що **сам Claude
   Code і `build-index.mjs` таке з'їдають** (у `build-index.mjs` власний наївний парсер
   frontmatter), тому ні агент, ні каталог сайту помилки не показують — падає лише строгий
   YAML-парсер в evals. Симптом заплутує ще й тим, що вказує на колонку (у нас 635), через що
   легко піти хибним слідом «завелика довжина». Довжина ні до чого: мінімальний
   `description: Explains the chain: db to model.` падає так само.
   **Правило: якщо в описі є двокрапка з пробілом — бери значення в подвійні лапки** (як
   `engineering-paved-path/skills/{fastify-best-practices,frontend-architecture}`). Лапки знімає
   `build-index.mjs`, тож `catalog.json` побайтово не змінюється.
   Швидка перевірка всіх файлів одразу:
   ```bash
   cd evals && node -e 'const m=require("gray-matter"),fs=require("fs"),{execSync}=require("child_process");
   for(const f of execSync("find ../plugins -name SKILL.md -o -path \"*/agents/*.md\"",{encoding:"utf8"}).trim().split("\n"))
     try{m(fs.readFileSync(f,"utf8"))}catch(e){console.log("THROW",f,e.message.split("\n")[0])}'
   ```
2. **`markdownlint` MD029** на нумерованих списках із код-блоками: якщо код-фенс стоїть на
   нульовому відступі, список розривається й нумерація «скидається». Лік — **відступ код-фенсу на
   3 пробіли**, щоб він належав пункту списку (виправляли в `task-bank.md`).
3. **Ліміт Instructions у ChatGPT рахується в БАЙТАХ, не символах** — кирилиця 2 байти/літеру.
   Старий пакет у `sql_agent` мав 8671 байт при ліміті 8000 і ChatGPT відхилив би його, хоча
   «5452 символи» виглядали безпечно. `chatgpt/build_gpt_package.ps1` міряє байти й друкує
   `OK`/`Warning` — дивись на це щоразу після правки `gpt_instructions.md`.

**Чого CI НЕ перевіряє:** `WARN: no eval file` є для **всіх 58 скілів** усіх плагінів (не лише
наших) — це не регресія й білд не валить. Написання eval-файлів — окрема незроблена задача
маркетплейсу, не борг цього плагіна.

**Два фактичні дефекти, знайдені тестами й виправлені одразу** (показові приклади того, чому
тестовий прогін окупається):

1. `vector-databases.md` §9 радив чанки 512–1024 токени й одночасно модель `all-MiniLM-L6-v2`,

   FTS-конфігурації `'ukrainian'`**.
2. `lakehouse-delta-glue.md` §4 давав лише легасі-синтаксис `APPLY CHANGES INTO` і стару назву
   продукту. Актуально: **`CREATE FLOW … AS AUTO CDC INTO`**, продукт — **Lakeflow Declarative
   Pipelines** (кол. Delta Live Tables/DLT).

## Прямий тест `lakehouse-delta-glue` (2026-07-19)

Останній довідник без **власного** прямого тесту нарешті прогнано: задача «lakehouse доставки
їжі» (S3-інджест зі зміною схеми + CDC статусів + SCD2 по курʼєрах + дві вітрини, одна ковзна).
Агент відпрацював за контрактом, підтягнув 6 довідників, зафіксував припущення таблицею,
звірився з документацією Databricks/Spark і чесно позначив, що код не прогнано на кластері.

**Фактичних дефектів не знайдено.** Результат — 3 «📌 Урок-кандидат» + 1 «📒 Запис у банк задач»;
усі чотири **схвалено замовником і внесено того ж дня**:

1. **As-of join на SCD2-вимір: анатомія предиката** → нова **§6c** у
   `analyze-sql-examples/references/fundamentals-and-gotchas.md` (підсекція §6 про JOIN — так
   уникнули перенумерації 21 секції та змісту). Чотири тихі точки відмови: `is_current = TRUE`
   замість темпорального предиката (звіт мутує заднім числом); `BETWEEN` замість напіввідкритого
   `[valid_from, valid_to)` (подвійний матч на межі версій); `NULL` як відкрита межа + наївне
   порівняння (трилогіка викидає поточну версію); `INNER JOIN` при `NULL` у FK факту. Плюс
   інваріант «після join на вимір `COUNT(*)` факту не змінюється» і перевірка перекриття версій.
   **Усі чотири режими відмови заміряно в `sqlite3`** — числа в таблиці довідника: еталон 4 рядки,
   `BETWEEN` → 5 (дубль на межі), `INNER` → 3 (зник факт із `NULL` у FK), а `is_current` і
   пропущений `COALESCE` **не змінюють кількості рядків** і ловляться лише звіркою значень.
   Крос-діалектне (додано ClickHouse `ASOF LEFT JOIN`), тому в загальний каталог граблів.
2. **Формула вікна перерахунку інкрементної вітрини** → нова підсекція в
   `build-data-projects/references/etl-pipeline-blueprint.md` («Ідемпотентне інкрементне
   оновлення»). `lookback = макс. затримка джерела + макс. тривалість життєвого циклу + ширина
   бізнес-вікна`; третій доданок забувають, а він множить радіус ураження в N разів для ковзної
   N-денної метрики. Внесено разом із двома суміжними правилами: метрики-відношення зберігати
   чисельником+знаменником (не адитивні), «незрілі» останні періоди позначати прапорцем.
3. **`addNewColumns` падає за задумом** → нова підсекція §3 у `lakehouse-delta-glue.md` (таблиця
   п'яти режимів + продакшн-рецепт `rescue` + `EXPECT (_rescued_data IS NULL)` + чому
   `schemaHints` не косметика) і **три нові граблі §7 (16–18)**: еволюція схеми, радіус ковзного
   вікна, односкладовий `SEQUENCE BY`. Граблів стало 18.
4. **📒 Задача №8** «Lakehouse доставки їжі: SCD2 з перезаписуваного джерела, as-of join, ковзна
   вітрина» — **банк тепер 8 кейсів**. Базову механіку medallion не дублює, а посилається на №6;
   несе три осі, яких там немає: вимір із джерела **без історії**, темпоральний join, ковзне вікно
   з радіусом ×7 (+ `TRACK HISTORY ON` проти monster dimension, `RANGE` vs `ROWS`, порт на Glue).

Побічно (щоб уроки не лишились ізольованими) розширено `design-dwh-etl/references/dimensional-modeling.md`
§SCD трьома пунктами: тип фіксується **по атрибуту** (monster dimension), перехресне посилання на
§6c із зауваженням, що сентинел `'9999-12-31'` робить пастку з `NULL` неможливою конструктивно, і
«перш ніж проєктувати SCD2 — спитай, звідки візьметься історія».

## Порція «ClickHouse + Cassandra best practices» (2026-07-19, гілка `feat/db`)

Замовник дав 6 посилань (офіційні ClickHouse Best Practices і Operations→Usage Recommendations;
Apache Cassandra Data Modeling; DataStax CQL best practices; Redgate Simple Talk). Усі шість
доступні й опрацьовані. Це **поповнення, а не виправлення**: фактичних дефектів у наявних
довідниках звірка не виявила, тож нічого не спростовано — лише додано.

**`analyze-sql-examples/references/clickhouse.md`** (67 → ~200 рядків), нові секції:

- **Як обирати `ORDER BY`** — 5 правил (частота фільтрації → кореляція → 4–5 стовпців →
  спершу низька кардинальність → огрублення типу `toDate()`), і те, що ключ **незмінний після
  створення таблиці**.
- **Ключ партиціювання** — офіційна теза «це техніка керування даними, **а не** оптимізації
  запитів»; орієнтир **< 100–1000 різних значень**; парти **не зливаються між партиціями**;
  `max_parts_in_total` / `parts_to_throw_insert` → `Too many parts`.
- **Стратегія вставки** — батчі **1 000 мін / 10k–100k опт**, ~1 `INSERT`/с; sync vs async;
  таблиця дефолтів `async_insert_*` (100 MiB / 200 мс / 450); і головне — **async вимикає
  дедуплікацію**, тобто «async + at-least-once продюсер» = дублікати.
- **Мутації та `OPTIMIZE FINAL`** — окрема секція. Мутація переписує **цілі парти**, не
  відкочується, переживає рестарт (`KILL MUTATION`). `OPTIMIZE FINAL` ігнорує запобіжник
  **~150 GB** (`max_bytes_to_merge_at_max_space_in_pool`) → роздутий парт далі не зливається →
  **у `ReplacingMergeTree` дублікати накопичуються саме через те, що їх «прибирали»**.
- **JOIN** — порядок таблиць (менша праворуч; **з 24.12 планувальник робить це сам**), таблиця
  п'яти алгоритмів, `join_algorithm` = `direct,parallel_hash,hash`, пролив на диск.
- **Тиха пастка словників** — словник мовчки лишає **лише останнє** значення ключа, тож
  придатний тільки для 1:1 і N:1; підміна `JOIN` словником на 1:N — тиха втрата рядків.
- **Типи** — `LowCardinality` до **~10 000** унікальних; `Enum8/16`; `FixedString` лише для
  фіксованої довжини; `Date`/`DateTime` замість `DateTime64`; ціна `Nullable`.
- **Індекси пропуску** (нове) — minmax/set(N)/bloom_filter/text; **`tokenbf_v1` і `ngrambf_v1`
  застарілі**; умова корисності — кореляція з первинним ключем; `EXPLAIN indexes = 1`.
- **Матеріалізовані подання** — інкрементні vs `REFRESHABLE` (`REPLACE`/`APPEND`), правило
  розкладу «не частіше, ніж запит рахується».
- **JSON-тип** (нове) — коли доречний, коли `String`; `max_dynamic_paths` < 10 000.
- **Експлуатація та залізо** (нове) — governor `performance`, THP `madvise`, ext4+`noatime`,
  RAM 32/128 GB, режим для <16 GB, RAID, мережа 10 Gbps, Keeper > ZooKeeper, антивірус.

**`analyze-sql-examples/references/cassandra-cql.md`** (79 → ~200 рядків), нові секції:

- **Методологія проєктування** — 5 кроків (концептуальна → **запити** → логічна → фізична →
  оцінка); конвенція назв **`<сутність>_by_<атрибут>`**; дві цілі, що тягнуть у різні боки
  (рівномірний розподіл ↔ мінімум партицій на запит), і те, що дизайн — це вибір точки між ними.
- **Нотація Чеботка** — K / C↑C↓ / S.
- **Розрахунок розміру партиції** — формула комірок **`Nv = Nr·(Nc − Npk − Ns) + Ns`**, формула
  розміру на диску, метадані комірки ≈ **8 байт**; межі: жорсткий ліміт **2 млрд комірок**,
  практична ціль **< 100 MB**. Це найцінніше з порції: дає перевірюваний критерій замість «на око».
- **Бакетування** — як розбивати завеликі партиції (`PRIMARY KEY ((hotel_id, month), date)`),
  чому доба часто надто дрібно, а рік надто велико.
- **Практики запису й читання** — **`BATCH` не прискорює** (мультипартиційний зазвичай
  повільніший за окремі async-запити; атомарність без ізоляції; `batch_size_fail_threshold_in_kb`);
  prepared statements і **`NULL` у бінді створює tombstone** (треба `unset`); пагінація через
  paging state; `LOCAL_QUORUM` і `R + W > RF`.
- **Індекси й MV** — **SAI** рекомендований у 5.0, але таблиця-під-запит лишається основним
  способом; MV — свідоме рішення з ціною, не дефолт.
- **Джерела tombstone** — не лише `DELETE`: `NULL`, заміна колекції цілком, TTL, оновлення MV.
- **BLOB-и** — в об'єктне сховище/CDN, у таблиці лише URI.
- **Коли Cassandra не підходить** — ACID/JOIN, ad-hoc аналітика, **черга повідомлень**
  (лавина tombstone саме там, де читають), важке конкурентне читання з довільними фільтрами.

Дрібне: два пункти в чек-лист `analyze-sql-examples/SKILL.md` (ClickHouse / Cassandra) і уточнення
рядка про індекси Cassandra в `db-connectivity-cloud/references/db-families-cap.md`.

### Тестовий прогін `@sql-mentor` (той самий день, до мерджу)

Прогнано **дві** реалістичні задачі, кожна навмисне сконструйована так, щоб зачепити саме нові
секції. Обидві — «поганий прод, знайди що не так», бо це формат, у якому дефекти довідника
спливають найшвидше.

| Задача | Що перевіряла |
|---|---|
| ClickHouse: 300 млн подій/добу, вставка по одному рядку, нічний `OPTIMIZE FINAL`, мутації, повільний звіт, `Too many parts` | ключ сортування, ключ партиціювання, стратегія вставки, мутації, `OPTIMIZE FINAL`, типи |
| Cassandra 5.0: бронювання готелів, 4 запити; колега пропонує одну таблицю + вторинні індекси + `ALLOW FILTERING`, бо «у 5.0 є SAI» | методологія, розрахунок партиції, бакетування, межі SAI |

**Знайдено й одразу виправлено 2 фактичні дефекти** (це дефекти, не уроки — схвалення не потребують):

1. **`clickhouse.md` § «Стратегія вставки» стверджував неправду про ідемпотентність.** Було:
   «синхронна вставка дає ідемпотентні ретраї, MergeTree дедуплікує ідентичний батч, повтор після
   таймауту безпечний». Насправді дедуплікація вставок увімкнена за замовчуванням **лише для
   `Replicated*MergeTree`**; для звичайного `MergeTree` це окреме налаштування
   `non_replicated_deduplication_window`, і воно **вимкнене за замовчуванням** (зворотна
   сумісність). Ціна помилки практична: на одновузловій таблиці ретрай **тихо дублює батч** —
   помилки немає, є завищені суми. Звірено з офіційною документацією ClickHouse
   («Deduplicating inserts on retries») і Altinity KB. Замінено на розгорнуту засторогу з
   переліком того, що робити натомість.
2. **`cassandra-cql.md` § «Практики запису» називав параметр іменем до 4.1.** Було
   `batch_size_fail_threshold_in_kb`; після стандартизації конфігурації у **4.1** одиниці переїхали
   з імені у значення: **`batch_size_fail_threshold: 50KiB`**, попередження —
   `batch_size_warn_threshold: 5KiB`. Старі імена ще приймаються, але в дефолтному `cassandra.yaml`
   від 5.0 їх немає — а довідник орієнтований саме на 5.0. Додано обидва імені й дефолти.

Плюс косметика: у таблиці алгоритмів JOIN рядок `parallel_hash` був підписаний «дефолт», хоча
дефолт — це список із трьох із порядком проб `direct → parallel_hash → hash`.

**Решта звірки — чисто.** Обидва агенти прицільно перевіряли по офіційних джерелах ключові числа,
і всі підтвердилися дослівно: пороги `async_insert_*` (100 MiB / 200 мс / 450), ~150 GB у
`max_bytes_to_merge_at_max_space_in_pool`, дефолт `join_algorithm`, deprecated-статус
`tokenbf_v1`/`ngrambf_v1`, `LowCardinality` ~10 000, «< 100–1000» для ключа партиціювання, обидві
формули Cassandra (`Nv` і `St`) — **дослівний збіг з документацією**, ліміт 2 млрд комірок,
~8 байт на комірку, `NULL`→tombstone, статус SAI у 5.0.

**Що прогін підтвердив як найкорисніше:** секція про `OPTIMIZE FINAL` (парадокс «дублікати
накопичуються через чистку» дав найсильніший висновок усього розбору) і рядок про межі SAI —
він виявився **кориснішим за первинне джерело**, бо офіційна документація SAI сформульована
м'якше, а довідник прямо називає межу застосування.

**Прогалини, виявлені прогоном** (не помилки — просто не покрито; винесено в наступні кроки):

- `clickhouse.md`: **семантика `ReplacingMergeTree` розкрита на чверть** — найбільша діра. Не
  сказано, що ключ дедуплікації це `ORDER BY` (а не `PRIMARY KEY`), що дедуплікація обмежена
  партицією і що колонка версії всередині `ORDER BY` вироджується. Також немає `PROJECTION`,
  `REPLACE PARTITION`, `Decimal` для грошей, числа `parts_to_throw_insert`, і не сказано, що
  інкрементне MV рахує рядок **до** дедуплікації (вітрина тихо розходиться з фактами).
- `cassandra-cql.md`: не згадано, що **`counter` не підтримує TTL** (найдорожча прогалина — на
  задачі з ретенцією це визначає структуру ключа); немає правила «~100 тис. рядків на партицію» як
  проксі до 100 MB; не сказано, що **умовні операції працюють лише в межах однієї партиції**
  (а це впливає на вибір partition key); секція бакетування дає гранулярність шаблоном
  («місяць/тиждень»), хоча на реальних числах місяць дав 214 MB — удвічі над ціллю.

### Кураторський цикл: схвалено й внесено того ж дня

Замовник схвалив **усі п'ять** пропозицій прогону, тож вони внесені (це і є штатний цикл:
агент пропонує → людина схвалює → асистент вносить).

**3 уроки:**

1. **Семантика `ReplacingMergeTree`** → нова підсекція в `clickhouse.md` § «Родина MergeTree»,
   чотири пункти: ключ дедуплікації — це **`ORDER BY`, а не `PRIMARY KEY`** (звірено дослівно:
   «uniqueness of rows is determined by the `ORDER BY` table section, not `PRIMARY KEY`»), і звідси
   симетрична пара помилок — заширокий ключ не склеює ретраї, **завузький склеює різні події, і це
   втрата даних**; колонка версії **не сміє входити в `ORDER BY`**, інакше вона стала всередині
   групи й правило «максимальна версія» вироджується; дедуплікація лише при злитті, а злиття не
   перетинають межу партиції → партиціювати за часом **події**, не приймання; до злиття дублікати
   у вибірці присутні, і **інкрементне MV рахує рядок ДО дедуплікації** — вітрина завищується
   назавжди. Плюс `is_deleted` (лише разом із `ver`; версію ClickHouse, у якій з'явився, свідомо
   не називаю — документація її не дає).
2. **`Nullable` → `DEFAULT 0` не є семантично нейтральною** → врізка в `clickhouse.md`
   § «Вибір типів даних». `SUM` не змінюється (тому й не помічають), а `AVG`/`COUNT(col)` — так.
   Перехресне посилання на `fundamentals-and-gotchas.md` §3.
3. **Бакет не можна робити дрібнішим за одиницю атомарної операції** → доповнення до
   `cassandra-cql.md` § «Бакетування», з робочим `BEGIN BATCH … IF … APPLY BATCH`. LWT працює лише
   в межах однієї партиції, тож перенесення `date` у partition key **тихо ламає атомарність
   бронювання** через межу бакета. Заодно секція перестала подавати гранулярність шаблоном:
   на 50 тис. подій/добу «типовий» місяць дає 214 MB — удвічі над ціллю.

**2 записи в банк задач** (`task-bank.md`, тепер **10 кейсів**): №9 ClickHouse-операційка,
№10 моделювання Cassandra з розрахунком партицій. Обидві теми в банку були відсутні повністю.

**Заодно закрито дві прогалини, без яких нові записи були б неузгоджені з довідниками:**
`counter` **не підтримує TTL** (Cassandra відхиляє `USING TTL`) — тому в counter-таблиці бакет
потрібен як механізм життєвого циклу навіть тоді, коли за розміром не потрібен; і правило
«~100 тис. рядків на партицію» назване тим, чим воно є, — **дешевим проксі до 100 MB**, який за
конфлікту з прямим розрахунком байтів програє.

**Найближче — те, заради чого плагін і робився:** замовник надає нові порції матеріалів

нові уроки винести на схвалення → перезібрати ChatGPT-архів → прогнати 4 гейти CI.

Решта, за запитом:

- **Залишок прогалин прогону ClickHouse/Cassandra** (великі — `ReplacingMergeTree` і TTL на
  `counter` — уже закриті разом зі схваленими уроками). Лишилися дрібніші, всі по `clickhouse.md`:
