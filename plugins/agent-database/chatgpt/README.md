# agent-database у Custom GPT (ChatGPT)

Той самий викладач-аналітик баз даних (умова → діалект → еталонний розв'язок → ризики →
критерії оцінювання), що й у Claude, запакований для **Custom GPT** у ChatGPT. Інші способи
розгортання — у кореневому [README.md](../README.md).

Пакет — це два файли, що йдуть у пару:

- **[gpt_instructions.md](gpt_instructions.md)** — вставляється в поле «Instructions» Custom
  GPT (стиснута інструкція: bootstrap, маршрутизація на 7 скілів, контракт виводу, обов'язкові
  правила, кураторський режим, обмеження ChatGPT-версії).
- **`dist/agent_database_knowledge.zip`** — методологічний зміст (усі 7 `SKILL.md` +
  43 `references/*.md`), який GPT розпаковує собі через Code Interpreter і читає на вимогу.
  Будується скриптом **[build_gpt_package.ps1](build_gpt_package.ps1)**.

Чому саме так: поле Instructions має практичний ліміт **8000 UTF-8 байтів** (перевірте живий
ліміт у редакторі GPT на момент встановлення), тому Instructions несуть лише **процедуру**
(що робити і в якому порядку), а Knowledge-архів — **зміст**, який GPT читає вибірково.

> **Ліміт рахується в БАЙТАХ, не в символах** — для української інструкції це принципово:
> кирилиця займає 2 байти на літеру, тож текст на 5 000 символів це вже ~8 000 байтів.
> `build_gpt_package.ps1` міряє саме байти (`UTF8.GetByteCount`) і друкує поточне значення
> проти ліміту — перед вставкою в ChatGPT переконайтеся, що бачите `OK`.

## Встановлення (перший раз)

1. **Зібрати пакет** (шлях залежить від того, де ви зараз):

   ```
   # якщо ви в корені плагіна agent-database\:
   powershell -ExecutionPolicy Bypass -File chatgpt\build_gpt_package.ps1

   # якщо ви вже всередині agent-database\chatgpt\:
   powershell -ExecutionPolicy Bypass -File build_gpt_package.ps1
   ```

   Скрипт збирає `dist/agent_database_knowledge.zip` із 7 скілів і друкує довжину
   `gpt_instructions.md` у UTF-8 байтах проти ліміту 8000. Якщо `Warning` — інструкцію треба
   скоротити **до** вставки в ChatGPT, інакше він її відхилить.

2. **ChatGPT → Explore GPTs → Create → вкладка Configure** (не Create-чат-режим).
   Створення GPT доступне на платних планах (Plus/Team/Enterprise).

3. **Name / Description** — довільні, напр. «Database Mentor» / «Викладач-аналітик баз даних:
   розбір умов, рецензія SQL/CQL, SQLite і MongoDB, DWH, ETL, BI».

4. **Instructions** — відкрити [gpt_instructions.md](gpt_instructions.md), скопіювати весь
   текст, вставити в поле Instructions.

5. **Capabilities** — увімкнути **Code Interpreter & Data Analysis** (обов'язково: без нього
   GPT не розпакує архів і не зможе прогнати перевірку прикладу в `sqlite3`). **Web Browsing**
   — увімкнути теж: агент звірятиме синтаксис із чинною документацією замість цитування з
   пам'яті. DALL·E не потрібен.

6. **Knowledge** — завантажити `dist/agent_database_knowledge.zip`.

7. **Conversation starters** — чотири підказки, що вмикають різні гілки маршрутизації:

   ```
   Розбери умову завдання й дай еталонний розв'язок для PostgreSQL
   Проаналізуй цей SQL-запит: знайди помилки, ризики та кращий варіант
   Поясни, як зібрати агрегацію в MongoDB (aggregation pipeline) з нуля
   Спроєктуй DWH під цей бізнес-процес: факти, виміри, DDL для PG/MySQL/ClickHouse
   ```

8. **Save → Create** (доступ: *Only me* / *Anyone with the link* / *GPT Store*).

9. **Перевірка:** у новому чаті написати «Розпакуй knowledge і перелічи файли в
   `analyze-sql-examples/references`». GPT має відповісти списком `.md`-файлів (там їх 10).
   Якщо каже, що не бачить архів — переконатися, що Code Interpreter увімкнено і файл справді
   завантажений у Knowledge.

## Як формулювати завдання

Мінімум, щоб почати — три речення: (1) що треба зробити (розібрати умову / порецензувати
запит / спроєктувати схему), (2) цільова СУБД або діалект, (3) вхідні дані — схема таблиць,
приклад рядків, обсяги. Якщо діалект не назвати, GPT обере найдоречніший і **позначить це
припущенням** — але краще назвати.

**Стартові фрази за метою:**

| Мета | Фраза |
|---|---|
| Розбір умови | «Розбери умову й дай еталонний розв'язок для [СУБД]: …» |
| Рецензія запиту | «Знайди помилки в цьому запиті ([діалект]): …» |
| Крос-діалектність | «Як зробити top-N у кожній групі в MySQL, PostgreSQL і ClickHouse?» |
| SQLite / MongoDB | «Поясни з нуля, як зібрати цей aggregation pipeline» |
| Під'єднання | «Як під'єднатися до ClickHouse з Python і залити CSV?» |
| Вибір типу БД | «Яка БД під [кейс]: реляційна, документна, колонкова чи векторна? Обґрунтуй» |
| Векторний пошук | «Спроєктуй семантичний пошук на pgvector із гібридом BM25 + RRF» |
| ETL / lakehouse | «Побудуй medallion на Delta Lake з інкрементним інджестом і CDC» |
| Проєктування DWH | «Спроєктуй зірку під [бізнес-процес]: факти, виміри, грануляція, SCD» |
| Якість даних | «Які DQ-перевірки поставити на цю таблицю і які пороги?» |
| Power BI / Tableau | «Міра показує не ті числа на зрізі місяців — знайди причину» |

## Оновлення (коли скіли зміняться тут, у Claude)

### A. Змінився лише вміст `references/*.md`

Instructions **не чіпати**. Зібрати пакет заново (див. команду вище) → у ChatGPT → редагувати
GPT → Knowledge → видалити старий `agent_database_knowledge.zip` → завантажити новий → Save.

### B. Змінився сам `SKILL.md` або склад скілів

Синхронізувати й **gpt_instructions.md**. Таблиця відповідності:

| Змінили в плагіні | Правити в `gpt_instructions.md` |
|---|---|
| Контракт виводу в `agents/sql-mentor.md` | «Контракт виводу» |
| Принципи якості в `agents/sql-mentor.md` | «Обов'язкові правила» |
| Кураторський режим (формат 📌/📒) | «Кураторський режим» |
| Новий/перейменований скіл | «Сім скілів» + список `$skills` у `build_gpt_package.ps1` |
| Новий діалект у `analyze-sql-examples` | перелік діалектів під таблицею маршрутизації |
| Новий референс у наявному скілі | Зазвичай **нічого правити** — GPT читає `SKILL.md` розпакованого архіву, а там перелік довідників |

Після правки `gpt_instructions.md`: перезапустити `build_gpt_package.ps1` (переконатися в
`OK`), вставити оновлений текст в Instructions, замінити Knowledge-архів як у випадку A.

## Відомі обмеження цієї версії

- **Пісочниця Code Interpreter скидається між розмовами** — архів доводиться розпаковувати в
  кожній новій розмові, і ніщо, записане GPT під час чату, не переживає її.
- **Немає скілів і субагентів** — маршрутизація зведена до таблиці в Instructions, усе
  послідовно.
- **Кураторський режим працює як пропозиції**: GPT видає «📌 Урок-кандидат» / «📒 Запис у банк
  задач», але власні файли не редагує — ви вносите правку в довідник тут, у плагіні, і
  перезбираєте архів. Це та сама межа, що й у Claude-версії, і вона навмисна.
- **Перевірка прикладів — лише SQLite** (модуль `sqlite3` у Code Interpreter). Для решти СУБД
  приклади ілюстративні; без увімкненого Web browsing GPT не звірить синтаксис із чинними
  доками — він має прямо про це сказати.

## Приватність

Завантажений Knowledge-архів стає частиною GPT. Тут він містить лише навчальні матеріали з баз
даних — без секретів і персональних даних. Перед публікацією в GPT Store перевірте, що до
`references/` не потрапило приватного.
