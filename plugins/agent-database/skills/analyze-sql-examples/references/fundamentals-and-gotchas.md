## 1. NULL і трилогіка

У SQL `NULL` означає не «немає значення», а «значення невідоме». Тому будь-яке порівняння з `NULL` дає не `TRUE`/`FALSE`, а **`UNKNOWN`**, а `WHERE`/`ON`/`HAVING` пропускають лише рядки, де умова саме `TRUE`.

```sql
2 =  NULL   -- UNKNOWN
2 <> NULL   -- UNKNOWN  (а не TRUE!)
NULL = NULL -- UNKNOWN
```

Правильно перевіряти на `NULL` — операторами `IS NULL` / `IS NOT NULL`, а «безпечну» рівність — `IS DISTINCT FROM` / `IS NOT DISTINCT FROM` (PostgreSQL; у T-SQL — `EXISTS`/`INTERSECT`-трюки, у MySQL — оператор `<=>`).

**`NOT IN` з `NULL` у списку — класична пастка:**
```sql
3 IN     (1, 2, NULL)  -- (3=1) OR (3=2) OR (3=NULL) = UNKNOWN → не проходить
3 NOT IN (1, 2, NULL)  -- (3<>1) AND (3<>2) AND (3<>NULL) = ... AND UNKNOWN = UNKNOWN → теж не проходить!
```
Тобто `NOT IN (…, NULL)` **ніколи не поверне рядок**. Фікс: прибрати `NULL` зі списку (`WHERE col IS NOT NULL`), або використати `NOT EXISTS` / `LEFT JOIN … IS NULL`, які поводяться коректно з `NULL`.

## 2. Фільтрація з NULL

Найпоширеніший «тихий» баг вибірки. Задача: відібрати користувачів, у яких `company_id != 1`, при цьому в багатьох `company_id IS NULL`.
```sql
SELECT * FROM users WHERE company_id != 1;   -- губить УСІ рядки з NULL!
```
Рядки з `company_id IS NULL` дають `UNKNOWN` і випадають — на реальних даних так «зникає» до 80% записів. Фікс:
```sql
SELECT * FROM users WHERE COALESCE(company_id, -1) != 1;   -- -1 гарантовано відсутнє в стовпці
-- або явно:
SELECT * FROM users WHERE company_id != 1 OR company_id IS NULL;
```

## 3. Агрегація й NULL

Агрегатні функції (`SUM`, `AVG`, `MIN`, `MAX`, `COUNT(col)`) **ігнорують `NULL`**, а `COUNT(*)` рахує всі рядки. Явно:
- `COUNT(*)` — усі рядки (враховує NULL);
- `COUNT(col)` — лише рядки, де `col IS NOT NULL` (тому часто **менше** за `COUNT(*)`);
- `COUNT(DISTINCT col)` — кількість різних не-NULL значень. Через це «середнє» можна порахувати двома способами й отримати різне:
```sql
WITH a AS (
  SELECT 1 AS sales UNION ALL SELECT 2 UNION ALL SELECT NULL UNION ALL SELECT 3
)
SELECT AVG(sales),                    -- 2.0  (NULL пропущено: (1+2+3)/3)
       SUM(sales) * 1.0 / COUNT(*)    -- 1.5  ((1+2+3)/4, бо рядків 4)
FROM a;
```
Якщо за бізнес-логікою `NULL` має рахуватися як 0 (напр. «середня кількість покупок на користувача»), приводь явно: `AVG(COALESCE(sales, 0))`. Правило: завжди перепровіряй середнє двома способами — на великих даних розбіжність непомітна, а рішення на її основі коштують дорого.

## 4. Логічний порядок виконання запиту

Ти пишеш `SELECT` першим, а логічно він виконується шостим. Порядок (8 кроків):
```
FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT
```
Це пояснює три «собеседні» питання:
- **Псевдонім зі `SELECT` не можна використати у `WHERE`** — `WHERE` відпрацьовує раніше за `SELECT`. Псевдоніми доступні у `GROUP BY`/`HAVING`/`ORDER BY`.
  ```sql
  -- НЕ працює:
  SELECT user_name, YEAR(user_birth_date) AS year_of_birth
  FROM users WHERE year_of_birth = 2000;
  -- Треба повторити вираз:
  SELECT user_name, YEAR(user_birth_date) AS year_of_birth
  FROM users WHERE YEAR(user_birth_date) = 2000;
  ```
  (Зауваж: `WHERE YEAR(col) = 2000` не sargable — краще `WHERE col >= '2000-01-01' AND col < '2001-01-01'`, див. §5.)
- **Для фільтра за агрегатом потрібен `HAVING`, а не `WHERE`** (агрегати рахуються після `GROUP BY`).
- **`ORDER BY` з псевдонімом — ок** (виконується після `SELECT`).

## 5. `BETWEEN` + timestamp

`BETWEEN` включає обидві межі. Для стовпця-`timestamp` дата без часу тлумачиться як опівніч:
```sql
WHERE created_at BETWEEN '2024-01-01' AND '2024-01-31'
-- '2024-01-31' = 2024-01-31 00:00:00 → усі замовлення 31-го після опівночі ВИПАДАЮТЬ
```
Фікс — **напівінтервал** (включно знизу, виключно зверху):
```sql
WHERE created_at >= '2024-01-01' AND created_at < '2024-02-01'
```
Не «лікуй» це через `<= '...23:59:59'` — у зазорі до півночі ховаються частки секунди. Правило для дат із часом: `>= початок_періоду AND < початок_наступного`. Працює однаково в PostgreSQL і MySQL.

## 6. `LEFT JOIN`, що тихо стає `INNER`

```sql
SELECT c.*, o.status
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id
WHERE o.status = 'paid';       -- клієнти без замовлень ЗНИКАЮТЬ
```
`WHERE` виконується **після** JOIN: для незіставлених клієнтів `o.status` це `NULL`, умова `NULL = 'paid'` → `UNKNOWN` → рядок відкидається, і `LEFT JOIN` фактично перетворюється на `INNER`. Фікс — умова по «правій» таблиці переїжджає в `ON`:
```sql
SELECT c.*, o.status
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id AND o.status = 'paid';
```
Правило: умова на **ліву** (провідну) таблицю — у `WHERE`; умова на **праву** (опційну) — в `ON`.

### 6a. JOIN роздуває `SUM`/`COUNT` (fan-out, подвійний рахунок)
Агрегуючи по «батьку» після JOIN до «дітей», кожен батьківський рядок множиться на кількість дочірніх — і `SUM` роздувається (звіт по виручці «завищений удвічі», і його віддають фінансам):
```sql
-- НЕ те: order з 3 позиціями потрапить у SUM тричі
SELECT o.id, SUM(o.total) FROM orders o JOIN order_items i ON i.order_id=o.id GROUP BY o.id;
-- Фікс: спершу агрегуй дітей у підзапиті, потім join
SELECT o.id, o.total, x.items_cnt
FROM orders o
JOIN (SELECT order_id, COUNT(*) AS items_cnt FROM order_items GROUP BY order_id) x
  ON x.order_id = o.id;
```
Правило: не змішуй агрегацію батька з рядок-множним JOIN до дітей; агрегуй кожен рівень окремо (підзапит/CTE), або рахуй `COUNT(DISTINCT o.id)`.

### 6b. Self-JOIN і анти-join (`LEFT JOIN … IS NULL`)
Одну таблицю можна використати **двічі** (self-join) — напр. «співробітники без керівника у своєму відділі»:
```sql
SELECT e.name
FROM employee e
LEFT JOIN employee chief
  ON e.chief_id = chief.id AND e.dept_id = chief.dept_id
WHERE chief.id IS NULL;         -- анти-join: лишаємо тих, у кого НЕМАЄ збігу
```
Патерн **`LEFT JOIN … WHERE right.key IS NULL`** = анти-join (рядки без пари). Альтернатива — `NOT EXISTS` (часто читабельніше й коректніше з `NULL`, ніж `NOT IN`).

### 6c. As-of join на версійний (SCD2) вимір — чотири тихі точки відмови

Задача: зіставити факт із виміром **на момент події**, а не на «зараз» («у якому місті був курʼєр, коли створили замовлення»). Вимір версійний: `valid_from` / `valid_to` (у Delta/Databricks — `__START_AT` / `__END_AT`), відкрита версія має `valid_to IS NULL`. Еталонний предикат:

```sql
-- PostgreSQL
SELECT f.order_id, f.amount, COALESCE(d.city, 'Unknown') AS city
FROM fact_orders f
LEFT JOIN dim_courier d
       ON d.courier_id = f.courier_id
      AND f.created_at >= d.valid_from
      AND f.created_at <  COALESCE(d.valid_to, TIMESTAMP '9999-12-31');
```

Чотири помилки, за спаданням «тихості» — тобто того, як довго вони живуть непоміченими:

| # | Помилка | Наслідок |
|---|---|---|
| а | `AND d.is_current = TRUE` замість темпорального предиката | бере значення виміру **сьогодні**; звіт за березень **мовчки змінюється** щоразу, коли атрибут оновлюють. Помилки немає, логів немає — аналітик двічі виконує той самий запит і бачить різні числа |
| б | `BETWEEN d.valid_from AND d.valid_to` | `BETWEEN` включає **обидві** межі, а межі суміжних версій збігаються → факт рівно на межі матчиться на **дві** версії → рядок дублюється → `SUM` завищено (той самий механізм, що §5 і §6a) |
| в | `f.created_at < d.valid_to` без обробки `NULL` | для поточної версії `valid_to IS NULL` → порівняння дає `UNKNOWN` → **усі свіжі факти тихо зникають** (§1–2). `COALESCE` до сентинела знімає це конструктивно й краще джойниться, ніж `(valid_to IS NULL OR …)` |
| г | `INNER JOIN`, коли у факті `courier_id IS NULL` | зникають саме «неповні» факти (замовлення, скасоване до призначення курʼєра) — а це часто рівно те, що метрика має рахувати. Потрібен `LEFT JOIN` + учасник-заглушка `'Unknown'`/`-1` (стандарт Кімбола замість `NULL` у вимірі) |

**Заміряно в `sqlite3`** на мінімальному наборі (курʼєр із двома версіями — Львів до 2024-03-10, далі Київ із відкритою межею; 4 факти, з них один рівно **на межі** версій і один із `courier_id IS NULL`):

| Варіант join | Рядків на виході | Що конкретно зламалося |
|---|---:|---|
| еталон (`LEFT` + напівінтервал + `COALESCE`) | **4** | — (о1→Львів, о2→Київ, о3→Київ, о4→Unknown) |
| (а) `is_current = TRUE` | 4 | о1 отримав **Київ** замість Львова — лютневе замовлення переїхало разом із курʼєром |
| (б) `BETWEEN` | **5** | факт на межі зматчився на **обидві** версії (о2 → і Київ, і Львів) |
| (в) без `COALESCE` | 4 | о2 і о3 отримали `NULL` — **обидва факти поточної версії** втрачено |
| (г) `INNER JOIN` | **3** | о4 (`courier_id IS NULL`) зник повністю |

Зверніть увагу: помилки (а) і (в) **не змінюють кількість рядків** — їх не спіймати підрахунком, лише звіркою значень. Тому інваріант нижче необхідний, але не достатній.

**Обовʼязкова перевірка, яку майже ніколи не пишуть.** Перекриття інтервалів у виміри (наслідок кривої збірки SCD2) розмножує факти й завищує суми **без єдиної помилки**:

```sql
-- 1. перекриття версій усередині ключа — має бути 0 рядків
SELECT a.courier_id, a.valid_from, b.valid_from
FROM dim_courier a
JOIN dim_courier b
  ON a.courier_id = b.courier_id
 AND a.valid_from < b.valid_from
 AND b.valid_from < COALESCE(a.valid_to, TIMESTAMP '9999-12-31');

-- 2. join не змінив кількість фактів — має бути 0
SELECT (SELECT COUNT(*) FROM fact_orders_enriched)
     - (SELECT COUNT(*) FROM fact_orders) AS delta;
```

**Правило:** після join на вимір кількість рядків факту **не має змінюватися** — перевіряй це assertion-ом, а не очима. Діалектне: у ClickHouse для цього є `ASOF LEFT JOIN` (вимагає нерівності останньою умовою й сам бере найближче попереднє значення) — коротше й швидше за ручний напівінтервал, але семантика «остання версія ≤ моменту» вже вшита, тож відкриту межу з `NULL` перевіряй окремо. Моделювання самого SCD2 (сурогати, типи 1/2/3) — у скілі `design-dwh-etl`, `references/dimensional-modeling.md`.

## 7. `SELECT *` — чому уникати в коді

Три причини, чому сеньйори майже не пишуть `SELECT *` у продуктовому коді:
1. Тягне всі стовпці, коли треба два — зайвий трафік і пам'ять.
2. Ламає **Index Only Scan**: СУБД могла б відповісти лише з індексу, але через `*` мусить іти в таблицю (heap fetch).
3. Найпідступніше: колега додав стовпець — `INSERT ... SELECT *` або код, що покладається на порядок/кількість колонок, тихо ламається.

Перелічуй лише потрібні стовпці; `*` лиши для ручних запитів у консолі.

## 8. `GROUP BY` й агрегація

- **Неагрегований стовпець поза `GROUP BY`** — поведінка залежить від СУБД:
  - **MySQL** із вимкненим `ONLY_FULL_GROUP_BY` (історично) міг «дописати» стовпець і повернути довільне значення з групи — тихий баг. Сучасний MySQL 5.7+/8.0 за замовчуванням із `ONLY_FULL_GROUP_BY` → помилка.
  - **MS SQL Server / PostgreSQL / Oracle** — сувора помилка: стовпець має бути або в `GROUP BY`, або під агрегатом.
  ```sql
  SELECT order_id, order_code, SUM(order_value)   -- order_code ні в GROUP BY, ні під агрегатом
  FROM orders GROUP BY order_id;                  -- MySQL: може відпрацювати; MSSQL/PG: помилка
  ```
- **Зайве поле в `GROUP BY` роздуває результат.** Треба «кількість транзакцій на клієнта»:
  ```sql
  -- НЕ те: замість 1 рядка на клієнта — багато
  SELECT client_id, SUM(amount) FROM transactions GROUP BY client_id, transact_id;
  -- Правильно — групувати лише за тим, що в SELECT без агрегату:
  SELECT client_id, SUM(amount) FROM transactions GROUP BY client_id;
  ```
- **Вираз замість поля в `GROUP BY`.** Якщо в `SELECT` є `CASE`-прапор, групувати треба за всім виразом, а не за «сирим» полем:
  ```sql
  -- Групує неправильно (за start_date, а не за прапором):
  SELECT client_id,
         CASE WHEN start_date >= DATE '2026-01-01' THEN 1 ELSE 0 END AS new_client_flag,
         SUM(amount)
  FROM transactions GROUP BY client_id, start_date;
  -- Правильно:
  SELECT client_id,
         CASE WHEN start_date >= DATE '2026-01-01' THEN 1 ELSE 0 END AS new_client_flag,
         SUM(amount)
  FROM transactions
  GROUP BY client_id, CASE WHEN start_date >= DATE '2026-01-01' THEN 1 ELSE 0 END;
  ```

## 9. `WHERE` vs `HAVING`

Обидва фільтрують за умовою, але:
- `WHERE` фільтрує **окремі рядки до групування** (і не може містити агрегатів).
- `HAVING` фільтрує **вже згруповані значення** й має сенс лише з `GROUP BY` (може містити агрегати, напр. `HAVING COUNT(*) > 5`).

## 10. `UNION` vs `UNION ALL`

Обидва об'єднують результати запитів (однакова кількість/типи стовпців). `UNION` **прибирає дублікати** (робить сортування/хешування — дорожче), `UNION ALL` — **лишає все як є** (швидше). Якщо дублів свідомо немає або вони потрібні — бери `UNION ALL`.

## 11. Віконні функції й top-N у групі

Класична задача: «топ-3 товари за продажами в кожній категорії». Корельований підзапит працює, але повільний і нечитабельний. Ідіома — віконна функція:
```sql
SELECT * FROM (
  SELECT product_id, category_id, sales_amount,
         ROW_NUMBER() OVER (PARTITION BY category_id ORDER BY sales_amount DESC) AS rn
  FROM sales
) t
WHERE rn <= 3;
```
Дві пастки:
- **`WHERE rn <= 3` у тому самому рівні запиту не працює** — вікно рахується на кроці `SELECT`, а `WHERE` — раніше (§4). Тому потрібна обгортка підзапитом/CTE.
- **Вибір функції при нічиїх:**
  - `ROW_NUMBER()` — унікальна нумерація, нічию рве довільно (1,2,3,4) → рівно N рядків.
  - `RANK()` — однаковим значенням однаковий ранг, наступний **пропускається** (1,2,2,4) → «топ-3» може дати менше/більше очікуваного.
  - `DENSE_RANK()` — однаковим однаковий ранг, без пропусків (1,2,2,3) → «топ-3 різних значень».

  Приклад: суми 1000,800,800,700 → `RANK<=3` поверне рядки з рангами 1,2,2 (700 має ранг 4, випадає); `DENSE_RANK<=3` включить і 700.

Наявність віконних функцій: PostgreSQL, SQL Server, **MySQL 8.0+**, SQLite 3.25+, ClickHouse. У **MySQL ≤ 5.7** їх немає — потрібні змінні сесії або self-JOIN із підрахунком.

## 12. `TRUNCATE` vs `DELETE`

Обидва можуть видалити всі рядки, але:
1. `TRUNCATE` скидає таблицю цілком (як пере-створення) — значно швидше за порядкове `DELETE`.
2. `TRUNCATE` зазвичай **не запускає тригери** й не робить порядкових перевірок FK-каскадів (правила відрізняються за СУБД — у PostgreSQL `TRUNCATE` враховує FK і може каскадити з `CASCADE`, у MySQL/InnoDB заборонений за наявності FK-посилань).
3. Транзакційність: у **PostgreSQL/SQL Server `TRUNCATE` транзакційний** (можна відкотити), в **Oracle/MySQL — це DDL з неявним commit** (відкотити не можна). Не переноси «Oracle-інтуїцію» на всі СУБД — перевіряй для цільової.

`DELETE ... WHERE …` — вибіркове видалення, повністю транзакційне, запускає тригери.

## 13. Рядкові типи: `CHAR` / `VARCHAR` / `NVARCHAR`

- `CHAR(n)` — **фіксована** довжина; коротший рядок доповнюється пробілами (`'SQL'` у `CHAR(10)` → `'SQL       '`). Доречний для справді фіксованих кодів (хеші, інвентарні номери).
- `VARCHAR(n)` — **змінна** довжина, пам'ять за фактом.
- `NVARCHAR` (T-SQL/SQL Server) — рядок змінної довжини в **Unicode** (історично 2 байти/символ, UCS-2/UTF-16), підтримує будь-які мови/ієрогліфи. `VARCHAR` у SQL Server залежить від кодування (collation) — для не-ASCII може губити символи. Бери `NVARCHAR`, якщо можливі багатомовні дані.
- У PostgreSQL це неактуально: `text`/`varchar` завжди в кодуванні БД (зазвичай UTF-8); окремого `NVARCHAR` немає, а `char(n)`/`varchar(n)` радше уникають на користь `text`.
- Ширший контекст — стаття Joel Spolsky «The Absolute Minimum Every Software Developer Absolutely, Positively Must Know About Unicode and Character Sets».

## 14. Порядок стовпців у складеному індексі

Порядок має значення: індекс `(a, b)` — не те саме, що `(b, a)`.
```sql
CREATE INDEX ix1 ON users (user_name, user_birth_date);   -- ≠
CREATE INDEX ix2 ON users (user_birth_date, user_name);
```
Складений B-tree індекс корисний для умов на **лівий префікс**: `WHERE user_name = ?` (і `AND user_birth_date = ?`) використає `ix1`, а `WHERE user_birth_date = ?` — ні. Правило: стовпці, за якими фільтруєш найчастіше/найточніше (рівність), став першими; для діапазонного фільтра (`created_at > ?`) — після рівностей (напр. `(status, created_at)`).

**Селективність і ціна:** індекс найкорисніший на стовпцях із **високою кардинальністю** (email, id — значення рідко повторюються); на стовпці з 2–3 значеннями (стать, boolean) звичайний B-tree малокорисний (для «гарячого» підмножини — частковий індекс). **Кожен індекс уповільнює запис** (`INSERT`/`UPDATE`) і займає місце — не індексуй усе підряд, лише реальні шляхи доступу.

## 15. Монотонність ключа й роздування індексу (UUID v4 vs v7)

Випадковий ключ (UUID v4) при вставці потрапляє у **випадкове** місце B-дерева індексу → сторінки рвуться навпіл (**page splits**) → вставки гальмують, індекс розпухає. Монотонний ключ (лічильник, timestamp-based **UUID v7**) дозаписується в кінець дерева без сплітів.
- Бенчмарк (10M вставок, орієнтовно): UUID v4 ≈ 42с / 440MB; монотонний лічильник ≈ 18с / 200MB; UUID v7 ≈ 22с / 210MB.
- У **PostgreSQL 18** — вбудований `uuidv7()` (`... DEFAULT uuidv7()`); для старших версій — `gen_random_uuid()` (v4) або генерація v7 у застосунку.
- У **MySQL/InnoDB** ефект сильніший: PK кластерний, тож випадковий PK роздуває саму таблицю.
- Нюанс: UUID v7 містить час створення — враховуй, якщо `id` видимий ззовні (витік метаданих про час).

## 16. Змішування діалектів (портативність дат/змінних)

Код нерідко змішує синтаксис різних СУБД і не виконається як єдине ціле. Приклад:
```sql
DECLARE @StartDate DATE = add_months(trunc(sysdate,'mm'), -2);  -- @var це T-SQL, а sysdate/add_months/trunc — Oracle
DECLARE @EndDate   DATE = last_day(trunc(sysdate));
```
Еквіваленти «перше число N місяців тому … останній день місяця»:
- **T-SQL (SQL Server):** змінні `@var`; `DATEADD(MONTH,-2, DATEFROMPARTS(YEAR(GETDATE()),MONTH(GETDATE()),1))`; кінець місяця — `EOMONTH(GETDATE())`.
- **PostgreSQL / PL/pgSQL:** `DECLARE` без `@`; `date_trunc('month', now()) - interval '2 month'`; кінець місяця — `date_trunc('month', now()) + interval '1 month' - interval '1 day'`.
- **MySQL:** `DATE_SUB(...)`, `LAST_DAY(CURDATE())`, `CURDATE()`.
- **Oracle:** `sysdate`, `trunc(sysdate,'mm')`, `add_months`, `last_day`.

Коли бачиш мішанину — познач її й дай чистий варіант під цільовий діалект.

## 17. Продуктивність: `LIKE`, пагінація, N+1

Крос-діалектні пастки продуктивності (стосуються PostgreSQL, MySQL та ін.):
- **`LIKE '%foo%'` (ведучий `%`) вимикає B-tree** — повний скан (мс проти секунд на мільйоні рядків). `LIKE 'foo%'` (тільки хвостовий `%`) використовує індекс. Для пошуку підрядка/по опечатках — спеціальні індекси: у PostgreSQL `pg_trgm` (GIN), у MySQL — повнотекстові індекси.
- **Глибокий `OFFSET` у пагінації** — `LIMIT 20 OFFSET 1000000` сканує й відкидає мільйон рядків щоразу (складність росте з номером сторінки), може «покласти» сервер. Фікс — **keyset/cursor-пагінація** за індексованим ключем: `WHERE id > :last_seen_id ORDER BY id LIMIT 20` (стала швидкість на будь-якій сторінці).
- **N+1 (антипатерн ORM)** — код робить 1 запит за списком + по 1 запиту на кожен елемент (N) для пов'язаних даних → сотні дрібних запитів замість одного `JOIN`. ORM це «ховає». Фікс — жадібне завантаження одним запитом: Django `select_related`/`prefetch_related`, SQLAlchemy `joinedload`/`selectinload`, або явний `JOIN`. Прискорення на порядки (5с→50мс). Споріднене — N+1 у GraphQL-resolverах (там рятує DataLoader).

## 18. Обмеження цілісності й data quality (`CHECK`, природний ключ)

Бізнес-правила краще тримати **поруч із даними** — менше «сміття» на проді, ніж покладатися лише на код застосунку:
- **`CHECK`** — валідація на рівні БД: `amount NUMERIC NOT NULL CHECK (amount >= 0)`, частка `CHECK (ratio BETWEEN 0 AND 1)`, дати `CHECK (end_date >= start_date)`, непорожні рядки `CHECK (length(trim(name)) > 0)`. Пам'ятай: `CHECK` пропускає `NULL` (трилогіка) — комбінуй із `NOT NULL`, коли значення обов'язкове.
- **Нормалізоване зберігання + `CHECK` на формат:** напр. телефон — лише цифри + перевірка довжини (`CHECK (phone ~ '^[0-9]{10,15}$')`), форматування вже на виводі.
- **Природний складений ключ замість сурогату:** якщо унікальна сама **комбінація** — вона і є PK; окремий `id` не потрібен. Напр. `monthly_reports(year, month, ... , PRIMARY KEY (year, month))` (плюс `CHECK (month BETWEEN 1 AND 12)`). Сурогатний ключ додавай, коли природний громіздкий, змінний або потрібен зовнішнім посиланням.

## 19. `DISTINCT` vs `GROUP BY`

- `SELECT DISTINCT col1, col2 FROM t` — прибирає **дублікати рядків**; наміру «унікальні комбінації» читається чіткіше.
- `SELECT col1, col2 FROM t GROUP BY col1, col2` без агрегатів дає той самий результат, але `GROUP BY` призначений для **агрегації** (`SUM`/`COUNT`/…). Якщо агрегатів немає — бери `DISTINCT` (ясніший намір); якщо потрібні агрегати по групах — `GROUP BY`.
- Не поєднуй `DISTINCT` з агрегатами «про всяк випадок» — це часто ознака помилки в моделі запиту (напр. дублі від JOIN, §6a: краще прибрати причину, ніж «замаскувати» `DISTINCT`).

## 20. Ділення на нуль і `NULLIF`

Вираз `a / b` падає, коли `b = 0` — але **по-різному в різних СУБД**: PostgreSQL / SQL Server / Oracle кидають помилку (`division by zero`), а MySQL за замовчуванням **тихо повертає `NULL`** з warning (поведінку змінює sql_mode `ERROR_FOR_DIVISION_BY_ZERO` + strict). Тихий `NULL` небезпечніший за помилку — звіт «порахувався», але з дірками.

Ідіома захисту — **`NULLIF(b, 0)`**: повертає `NULL`, якщо `b = 0`, інакше `b`; ділення на `NULL` дає `NULL` замість падіння:
```sql
SELECT paid_cnt * 1.0 / NULLIF(total_cnt, 0) AS conv_rate ...
-- за потреби «0 замість NULL»: COALESCE(paid_cnt*1.0/NULLIF(total_cnt,0), 0)
```
Працює однаково в усіх основних СУБД (стандартна функція). Особливо важливо в знаменниках агрегатів (`SUM(...)/NULLIF(COUNT(...),0)`), де порожня група — нормальний крайовий випадок.

## 21. Сумнівні «поради-штампи», які треба перевіряти

Студенти (і деякі книжки) приносять поради, що були правдою колись, частково, або ніколи. Розбирай їх з доказами (план запиту), а не приймай на віру:

- **«Уникай підзапитів, використовуй JOIN».** Сучасні оптимізатори (PG, MySQL 8, MSSQL) розкривають некорельовані підзапити в join самостійно — план однаковий. Реальна проблема вужча: **корельований** підзапит у `SELECT`-списку на великій вибірці (виконується на кожен рядок) — ось його міняй на JOIN/віконну функцію. `EXISTS` — часто найкращий вибір для «чи є пов'язане» (§ `sql-fundamentals.md`).
- **«DISTINCT повільний — використовуй GROUP BY».** У сучасних СУБД план ідентичний. Справжнє питання — *чому* з'явилися дублікати (часто fan-out від JOIN, §6a): прибери причину, а не симптом.
- **«Index на кожен стовпець»** — див. §14: кожен індекс сповільнює запис; індексуй реальні шляхи доступу.
- **«INT-ключі завжди швидші за рядкові»** — зерно правди (коротший ключ = щільніший індекс, дешевші порівняння), але висновок «ніколи не роби рядковий PK» хибний: короткі стабільні natural keys (ISO-коди) — нормальна практика; проблема не «рядок», а **довжина, змінюваність і випадковість** ключа (§15).
- **«Тригер на логування кожного запиту»** — DML-тригери бачать лише INSERT/UPDATE/DELETE (не SELECT) і дорого коштують на кожній операції; для аудиту повільних запитів існують штатні механізми: `slow_query_log` (MySQL), `pg_stat_statements`/`log_min_duration_statement` (PG), Query Store/Extended Events (MSSQL).

Правило: будь-яке «завжди роби X» перевіряється за 5 хвилин на `EXPLAIN` — привчай до цього студентів.
