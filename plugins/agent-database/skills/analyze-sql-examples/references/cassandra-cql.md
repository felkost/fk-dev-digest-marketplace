# Cassandra (CQL) — довідник для аналізу запитів

CQL синтаксично схожий на SQL, але Cassandra — розподілена wide-column БД (розподіл за мотивами Amazon Dynamo, модель даних — за Google Bigtable). Головне під час аналізу: **модель даних диктується запитами**, і багато «звичного» SQL тут **невиразне**. Це часто найважливіший пункт у блоці «Обмеження».

## Архітектура (чому CQL такий, який є)
- **Masterless-кільце:** будь-який вузол рівноцінний, немає single point of failure; вузли спілкуються **gossip**-протоколом. Горизонтальне масштабування додаванням вузлів без простою.
- **Розподіл даних:** рядки розкидані по вузлах за **token** від хешу **partition key** (partitioner). Віртуальні вузли (vnodes) рівномірять розподіл.
- **Реплікація:** фактор реплікації (RF) — скільки копій; топологію задає snitch (multi-DC реплікація «з коробки»).
- **Налаштовувана узгодженість (tunable consistency):** рівень задається **на запит** — `ONE`, `QUORUM`, `LOCAL_QUORUM`, `ALL` тощо. За замовчуванням це **AP**-система (доступність + стійкість до розділів); сильну узгодженість «докуповують» рівнем `QUORUM` (читання+запис).
- **Самолікування:** hinted handoff (тимчасово недоступний вузол «дожене» пропущені записи), read repair, anti-entropy repair (Merkle-дерева).
- **LWT (легкі транзакції)** через Paxos: `INSERT ... IF NOT EXISTS`, `UPDATE ... IF ...` — узгоджені, але **дорогі**; лише для рідкісних випадків, не для гарячого шляху.

## Модель даних (термінологія знизу вгору)
- **column** — пара «ім'я–значення» (насправді триплет із мікросекундною **timestamp** для розв'язання конфліктів за принципом *last-write-wins*).
- **row** — набір колонок під первинним ключем.
- **partition** — група пов'язаних рядків, що зберігаються разом на одних вузлах.
- **table** — контейнер рядків, організованих по партиціях.
- **keyspace** — контейнер таблиць (≈ база; тут задають клас реплікації та RF).
- **cluster (ring)** — контейнер keyspace-ів на кількох вузлах.

### Первинний ключ = partition key + clustering columns
```sql
CREATE TABLE my_keyspace.user (
  last_name  text,
  first_name text,
  title      text,
  PRIMARY KEY (last_name, first_name)   -- partition key = last_name; clustering = first_name
) WITH CLUSTERING ORDER BY (first_name ASC);
```
- **Partition key** (перша частина, може бути складеним `((a, b), c)`) визначає **вузол** зберігання.
- **Clustering columns** визначають **порядок сортування** рядків усередині партиції.
- **Static column** — значення, спільне для всіх рядків партиції (поза первинним ключем).
- Якщо clustering-стовпців немає — партиція = один рядок.
- **Первинний ключ незмінний** після створення таблиці (він керує розподілом і зберіганням на диску).

### Доступ вимагає ключа
- `INSERT`/`UPDATE` — треба задати **всі** стовпці первинного ключа у `WHERE`.
- `SELECT`/`DELETE` — можна за партицією повністю, за конкретним рядком, або за кількома партиціями через `IN`.
- `INSERT` = `UPDATE` = **upsert** (той самий первинний ключ перезаписує рядок; неіснуючий — створює). Виняток — LWT.

## Методологія проєктування: від запитів до таблиць

У реляційній БД спершу нормалізують сутності, а запити пишуть потім. **У Cassandra навпаки: спершу запити, потім таблиці.** Офіційна методологія — п'ять кроків:

1. **Концептуальна модель** — сутності, атрибути, зв'язки (звичайна ER-діаграма; цей крок такий самий, як у реляційному світі).
2. **Запити застосунку** — виписати всі запити, які система має обслуговувати, і **впорядкувати за важливістю**. Це головний вхід для дизайну, а не другорядна деталь.
3. **Логічна модель** — по одній таблиці **на запит**.
4. **Фізична модель** — типи CQL, UDT, статичні стовпці, індекси.
5. **Оцінка й уточнення** — розрахунок розміру партиції, тестування, коригування.

### Крок 3: логічна модель

- **Назва таблиці = `<сутність>_by_<атрибут>`**: `hotels_by_poi`, `available_rooms_by_hotel_date`. Назва одразу каже, який запит таблиця обслуговує; якщо назву не вдається сформулювати — запит ще не визначено як слід.
- **Первинний ключ** збирається так: partition key — те, за чим фільтрує запит; clustering columns — те, що дає **унікальність** і **потрібний порядок сортування**.
- Решта атрибутів запиту йде звичайними стовпцями; те, що повторюється в межах партиції, — як `static`.
- **Одні й ті самі дані свідомо лежать у кількох таблицях із різними ключами.** Це не порушення нормалізації — це і є дизайн.

**Дві цілі, між якими шукають баланс:**

1. **Рівномірно розподілити дані по кільцю** — щоб не було гарячих вузлів (проти цього працює низькокардинальний або перекошений partition key).
2. **Мінімізувати кількість партицій, які читає один запит** — ідеал: **одна партиція на запит**. Кожна зайва партиція — ще один координований похід по вузлах.

Ці цілі тягнуть у різні боки: найрівномірніший розподіл дає ключ, унікальний на рядок, але тоді будь-яка вибірка «список за …» читає багато партицій. Проєктування — це вибір точки між ними, і саме його треба вміти пояснити.

### Нотація Чеботка (Chebotko diagram)

Стандартний спосіб малювати логічну й фізичну модель Cassandra:

- **K** — стовпець partition key;
- **C↑ / C↓** — clustering-стовпець із напрямком сортування (ASC/DESC);
- **S** — static-стовпець;
- стрілки від запитів до таблиць показують, який запит обслуговує яка таблиця.

Фізична діаграма додає до цього типи CQL, keyspace, позначки колекцій/UDT та індексів.

### Крок 4: фізична модель

- Типи беруться з CQL; для атрибутів із міжнародною варіативністю (телефон, адреса) — `text`, а не «жорсткий» формат.
- **UDT** прибирає дублювання спільних структур (напр. `address`, що використовується в кількох таблицях).
- **UUID як ідентифікатор** зменшує зв'язність між сутностями — зручно в мікросервісах, де немає спільного лічильника.

### Крок 5: розрахунок розміру партиції (робити ДО створення таблиці)

Партиція — одиниця зберігання й читання, тож її розмір **рахують, а не вгадують**.

**Кількість комірок (cells) у партиції:**

> `Nv = Nr · (Nc − Npk − Ns) + Ns`

де `Nr` — рядків у партиції, `Nc` — усього стовпців, `Npk` — стовпців первинного ключа, `Ns` — статичних стовпців.
Сенс: стовпці первинного ключа й статичні **не зберігаються в кожному рядку**, тому їх віднімають із добутку; статичні додаються один раз на партицію.

**Розмір на диску** — та сама логіка плюс метадані:

> `St = Σ sizeOf(ck_i) + Σ sizeOf(cs_j) + Nr · (Σ sizeOf(cr_k) + Σ sizeOf(cc_l)) + Nv · sizeOf(t_avg)`

тобто ключі партиції та статичні стовпці (один раз) + звичайні й clustering-стовпці (на кожен рядок) + метадані кожної комірки (`t_avg` ≈ **8 байт** — та сама мітка часу).

**Межі, з якими звіряєшся:**

| Орієнтир | Значення |
|---|---|
| Жорсткий ліміт Cassandra | **2 млрд комірок** на партицію |
| Практична ціль | **< 100 MB** на партицію |
| Реальність | деградація латентності починається **задовго до** жорсткого ліміту |

**Головний драйвер розміру — кількість рядків у партиції.** Тому питання «скільки рядків тут накопичиться за рік?» ставлять на етапі проєктування, а не в проді.

### Розбиття завеликих партицій: бакетування

Якщо розрахунок дає завелику партицію — **додай стовпець до partition key**. Класичний прийом — бакет за часом:

```sql
-- було: партиція росте вічно
PRIMARY KEY (hotel_id, date)

-- стало: партиція обмежена одним місяцем
PRIMARY KEY ((hotel_id, month), date)
```

Гранулярність бакета підбирають так, щоб партиції були **помірними**: доба часто дає надто дрібні партиції (багато походів по вузлах), рік — надто великі; місяць або тиждень — типовий компроміс. Бакет не обов'язково часовий: якщо перекіс не за часом, беруть хеш-суфікс (`bucket = hash(id) % N`).

## Чого CQL НЕ вміє (типова невиразність)
- **Немає JOIN** — жодних. Зв'язки моделюють денормалізацією/дублюванням (одна таблиця на патерн доступу).
- **Немає довільних агрегатів по всій таблиці** ефективно; `GROUP BY` — лише в межах partition key.
- **Немає віконних функцій** (`ROW_NUMBER`/`RANK`).
- **Немає довільного `ORDER BY`** — лише за clustering-стовпцями й у напрямку з `CLUSTERING ORDER BY`.
- **`WHERE` обмежений:** фільтр — за повним partition key і clustering-стовпцями по порядку. Фільтр за неключовим стовпцем вимагає вторинного індексу (обережно) або `ALLOW FILTERING` (антипатерн на проді — скан багатьох вузлів).

> Приклад невиразності: «розворот стовпця через `ROW_NUMBER` + self-JOIN» чи «топ-3 у категорії через віконну функцію» в CQL **не розв'язуються серверно** — це роблять на боці застосунку або іншою моделлю даних. Прямо зазнач це в «Обмеженнях».

## CQL-типи
- **Числові:** `int` (32), `bigint` (64), `smallint` (16), `tinyint` (8), `varint` (довільна точність), `float`, `double`, `decimal`. `enum` немає — зберігають як `text`.
- **Текст:** `text`/`varchar` (синоніми, UTF-8), `ascii`. Для нових даних — `text`.
- **Час/ідентичність:** `timestamp` (ISO-8601, завжди з таймзоною — best practice), `date`, `time`; `uuid` (Type 4, випадковий; функція `uuid()`), `timeuuid` (Type 1, час+MAC+лічильник — безконфліктна мітка часу; `now()`, `dateOf()`).
- **Інші:** `boolean`, `blob` (двійкове), `inet` (IPv4/IPv6), **`counter`** (лише інкремент/декремент; не може бути в первинному ключі; у таблиці з counter усі неключові стовпці — counter-и).

## Колекції, tuple, UDT
- **`set<t>`** — унікальні, повертаються відсортовано; додавання без читання (`col = col + {..}`).
- **`list<t>`** — впорядкований; **операції за індексом дорогі** (читає-переписує весь список) і `append` **неідемпотентний** — часто краще `set`/`map`.
- **`map<k,v>`** — пари ключ-значення.
- **`tuple<...>`** — фіксований набір значень різних типів (оновлюється лише цілком; на практиці рідко).
- **UDT** (`CREATE TYPE address (...)`) — іменовані поля; вкладення колекцій потребує **`frozen<...>`** (зберігається як blob, читається/пишеться цілком).

## Timestamp і TTL
- Кожне значення має мікросекундну **timestamp** (last-write-wins); `WRITETIME(col)` показує її (не можна для стовпців первинного ключа). `USING TIMESTAMP` — задати вручну (обережно).
- **TTL** на рівні окремого значення: `USING TTL 3600` (сек) → значення саме зникає; `TTL(col)` показує залишок. На рівні рядка — лише переінсертом усіх неключових стовпців.

## Практики запису й читання

- **`BATCH` — не прискорювач і не транзакція в звичному сенсі.** У SQL батч зменшує кількість круговертей і зазвичай пришвидшує; у Cassandra **навпаки**: батч на кілька партицій змушує координатор працювати з багатьма вузлами й часто **повільніший** за окремі асинхронні запити.
  - Виправданий сценарій — **батч у межах однієї партиції**: тоді це фактично один запис, і він справді атомарний та дешевий.
  - Логований батч дає **атомарність** (усе або нічого), але **не ізоляцію** — інші читачі можуть побачити частину змін.
  - Розмір обмежений у `cassandra.yaml`: **`batch_size_fail_threshold`** (дефолт `50KiB`), попередження — **`batch_size_warn_threshold`** (`5KiB`). Це імена **з 4.1**, після стандартизації конфігурації (одиниці переїхали з імені у значення). Старі `batch_size_fail_threshold_in_kb` / `batch_size_warn_threshold_in_kb` ще приймаються заради сумісності, але в дефолтному `cassandra.yaml` від 5.0 їх уже немає — не шукай.
- **Підготовлені запити (prepared statements)** компілюються й кешуються, тож повторні виконання не парсяться заново — дефолт для будь-якого продакшн-драйвера.
  - **Не біндь `NULL` у підготовлений запит:** вставка `NULL` створює **tombstone**. Якщо значення просто «не задане» — використовуй `unset` (драйвери це вміють). Це одна з найчастіших прихованих причин лавини надгробків.
- **Пагінація** — офсетів немає; не імітуй їх через `LIMIT`. Користуйся paging state драйвера (`fetch_size`).
- **Рівень узгодженості задається на запит.** Типова прод-конфігурація для мультирегіону — `LOCAL_QUORUM` на читання й на запис: сильна узгодженість у межах ЦОД без крос-регіонної латентності. Умова «прочитати власний запис» — `R + W > RF`.

## Індекси й матеріалізовані подання (коли таблиці-під-запит замало)

- **SAI (Storage-Attached Index)** — рекомендований механізм вторинних індексів у Cassandra 5.0 і сучасних збірках DataStax (для DSE 5.1, де SAI немає, — класичні `2i`). Дешевші й функціональніші за старі `2i`/`SASI`, підтримують числові діапазони й колекції, і саме вони — основа векторного пошуку (`VECTOR` + `ORDER BY … ANN OF`).
- Навіть із SAI **основний спосіб лишається той самий — таблиця під запит**. Індекс — це уточнення в межах партиції або низькоселективний фільтр, а не заміна дизайну.
- **Матеріалізовані подання** (`CREATE MATERIALIZED VIEW`) автоматично підтримують денормалізовану копію під іншим ключем. Зручно, але ціна: додатковий запис на кожне оновлення базової таблиці, породження tombstone і історично нестабільна репутація (у низці версій позначені як експериментальні). Дефолтний вибір — **денормалізація руками** через таблицю-під-запит; MV — свідоме рішення з розумінням ціни.
- **`ALLOW FILTERING` у проді — заборонено.** Це не «фікс помилки», а зізнання, що модель не відповідає запиту: замість індексного пошуку Cassandra перебирає дані на багатьох вузлах.

## Продуктивність, зберігання, граблі
- **Ідемпотентність:** `counter`-інкремент і `list`-append **неідемпотентні** — небезпечно при ретраях у розподіленій системі (значення може змінитися двічі).
- **Гарячі / необмежені партиції** — головний антипатерн; закладай bucket у partition key (напр. за днем).
- **Tombstones:** видалення й TTL лишають «надгробки»; вони фізично прибираються лише після `gc_grace_seconds` (за замовчуванням 864000 = 10 днів) при компакції. Багато tombstone у партиції різко гальмує читання.
  Джерел tombstone більше, ніж здається, і `DELETE` серед них не найчастіше: вставка **`NULL`** у стовпець, **заміна колекції цілком** (`col = {…}` спершу ставить надгробок на стару), **закінчення TTL**, **оновлення матеріалізованого подання**. Найчастіша несподіванка — саме `NULL`, прибинджений у підготовлений запит.
- **Великі бінарні об'єкти (BLOB) — не в Cassandra:** зображення й відео тримають в об'єктному сховищі або CDN, а в таблиці лишають **URI**. Інакше партиції перекошуються за розміром, і жоден розрахунок з `Nv` вже не рятує.
- `ALLOW FILTERING` — уникай на проді.
- Зберігання: memtable → SSTable (+ commit log), фонова компакція (напр. SizeTieredCompactionStrategy), Bloom-фільтри.
- Не покладайся на «унікальність PK з OLTP-світу» — семантика інша (upsert).

## Коли Cassandra — правильний вибір, а коли ні

**Підходить:** дуже високий темп запису; лінійне горизонтальне масштабування; мультирегіональна реплікація «з коробки»; наперед відомі патерни доступу; допустима eventual consistency.

**Не підходить** (часті помилки вибору, які варто називати прямо в блоці «Обмеження»):

- потрібні **ACID-транзакції між сутностями** або **JOIN** — це до реляційної БД;
- **ad-hoc аналітика** й довільні агрегати — це до ClickHouse чи DWH;
- **черга повідомлень** — класичний анти-патерн: черга означає «читай і видаляй голову», тобто лавину tombstone саме там, де найчастіше читають; бери Kafka/RabbitMQ;
- **важке конкурентне читання** з довільними фільтрами — модель під це не проєктується.

## Розгортання / робота
- Оболонка **`cqlsh`** (порт CQL native 9042). Docker: образ `cassandra`, том `/var/lib/cassandra`.
- Драйвери: DataStax (Java, Python `cassandra-driver` — див. скіл `db-connectivity-cloud`). Керовані: **AWS Keyspaces**, **Azure Cosmos DB (Cassandra API)**.
- Історична примітка: старі матеріали (до ~2012, Thrift-епоха) описують `cassandra-cli`, `column family`, `set/get` — це **застаріло**; сучасний доступ — через CQL/`cqlsh`.
- Поточна стабільна лінійка на 2026-07 — **Cassandra 5.0** (SAI, векторний пошук, dynamic data masking).
- Джерела: «Cassandra: The Definitive Guide» (3-тє вид., O'Reilly); офіційна документація cassandra.apache.org (розділ Data Modeling — методологія, нотація Чеботка, формули розміру партиції); DataStax CQL — Data modeling best practices; Redgate Simple Talk — «Apache Cassandra Data Modeling and Query Best Practices».
