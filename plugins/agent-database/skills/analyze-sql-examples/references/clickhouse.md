# ClickHouse — довідник для аналізу запитів

ClickHouse — колонкова OLAP-СУБД для аналітики на великих обсягах. Синтаксис близький до SQL, але модель зберігання й ідіоми продуктивності інші. Не плутай із транзакційними (OLTP) БД: ClickHouse оптимізований під масові `INSERT` і аналітичні `SELECT`, а не під точкові оновлення.

## Родина MergeTree (основа)
Головний рушій таблиць — `MergeTree` та похідні (`ReplacingMergeTree`, `SummingMergeTree`, `AggregatingMergeTree`, `ReplicatedMergeTree`).
```sql
CREATE TABLE trips (
  pickup_datetime DateTime,
  pickup_date     Date,
  fare            Float64,
  passenger_count UInt8
)
ENGINE = MergeTree
PARTITION BY toYYYYMM(pickup_date)     -- партиціювання (зазвичай за часом)
ORDER BY (pickup_datetime);            -- ключ сортування = «первинний ключ» (розріджений індекс)
```
- **`ORDER BY` = ключ сортування** і водночас первинний (розріджений) індекс: дані фізично впорядковані, індекс — не на кожен рядок, а на «гранули». Це не унікальний PK; дублікати можливі.
- **`PARTITION BY`** ріже дані на частини (partition pruning при фільтрі за ключем партиції). Не роби надто дрібні партиції.
- Вставки створюють парти, які фоново зливаються (merge). Тому `ReplacingMergeTree`/`SummingMergeTree` дедуплікують/агрегують **асинхронно** — до злиття можуть бути «зайві» рядки; для гарантії — `FINAL` (дорого) або агрегація в запиті.

### `ReplacingMergeTree`: чотири речі, які треба знати до того, як його обрати

Найпопулярніший вибір під подієві таблиці — і джерело найтихіших багів, бо поводиться не так, як підказує OLTP-інтуїція.

1. **Ключ дедуплікації — це `ORDER BY`, а не `PRIMARY KEY`.** Документація формулює прямо: унікальність рядків визначає секція `ORDER BY`, не `PRIMARY KEY`. Звідси симетрична пара помилок: ключ **заширокий** → справжні ретраї не склеюються; ключ **завузький** → склеюються **різні** події, і це вже **втрата даних**. Класика — `ORDER BY (user_id, event_time)`: два різні кліки в ту саму мілісекунду тихо стають одним. Правило: ключ сортування має бути **рівно ідентифікатором події**; якщо такого стовпця немає, додай `event_id` **останнім** (він високої кардинальності, тому в кінці ключа дешевий).
2. **Колонка версії не сміє входити в `ORDER BY`.** `ReplacingMergeTree(ver)` лишає рядок із максимальним `ver` у групі дублікатів. Але якщо `ver` уже є частиною ключа сортування, то всередині групи він **сталий за побудовою**, і правило вироджується в «лишити будь-який». `ReplacingMergeTree(event_time)` при `ORDER BY (…, event_time)` — саме цей випадок: виглядає як коректний код і не робить нічого.
3. **Дедуплікація відбувається лише при злитті, а злиття не перетинають межу партиції.** Практичний наслідок: партиціюй за часом **події**, а не за часом приймання — інакше ретрай, що дійшов наступної доби, потрапляє в іншу партицію й не склеїться з оригіналом **ніколи**.
4. **До злиття дублікати присутні у вибірці** («merging occurs in the background at an unknown time»). Або `FINAL`, або агрегація в запиті (`argMax`), або усвідомлене «читаємо як є». Найтихіший наслідок: **інкрементне матеріалізоване подання рахує рядок на вставці, тобто ДО дедуплікації** — сира таблиця з часом стане чистою, а вітрина лишиться завищеною назавжди. Дедуплікуй до ClickHouse (ідемпотентний `event_id` на боці продюсера) або періодично перебудовуй вітрину через `REPLACE PARTITION`.

Для «м'якого» видалення є параметр `is_deleted` (працює лише разом із `ver`): `1` — рядок-видалення, `0` — рядок-стан.

### Як обирати `ORDER BY` (офіційні правила)

Ключ сортування визначає і продуктивність, і стиск, і **його не можна змінити після створення таблиці** — це рішення «на все життя таблиці».

1. **Частота фільтрації — головний критерій.** Першими став стовпці, що найчастіше стоять у `WHERE` і **відсікають найбільше рядків**.
2. **Кореляція з рештою даних.** Стовпці, корельовані з іншими, дають суміжне зберігання → кращий стиск і менше пам'яті на `GROUP BY`/`ORDER BY`.
3. **4–5 стовпців зазвичай достатньо** — далі виграш падає, а індекс росте.
4. **Спершу низька кардинальність.** Типовий порядок: `(низькокардинальний_фільтр, дата, висококардинальний_id)`.
5. **Огрублюй тип у ключі:** `toDate(created_at)` замість `DateTime` — дата вкладається в 16 біт, індекс менший, фільтрація швидша.

У прикладі офіційної документації правильний `ORDER BY` дав різницю в **два порядки прочитаних рядків** на тому самому запиті (≈197 тис. проти ≈59,8 млн) — це не «кілька відсотків», а різниця між інтерактивним і неінтерактивним звітом.

### Ключ партиціювання — це про керування даними, а не про швидкість

- Офіційна теза: **партиціювання — насамперед техніка керування життєвим циклом даних, а не інструмент оптимізації запитів.** Prunning буває, але це не привід партиціювати.
- **Низька кардинальність, орієнтир < 100–1000 різних значень.** `toYYYYMM(date)` — так; `user_id` — категорично ні.
- ClickHouse **не зливає парти між партиціями** — дрібні партиції назавжди лишаються дрібними партами.
- Перевищення `max_parts_in_total` / `parts_to_throw_insert` → помилка **`Too many parts`**.
- Партиціюй тоді, коли потрібні `DROP PARTITION` або `TTL`-ретеншн саме за цим ключем.

## Запити
- Стандартний `SELECT ... FROM ... WHERE ... GROUP BY ... ORDER BY ...`, плюс тисячі функцій (масиви, дати, статистика, `arrayJoin`, `uniqExact`/`uniq`, `quantile`).
- **`FORMAT`** — понад 70 форматів вводу/виводу: `SELECT ... FORMAT TabSeparated | JSONEachRow | CSV ...`.
- Вставка з зовнішніх джерел прямо в запиті: `INSERT INTO trips SELECT * FROM s3('https://.../*.tsv.gz', 'TSV', 'схема')`.
- `clickhouse-local` — виконання запитів над файлами без сервера.

## Словники (dictionaries) замість JOIN
Довідкові дані тримають у in-memory словнику й тягнуть через `dictGet` — швидше за JOIN для «зіркових» довідників:
```sql
SELECT dictGet('zone_dict', 'borough', toUInt64(location_id)) AS borough, count()
FROM trips GROUP BY borough;
```
- ClickHouse може й сам оптимізувати JOIN зі словником. Для великих таблиць-фактів `JOIN` дорогий — оцінюй, чи не краще словник або денормалізація.

> **Тиха пастка словників:** словник **мовчки дедуплікує ключі**, лишаючи тільки **останнє** значення. Тому він придатний лише для зв'язків **1:1 і N:1**. Підставити словник замість `JOIN` на зв'язку **1:N** — це тиха втрата рядків: помилки не буде, будуть неправильні числа.

## JOIN: спершу мінімізуй, потім обирай алгоритм

- **Спершу денормалізація.** `JOIN` дорожчий за читання з однієї широкої таблиці. Денормалізація виправдана, коли довідник змінюється рідко, зв'язок не M:N і потрібні лише окремі стовпці. Альтернатива — матеріалізоване подання з інкрементним агрегатом.
- **Порядок таблиць: менша — праворуч** (саме з правої будується хеш-таблиця). **Починаючи з 24.12 планувальник переставляє сам**; на старіших версіях це роблять руками, і це класична причина «запит з'їв усю пам'ять».
- **Алгоритми** — налаштування `join_algorithm`, дефолт `direct,parallel_hash,hash` (пробує зверху вниз):

| Алгоритм | Коли доречний |
|---|---|
| `direct` | права сторона — словник; хеш-таблиця не будується взагалі (point lookup) |
| `parallel_hash` | робоча конячка дефолтного списку: мала/середня права таблиця, що влазить у пам'ять |
| `full_sorting_merge` | обидві таблиці вже відсортовані за ключем з'єднання |
| `partial_merge` | великі таблиці при дефіциті пам'яті — економно, але повільно |
| `grace_hash` | великі обсяги, коли пам'ять треба тримати в межах |

- `grace_hash`, `partial_merge` і `full_sorting_merge` вміють **проливатися на диск**, коли хеш-таблиця не влазить у пам'ять. Це рятує від OOM, але сильно гальмує — краще переписати запит, ніж закладатися на пролив.

## Розподілений і паралельний GROUP BY
- Агрегація сильно паралелиться (по ядрах) і розподіляється (по шардах через `Distributed`-таблиці): часткові агрегати рахуються на шардах, потім зливаються. Це те, що дає ClickHouse швидкість на мільярдах рядків (десятки–сотні млн рядків/с).
- Для важких `GROUP BY` за рядковими ключами стеж за пам'яттю (`max_memory_usage`, `max_bytes_before_external_group_by` для «проливання» на диск).

## Стратегія вставки

**Батчі.** Мінімум **1 000 рядків**, оптимально **10 000–100 000**; орієнтир — приблизно **один `INSERT` на секунду**. Кожен `INSERT` створює новий парт із власними метаданими й індексом, тож дрібні часті вставки заганяють фонові злиття в борг і дають **`Too many parts`**.

**Синхронна вставка (за замовчуванням)** — коли батчити можна на клієнті. Кожен `INSERT` одразу створює парт.

> **Ідемпотентність ретраю залежить від рушія — і це не дрібниця.** Дедуплікація вставок увімкнена за замовчуванням **лише для `Replicated*MergeTree`** (`replicated_deduplication_window`, `replicated_deduplication_window_seconds`). Для **звичайного, нереплікованого `MergeTree`** її контролює окреме налаштування `non_replicated_deduplication_window`, і **за замовчуванням вона вимкнена** — заради зворотної сумісності. Тобто на одновузловій таблиці `MergeTree` повтор після таймауту **тихо задублює батч**: помилки не буде, будуть завищені суми. Або вмикай `non_replicated_deduplication_window` явно, або став власний `insert_deduplication_token`, або тримай ідентифікатор події в даних і чисть `ReplacingMergeTree`.

**Асинхронна вставка** — коли батчити на клієнті не виходить (обсервабіліті, багато дрібних подій від різних продюсерів): сервер сам накопичує буфер і скидає його за порогом.

```sql
SET async_insert = 1, wait_for_async_insert = 1;
```

| Налаштування | Дефолт | Що робить |
|---|---|---|
| `async_insert_max_data_size` | 100 MiB | розмір буфера до скидання |
| `async_insert_busy_timeout_ms` | 200 мс (1000 у Cloud) | час до скидання |
| `async_insert_max_query_number` | 450 | кількість запитів до скидання |

- **`wait_for_async_insert = 1` — рекомендований дефолт:** сервер підтверджує лише після запису на диск. З `0` клієнт отримує «ок» ще до фіксації — це втрата даних при збої й неочевидна обробка помилок.
- **Граблі:** асинхронна вставка **вимикає дедуплікацію** (вмикається окремо, і не варто вмикати за наявності залежних матеріалізованих подань). Тобто «async + продюсер з at-least-once» = дублікати, і жоден ретрай не є безпечним за замовчуванням.
- Висока кардинальність ключа партиціювання дає `Too many parts` **навіть з async-вставками** — async лікує дрібні вставки, а не поганий ключ.
- **Інтерфейс:** нативний протокол (порт 9000, формат Native, поблокове стиснення) швидший, бо перекладає парсинг на клієнта; HTTP (8123) сумісніший і зручніший за балансувальником, підтримує 70+ форматів.
- **Стиск:** `LZ4` за замовчуванням (дешевий по CPU, зменшує обсяг більш ніж удвічі), `ZSTD` — коли платиш за трафік (крос-регіон, хмарний egress).

## Мутації та `OPTIMIZE FINAL` — дві найпоширеніші «самострільні» операції

### Уникай мутацій (`ALTER TABLE … UPDATE/DELETE`)

- Мутація **не змінює рядки на місці**: вона асинхронно **переписує цілі парти**, яких торкнулася зміна. Оновлення одного рядка може переписати гігабайти — write amplification.
- Нові парти атомарно замінюють старі, коли будуть готові. **Мутацію не можна відкотити**, і вона продовжиться навіть після рестарту сервера — зупинити можна лише через `KILL MUTATION`.
- Замість мутацій:
  - `ReplacingMergeTree` / `CollapsingMergeTree` — виправлення даних при злитті або в запиті;
  - **lightweight delete** (`DELETE FROM …`) — дешевше за `ALTER TABLE … DELETE`;
  - `DROP PARTITION` — коли викидається цілий період;
  - `TTL` — автоматичний ретеншн замість ручних чисток.

### Уникай `OPTIMIZE TABLE … FINAL`

- Змушує злити **всі активні парти в один**: розпакувати → злити → запакувати → записати. Дуже дорого по CPU й I/O.
- У штатному режимі ClickHouse **не зливає парти, більші за ~150 GB** (`max_bytes_to_merge_at_max_space_in_pool`). **`OPTIMIZE FINAL` цей запобіжник ігнорує** → довгі злиття, тиск на пам'ять, OOM.
- **Наслідок парадоксальний і тому показовий:** роздутий парт далі вже майже не зливається, тож у `ReplacingMergeTree` **дублікати починають накопичуватися саме через те, що їх «прибирали» через `OPTIMIZE FINAL`**.
- Правильно: довіряти фоновим злиттям (вони інкрементні й зважають на ресурси), а для коректного результату «тут і зараз» — модифікатор `FINAL` у `SELECT` (працює найкраще, коли фільтр збігається зі стовпцями первинного ключа) або агрегація в самому запиті.

## Time-series / IoT (типовий сценарій ClickHouse)
ClickHouse часто беруть під метрики/телеметрію/IoT. Ключові прийоми:
- **Схема й сортування:** `ENGINE = MergeTree PARTITION BY toYYYYMM(ts) ORDER BY (device_id, ts)` — сортування за (сутність, час) різко пришвидшує діапазонні запити по часу.
- **Кодеки стиснення (величезна економія на впорядкованих даних), ланцюжкові:**
  - `Delta`, `DoubleDelta` — для **впорядкованих таймстампів**/монотонних лічильників;
  - `Gorilla` — для **float-метрик (gauge)**;
  - `T64` — для цілих із обмеженим діапазоном.
  ```sql
  created_at DateTime CODEC(DoubleDelta, LZ4),
  usage_user Float64  CODEC(Gorilla, LZ4)   -- кодек + загальний компресор (LZ4/ZSTD)
  ```
- **`LowCardinality(String)`** для тегів/імен метрик з малою кардинальністю.
- **TTL для ретеншну й даунсемплінгу:** `TTL ts + INTERVAL 30 DAY DELETE` (видалення старого) або `TTL ts + INTERVAL 7 DAY GROUP BY device_id, toStartOfHour(ts)` (агрегація «сирих» у погодинні).
- **Матеріалізовані подання** (`SummingMergeTree`/`AggregatingMergeTree`) для інкрементних роллапів (хвилина→година→доба) на льоту при вставці.
- Порівняно зі спеціалізованими TSDB (InfluxDB): стиснення трохи гірше, зате повноцінний SQL, JOIN-и і масштабування.

## Вибір типів даних (прямий вплив на стиск і швидкість)

- **Найвужчий числовий тип, що вміщає діапазон:** `UInt16` замість `Int32`, якщо від'ємних значень немає, а діапазон 0–65535. Числа не тримати в `String` — це ламає і семантику, і стиск.
- **`LowCardinality(String)` — орієнтир до ~10 000 унікальних значень** (словникове кодування). Вище цієї межі зазвичай уже шкодить.
- **`Enum8`/`Enum16`** для скінченної множини значень: 8 або 16 біт залежно від кількості варіантів, плюс безкоштовна валідація на вставці.
- **`FixedString(N)`** — лише для справді фіксованої довжини (коди країн, валют); інакше `String`/`LowCardinality(String)`.
- **Дата/час — якомога грубіше:** `Date`/`Date32` для дат без часу; `DateTime` замість `DateTime64`, якщо мілісекунди не потрібні насправді.
- **`Nullable` має ціну:** ClickHouse тримає окремий службовий стовпець `UInt8` з масками null. Де можливо — дефолт (`0`, `''`) замість `NULL`.

  > **Але заміна `Nullable` на `DEFAULT 0` не є семантично нейтральною.** `SUM` не зміниться (агрегати ігнорують `NULL`, а нулі нічого не додають) — і саме тому помилку не помічають. Зате `AVG`, `COUNT(col)` і будь-яка метрика, де `col` стоїть у знаменнику, **зміняться**: `NULL` з агрегації випадав, а `0` у неї входить. На таблиці замовлень знаменник стрибає з «замовлень із виручкою» на «всі замовлення», і середній чек обвалюється в рази — без жодного повідомлення про помилку. Якщо треба відрізняти «значення 0» від «значення немає» — лишай `Nullable` або додай окремий прапорець `has_value UInt8`. Механіка — `fundamentals-and-gotchas.md` §3.

## Індекси пропуску даних (data skipping)

Вторинний індекс у ClickHouse **не шукає рядок** — він дозволяє **пропустити блок гранул**, у якому шуканого значення точно немає.

```sql
ALTER TABLE logs ADD INDEX idx_err error_code TYPE bloom_filter GRANULARITY 4;
```

| Тип | Для чого |
|---|---|
| `minmax` | діапазонні фільтри по «майже впорядкованому» стовпцю |
| `set(N)` | до N різних значень на блок — низька кардинальність |
| `bloom_filter` | «голка в копиці сіна»: рідкісні значення, точний збіг |
| `text` | інвертований індекс для повнотекстового пошуку по вільному тексту |

- **`tokenbf_v1` / `ngrambf_v1` — застарілі (deprecated);** для нового коду бери `text`.
- **Головна умова корисності — кореляція з первинним ключем** (або те, що схожі значення потрапляють в одну вставку). Індекс по стовпцю, значення якого рівномірно «розмазані» по всіх блоках, **не пропустить жодного блоку** — це чисті накладні витрати на розмір і на вставку.
- `GRANULARITY` — скільки гранул покриває один запис індексу: менше → точніше відсікання, але більший індекс.
- **Не вгадуй — заміряй:** `EXPLAIN indexes = 1 SELECT …` і `SET send_logs_level = 'trace'`; порівнюй прочитані рядки/гранули до і після на реальних даних і реальних запитах.

## Матеріалізовані подання: інкрементні vs оновлювані

| | Інкрементне (класичне) | Оновлюване (`REFRESHABLE`) |
|---|---|---|
| Тригер | на кожній вставці в джерело | за розкладом |
| Що робить | виконує запит **на новому блоці** й дописує результат у цільову таблицю | **перевиконує весь запит** і перезаписує результат |
| Ціна | зсуває обчислення на час вставки | періодичний сплеск навантаження |
| Коли | реалтайм-агрегати, фільтри, прості перетворення однієї таблиці | «топ-N», довідники, складні join-и й денормалізація, batch-конвеєри в стилі dbt DAG |

- Оновлювані подання пишуть у режимі `REPLACE` (дефолт — перезапис поточного стану) або `APPEND` (накопичення знімків, тобто історія).
- **Правило розкладу:** не став оновлення частіше, ніж запит встигає порахуватися — «раз на 10 с» для запиту тривалістю 10 с з'їсть кластер.
- Результат оновлюваного подання має бути **обмеженим за розміром**; необмежене зростання деградує все.

## JSON-тип: коли доречний

- **Бери `JSON`**, коли структура справді непередбачувана (ключі різняться між записами, типи змінюються) **і ти фільтруєш/агрегуєш по конкретних шляхах усередині**: логи, події, UGC. Тип розкладає об'єкт на **підстовпці** — саме тому це й працює.
- **Не бери `JSON`**, коли схема відома й стабільна (звичайні стовпці, `Tuple`, `Array`) або коли документ **непрозорий**: якщо його лише кладуть і віддають цілим, не аналізуючи всередині БД — це `String`.
- **Ціна:** вставки повільніші (розбір, виведення типів, гнучке зберігання). Підказки типів (type hints) прибирають виведення типів і зайву непряму адресацію.
- Тримай `max_dynamic_paths` **нижче 10 000**.

## Типові граблі / особливості
- **Не для OLTP:** точкові `UPDATE`/`DELETE` — важкі мутації (`ALTER TABLE ... UPDATE`), асинхронні; не роби їх часто.
- **`Nullable`** має накладні витрати — уникай, якщо можна дефолтом (0/'').
- Типи явні й розмірні: `UInt8/32/64`, `Int*`, `Float64`, `Decimal`, `LowCardinality(String)`, `DateTime`/`DateTime64` — правила вибору див. «Вибір типів даних» вище.
- `SELECT *` на широкій колонковій таблиці особливо марнотратний — читай лише потрібні колонки (тут це критично для продуктивності).
- Первинний ключ не забезпечує унікальність — не покладайся на нього як на PK з OLTP-світу.

## Експлуатація та залізо (вимоги під прод)

Не «тюнінг заради тюнінгу»: без цих речей кластер дає рвану латентність на рівному місці, а на співбесідах саме про них і питають.

- **CPU:** governor у `performance`; стежити за тротлінгом (`dmesg`, `turbostat`).

  ```bash
  echo 'performance' | sudo tee /sys/devices/system/cpu/cpu*/cpufreq/scaling_governor
  ```

- **Віртуалізація:** прокидати CPUID — `cpu_mode=host-passthrough` (OpenStack/nova) або `<cpu mode='host-passthrough'/>` (libvirt). Інакше ClickHouse не побачить AVX і втратить швидкість на рівному місці.
- **RAM:** від **32 GB**; для інтерактивної аналітики на великих даних — **128 GB+**. `vm.overcommit_memory` = 0 або 1. Вивільнити кеш алокатора — `SYSTEM JEMALLOC PURGE`.
- **Малі системи (<16 GB)** — окремий режим: mark cache ~500 MB, 1 потік на запит, `max_block_size` 8192 (можна до 1024), `max_download_threads` 1, вимкнути паралельний парсинг/форматування й лог-таблиці (`metric_log`, `text_log`, `trace_log`, `asynchronous_metric_log`).
- **Transparent Huge Pages → `madvise`** (не `always`): класична причина рваних затримок.

  ```bash
  echo 'madvise' | sudo tee /sys/kernel/mm/transparent_hugepage/enabled
  ```

- **Диски:** SSD; для HDD — 7200 RPM. RAID-10/5/6/50 для HDD; RAID-0 для SSD **лише за наявності реплікації**. Блок 64 KB, страйп ~1 MB, підняти `stripe_cache_size` (напр. 4096).
- **ФС:** `ext4` з `noatime` (XFS теж прийнятна). **Не** використовувати стиснені/шифровані ФС, FAT-32, NFS.
- **Планувальник I/O:** `mq-deadline`/CFQ для HDD, `noop` для SSD; увімкнути NCQ і write cache для HDD, `fstrim` для NVMe/SSD.
- **Мережа:** від **10 Gbps** — реплікація й розподілені запити впираються саме в неї (1 Gbps працює, але це вузьке місце).
- **Координація:** віддавай перевагу **ClickHouse Keeper** над ZooKeeper. Якщо все ж ZooKeeper — 3.4.9+, на **окремих серверах** (або хоча б окремих дисках), heap `-Xms128M -Xmx1G`, G1GC з `-XX:MaxGCPauseMillis=50`, `autopurge.purgeInterval=1`, `snapCount=3000000`, `maxSessionTimeout=60000000`.
- **Антивірус** має виключати `/var/lib/clickhouse` — інакше деградація без видимої причини.
