# MySQL — довідник для аналізу запитів

Стосується MySQL 8.0+ (де відрізняється від 5.7 — позначено). Багато спільного з MariaDB, але не все.

## Ключовий синтаксис і особливості
- **Ліміт:** `LIMIT n` або `LIMIT offset, n` (не `TOP`). Пагінація: `LIMIT n OFFSET m`.
- **Ідентифікатори** беруться у зворотні лапки: `` `order` ``. Рядки — одинарні лапки; за замовчуванням `"..."` теж рядок (якщо не ввімкнено `ANSI_QUOTES`).
- **Автоінкремент:** `AUTO_INCREMENT` на PK. UPSERT: `INSERT ... ON DUPLICATE KEY UPDATE ...`.
- **Рушії:** `InnoDB` (за замовчуванням, транзакції, FK, кластерний PK) vs `MyISAM` (без транзакцій/FK). Аналізуючи продуктивність/блокування — уточнюй рушій.
- **Типи:** `TINYINT(1)`/`BOOLEAN` (немає справжнього boolean — це `TINYINT`); `DATETIME` vs `TIMESTAMP` (останній зберігається в UTC і має вужчий діапазон 1970–2038); `JSON`; `ENUM`. Для грошей — `DECIMAL`, не `FLOAT`.
- **Рядки/кодування:** використовуй `utf8mb4` (а не застарілий `utf8`, що лише 3 байти й не вміщає емодзі та частину символів).

## Віконні функції й CTE
- З'явилися у **MySQL 8.0** (`ROW_NUMBER/RANK/DENSE_RANK/LAG/LEAD`, `WITH` CTE, рекурсивні CTE). У **5.7 їх немає** — top-N роблять через змінні сесії або self-JOIN:
  ```sql
  -- Емуляція ROW_NUMBER у MySQL 5.7:
  SELECT t.*, @rn := @rn + 1 AS rn
  FROM my_table t, (SELECT @rn := 0) x
  ORDER BY sort_col;
  ```

## Типові граблі (крім спільних із fundamentals-and-gotchas.md)
- **`ONLY_FULL_GROUP_BY`:** у 8.0 увімкнено за замовчуванням — неагрегований стовпець поза `GROUP BY` дає помилку. Історичний код із 5.x, що на це покладався, «мовчки» повертав довільне значення групи.
- **Неявні перетворення типів:** `WHERE varchar_col = 123` порівнює через приведення й може ігнорувати індекс; `'10 apples' = 10` → `TRUE` (число парситься з початку рядка).
- **`GROUP_CONCAT`** обрізається за `group_concat_max_len` (за замовчуванням 1024 байти) — тихо втрачає дані.
- **`LIMIT` без `ORDER BY`** не гарантує порядок.
- Порівняння рядків за замовчуванням **регістронезалежне** (залежить від collation, напр. `utf8mb4_0900_ai_ci`).

## Партиціювання (і чому воно виключає FK)

`PARTITION BY RANGE / LIST / HASH / KEY` — є, але з обмеженнями, які найчастіше «вистрілюють» при перенесенні DDL фактових таблиць із PostgreSQL:

- **Партиційовані InnoDB-таблиці не підтримують `FOREIGN KEY`** — ні як сторона, що посилається, ні як та, на яку посилаються. Тобто вибір бінарний: **або партиції без FK** (посилальну цілісність забезпечує ETL), **або FK без партицій**. Спроба створити партиційовану таблицю з FK падає з помилкою — типове «чому не створюється таблиця».
- **Кожен унікальний ключ (зокрема PK) мусить містити всі стовпці виразу партиціювання.** Те саме правило діє в PostgreSQL, тож `PRIMARY KEY (date_key, payment_id)` для партиціонованого за `date_key` факту — коректно в обох СУБД.
- Вираз партиціювання має повертати ціле: `PARTITION BY RANGE (YEAR(sale_date))` або `RANGE COLUMNS(sale_date)` для дат напряму.
- `MAXVALUE`-секція для «хвоста» (як у range-партиціюванні загалом); для `LIST` spillover-секції не існує — нове значення потребує `ALTER TABLE`.

Порівняння з іншими: PostgreSQL — декларативне партиціювання з FK-підтримкою (обмеження на унікальність те саме); ClickHouse — `PARTITION BY` є завжди, але FK/UNIQUE немає в принципі.

## Продуктивність
- `EXPLAIN` / `EXPLAIN ANALYZE` (8.0.18+). Дивись `type` (уникай `ALL` — повний скан), `key`, `rows`.
- Функція на стовпці в `WHERE` вбиває індекс: `WHERE DATE(created_at) = '2024-01-01'` → повний скан; краще діапазон.
- InnoDB: вторинні індекси містять значення PK — короткий PK вигідний; випадковий PK (UUID v4) роздуває кластерний індекс (див. §15 каталогу).

## Інструменти
- **MySQL Workbench** — SQL-редактор, EER-моделювання, адміністрування, Migration Wizard (з MS Access/SQL Server/PostgreSQL). Годиться для скріншотів планів і схем у навчальних матеріалах.
