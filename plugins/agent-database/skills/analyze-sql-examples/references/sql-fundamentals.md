# Фундаментальні будівельні блоки SQL

Канонічні конструкції (з класичних підручників: Beaulieu «Learning SQL», Graeber, Kulikov). Крос-діалектно; для «граблів» див. `fundamentals-and-gotchas.md`, для розгорнутих рецептів «одна задача — п'ять діалектів» (пагінація, NULL-сортування, дати, pivot, ROLLUP, ієрархії) — `dialect-recipes.md`.

## Типи з'єднань (JOIN)
- **`INNER JOIN`** — лише рядки зі збігом в обох таблицях.
- **`LEFT/RIGHT [OUTER] JOIN`** — усі рядки з лівої/правої + збіги з іншої (де немає — `NULL`).
- **`FULL OUTER JOIN`** — усі рядки обох (у MySQL відсутній → емуляція через `LEFT UNION RIGHT`).
- **`CROSS JOIN`** — декартів добуток.
- **Self-JOIN** — таблиця з собою (ієрархії «співробітник→керівник», порівняння рядків).
- **Анти-join** — `LEFT JOIN … WHERE right.key IS NULL` або `NOT EXISTS` (рядки без пари).
- Умова з'єднання — в `ON`; пам'ятай про пастку «зовнішній join → внутрішній» (див. §6 каталогу).

## Підзапити
- **Некорельований** — виконується раз, результат підставляється (`WHERE x IN (SELECT …)`).
- **Корельований** — залежить від зовнішнього рядка, виконується для кожного (`WHERE EXISTS (SELECT 1 … WHERE t.id = outer.id)`).
- **`EXISTS` vs `IN`:** `EXISTS`/`NOT EXISTS` коректно працюють із `NULL` і часто ефективніші; `NOT IN` із `NULL` у списку **ніколи не повертає рядок** (див. §1). За замовчуванням для «чи є пов'язане» — `EXISTS`.
- Підзапит у `FROM` (похідна таблиця) і `WITH` (CTE) — для читабельності й повторного використання; рекурсивні CTE — для ієрархій/графів.

## Операції над множинами
- **`UNION`** (прибирає дублі) / **`UNION ALL`** (лишає) — об'єднання.
- **`INTERSECT`** — спільні рядки; **`EXCEPT`** (в Oracle — `MINUS`) — різниця.
- Вимоги: однакова кількість і сумісні типи стовпців; імена беруться з першого запиту.

## Представлення (VIEW)
- Збережений іменований `SELECT` — інкапсуляція складної логіки, обмеження доступу до стовпців. Звичайний VIEW — лише текст запиту (рахується щоразу); матеріалізований — зберігає результат (див. `postgresql.md`).
- Оновлюваність VIEW обмежена (прості — так; з агрегатами/JOIN — зазвичай ні).

## Транзакції та ACID
- **ACID:** Atomicity (усе або нічого), Consistency (правила БД дотримані), Isolation (паралельні транзакції не заважають), Durability (закоммічене збережене).
- `BEGIN … COMMIT/ROLLBACK`; `SAVEPOINT` для часткового відкату.
- **Рівні ізоляції** (від слабшого до сильнішого) і які аномалії допускають:
  | Рівень | Dirty read | Non-repeatable read | Phantom |
  |---|---|---|---|
  | READ UNCOMMITTED | так | так | так |
  | READ COMMITTED | ні | так | так |
  | REPEATABLE READ | ні | ні | так* |
  | SERIALIZABLE | ні | ні | ні |

  \* У PostgreSQL REPEATABLE READ (snapshot) не має і фантомів; дефолт PG/Oracle — READ COMMITTED, MySQL/InnoDB — REPEATABLE READ, MS SQL — READ COMMITTED.

## Крос-діалектна шпаргалка (MySQL / MS SQL / PostgreSQL / Oracle)
| Задача | MySQL | MS SQL (T-SQL) | PostgreSQL | Oracle |
|---|---|---|---|---|
| Обмежити рядки | `LIMIT n` | `TOP (n)` / `OFFSET…FETCH` | `LIMIT n` | `FETCH FIRST n ROWS` / `ROWNUM` |
| Автоключ | `AUTO_INCREMENT` | `IDENTITY(1,1)` | `GENERATED … IDENTITY` | `SEQUENCE`+`NEXTVAL` / `IDENTITY` (12c+) |
| NULL→значення | `IFNULL`/`COALESCE` | `ISNULL`/`COALESCE` | `COALESCE` | `NVL`/`COALESCE` |
| Поточний час | `NOW()`/`CURDATE()` | `GETDATE()` | `now()` | `SYSDATE`/`CURRENT_DATE` |
| Рядковий тип | `VARCHAR` | `NVARCHAR` | `TEXT` | `VARCHAR2` |
| «Порожня» таблиця для `SELECT` виразу | (не потрібна) | (не потрібна) | (не потрібна) | `DUAL` (`SELECT SYSDATE FROM dual`) |
| UPSERT | `ON DUPLICATE KEY UPDATE` | `MERGE` | `ON CONFLICT DO UPDATE` | `MERGE` |
| Конкатенація | `CONCAT()` | `+` / `CONCAT` | `\|\|` / `CONCAT` | `\|\|` / `CONCAT` |
| +N днів до дати | `d + interval 5 day` | `DATEADD(day,5,d)` | `d + interval '5 day'` | `d + 5` |
| Різниця дат (дні) | `DATEDIFF(d2,d1)` | `DATEDIFF(day,d1,d2)` | `d2 - d1` | `d2 - d1` |
| NULL при сортуванні | CASE-прапор / `IS NULL` | CASE-прапор | `NULLS FIRST/LAST` | `NULLS FIRST/LAST` |
| Випадкові n рядків | `ORDER BY RAND()` | `ORDER BY NEWID()` | `ORDER BY random()` | `ORDER BY dbms_random.value()` |
| Список із групи | `GROUP_CONCAT` | `STRING_AGG` (2017+) | `STRING_AGG` | `LISTAGG` |
| Підсумки ROLLUP | `GROUP BY j WITH ROLLUP` | `GROUP BY ROLLUP(j)` | `GROUP BY ROLLUP(j)` | `GROUP BY ROLLUP(j)` |

> Oracle **не входить** до основних діалектів агента, але часто трапляється в матеріалах — цю шпаргалку використовуй, щоб розпізнати Oracle-синтаксис і перекласти під цільову СУБД (пор. §16 каталогу — змішування діалектів). Розгорнуті варіанти кожного рядка (з пастками) — у `dialect-recipes.md`.

## Великі таблиці: партиціювання, шардинг (за Beaulieu, гл. 17)

Таблиця стає «завеликою», коли рутинні операції (повний скан, перебудова індексу, архівація, бекап) перестають вкладатися у вікна обслуговування — зазвичай від кількох десятків мільйонів рядків.

- **Горизонтальне партиціювання** — таблиця ділиться на секції з **неперетинними** підмножинами рядків за **ключем партиціювання**; для користувача це досі одна таблиця, оптимізатор читає лише потрібні секції (**partition pruning**), а адмін-операції (архівація, `DROP` старої секції) працюють посекційно.
- Методи: **range** (найчастіше за датою: секція = тиждень/місяць; тримай «хвостову» секцію `MAXVALUE`, інакше вставка майбутньої дати впаде), **list** (перелік значень: регіони, валюти; spillover-секції немає — нове значення вимагає `ALTER`), **hash** (рівномірний розподіл за хешем ключа з високою кардинальністю), **композитне** (range + hash-підсекції).
- **Індекси секціонованої таблиці:** *локальний* (окремий на кожну секцію) vs *глобальний* (спільний; потрібен запитам **без** фільтра за ключем партиціювання).
- **Діалектні обмеження, що ламають перенесення DDL:** унікальні ключі (зокрема PK) мусять містити всі стовпці ключа партиціювання (PostgreSQL і MySQL); у **MySQL/InnoDB партиційовані таблиці взагалі не підтримують `FOREIGN KEY`** — доводиться обирати «партиції або FK» (деталі — `mysql.md`); у ClickHouse FK/UNIQUE немає в принципі, цілісність — на боці ETL.
- Підтримка: PostgreSQL (`PARTITION BY RANGE/LIST/HASH`, декларативно з PG 10), MySQL (`PARTITION BY RANGE/LIST/HASH`), MSSQL (partition function/scheme), Oracle (найбагатша реалізація), ClickHouse (`PARTITION BY toYYYYMM(...)` — базовий патерн).
- **Шардинг** — те саме розбиття, але **між серверами** (кожен шард — окрема БД). Це вже архітектура застосунку: маршрутизація запитів, крос-шардові запити й транзакції — твоя відповідальність (пор. Vitess, Citus). Партиціювання — «шардинг усередині одного сервера» і має бути першим кроком.
- NoSQL-системи (Cassandra, MongoDB) будують шардинг у ядро (partition key / shard key) — тому вони «легко масштабуються горизонтально», але ціною гнучкості запитів (див. відповідні діалектні довідники).

## Еволюція стандартів SQL (за Machado/Russa, «Advanced SQL», O'Reilly)

Корисний навчальний каркас «звідки взялася фіча» (і чому в старих СУБД її немає):

| Стандарт | Ключове |
|---|---|
| SQL-86/89 | базові SELECT/типи; **без** JOIN-синтаксису (лише декартів добуток + фільтр), PK/GRANT — у 89 |
| **SQL-92** | «перший сучасний SQL»: JOIN-синтаксис, підзапити, UNION/INTERSECT, DATE/TIME/TIMESTAMP, FK/CHECK, транзакції та рівні ізоляції |
| **SQL:1999** | об'єктно-реляційне, тригери, збережені процедури, **CTE і WITH RECURSIVE** (фундамент dbt-стилю) |
| **SQL:2003** | **віконні функції** (RANK, кадри вікна), SEQUENCE/IDENTITY, SQL/XML |
| SQL:2008 | TRUNCATE, INSTEAD OF-тригери |
| **SQL:2011** | темпоральні таблиці (`FOR SYSTEM_TIME AS OF …` — «машина часу» для аудиту; пор. time travel у Snowflake/BigQuery/Delta) |
| **SQL:2016** | нативний **JSON** (JSON_VALUE/JSON_QUERY/JSON_TABLE) |
| **SQL:2023** | **property graph queries (SQL/PGQ)**: `MATCH (a)-[:REL*2..4]->(b)` поверх реляційних таблиць; поглиблений JSON-тип |

Конвергенція діалектів: більшість «фірмових» фіч (CTE, вікна, MERGE, JSON) тепер усюди; лишаються розбіжності в датах/рядках/регістрі ідентифікаторів (PostgreSQL приводить некв. ідентифікатори до нижнього регістру, Oracle — до верхнього, BigQuery/MySQL — case-insensitive) і в нестандартизованих зонах (повнотекст, геодані). Хмарні DW не «переписали» SQL: BigQuery — ANSI-подібний, Snowflake — близький до Oracle/ANSI (+`VARIANT` для JSON, зручний **`QUALIFY`** для фільтра за вікном без обгортки), Redshift — форк PostgreSQL.

## Рекурсивні CTE (ієрархії та ітерації в чистому SQL)

```sql
WITH RECURSIVE subordinates AS (
  SELECT id, name, manager_id, 0 AS level          -- якір: корінь ієрархії
  FROM employees WHERE manager_id IS NULL
  UNION ALL
  SELECT e.id, e.name, e.manager_id, s.level + 1   -- рекурсивний член: діти вже знайдених
  FROM employees e JOIN subordinates s ON e.manager_id = s.id
)
SELECT * FROM subordinates ORDER BY level, manager_id;
```

- Структура: **якірний член** ∪ **рекурсивний член** (посилається на власне ім'я CTE) через `UNION ALL`; ітерації тривають, поки рекурсивний SELECT повертає нові рядки.
- Застосування: оргструктури, bill-of-materials, обхід графів (усі досяжні вузли), генерація послідовностей чисел/дат без циклів.
- Завершення: гарантуй, що join зрештою не знаходить нових рядків; захисти від циклів у даних (PostgreSQL 14+: `CYCLE col SET is_cycle USING path`; SQL Server: `OPTION (MAXRECURSION n)`, дефолт 100; MySQL: `cte_max_recursion_depth`, дефолт 1000).
- Продуктивність: кожна ітерація — додаткові скани; індекс на стовпці зв'язку (manager_id) обов'язковий; на великих/глибоких ієрархіях вартість росте швидко.
- Діалекти: `WITH RECURSIVE` — PostgreSQL/MySQL 8+/SQLite/BigQuery; SQL Server і Oracle — просто `WITH` (без слова RECURSIVE); у Oracle історично `CONNECT BY PRIOR`.

## Мультипарадигмальний SQL сьогодні (куди вміє «дотягтися» одна СУБД)

- **JSON/XML:** зберігання + запити (`JSON_VALUE`, `JSON_QUERY`, `JSON_EXISTS`, `JSON_TABLE`; PostgreSQL — `jsonb` з GIN-індексом) — гібрид «реляційне + документне» без окремої NoSQL. Менше ETL: фільтрувати/проєктувати JSON може сама база.
- **Темпоральність:** system-versioned таблиці (SQL Server, Oracle, MariaDB) — «як виглядав рядок на дату X» одним `FOR SYSTEM_TIME AS OF`; у DW — time travel (Snowflake до 90 днів, BigQuery 7 днів, Delta Lake — версії таблиці).
- **Графи:** SQL/PGQ (SQL:2023) — патерни шляхів/сусідств у реляційній базі; для глибоких графових алгоритмів — усе ще спеціалізовані БД (Neo4j/Neptune).
- **Повнотекст:** PostgreSQL `to_tsvector @@ to_tsquery` + GIN; SQL Server `CONTAINS`; синтаксиси принципово різні (нестандартизована зона).
- **ML/LLM у SQL:** BigQuery ML (`CREATE MODEL`, `ML.PREDICT`, `ML.GENERATE_TEXT` через Vertex AI), Snowflake Cortex (`COMPLETE()`, `CLASSIFY_TEXT()` тощо — LLM як SQL-функції; дані не покидають периметр), PostgreSQL — розширення (PostgresML, pgvector — див. `db-connectivity-cloud/references/vector-databases.md`). Тренд — «ML до даних», а не дані до ML.
- **SQL як мова трансформацій (dbt/ELT):** трансформації = SQL-моделі (кожна — SELECT), версіонування в git, тести, документація, автоматичний порядок виконання за залежностями; семантичний шар — метрика визначається один раз. Інженерні практики (код-рев'ю, CI/CD) приходять в аналітику. Див. `design-dwh-etl`.
- **DuckDB** — «SQLite для аналітики»: in-process OLAP, колонкове зберігання, вміє SQL прямо по Parquet/CSV-файлах і Pandas DataFrame без завантаження; десятки мільйонів рядків на ноутбуці. Чудовий інструмент для навчальних демо й локального прототипування.

## Оптимізатор, EXPLAIN і стратегії JOIN

- СУБД обирає план **за вартістю** на основі **статистики** (кількість рядків, гістограми, селективність). Застаріла статистика → хибна оцінка кардинальності → поганий план. Після масових завантажень: `ANALYZE` (PG), авто-статистика в хмарних DW.
- **`EXPLAIN`** — план без виконання; **`EXPLAIN ANALYZE`** — план + фактичні часи й рядки. Головний діагностичний прийом: порівняй `rows=` (оцінка) з фактом — розбіжність на порядки = проблема статистики/виразів.
- Три стратегії з'єднання, які видно в плані: **nested loop** (одна сторона мала або є селективний індекс), **hash join** (великі таблиці без індексів; потребує пам'яті — стеж за spill на диск), **merge join** (обидві сторони відсортовані за ключем). Порядок з'єднань таблиць оптимізатор перебирає сам — допомагай йому ранніми фільтрами.
- Seq scan замість index scan — не завжди помилка: якщо фільтр повертає велику частку таблиці, повний скан дешевший за мільйони точкових звернень.
- Типи індексів PostgreSQL поза B-tree: **GIN** (масиви, `jsonb`, повнотекст), **GiST** (геометрія, діапазони), **BRIN** (величезні природно-впорядковані таблиці — метадані на блок, майже безплатний), hash (рідко виграє в B-tree). Плюс: **covering** (`INCLUDE(col)` — index-only scan), **часткові** (`WHERE status='active'`), **функціональні** (`ON t (lower(email))` — відповідь на не-sargable фільтри).
- Анти-патерни виконання (доповнення до каталогу): **RBAR** (row-by-agonizing-row — цикл у застосунку з одиночними UPDATE замість одного set-based `UPDATE … WHERE id IN (…)`); відсутність **prepared statements** при масових однотипних запитах (повторний парсинг+планування); **одна гігантська транзакція** на мільйони рядків (роздуті журнали, блокування — розбивай на батчі); guess-тюнінг без погляду в план.

## Тюнінг у хмарних DW: індексів немає

- **BigQuery/Snowflake не мають користувацьких індексів.** Еквівалент: **партиціювання** (BigQuery: за датою/int-діапазоном) + **кластеризація** (сортування в межах партицій за 1–4 стовпцями); Snowflake — автоматичні **мікропартиції** з метаданими (min/max) і pruning, опційно clustering key. Правило: *PostgreSQL/MySQL — став індекси; BigQuery/Snowflake — проєктуй партиції та кластеризацію і **фільтруй за ключем партиціювання***.
- **Кеш результатів:** Snowflake повертає кешований результат ідентичного запиту (ігнорує коментарі/пробіли), BigQuery — до 24 год за точного збігу тексту; будь-яка зміна даних інвалідовує. Для BI-запитів — тримай текст запиту стабільним.
- **Матеріалізовані представлення:** передобчислений результат «як таблиця». BigQuery/Snowflake **автоматично переписують** запит на MV, якщо він може бути відповіданий нею (навіть якщо запит іде до базової таблиці!); PostgreSQL — MV є, але без автопереписування і з ручним `REFRESH MATERIALIZED VIEW`. Компроміс: місце + вартість підтримки vs швидкість; ідеально для append-mostly фактів з важкими агрегаціями (типовий патерн: денний/місячний rollup великої факт-таблиці).
- Compute-моделі й вартість: BigQuery — слоти (on-demand платиш за **проскановані байти** → `SELECT *` буквально коштує грошей; dry-run для оцінки; квоти), Snowflake — віртуальні склади (кредити за час × розмір; **auto-suspend 1–5 хв**, окремі склади під ETL/BI = ізоляція навантажень, multi-cluster для конкурентності, Resource Monitors проти сюрпризів у рахунку).
- Діагностика: BigQuery — execution graph, `INFORMATION_SCHEMA.JOBS` (шукай високі slot-ms за малого виходу — декартові добутки/повторні скани); Snowflake — Query Profile (вузол «Most Expensive»; багато Exchange-операторів = перерозподіл даних через погані ключі; spill на диск = замалий склад).
