# AWS — керовані сервіси баз даних

Зіставлення кожної підтримуваної БД із сервісом AWS + як під'єднатися з Python і на що зважати.

## Реляційні (MySQL, PostgreSQL, SQL Server)
- **Amazon RDS** — керований MySQL/PostgreSQL/SQL Server/MariaDB/Oracle: бекапи, патчі, Multi-AZ, репліки читання.
- **Amazon Aurora** — MySQL/PostgreSQL-сумісний, розділене сховище, авто-масштабування, швидші відмовостійкість і репліки. Дорожче, але продуктивніше.
- Під'єднання: звичайні драйвери (`psycopg`, `pymysql`, `pyodbc`) до endpoint-а RDS/Aurora. Пароль — зі **Secrets Manager**; краще — **IAM database authentication** (тимчасові токени замість пароля).
- PL/pgSQL працює як у звичайному PostgreSQL (RDS/Aurora PostgreSQL).

### RDS: експлуатаційний мінімум (за «AWS in Action», гл. 10)
- **Бекапи.** *Автоматичні снапшоти* — задаєш вікно (`PreferredBackupWindow`) і строк зберігання (`BackupRetentionPeriod`, 0 = вимкнено); видаляються разом з інстансом. *Ручні снапшоти* живуть, доки не видалиш сам (важливий автоснапшот копіюй у ручний). Завдяки журналу змін можливий **point-in-time restore** до будь-якої хвилини в межах retention. Відновлення завжди створює **новий** інстанс (не «в наявний») — застосунок треба перемкнути на новий endpoint. Снапшот можна скопіювати в інший регіон (DR/переїзд; зваж на комплаєнс).
- **Multi-AZ (висока доступність).** `MultiAZ: true` → primary + standby у **різних зонах доступності**, синхронна реплікація, автоматичний failover через перемикання DNS-імені; заодно обслуговування проходить без даунтайму (перемикання на standby). Для продуктиву — обов'язково; варіант із **двома** standby (PG/MySQL) — standby читаються, failover < 60 с. В Aurora інакше: спільний кластерний том на кілька дисків, репліка промоутиться ~за хвилину.
- **Репліки читання** — **асинхронна** реплікація на окремі read-only інстанси (MySQL/MariaDB/PG/Oracle/MSSQL; потрібні ввімкнені бекапи). Розподіл читання/запис — на боці застосунку. Репліку можна **промоутити** в самостійну БД (міграція, зняття навантаження перед важким DDL) і створювати в іншому регіоні. Не плутай: Multi-AZ — це доступність (синхронно; класичний одиночний standby не читається), read replica — масштабування читань (асинхронно, можливий лаг).
- **Три рівні контролю доступу:** (1) IAM-політики — хто керує *конфігурацією* RDS (create/modify/delete; окремою політикою забороняй `rds:Delete*`/`rds:Remove*` від людських помилок); (2) security groups — хто має *мережевий* доступ (тільки SG застосунку на порт БД); (3) користувачі/GRANT самої СУБД — хто що бачить у *даних*. IAM db auth замінює паролі токенами (15 хв, MySQL/MariaDB/PG).
- **Масштабування:** вертикально (клас інстансу; коротка перерва — з Multi-AZ непомітна) і сховище (gp3/io1 для продуктиву). Стеля вертикалі скінченна — на відміну від S3/DynamoDB, які масштабуються горизонтально «без меж».
- **Моніторинг:** метрики в CloudWatch (CPU, freeable memory, free storage) + алярми на пороги; повільні запити — штатними засобами двигуна (slow query log / pg_stat_statements).

## Кешування (за «AWS in Action», гл. 11)
- **Amazon ElastiCache** — керований **Redis** або **Memcached** (Redis багатший: складні типи, транзакції, Lua, реплікація/failover, бекапи). AWS ставить, патчить, моніторить (CloudWatch) і перемикає репліки за тебе; ти обираєш `cache.<тип>` вузли й subnet group, як у RDS.
- **Amazon MemoryDB for Redis** — Redis-сумісна **основна** БД (довговічність через розподілений журнал транзакцій), коли in-memory сховище має бути не лише кешем.
- Стратегії кешування (lazy loading vs write-through, TTL, LRU-виселення) — див. `redis.md`.

## MongoDB-сумісна
- **Amazon DocumentDB** — сумісний із MongoDB API (не справжній MongoDB; підтримує підмножину можливостей/версій). Драйвер `PyMongo` з TLS-сертифікатом Amazon. Для повного MongoDB — MongoDB Atlas у AWS-регіоні.

## Cassandra-сумісна
- **Amazon Keyspaces** — керований, сумісний з Apache Cassandra (CQL), serverless-оплата. Драйвер `cassandra-driver` з SigV4/сервісними обліковими даними. Деякі можливості Cassandra обмежені.

## Ключ-значення / документна нативна
- **DynamoDB** — нативний serverless NoSQL (ключ-значення/документ), не сумісний з CQL/Mongo. Python — `boto3`. Модель доступу як у Cassandra: проєктуй під запити (partition/sort key, GSI).
- Чому NoSQL масштабується там, де реляційній важко: ACID-транзакції на кластері вимагають двофазного коміту між вузлами — координація дорожчає з кожним вузлом; DynamoDB жертвує цими гарантіями заради лінійного горизонтального масштабування (від 1 запиту/с до десятків тисяч).
- Модель даних: таблиця → item-и (без фіксованої схеми, атрибути в кожного свої) → обов'язковий лише **первинний ключ**: або самий **partition key** (`HASH`), або **partition + sort key** (`HASH`+`RANGE`; комбінація унікальна, sort key відсортований усередині партиції — патерн «усі задачі користувача за часом»). Пропускна здатність — on-demand або provisioned RCU/WCU.
- Це БД для застосунку, а не для ad-hoc аналітики: **застосунок треба писати під DynamoDB** (не можна «перевести» готовий SQL-застосунок). Інші NoSQL-сервіси AWS: Keyspaces (CQL), DocumentDB (Mongo API), Neptune (графова), MemoryDB (Redis).

## Колонкова аналітика (для ClickHouse-подібних задач)
- **Amazon Redshift** — MPP-сховище для аналітики (колонкове). **Athena** — SQL над файлами в S3 (Presto/Trino). ClickHouse у AWS — самостійно на EC2/EKS або **ClickHouse Cloud** (керований, у т.ч. на AWS).

## Інші purpose-built двигуни (за «AWS Certified Database Study Guide», оглядово)

- **Neptune** — графова (Gremlin/openCypher/SPARQL): соцзв'язки, рекомендації, fraud-графи.
- **Timestream** — часові ряди (IoT/метрики): автотиринг пам'ять→магнітне сховище, time-series функції.
- **QLDB** — ledger: незмінний криптографічно верифікований журнал (аудит володіння/транзакцій), централізований (не блокчейн).
- **OpenSearch Service** — повнотекст + логи + k-NN векторний пошук (пара для DynamoDB через zero-ETL).
- **ElastiCache/MemoryDB** — див. вище. Правило вибору сервісу — PIE-теорема й опорні питання: `nosql-aws-best-practices.md`.

## Міграція БД у AWS: SCT + DMS

- **SCT (Schema Conversion Tool)** — конвертує **схему й код** (DDL, процедури) між двигунами; звіт покаже, що не конвертується автоматично (типова причина відмови від гетерогенної міграції). Гомогенно (Oracle→RDS Oracle) SCT не потрібен.
- **DMS (Database Migration Service)** — переносить **дані**: full load, full load + CDC (безперервна реплікація до перемикання; мінімальний даунтайм), або CDC-only. Патерн гетерогенної міграції: SCT (схема) → DMS full load + CDC → перемикання застосунку. DMS також уміє постійну реплікацію для аналітики (OLTP → Redshift/S3).

## Disaster Recovery: RPO/RTO і чотири стратегії

- **RPO** — скільки даних можна втратити (час від останньої відновлюваної точки); **RTO** — за скільки треба відновитися. Визначаються бізнесом *до* вибору технології; що агресивніші — то дорожче.
- Спектр стратегій (за зростанням вартості): **backup & restore** (RPO = вік бекапа, RTO = час відновлення; автоснапшоти RDS реплікуй в інший регіон) → **pilot light** (мінімальна копія: репліка БД в іншому регіоні, компут вимкнено) → **warm standby** (зменшена, але робоча копія стека) → **multi-site active-active** (Aurora Global Database — вторинний регіон читає, промоушн ~хвилина; DynamoDB global tables — RPO≈0). Усередині регіону Multi-AZ ≠ DR: це висока доступність, а DR — це інший **регіон**.
- Особливість Aurora: сховище на 6 копій/3 AZ — відновлення після збою вузла зазвичай без відновлення з бекапа (RPO=0 в межах регіону).

## Мережа й безпека
- Тримай БД у приватних підмережах VPC; доступ — через security groups; уникай публічного endpoint-а.
- TLS обов'язково; секрети — Secrets Manager/Parameter Store; доступ додатків — через IAM-ролі, не статичні ключі.

## Приклад під'єднання (RDS PostgreSQL, IAM-токен — ескіз)
```python
import os, boto3, psycopg
token = boto3.client("rds").generate_db_auth_token(
    DBHostname=os.environ["RDS_HOST"], Port=5432, DBUsername="app_user")
conn = psycopg.connect(host=os.environ["RDS_HOST"], user="app_user",
                       password=token, dbname="shop", sslmode="require")
```
