# Родини (жанри) баз даних і теорема CAP

За «Сім баз даних за сім тижнів» (Е. Редмонд, Дж. Вілсон) з осучасненням. Використовуй, коли треба пояснити, **який тип БД обрати під задачу**, або чому в курсі поруч живуть PostgreSQL, Cassandra, ClickHouse, MongoDB і Redis.

## П'ять жанрів: сильні/слабкі сторони

| Жанр | Приклади | Добре для | Погано для |
|---|---|---|---|
| **Реляційні** | PostgreSQL, MySQL, MS SQL, SQLite | структура даних відома наперед, а майбутні запити — ні («плати складністю схеми авансом за гнучкість запитів»): замовлення, склад, облік; транзакції, цілісність | змінна/глибоко вкладена структура; горизонтальне масштабування записів |
| **Ключ-значення** | Redis, DynamoDB, Aerospike, Riak | максимальна швидкість і масштабованість простих операцій за ключем: сесії, кеші, лічильники — дані без зв'язків між собою | будь-які запити, складніші за CRUD за ключем (немає індексів за значенням/сканування) |
| **Стовпцеві (wide-column)** | Cassandra, HBase, (аналітичні колонкові: ClickHouse, Redshift) | «великі дані» на кластерах із десятків/сотень вузлів; запис-інтенсивні потоки; схема проєктується **під відомі запити** | ad-hoc запити, коли шаблони доступу заздалегідь невідомі |
| **Документні** | MongoDB, CouchDB, DocumentDB | змінна предметна область, документ ≈ об'єкт застосунку; легко шардуються | звички реляційного світу: складні JOIN-и по нормалізованій схемі; денормалізація тут — норма, а не гріх |
| **Графові** | Neo4j, Neptune | сильно зв'язані дані: соцмережі, рекомендації, права доступу (ACL), маршрути — «все, що малюється на дошці кружечками й стрілками» | великі обсяги «пласких» даних; шардинг графа між машинами принципово важкий |
| **Векторні** *(новий жанр, era GenAI)* | Milvus/Zilliz, Pinecone, Qdrant; інтегровано: pgvector, Atlas, Cosmos DB, Cassandra 5 | семантичний пошук по неструктурованому (текст/зображення/аудіо через ембединги), RAG, рекомендації, семантичний кеш LLM | точні обчислення/транзакції; пошук за точним збігом (це до реляційних/FTS); детально — `vector-databases.md` |

Нюанси вибору поза жанром: гранична ємність (стовпцеві — найбільша, графові — найменша), потрібні ad-hoc запити чи вистачає mapreduce/агрегацій, інтерфейс (HTTP/REST vs бінарний драйвер), інструменти масової загрузки, вимоги до довговічності/безпеки.

## Теорема CAP (для розподілених систем)

Із трьох властивостей одночасно гарантуються лише **дві**:
- **C (Consistency)** — після запису всі читання бачать нове значення;
- **A (Availability)** — система відповідає, поки живий хоч один вузол;
- **P (Partition tolerance)** — система працює попри втрату зв'язності між вузлами.

Для **розподіленої** системи P — не опція, а факт життя (мережа рветься) → реальний вибір: **CP** (при розриві жертвуємо доступністю: частина запитів відхиляється, зате без суперечностей — MongoDB, HBase у типовій конфігурації) або **AP** (відповідаємо завжди, але, можливо, старими даними — CouchDB, Cassandra з низьким рівнем узгодженості). Нерозподілені системи (одновузлові PostgreSQL/Redis/Neo4j) — «CA»: для них CAP не має сенсу.

- **Eventual consistency (узгодженість у кінцевому підсумку):** вузли відповідають завжди, а зміни розповсюджуються у фоні — у кожен момент можлива несвіжість, але зрештою всі зійдуться (класичний приклад — DNS). Cassandra дозволяє **налаштовувати** узгодженість на рівні запиту (`ONE`/`QUORUM`/`ALL`) — рухаючи систему по осі C↔A (див. `analyze-sql-examples/references/cassandra-cql.md`).
- CAP — не вся картина: на практиці не менш важлива **затримка** (latency); системи типу Dynamo проєктували насамперед під гарантований час відповіді.

## Кейс порівняння всередині жанру: Cassandra vs Aerospike

Приклад того, що вибір не закінчується жанром (обидві — розподілені KV/wide-column). Джерело — порівняння Aerospike (вендорське! маркетингові тези фільтруй), технічні факти перевіряються:

| Вісь | Cassandra | Aerospike |
|---|---|---|
| Модель | wide-column, кластерні ключі | multi-model: KV + документи (bins зі вкладеними list/map) |
| Узгодженість | лише AP; 9 рівнів consistency на запит | AP **або** CP (strong consistency пройшла Jepsen-тест 2018) |
| Реплікація | типово RF=3, read-repair | типово RF=2, roster-based, self-healing failover |
| Двигун зберігання | LSM-дерево (JVM → GC-паузи в p99) | лог-структурований, написаний на C, гібрид RAM+SSD (працює з raw-блоками NVMe) |
| Індекси | primary сильний; secondary обмежені (SAI/SASI) | primary+secondary production-ready, але **один індекс на запит** |
| CDC | складний, дублікати | вбудований, батчами, тільки остання версія |

Мораль для вибору: усередині жанру порівнюй узгодженість (AP vs CP), хвости латентності (GC!), вартість заліза (RF, RAM vs SSD), зрілість вторинних індексів і CDC. Вендорські бенчмарки «у нас удвічі менше серверів» — перевіряй незалежно.

## Полістор (polyglot persistence)

Реальні системи дедалі частіше комбінують БД: реляційна — транзакційне ядро, Redis — кеш/черги, колонкова — аналітика, документна — контент, графова — зв'язки. Плата — оркестрація й синхронізація між сховищами (ETL/CDC), тож комбінуй за виміряною потребою, а не «бо модно». Канонічна помилка — тягнути один інструмент на всі задачі (реляційну СУБД як чергу повідомлень, Mongo як аналітичне сховище, Redis як єдине джерело правди).

## Як застосовувати при виборі БД під задачу

1. Визнач природу даних: регулярна структура? документи? пари ключ-значення? граф зв'язків? часові ряди/події (→ колонкова аналітика)?
2. Визнач патерни доступу: відомі наперед (можна проєктувати під запити — Cassandra/DynamoDB) чи ad-hoc (реляційна/ClickHouse для аналітики)?
3. Оціни масштаб: влазить в один сервер (+ репліки) → реляційна майже завжди виграє; потрібен кластер на запис → стовпцеві/документні/KV.
4. Вимоги до узгодженості: суворі транзакції між сутностями → реляційна (або обмежено Mongo 4+); допустима eventual consistency → ширший вибір.
5. Зафіксуй компроміс CAP/latency явно в «Обмеженнях» відповіді.
