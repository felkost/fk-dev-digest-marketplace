# Redis (кеш / key-value, суміжна тема)

> Redis — **не одна з 8 основних СУБД** курсу, але часто з'являється поруч (кеш, черги). Тут — оглядово: що це, коли брати, і коли PostgreSQL/Mongo вистачає без нього.

## Що це
Redis — **in-memory** сховище ключ-значення (дані в RAM → доступ за мікросекунди). Однопотокова обробка команд (атомарність окремої команди «безкоштовна»). Опційна довговічність: **RDB** (періодичні снапшоти) і/або **AOF** (журнал операцій).

## Структури даних (головна «фішка»)
- **String** — рядок/число/байти; лічильники (`INCR`), кеш-значення, флаги.
- **Hash** — поле→значення (об'єкт: `HSET user:1 name Ann`).
- **List** — впорядкований список; черги/стеки (`LPUSH`/`RPOP`).
- **Set** — унікальні елементи; членство, перетини (теги, унікальні відвідувачі).
- **Sorted Set (ZSET)** — елементи з `score` (double); **лідерборди**, топ-N, черги з пріоритетом, часові вікна.
- **Stream** — журнал подій (черги повідомлень зі споживчими групами).
- Плюс **Pub/Sub**, TTL на ключ (`EXPIRE`), транзакції (`MULTI/EXEC`), скрипти Lua (атомарні складні операції).

## Типові застосування
Кеш важких запитів; сесії; **rate limiting** (лічильник з `EXPIRE`); лідерборди (ZSET); черги задач/повідомлень (List/Stream); розподілені локи (обережно — краще Redlock/бібліотеки); лічильники в реальному часі.

## Python
```python
import redis, os
r = redis.Redis.from_url(os.environ["REDIS_URL"])   # "redis://:pass@host:6379/0"
r.set("greeting", "hi", ex=60)          # TTL 60с
r.incr("page:views")
r.zadd("leaderboard", {"ann": 100, "bob": 90})
top = r.zrevrange("leaderboard", 0, 2, withscores=True)
```

## Кеш-стратегії (за «AWS in Action», гл. 11 — застосовно до будь-якого Redis)

- **Lazy loading (кеш на вимогу):** застосунок читає з кешу → промах → читає з БД → кладе в кеш із TTL. Просто, кешується лише те, що справді читають. Мінус: перший запит повільний; дані застарівають на строк до TTL.
- **Write-through (кеш наперед):** застосунок пише в БД **і** в кеш (або кеш наповнює фоновий job). Читання завжди свіже і швидке. Мінус: кеш має вмістити всі дані; пам'ять скінченна → **виселення** (типово LRU — найдавніше використане).
- **Вибір TTL — компроміс:** коротший TTL = свіжіші дані, але більше навантаження на БД; довший = навпаки. Навіть TTL у кілька секунд знімає з БД тисячі повторних запитів «гарячого» ключа — «малий TTL безсенсовний» це міф.
- Ключ кешу для результату SQL-запиту — сам текст запиту (або його md5/sha-хеш, щоб не носити довгі ключі). Але якщо дані мають структуру (топ-N) — краще рідна структура Redis (ZSET: `ZREVRANGE leaderboard 0 9` ≈ `SELECT … ORDER BY score DESC LIMIT 10` за мікросекунди).
- Кеш — **не** основне сховище: копія даних завжди живе в БД із дисковою довговічністю; втрата кешу має означати «повільніше», а не «втратили дані».

## Довговічність, реплікація, масштабування

- **Довговічність налаштовувана:** RDB-снапшоти (швидко, але втрачаєш хвости) і/або AOF-журнал (до fsync на кожну операцію — надійніше, повільніше). Все в RAM: датасет обмежений пам'яттю вузла.
- **Реплікація primary→replica:** репліки читаються (розвантаження читань) і дають failover (Sentinel / керовані сервіси роблять це автоматично).
- **Redis Cluster** — шардинг простору ключів між вузлами, коли не влазиш в один вузол.
- Транзакції `MULTI/EXEC` — це **черга команд без відкату** (усі виконаються атомарно, але помилка всередині не відкочує попередні) — не плутати з ACID реляційних БД; складна атомарність — Lua-скрипт.

## Коли Redis, а коли ні (важливо для навчання)
- **Так:** гарячий кеш, ефемерні дані, лічильники/лідерборди/rate-limit, черги, pub/sub — там, де потрібні мікросекунди й прості структури.
- **Ні/надлишково:** якщо потрібен **лише** кеш одного важкого агрегату — у PostgreSQL часто вистачає **MATERIALIZED VIEW** (можна індексувати/JOIN-ити, не тягнути окремий сервіс на 16GB RAM). Складна вибірка з фільтрами/зв'язками — це робота реляційної БД, не Redis.
- Redis тримає активний набір **у пам'яті** — не «безмежне» сховище; для великих холодних даних потрібна основна БД.

**Альтернативи Redis для write-heavy лічильника** (перегляди, кліки — гарячий рядок, часто інкрементується, точність некритична), коли додавати ElastiCache «лише заради лічильника» не хочеться:

| Варіант | Латентність запису | Новий сервіс? | Компроміс |
|---|---|---|---|
| **ElastiCache/Redis** `INCR` | мікросекунди | так, always-on вузол (плата навіть у простої) | найшвидше; виправдано, якщо Redis уже потрібен під щось інше (кеш, сесії) |
| **DynamoDB** `UpdateItem` з `ADD counter :inc` (atomic) | одиниці мс | так, але **serverless on-demand** (плата за факт запитів, нуль у простої) | бюджетно чутливий пульсуючий трафік; ліміт партиції ~3000 RCU/1000 WCU на «гарячий» ключ |
| **`UNLOGGED`-таблиця в наявній RDBMS** + батчування у застосунку (акумулюй у пам'яті процесу, `flush` раз на N секунд) | одиниці мс (за рахунок батчування, не технології) | ні | нуль нового сервісу, але: **не потрапляє у WAL** → не реплікується на read-репліки й **обнуляється** після crash recovery; буфер у пам'яті процесу губиться при рестарті/масштабуванні — прийнятно лише для «приблизного» лічильника |

Правило вибору: рахуй не «скільки коштує сервіс», а «скільки коштує простій сервісу» — DynamoDB on-demand часто дешевший за always-on Redis-вузол саме через нульову плату в тиші. Якщо Redis уже в стеку для іншої задачі (напр. кеш результатів рекомендацій) — лічильник туди ж, він стає «безкоштовним» додатковим use-case того самого сервісу, а не окремим рішенням.

## Розгортання
- Docker: образ `redis`, порт `6379`. Керовані: **AWS ElastiCache**/**MemoryDB**, **GCP Memorystore**, **Azure Cache for Redis**.
