# Векторні бази даних: ембединги, ANN-індекси, гібридний пошук, RAG

Джерела: N. Borwankar «Vector Databases: A Practical Introduction» (O'Reilly, 2025), Ло Юнь «Векторные базы данных» (ДМК, 2026), серія Zilliz «Vector Database 101», Microsoft Learn (Cosmos DB Integrated Vector Database, оновл. 2026-05), документація pgvector 0.8. Використовуй, коли питають про семантичний пошук, ембединги, RAG, вибір векторної БД або векторні можливості курсових СУБД.

## 1. Навіщо векторна БД

- **Неструктуровані дані** (текст, зображення, аудіо, відео) — за прогнозами IDC, ~80 % нових даних до 2028. Класичні СУБД шукають за **синтаксисом** (точний збіг, LIKE, FTS за термінами); людський запит — за **семантикою** («laptop won't turn on» має знайти «device fails to boot»). Це «семантичний розрив».
- Рішення: **ембединг** — модель відображає фрагмент даних у вектор із десятків–тисяч float-вимірів так, що семантично близькі об'єкти опиняються поруч у векторному просторі. Пошук стає **k-NN / ANN** задачею — «знайди k найближчих векторів».
- До ембедингів неструктуроване зберігали як BLOB/файл + метадані — шукати можна було лише за метаданими.
- Векторна БД ≠ векторний індекс-бібліотека (FAISS, hnswlib): бібліотека дає лише індекс+пошук, без CRUD, метаданих, фільтрів, персистентності, реплікації. БД додає повний життєвий цикл даних.

## 2. Ембединги (стисло; акцент на застосуванні)

- **Dense** (більшість значень ≠ 0): Word2Vec, GloVe, BERT/сучасні sentence-трансформери, CLIP (мультимодальний). Типові розмірності: 384 (all-MiniLM-L6-v2), 768 (BERT-base), 1536 (OpenAI text-embedding-3-small), до 3072+.
- **Sparse** (більшість нулі; ненульові = вага терма): TF-IDF, BM25, SPLADE, BGE-M3 — «навчений keyword-пошук», добре ловить точні терміни.
- **Binary** (1 біт/вимір, Hamming-відстань): компактно і швидко, з втратою точності — дедуплікація зображень, фінгерпринти.
- Класика векторної арифметики: Vec("Queen") ≈ Vec("King") − Vec("Man") + Vec("Woman") (Mikolov et al., 2013).
- **Критично:** запити ембедяться **тією самою моделлю**, що й дані. Змішування моделей/версій в одній колекції руйнує recall. Заміну моделі роби через нову колекцію + **alias-переключення** (див. §5).

## 3. Метрики відстані/схожості

| Метрика | Формула-суть | Коли застосовувати |
|---|---|---|
| **Cosine** | косинус кута; ігнорує довжину вектора | текст/семантика (напрям важливий, норма — ні); контент різної довжини — без bias «довший виграє» |
| **L2 (Euclidean)** | пряма відстань | коли значущі абсолютні величини; k-means-подібні задачі; частина image-ембедингів |
| **Inner product (dot)** | вирівняння × величина | рекомендаційні системи (норма кодує впевненість/популярність); для **нормалізованих** векторів ≡ cosine |
| Manhattan (L1) | сума модулів різниць | рідше; стійкіша до викидів по осях |

Пастка: багато систем повертають **відстань** (0 = ідентичні), а ранжувати треба за **схожістю** — конвертуй `similarity = 1 − cosine_distance`. BM25-оцінки необмежені (0..20+), cosine — [0..1]: перед злиттям нормалізуй (див. §7).

## 4. ANN-індекси і квантизація

Точний k-NN = повний перебір: лінійно від n, «прокляття розмірності». ANN жертвує краплею recall заради порядків швидкості: приймаємо x, якщо d(q,x) ≤ (1+ε)·d(q, справжній найближчий).

| Індекс | Ідея | Плюси | Мінуси | Типове застосування |
|---|---|---|---|---|
| **Flat (brute force)** | перебір усіх | 100 % recall, просто | повільно, все в RAM | < ~1 млн векторів; еталон для вимірювання recall |
| **IVF** | k-means кластери + інвертований список; шукаємо в nprobe найближчих кластерах | швидко, менше пам'яті (з PQ/SQ) | треба тренування (центроїди); recall < 100 % | великі набори; `nlist` (кластерів), `nprobe` (скільки сканувати: більше = точніше/повільніше) |
| **HNSW** | багатошаровий граф «малих світів»: верхні шари — «магістралі», нижні — «локальні вулиці»; жадібна навігація | найкращий баланс швидкість/точність (recall до ~99 %), динамічні вставки | багато RAM, повільна побудова | десятки млн векторів; стандарт де-факто для тексту |
| **LSH** | хешування схожих у спільні кошики | просто | чутливий до параметрів, слабший на високих розмірностях | нішеве |
| **DiskANN (Vamana)** | граф здебільшого на NVMe SSD, не в RAM | масштаб 100 млн+ на дешевому залізі; +2–3 мс латентності | складніше | Cosmos DB (NoSQL API), Milvus; бюджетний масштаб |

**Параметри HNSW:** `M` — зв'язків на вузол (більше = точніше, RAM ≈ лінійно від M; старт: 16); `efConstruction` — ширина пошуку при побудові (якість індексу vs час побудови; 64–200); `efSearch` — бюджет обходу при запиті (головний важіль recall vs латентність). **Нюанс складності:** «HNSW = O(log n)» — евристика, не теорема; за фіксованого efSearch час ≈ лог-подібний, але щоб **утримати** recall зі зростанням n чи розмірності, efSearch доводиться нарощувати — час дрейфує до O(n^α). Гірше на: дуже високій внутрішній розмірності, дублікатах/«hubness», вимогах recall ≥ 0.99.

**Квантизація** (стискання векторів): **SQ** — float32 → int8/4-біт/fp16 (−75 % RAM, мінімальна втрата); **PQ** — вектор ріжеться на m підвекторів, кожен кодується індексом центроїда свого кодбука (−90 %+ пам'яті, помітніша втрата; відстані рахуються за таблицями поглядів). Комбінації: IVF+PQ (IndexIVFPQ у FAISS) — робочий кінь великих наборів; OPQ — PQ з попереднім поворотом простору.

**Метрики якості системи:** **recall** (частка справжніх сусідів у відповіді; цільово 80–99 % залежно від задачі), латентність (p50/p99), **QPS**. Бенчмарки: ANN-Benchmarks, VectorDBBench. Оцінюй продуктивність **лише разом із recall** — «швидко» без recall безглуздо. Орієнтир з індустрії (Tencent Cloud, 10 млн × 768d, recall 95 %): середня латентність < 10 мс, p99 < 20 мс.

## 5. Логічна структура векторної БД (за Ло Юнем; Tencent VectorDB ≈ Milvus-подібні)

Ієрархія: **екземпляр → база → колекція (≈ таблиця) → документ (≈ рядок) → поле** (скалярне або векторне). Ізоляція: відділи — екземплярами, системи — базами, модулі — колекціями. При створенні колекції фіксуються: тип індексу + метрика, `shardNum` (сегменти — горизонтальний поділ даних), `replicaNum` (репліки — відмовостійкість і читання).

Три типи індексів: **primary key** (точковий доступ за id), **векторний** (ANN), **фільтрувальний** (за скалярними полями — для гібридних запитів). Особливості, які відрізняють від реляційних СУБД:

- **Динамічна схема:** окрім проіндексованих полів, документи можуть мати довільні поля без ALTER TABLE (як у NoSQL).
- **Alias-механізм:** онлайн-заміна моделі ембедингів — заливаємо нову колекцію B, `alias/set` A→B; клієнтський код не змінюється. Ключовий операційний патерн.
- **Вбудована векторизація:** БД сама ембедить (`embedding: {field, vectorField, model}`) — пишеш сирий текст, не вектор (аналог: Atlas Vector Search + автоембединги, Cosmos DB).
- **Перебудова індексу** (`index/rebuild`) — потрібна при зростанні даних (перепідбір M/efConstruction); ризикована: з `dropBeforeRebuild` колекція недоступна, без — read-only на час побудови.

Внутрішня архітектура (для розуміння): 4 шари — storage / index / query / service; персистентність через WAL + снапшоти; кластер — Raft-реплікація, шардинг, розділення читання/запису; оптимізації — SIMD/AVX512, GPU (індекс CAGRA).

## 6. Pure vs integrated: яку векторну БД обрати

- **Спеціалізовані (pure):** Milvus/Zilliz Cloud, Pinecone, Weaviate, Qdrant, Chroma. Плюси: мільярдні масштаби, декомпозиція compute/storage, повний набір ANN-фіч. Мінуси: окрема інфраструктура, дублювання даних, синхронізація з основною БД (консистентність!).
- **Інтегровані:** pgvector, MongoDB Atlas, Cosmos DB, Cassandra 5, ClickHouse, Redis. Плюси: вектори **поруч із бізнес-даними** — один бекап/транзакції/фільтри без реплікації в окрему систему; знайомі інструменти. Аргумент Microsoft: менша вартість, вища консистентність даних; ChatGPT працює поверх Cosmos DB.
- Практичне правило для курсу: департаментський масштаб (до ~10 млн векторів) → інтегрована (найчастіше **PostgreSQL + pgvector**); мільярди векторів/окремий пошуковий сервіс → pure.

## 7. Гібридний пошук і фільтри

- **Semantic + keyword:** dense-пошук розуміє смисл, але «розмиває» точні терміни (артикули, назви функцій); keyword (BM25/sparse) ловить їх. Злиття: нормалізуй оцінки (BM25 поділи на max у видачі; cosine_distance → 1−d), зваж `final = 0.7·semantic + 0.3·keyword` (стандартний старт; пошук «розпливчастий» → підніми keyword-вагу), або **RRF** (Reciprocal Rank Fusion) без ваг. Бери кандидат-пул `limit × 2` з кожної гілки перед злиттям.

**RRF у чистому SQL** — робочий патерн, який знімає всю проблему нормалізації (працює на **рангах**, а не на оцінках, тому необмежений BM25 і cosine ∈ [−1..1] стають порівнянними):

```sql
WITH semantic AS (                       -- гілка 1: ANN
  SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> :qvec) AS rnk
  FROM chunks WHERE course_code = :course
  ORDER BY embedding <=> :qvec LIMIT 20                  -- кандидат-пул ≈ 2×k
), keyword AS (                          -- гілка 2: повнотекст
  SELECT c.id, ROW_NUMBER() OVER (ORDER BY ts_rank_cd(c.tsv, q) DESC) AS rnk
  FROM chunks c, websearch_to_tsquery('simple', :qtext) q   -- безпечний парсер сирого вводу
  WHERE c.tsv @@ q AND c.course_code = :course
  ORDER BY ts_rank_cd(c.tsv, q) DESC LIMIT 20
), fused AS (                            -- злиття: k = 60 (класичне значення з праці Cormack et al.)
  SELECT COALESCE(s.id, k.id) AS chunk_id,
         COALESCE(1.0 / (60 + s.rnk), 0) + COALESCE(1.0 / (60 + k.rnk), 0) AS rrf_score
  FROM semantic s FULL OUTER JOIN keyword k ON s.id = k.id  -- FULL OUTER: беремо й одногілкові
)
SELECT c.chunk_text, f.rrf_score
FROM fused f JOIN chunks c ON c.id = f.chunk_id
ORDER BY f.rrf_score DESC LIMIT 10;
```

Два обов'язкові елементи, які найчастіше ламають: **`FULL OUTER JOIN`** (не INNER — інакше втратиш усе, що знайшла лише одна гілка) і **`COALESCE(…, 0)`** для відсутнього рангу. Ефект перевірено: спільний кандидат обох гілок природно спливає нагору (≈0.033 проти ≈0.016 в одногілкових), при цьому жодних ваг і жодної нормалізації.
- **Фільтри за метаданими:** pre-filtering (спершу фільтр, потім ANN у підмножині), post-filtering (ANN, потім фільтр — ризик: після фільтра лишилося < k результатів), iterative filtering (добираємо, поки не набереться k). У pgvector 0.8 це `hnsw.iterative_scan = 'relaxed_order'` — рятує запити з селективним WHERE.
- Інші запити: range search (усі в радіусі THRESHOLD — «схожі, але не дублікати»), grouping (різноманітність видачі — по одному чанку з документа), batch search, multi-vector/negative-vector.

## 8. Векторна підтримка у курсових СУБД (стан 2026)

| СУБД | Механізм | Індекси | Нотатки |
|---|---|---|---|
| **PostgreSQL** | розширення **pgvector 0.8.x**: типи `vector`, `halfvec` (fp16, −50 % місця, до 16k вимірів), `sparsevec`, `bit`; оператори `<->` (L2), `<=>` (cosine), `<#>` (−inner product) | HNSW, IVFFlat; iterative index scans | мейнстрим курсу; приклад нижче |
| **SQLite** | розширення **sqlite-vec** (наступник sqlite-vss; віртуальні таблиці vec0), поруч — **FTS5/BM25** для keyword-гілки | brute-force (vec) | ідеально для навчальних RAG-прототипів без сервера |
| **MongoDB Atlas** | `$vectorSearch` у aggregation pipeline (тільки Atlas, не self-hosted) | HNSW | + `$search` (Lucene) для гібриду |
| **Cassandra 5.0** | тип `VECTOR<FLOAT, n>`, запит `ORDER BY v ANN OF [...]` | SAI (storage-attached) | перший кейс розширюваності SAI |
| **ClickHouse** | тип `Array(Float32/64)` + функції відстаней; **vector similarity index** (HNSW, на базі usearch) | HNSW (експериментальний → стабілізується) | сильний у комбінації «аналітика + вектори» |
| **SQL Server 2025** | нативний тип `VECTOR`, `VECTOR_DISTANCE()` | DiskANN-подібний векторний індекс | новинка 2025 року |
| **MySQL 9** | тип `VECTOR`, `STRING_TO_VECTOR`/`VECTOR_TO_STRING`, `DISTANCE` | ANN-індекси — лише в HeatWave | ком'юніті-версія: тип є, ANN-індексу немає — фактично перебір |
| **Redis** | Redis Query Engine (кол. RediSearch): `VECTOR`-поля у HASH/JSON; Redis 8 — vector sets | HNSW, Flat | типова роль: семантичний кеш LLM-відповідей |
| **Cosmos DB** | NoSQL API: vector embedding policy + `VectorDistance()`; Mongo vCore: `$search`/vectorIndex | **DiskANN**, quantizedFlat, flat | 99.999 % SLA, geo-реплікація; див. `azure.md` |
| **DynamoDB** | власного ANN немає → zero-ETL реплікація в OpenSearch Service (k-NN) | (в OpenSearch) | патерн «KV-основа + пошуковий сателіт» |

## 9. Практичний рецепт: pgvector (перевірений патерн із книги Borwankar)

```sql
CREATE EXTENSION IF NOT EXISTS vector;
CREATE EXTENSION IF NOT EXISTS pg_trgm;      -- нечіткий текстовий пошук для гібриду

CREATE TABLE papers (
  id SERIAL PRIMARY KEY,
  arxiv_id VARCHAR(50) UNIQUE NOT NULL,
  title TEXT NOT NULL,
  abstract TEXT,
  authors TEXT[],                            -- GIN-індекс для фільтрів
  categories TEXT[],
  published_date DATE,
  pdf_processed BOOLEAN DEFAULT FALSE        -- прапорці стану конвеєра
);

CREATE TABLE paper_chunks (
  id SERIAL PRIMARY KEY,
  paper_id INT REFERENCES papers(id) ON DELETE CASCADE,
  chunk_index INT NOT NULL,
  chunk_text TEXT NOT NULL,
  embedding vector(384),                     -- розмірність = модель (all-MiniLM-L6-v2)
  section_name VARCHAR(255),
  UNIQUE (paper_id, chunk_index)
);

CREATE INDEX ON paper_chunks USING hnsw (embedding vector_cosine_ops)
  WITH (m = 16, ef_construction = 64);       -- recall > 0.95 за розумного часу побудови
CREATE INDEX ON papers USING GIN (categories);

-- Гібридний запит: семантика + фільтр метаданих в одному SQL
SELECT p.title, c.chunk_text,
       1 - (c.embedding <=> :query_vec) AS similarity
FROM paper_chunks c JOIN papers p ON p.id = c.paper_id
WHERE p.categories @> ARRAY['cs.LG']
  AND p.published_date > '2024-01-01'
ORDER BY c.embedding <=> :query_vec
LIMIT 10;
```

Python-нюанси: `psycopg2` не адаптує numpy-масив у `vector` — обов'язково `from pgvector.psycopg2 import register_vector; register_vector(conn)` (у psycopg v3 — `from pgvector.psycopg import register_vector`).

**Чанкінг — і головне обмеження, про яке забувають.** Орієнтир «512–1024 токени (≈ 1–3 абзаци) зі sliding window і перекриттям ~20 %, зберігаючи межі речень» чинний **лише якщо стільки приймає сама модель**. Ліміт задає `max_seq_length` кодувальника, і надлишок **мовчки відкидається** — чанк у базі є, його хвіст непошуковуваний, помилки немає.

- `all-MiniLM-L6-v2` — **256 word-piece** (не 512: автори свідомо обмежили, бо на довших послідовностях якість падала). Тобто наведені вище 512–1024 для цієї моделі **недосяжні**.
- Кирилиця дає **2–3 word-piece на слово** проти ~1.3 для англійської → у 256 токенів влазить лише **~100–120 слів українського тексту**.
- Перевіряй у коді, а не на віру: `model.max_seq_length` і `model.get_sentence_embedding_dimension()` (друге — звірка з `vector(N)` у DDL).
- Моделі з довшим вікном: `intfloat/multilingual-e5-small` (**ті самі 384 виміри — DDL не змінюється**, 512 токенів, розуміє українську; вимагає префіксів `query:` / `passage:`), `paraphrase-multilingual-MiniLM-L12-v2`, OpenAI `text-embedding-3-*` (8191 токенів).

**Опкласс мусить відповідати оператору запиту.** `vector_cosine_ops` обслуговує лише `<=>`, `vector_l2_ops` — лише `<->`, `vector_ip_ops` — `<#>`. Індекс під одну метрику + запит з іншим оператором = **тихий seq scan без жодного попередження** (класика: скопійований з інтернету `vector_l2_ops` + запит `<=>`). Перевірка — `EXPLAIN`: має бути `Index Scan using …_hnsw`. Суміжне: `hnsw.ef_search` (дефолт 40) не лише керує recall, а й **обрізає видачу** — `LIMIT 50` при `ef_search = 40` поверне ≤ 40 рядків, теж без помилки.

**Повнотекстова гілка українською:** вбудованої конфігурації `'ukrainian'` у стандартному PostgreSQL **немає** (Snowball не має українського стемера) — `to_tsvector('ukrainian', …)` впаде з «text search configuration does not exist». Базовий варіант — `'simple'` (без стемінгу: «індекси/індексів» — різні терми), практичніший для морфології — `pg_trgm`, повноцінний — hunspell-словник `uk_UA` (потрібен доступ до `$SHAREDIR/tsearch_data`).

## 10. RAG-патерн і оцінка якості

Конвеєр: джерело → екстракція тексту → чанкінг → ембединг (батчами; синглтон моделі) → запис у БД → [запит користувача → ембединг тим самим кодувальником → гібридний пошук top-k → збирання контексту → промпт до LLM → відповідь]. RAG дає LLM актуальні/приватні знання без файнтюнінгу, обходить ліміт контексту і зменшує галюцинації.

- Локальний LLM (Ollama): temperature ~0.1 для фактологічності; **пастка** — дефолтний контекст 2048 токенів, довгі чанки мовчки обрізаються → задавай `num_ctx`.
- Оцінка: мінімум — spot-check за трьома осями: retrieval depth (чи знайдено відомий еталонний документ — recall), context relevance (відповідь спирається на знайдене чи генерик), faithfulness (LLM не приписує джерелу того, чого там немає). Систематично: RAGAS, LLM-as-a-Judge.
- LLM-кеш: пара (ембединг запиту → відповідь) у векторній БД — семантичний кеш зменшує кількість викликів LLM.

## 11. Коли не потрібен живий ANN-запит: precompute + кеш

Живий ANN-пошук на кожен запит користувача — не єдиний і часто не найкращий шлях до латентності «одиниці мс». Чесний орієнтир: індустрійний бенчмарк живого HNSW-запиту на 10 млн × 768 вимірів (recall 95%) дає середню латентність <10 мс, p99 <20 мс — **без** мережевого round-trip і acquire з'єднання з пулу, які завжди додаються зверху. Тобто «мілісекунди» для *живого* запиту — оптимістична ціль, не гарантія.

Ключовий прийом, коли результат ANN-пошуку **рідко змінюється відносно частоти читання** (класика — «схожі товари для X», рекомендації, кластеризація за змістом): рахувати офлайн, читати онлайн точковим ключем.

1. **Batch (cron/scheduled job, раз на годину/ніч):** для кожного об'єкта — ANN-запит top-N до векторної БД, результат матеріалізується (окрема легка таблиця `recommendations(entity_id, similar_ids, computed_at)` або кеш-запис).
2. **Онлайн (гарячий шлях):** застосунок читає готовий список за первинним ключем — це вже не ANN, а точкове читання; мікросекунди–одиниці мс легко досяжні.
3. **Живий ANN лишається fallback-ом**, а не основним шляхом: щойно доданий об'єкт без ще обрахованого precompute, довільний текстовий пошук користувача, персоналізований вектор на льоту.

Це і є практична відповідь на вимогу «мілисекунди для семантичного пошуку» — досяжна не тому, що сам ANN-індекс аж такий швидкий, а тому що гарячий шлях узагалі не виконує ANN. Той самий прийом, тільки термінами BI/DWH — Gold-вітрина замість перерахунку на кожен запит (`design-dwh-etl`), і кеш-стратегія write-through/lazy loading (`redis.md`) — це один і той самий архітектурний принцип: рахуй рідко, читай часто.

## 12. Типові помилки студентів (граблі)

1. **Різні моделі ембедингів** для даних і запитів (або мішанина версій в одній колекції) → recall падає «незрозуміло чому». Заміна моделі = нова колекція + alias.
2. Плутанина **відстань vs схожість**: ORDER BY similarity ASC або `WHERE distance > 0.8` замість `<`. Перевір напрям для конкретної метрики й системи.
3. **Ненормалізовані вектори з inner product** → ранжування спотворене нормою. Або нормалізуй, або бери cosine.
4. Пряме додавання **BM25 + cosine без нормалізації** → keyword-гілка завжди домінує.
5. **Post-filtering після top-k**: фільтр з'їв результати, повернулося 2 з 10. Використовуй pre-/iterative filtering (pgvector: iterative_scan).
6. Очікування **100 % точності від HNSW/IVF** — ANN апроксимаційний за визначенням; для еталонних порівнянь тримай Flat-індекс.
7. Вектор як `FLOAT[]` у звичайній таблиці без ANN-індексу «бо працює» — на 100k+ рядків це full scan.
8. Ігнорування **розмірності**: `vector(384)` у DDL ≠ модель на 768 вимірів → помилка вставки (у кращому разі) або тихе обрізання (в гіршому, залежно від клієнта).
9. Індекс будують **до** масового завантаження (повільно) — для великих обсягів: завантаж дані → побудуй HNSW.
10. Зберігання чанків без `chunk_index`/посилання на джерело → неможливо зібрати контекст і процитувати джерело у RAG.
