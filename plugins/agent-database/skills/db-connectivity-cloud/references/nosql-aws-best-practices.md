# Схеми даних і best practices: Cassandra, DynamoDB, Redshift, Aurora (+ Redis, MongoDB)

Практики моделювання для NoSQL і аналітичних/хмарних СУБД курсу. Джерела: офіційна документація (DataStax/Apache Cassandra, AWS), «AWS Certified Database Study Guide (DBS-C01)» (Wiley), типові теми обговорень спільнот (StackOverflow; прямий фетч SO блокується — теми звірено з документацією), огляди aws.amazon.com/products/databases і freshtech.global. Redis і MongoDB мають власні довідники — тут лише вказівники.

## PIE-теорема — рамка вибору «purpose-built» БД (доповнення до CAP)

З трьох властивостей обираєш дві: **P** (pattern flexibility — ad-hoc запити, оптимізатор сам знайде шлях), **I** (infinite scale — необмежене нарощування пропускної здатності), **E** (efficiency — передбачувана латентність завжди). **PE** = реляційні OLTP (RDS/Aurora), **IE** = NoSQL (DynamoDB/Keyspaces — масштаб і латентність, але патерни доступу фіксуй наперед), **PI** = аналітичні DW (Redshift — гнучкі запити на петабайтах, але латентність секунди–години). Опорні питання аналізу data store: патерни доступу відомі чи ad-hoc → латентність (мкс/мс/с) → масштабування (пропускна здатність + конкурентність: 2PL/MVCC vs OCC — DynamoDB використовує оптимістичний контроль) → транзакції ACID чи BASE → консистентність (eventual дешевша — бери, якщо можна) → обсяг на елемент (DynamoDB item ≤ 400 КБ, Keyspaces рядок ≤ 1 МБ — саме таке обмеження часто вирішує вибір) → довговічність (ephemeral → TTL/кеш) → доступність (RPO/RTO, мультирегіон) → комплаєнс → де живе бізнес-логіка → вартість (останньою, з можливістю послабити вимоги).

## Cassandra (wide-column) — «спершу запити, потім таблиці»

- **Query-first design:** таблиці проєктуються під конкретні запити, а не під сутності. Дублювання таблиці під другий запит — норма (write is cheap), JOIN-ів немає.
- **Первинний ключ = partition key + clustering key.** Partition key (через Murmur3-хеш) визначає вузол зберігання; clustering key — сортування рядків усередині партиції. Запит **мусить** задавати partition key повністю; фільтр без нього = розсилка на весь кластер.
- Правила партицій: рівномірний розподіл (без «гарячих» ключів на кшталт `country`), **обмежене зростання** — партиція не повинна рости безмежно (класичний прийом — bucketing за часом: `PRIMARY KEY ((sensor_id, month), event_time)`); орієнтир — тримати партиції до ~100 МБ.
- **`ALLOW FILTERING` — червоний прапорець**, а не «фікс»: означає повний скан; у проді заборонено — перепроєктуй таблицю/додай таблицю під запит. Вторинні індекси — лише для низькокардинальних полів у межах партиції; частий анти-патерн зі StackOverflow.
- **Tombstones:** DELETE (і INSERT NULL, і TTL) створюють надгробки, які живуть до compaction (`gc_grace_seconds`) — масові видалення/оновлення в одній партиції деградують читання аж до відмови запиту. Уникай патернів «черга на Cassandra».
- **BATCH** — не для продуктивності: логований батч між партиціями повільніший за окремі запити; доречний лише для атомарності записів в **одну** партицію.
- **Консистентність налаштовувана на запит:** `ONE`/`QUORUM`/`ALL`/`LOCAL_QUORUM`; сильна консистентність, коли **R + W > RF** (напр., RF=3, QUORUM на читання й запис). Компроміс затримка↔узгодженість — явна частина дизайну (CAP: AP-система).
- Синтаксис CQL і невиразні операції — `analyze-sql-examples/references/cassandra-cql.md`.

## Amazon DynamoDB (key-value/документна)

- Та сама філософія query-first: **partition key (+ sort key)**; комбінація унікальна; sort key дає діапазонні запити всередині партиції (`begins_with`, `between`).
- **Single-table design:** пов'язані сутності часто кладуть в одну таблицю з узагальненими ключами `PK`/`SK` (`USER#123` / `ORDER#2026-01-07`) — одна вибірка дістає граф об'єктів; ціна — складніше читати схему (документуй item types!).
- Доступ за іншими атрибутами — **GSI** (глобальний вторинний індекс; своя пропускна здатність, eventually consistent) / **LSI** (локальний, той самий partition key). Індекси проєктуй одразу — LSI після створення таблиці не додаси.
- Гарячі партиції: високочастотний однаковий ключ (напр., `date` як PK) впирається в ліміт партиції — розмазуй суфіксами/шардингом ключа. On-demand vs provisioned (RCU/WCU + auto scaling) — обирай за профілем навантаження.
- Сканів (`Scan`) у гарячому шляху бути не повинно; фільтр у `Scan/Query` виконується **після** читання (платиш за прочитане). Транзакції (TransactWriteItems) і TTL — є; для аналітики — експорт у S3/Athena, не сканування.
- **Жорсткі ліміти, які треба знати:** item ≤ **400 КБ**; одна фізична партиція — до **3 000 RCU / 1 000 WCU** (звідси й гарячі ключі); `ProvisionedThroughputExceededException` = впираєшся в capacity або в партицію.
- **Capacity modes:** **on-demand** («автопілот», миттєво тримає до 2× попереднього піка; старт нових проєктів — з нього, далі за метриками вирішуй) vs **provisioned** (RCU/WCU + auto scaling; дешевший за одиницю за передбачуваного профілю; для запланованого сплеску — підняти заздалегідь). 1 RCU = 1 сильно-консистентне читання ≤ 4 КБ/с (eventually — вдвічі дешевше), 1 WCU = запис ≤ 1 КБ/с.
- **Екосистема:** **DynamoDB Streams** — change log таблиці (тригер Lambda, реплікація, аудит; основа CDC-патернів); **DAX** — read/write-through кеш-кластер перед таблицею (мікросекундні читання, зняття RCU на read-heavy/спайкових навантаженнях; кластер на 3 AZ; сильно-консистентні читання йдуть повз кеш); **global tables** — мультирегіонна active-active реплікація (з 2025 — опція multi-Region strong consistency, RPO=0); **PITR** — відновлення на будь-яку секунду за останні 35 діб; TTL — «лінива» безплатна чистка застарілих item-ів.
- Вектори: власного ANN немає — патерн «DynamoDB як основа + zero-ETL у OpenSearch» (див. `vector-databases.md`).

## Amazon Keyspaces і DocumentDB (сумісні сервіси; коротко)

- **Keyspaces (for Apache Cassandra):** serverless CQL-сумісний; рядок ≤ 1 МБ (проти 400 КБ у DynamoDB — критерій вибору між ними); немає tombstone-проблем і compaction-тюнінгу самокерованої Cassandra, але й немає повного набору фіч (перевіряй UDT/матеріалізовані представлення). Моделювання — як у Cassandra (query-first, partition key).
- **DocumentDB (with MongoDB compatibility):** керований документний з Aurora-подібною архітектурою сховища (6 копій, 3 AZ, до 15 реплік); сумісність із MongoDB API **частková** (перевіряй версію API і список несумісностей) — код драйвера той самий, але це не Atlas. Вибір DynamoDB vs DocumentDB: перший — key-value доступ за відомими ключами й екстремальний масштаб; другий — документні запити/агрегації в стилі Mongo.

## Amazon Redshift (колонкове MPP-сховище)

- Модель — **зірка** з `design-dwh-etl/references/dimensional-modeling.md`; Redshift і задуманий під неї.
- **DISTSTYLE / DISTKEY** — як рядки розкладаються по вузлах: `KEY` (по стовпцю join-у — колокація факту з великим виміром), `ALL` (малі виміри копіюються на всі вузли), `EVEN`/`AUTO`. Неправильний dist key = перекачування даних між вузлами на кожен JOIN.
- **SORTKEY** — сортування на диску (аналог ORDER BY у ClickHouse): фільтри за часом → sort key за датою; zone maps відсікають блоки.
- Завантаження — **`COPY` із S3** (паралельний, файли розбиті на число зрізів), не INSERT-цикли; після масових операцій — `ANALYZE` (статистика), `VACUUM` (пересортування/повернення місця; сучасні версії багато роблять автоматично).
- Стискання стовпців (`ENCODE AUTO`), обмеження: PK/FK **декларативні** (не забезпечуються — як у ClickHouse, цілісність на ETL); конкурентність нижча за OLTP-СУБД — це сховище для аналітики, не бекенд застосунку. Serverless-варіант знімає адміністрування кластера.

## Amazon Aurora (реляційна, MySQL/PostgreSQL-сумісна)

- Сховище відділене від обчислень: кластерний том на 6 копій у 3 AZ; **writer + до 15 read replicas** на спільному томі (репліки без власної копії даних, лаг мінімальний).
- Використовуй **endpoints**: cluster (writer), reader (балансування читань), custom; failover = промоушн репліки (~30 с — швидше за класичний RDS Multi-AZ).
- Схеми/SQL — як у звичайних MySQL/PostgreSQL (усі правила з діалектних довідників чинні); специфіка Aurora — операційна (масштабування, Serverless v2 з ACU, Global Database для міжрегіонного DR, parallel query для аналітики).
- Коли Aurora замість RDS: потрібні >1 репліки з мінімальним лагом, швидкий failover, авто-масштабоване сховище; коли ні — малі/дешеві інстанси, точна сумісність із самостійним тюнінгом.
