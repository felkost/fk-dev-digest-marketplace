# MongoDB — як працювати й будувати запити

MongoDB — документна БД: дані у **колекціях** як BSON/JSON-**документи** (гнучка схема), а не рядки в таблицях. Запити — методом `find` (проста вибірка) і **aggregation pipeline** (конвеєр стадій для групувань/обчислень).

## Як під'єднатися (Python)
```python
from pymongo import MongoClient
client = MongoClient("mongodb://localhost:27017")   # рядок з env, не хардкод
db = client["shop"]
users = db["users"]
for u in users.find({"age": {"$gt": 18}}, {"name": 1, "_id": 0}):
    print(u["name"])
```
Асинхронно — `motor`. Кожен документ має `_id` (за замовчуванням `ObjectId`, монотонний за часом).

## `find` — проста вибірка
```js
db.users.find({ age: { $gt: 18 }, city: "Kyiv" },   // фільтр (WHERE)
              { name: 1, age: 1, _id: 0 })            // проєкція (SELECT колонок)
        .sort({ age: -1 }).limit(10)
```
Оператори: `$gt/$gte/$lt/$lte/$ne`, `$in/$nin`, `$and/$or/$not`, `$exists`, `$regex`, доступ до вкладених полів через крапку (`"address.city"`), масиви — `$elemMatch`, `$all`.

## Aggregation pipeline — групування й обчислення
Конвеєр стадій, дані «течуть» зверху вниз:
```js
db.sales.aggregate([
  { $match:   { date: { $gte: ISODate("2025-03-01") } } },   // фільтр РАНО (щоб узяти індекс)
  { $group:   { _id: "$category", total: { $sum: "$amount" },
                avg: { $avg: "$amount" }, cnt: { $sum: 1 } } },
  { $sort:    { total: -1 } },
  { $limit:   3 }
])
```
Ключові стадії: `$match` (WHERE), `$group` (GROUP BY + агрегати), `$project` (SELECT/обчислення), `$sort`, `$limit`/`$skip`, `$unwind` (розгортання масиву в рядки), `$lookup` (≈ LEFT JOIN до іншої колекції), `$setWindowFields` (віконні функції, 5.0+).

## Зіставлення з SQL

| SQL | MongoDB |
|---|---|
| таблиця / рядок / стовпець | колекція / документ / поле |
| `WHERE` | `$match` (або фільтр у `find`) |
| `SELECT col` | проєкція / `$project` |
| `GROUP BY` + `SUM/AVG` | `$group` з `$sum`/`$avg` |
| `ORDER BY` / `LIMIT` | `$sort` / `$limit` |
| `JOIN` | `$lookup` |
| `ROW_NUMBER/RANK` | `$setWindowFields` |
| `UNION ALL` | `$unionWith` |

## Типові помилки / поради
- **`find` vs `aggregate`:** для групувань/обчислень/join — лише `aggregate`.
- **`$group` мовчки «губить» поля, які не потрапили в `_id` чи акумулятор.** На відміну від строгого SQL (PostgreSQL/MS SQL Server: неагрегований стовпець поза `GROUP BY` — помилка компіляції) і навіть MySQL із `ONLY_FULL_GROUP_BY`, у MongoDB **немає жодного запобіжника** — документ після `$group` складається рівно з того, що перелічено, і решта полів просто зникає без жодного попередження. Класична пастка: групуючи лише за `product_id`, студент дивується, куди поділося поле `category` — воно не «загубилось по дорозі», його ніхто не замовляв. Фікс — включити потрібне поле в компаунд-`_id` (`_id: { product_id: "$product_id", category: "$category" }`) або явно перенести акумулятором (`$first`/`$push`).
- **`$match` — якомога раніше** в конвеєрі: інакше `$group` обробляє всю колекцію й не використає індекс.
- **Типи не приводяться:** число `5` і рядок `"5"` не рівні; дати мають бути `ISODate`, не рядки.
- **Індекси:** `db.coll.createIndex({ field: 1 })`; `$lookup`/`$sort`/`$match` на великих колекціях без індексу — повільно. Перевіряй план: `.explain("executionStats")`.
- **Схема гнучка, але не безсхемна:** узгоджуй структуру документів; моделюй під патерни запитів (вкладати чи посилатися — залежить від доступу й розміру).
- Трилогіка порівнянь відрізняється від SQL: відсутнє поле ≠ `null`; є оператори `$exists` і перевірка на `null` окремо.

## Приклад «топ-N у групі» (аналог віконної функції)
```js
db.sales.aggregate([
  { $setWindowFields: { partitionBy: "$category",
      sortBy: { amount: -1 },
      output: { rnk: { $rank: {} } } } },   // $rank ≈ RANK(); $documentNumber ≈ ROW_NUMBER()
  { $match: { rnk: { $lte: 3 } } }
])
```

## Проєктування схеми: вбудовувати чи посилатися
Центральне рішення в MongoDB (з «MongoDB: The Definitive Guide»):
- **Вбудовування (embedding)** — вкладати пов'язані дані в документ. Добре, коли їх читають разом, вони «належать» батьку і не ростуть безмежно (адреси користувача, позиції замовлення). Плюс: одне читання, атомарність на рівні документа. Мінус: дублювання, ліміт документа **16 МБ**, дорогі оновлення масивів, що ростуть.
- **Посилання (referencing)** — зберігати `_id` і з'єднувати через `$lookup`. Добре для M:N, великих/незалежних сутностей, даних, що часто змінюються окремо. Мінус: додаткові запити/`$lookup`.
- Правило: моделюй **під патерни доступу**, а не «нормалізовано за замовчуванням». Часте читання разом → вбудовуй; незалежне життя/зростання → посилайся.

## Індекси (докладніше)
- Одиночні, **складені** (порядок важливий — правило **ESR: Equality → Sort → Range**: спершу поля рівності, потім сортування, потім діапазон), багатоключові (по масивах), текстові, геопросторові (`2dsphere`), часткові, TTL (авто-видалення за часом), унікальні.
- **Покривний запит** — усі потрібні поля є в індексі (проєкція без `_id`), читання без звернення до документа.
- Перевіряй план: `db.coll.find(...).explain("executionStats")` — шукай `IXSCAN` (не `COLLSCAN`), `totalDocsExamined` близько до `nReturned`.
- Без індексу `$sort` у пам'яті обмежений (за замовчуванням 100 МБ) → потрібен індекс або `allowDiskUse`.

## Реплікація й шардинг (масштабування)
- **Набір реплік (replica set):** первинний (приймає записи) + вторинні (копії). Автоматичний failover при падінні первинного. `writeConcern` (скільки вузлів підтвердили запис, напр. `majority`) і `readPreference` (звідки читати) — керують надійністю/свіжістю vs швидкістю. Рушій сховища — **WiredTiger** (стиснення, документне блокування).
- **Шардинг (sharding):** горизонтальне розбиття колекції по кластерах за **ключем шардування** (shard key). Вибір ключа критичний: висока кардинальність + рівномірний розподіл + відповідність запитам; поганий ключ → «гарячий» шард. Хешований ключ — рівномірність; діапазонний — локальність діапазонних запитів. `mongos` маршрутизує запити.

## Коли MongoDB доречна, а коли ні (за «Сім баз даних за сім тижнів» + сучасний стан)

**Сильні сторони:**
- Горизонтальне масштабування великих обсягів і потоків запитів (replica sets + шардинг) закладене в ядро.
- Гнучка модель: не треба заздалегідь фіксувати схему; вкладені документи замінюють частину JOIN-ів; додати нове поле = просто почати його писати (без міграцій ALTER TABLE).
- Природно лягає на об'єктні моделі застосунків (MVC/ORM-стиль розробки): документ ≈ об'єкт; тому популярна там, де набір даних визначається застосунком.

**Слабкі сторони / коли не брати:**
- Гнучкість — двосічна: **опечатка в імені поля не є помилкою** — запис «успішно» піде в інше поле, а запит тихо поверне порожньо; дисципліна схеми переїжджає в код (шар валідації, schema validation у самій Mongo).
- Денормалізація за замовчуванням: якщо модель даних зріла й стабільна, а запити — складні ad-hoc з багатьма зв'язками, реляційна СУБД дасть більше «гнучкості запитів» (join/фільтри/представлення без попереднього планування).
- Кластер (шардинг) вимагає попереднього планування (вибір shard key, конфіг-сервери, mongos) — це не «увімкнув і забув».
- Історичний контекст: у класичних оглядах (книга 2012 р.) зазначено «немає JOIN і транзакцій» — **застаріло**: з 3.2 є `$lookup` (обмежений JOIN), з 4.0 — багатодокументні транзакції (з 4.2 — і в шардованих кластерах), але обидва дорожчі за «рідні» патерни; якщо вони потрібні постійно — це сигнал, що модель даних не документна.
- У термінах CAP реплікований Mongo — **CP**-система: за втрати зв'язності віддає перевагу узгодженості (первинний один; при failover частина запитів тимчасово відхиляється).

**Правило вибору:** дані «належать застосунку», схема еволюціонує, читання групуються навколо документа → Mongo. Дані спільні для багатьох застосунків, важливі ad-hoc запити/звітність, суворі транзакції між сутностями → реляційна.
