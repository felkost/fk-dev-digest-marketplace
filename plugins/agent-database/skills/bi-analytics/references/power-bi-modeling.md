# Power BI: просунуте моделювання даних

За С. Бахши «Power BI: моделирование на экспертном уровне» (гл. 8–10) і Еренмюллером-Єнсеном. Продовження базового `power-bi.md` (там — Power Query, зірка, основи DAX, діагностика фільтрів). Тут — техніки, які потрібні, коли проста зірка вже не покриває вимоги.

## 1. Зв'язки «багато-до-багатьох»: чому таблиця-міст

**Ключове розрізнення, яке плутають:**
- У **реляційній моделі** фізичного зв'язку «багато-до-багатьох» **не існує** — лише 1:1 і 1:N (PK на стороні «один», FK на стороні «багато»). M:N реалізується **таблицею-мостом** (bridge) з ключами обох таблиць.
- У **Power BI** є **кратність (cardinality) «багато-до-багатьох»** — коли таблиці з'єднані по **неключових** стовпцях з дублікатами з обох боків. Це не те саме, що класичний M:N, і саме тут виникають «неправильні підсумки», порожні значення й падіння продуктивності на великих моделях.

**Правило:** кратність M:N вмикай **лише за реальної потреби бізнес-логіки** й розуміючи наслідки; типовий M:N розв'язуй **мостом**.

У зірці роль моста фактично грає **таблиця фактів**: `Customer` ← `Internet Sales` → `Product` дає концептуальний M:N між клієнтами й товарами. Окремий міст потрібен, коли зв'язок існує **між вимірами** (клієнт ↔ причина покупки) або між двома фактами. Таблиця фактів, що містить **лише зовнішні ключі без метрик**, називається **factless fact** — це нормальний і корисний патерн (події без числових показників: відвідування, реєстрації, зв'язки).

Обмеження рушія (xVelocity/VertiPaq): **фізичний зв'язок за складеним ключем неможливий** — потрібен сурогатний конкатенований стовпець в обох таблицях:
```dax
SalesReasonsID = 'Internet Sales'[SalesOrderNumber] & 'Internet Sales'[SalesOrderLineNumber]
```
Той самий стовпець краще робити в **Power Query** (або взагалі в джерелі/DWH) — обчислювані стовпці DAX займають пам'ять і не стискаються так добре. Після побудови зв'язку **міст ховають** (Model → «око»), щоб не плутати авторів звітів.

## 2. Двонаправлені зв'язки: обережно

Двонаправлена крос-фільтрація — найчастіше зловживана можливість: вона дає заплутану модель, неправильні цифри й **неоднозначність**, через яку Power BI забороняє створювати нові зв'язки. Класична задача «два зрізи мають фільтрувати один одного» розв'язується **без** двонаправлених зв'язків:

```dax
Internet Sales Bidirectional =
CALCULATE (
    SUM ( 'Internet Sales'[SalesAmount] ),
    CROSSFILTER ( Customer[CustomerKey],  'Internet Sales'[CustomerKey], Both ),
    CROSSFILTER ( 'Product'[ProductKey],  'Internet Sales'[ProductKey],  Both )
)
```
Далі цю міру кладуть у **фільтр візуалізації** зрізу з умовою «is not blank» — зріз показує лише елементи, для яких є хоч один рядок фактів. Двонаправленість вмикається **тільки на час обчислення міри**, а не в моделі назавжди.

Компроміс чесно: без двонаправлених зв'язків вирази DAX стають громіздкішими й іноді повільнішими — рішення за автором моделі, але **за замовчуванням single-direction**.

## 3. Неактивні зв'язки й role-playing dimensions

Зв'язок стає **неактивним** із двох причин:
1. **Кілька шляхів фільтрації** до однієї таблиці (напр. `Sales Territory → Geography → Customer → Internet Sales` **і** прямий `Sales Territory → Internet Sales`) — рушій деактивує один зв'язок, щоб уникнути **неоднозначності**.
2. **Кілька прямих зв'язків** між тими самими таблицями — класика: `Date` і `Internet Sales` по `OrderDateKey`, `DueDateKey`, `ShipDateKey`. Активним може бути **лише один** зв'язок одночасно.

Другий випадок — це **role-playing dimension** (вимір у кількох ролях). Розв'язок — активувати потрібний зв'язок на час обчислення міри:
```dax
Internet Sales Due =
CALCULATE ( [Internet Sales], USERELATIONSHIP ( 'Internet Sales'[DueDateKey], 'Date'[DateKey] ) )

Internet Sales Shipped =
CALCULATE ( [Internet Sales], USERELATIONSHIP ( 'Internet Sales'[ShipDateKey], 'Date'[DateKey] ) )
```
Альтернатива — **фізичні копії виміру** (`Date Order`, `Date Ship`) з активними зв'язками: більше пам'яті, але зрозуміліше користувачам і не потребує окремих мір під кожну роль. Обирай за тим, що частіше: зрізи по різних датах у звіті (копії) чи окремі показники (USERELATIONSHIP).

**Тиха помилка:** міра «за датою відвантаження» без `USERELATIONSHIP` мовчки рахує за активним зв'язком (датою замовлення) — числа виглядають правдоподібно, а звіт неправильний. У рецензії перевіряй кожну «дато-специфічну» міру.

## 4. Конфігураційні таблиці й сегментація

Аналіз «по діапазонах» (цінові сегменти, вікові групи, ABC-класи) — **не обчислюваним стовпцем у факті**, а окремою **конфігураційною таблицею** з межами:

| Segment | Min | Max |
|---|---|---|
| Low | 0 | 50 |
| Medium | 51 | 450 |
| High | 451 | 1500 |
| Very high | 1501 | 999999 |

Чому так: межі змінюються бізнесом, і при вбудованому `CASE`-стовпці доведеться щоразу правити модель і перепубліковувати звіт. Конфігураційну таблицю тримають у **джерелі/DWH** (або Excel на SharePoint/OneDrive), звідки її оновлюють без правок моделі. Той самий патерн — для порогів фарбування візуалізацій (напр. «нижче середнього — червоним»).

Зв'язок із фактом тут не фізичний: сегмент визначається мірою через діапазон (`FILTER` по межах) або обчислюваним стовпцем із `LOOKUPVALUE`, якщо гранулярність дозволяє.

## 5. Ієрархії «батько-нащадок» (parent-child)

Самозв'язок (`employee → manager`) у Power BI розгортається у **пласкі рівні** через сімейство `PATH`:
```dax
Path        = PATH ( 'Parent Child'[ID], 'Parent Child'[Parent ID] )
Path Length = PATHLENGTH ( 'Parent Child'[Path] )
Level 2 ID  = PATHITEM ( 'Parent Child'[Path], 2, INTEGER )
Level 2 Name =
    LOOKUPVALUE ( 'Parent Child'[Name], 'Parent Child'[ID],
                  PATHITEM ( 'Parent Child'[Path], 2, INTEGER ) )
```
Кількість рівнів фіксується наперед (за максимальною `PATHLENGTH`) — це обмеження підходу.

**Нерівномірні (ragged) ієрархії** — коли гілки різної глибини, з'являються порожні рівні, що плутають користувачів у візуалізації. Лікується підстановкою значення попереднього рівня, коли `Path Length < N` (замість `BLANK`), або фільтром на порожні.

Паралель для SQL-студентів: у БД те саме роблять **рекурсивним CTE** (див. `analyze-sql-examples/references/dialect-recipes.md`, §12). Найкраще — розгорнути ієрархію **в ETL** і подати в Power BI уже пласкими рівнями: модель простіша, DAX швидший.

## 6. Агрегації та режими зберігання (великі дані)

Ідея агрегацій — та сама, що в OLAP: тримати **попередньо агреговану таблицю** на грубій гранулярності (напр. по датах), а деталь лишати в джерелі; рушій сам вирішує, звідки брати відповідь на запит.

**Режими зберігання таблиць:**
- **Import** — дані в пам'яті (VertiPaq), найшвидше;
- **DirectQuery** — запити йдуть у джерело (без обмеження обсягу);
- **Dual** — таблиця може працювати в обох режимах залежно від запиту.

**Патерн для великих обсягів:** факт-деталь у **DirectQuery**, агрегат в **Import**, спільні виміри в **Dual**. Тоді запит на рівні місяця/категорії обслуговує імпортований агрегат (мілісекунди), а drill-down до транзакцій — DirectQuery до джерела. Налаштовується через **Manage aggregations** (потрібне джерело з підтримкою DirectQuery: SQL Server, Synapse, PostgreSQL, ClickHouse-конектор тощо).

Якщо джерело **не** підтримує DirectQuery (Excel, CSV), агрегації робляться вручну: агрегована таблиця в Power Query → зв'язки з вимірами на відповідній гранулярності → міри для агрегату → **окремі міри-перемикачі**, що визначають рівень деталізації користувача (`ISINSCOPE`/`HASONEVALUE`) → приховати агрегат.

**Практична порада для курсу:** агрегат краще будувати **у джерелі** (view/матеріалізована вітрина в PostgreSQL або таблиця `SummingMergeTree`/`AggregatingMergeTree` у ClickHouse) — це той самий Gold-шар із `design-dwh-etl`, і його бачить не лише Power BI.

## Чек-лист просунутої моделі

- [ ] Кратність M:N застосована свідомо; типовий M:N реалізовано **мостом**, міст прихований.
- [ ] Двонаправлені зв'язки відсутні або обґрунтовані; де треба — `CROSSFILTER` у мірі, а не в моделі.
- [ ] Для role-playing дат є `USERELATIONSHIP`-міри **або** копії виміру; жодна дато-специфічна міра не покладається на активний зв'язок «за замовчуванням».
- [ ] Немає неоднозначних шляхів фільтрації (не «лікуємо» їх вимкненням потрібних зв'язків).
- [ ] Сегменти/пороги — у конфігураційній таблиці з джерела, не в `CASE` всередині моделі.
- [ ] Ієрархії parent-child розгорнуті (краще в ETL); ragged-рівні оброблені.
- [ ] Для великих фактів обрано Import/DirectQuery/Dual свідомо; агрегати живуть у DWH.
- [ ] Обчислювані стовпці DAX мінімізовані — їхнє місце в Power Query або в джерелі.
