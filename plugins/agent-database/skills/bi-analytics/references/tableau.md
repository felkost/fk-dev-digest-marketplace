# Tableau: візуальний аналіз поверх БД

Джерела: Murray «Tableau Your Data» (архітектура, підключення, обчислення), Sleeper «Practical Tableau» (практики візуалізації).

## Підключення й дані

- **Live** — запити йдуть у джерело в реальному часі (PostgreSQL/MySQL — нативні конектори; ClickHouse — ODBC/JDBC). Свіжі дані, але швидкість = швидкість джерела; кожен клік — запит.
- **Extract (Hyper)** — колонковий знімок даних у рушії Tableau; швидкий і автономний, оновлюється за розкладом. Аналог Import у Power BI. Правило те саме: агрегуй/фільтруй на вході (extract filters, aggregate to visible dims), не тягни всі сирі рядки.
- Кілька таблиць: **relationships** (логічний рівень, Tableau сам обирає join за контекстом — рекомендований шлях) або фізичні join-и; для різних джерел — cross-database join / blending (обмежений, «останній засіб»).
- Так само, як у Power BI: найкраще джерело — **готова зірка/вітрина з DWH**; складні трансформації — у SQL (custom SQL підтримується, але view в БД краще: план кешується, код версіюється).

## Основні поняття

- **Dimensions (сині)** — дискретні розрізи (категорії, дати-як-категорії); **Measures (зелені)** — неперервні агреговані числа. Синє/зелене = дискретне/неперервне — визначає, як Tableau малює (заголовки vs осі).
- Полиці Rows/Columns/Marks (Color/Size/Label/Detail) — «граматика графіки»: вимір на Detail підвищує деталізацію (гранулярність) візуалізації.
- **Обчислювані поля:** рядкові/числові/дати + агрегатні; табличні обчислення (running total, % of total, rank — рахуються по вже агрегованому результату, аналог віконних функцій).
- **LOD-вирази** — обчислення на іншому рівні деталізації, ніж візуалізація:
  - `{FIXED [customer_id] : MIN([order_date])}` — дата першої покупки клієнта (когортний аналіз!)
  - `{INCLUDE [product] : SUM([sales])}` — додати вимір у розрахунок (середнє по продуктах усередині категорії)
  - `{EXCLUDE [region] : SUM([sales])}` — прибрати вимір (частка регіону від загального)
  - Паралель для SQL-студента: `FIXED` ≈ підзапит із GROUP BY по вказаних полях; `INCLUDE/EXCLUDE` ≈ віконні функції з іншим PARTITION BY. Класика застосування — когорти: `FIXED` дата першої події → зсув у місяцях → retention-матриця (пор. SQL-рецепт у `build-data-projects`).

## Практики дашбордів (Sleeper)

- Спершу питання й аудиторія, потім тип графіка: тренд → лінія; порівняння категорій → bar; розподіл → гістограма/box; зв'язок двох мір → scatter; частини цілого → stacked bar/область (кругові — обережно, >3–4 сегментів не читається).
- Менше чорнила: прибирай рамки/сітки/дублікати підписів; колір — для сенсу (виділити, не розфарбувати); одиниці й контекст (порівняння з планом/минулим періодом) обов'язкові.
- Дашборд = відповідь на 3–5 пов'язаних питань з інтерактивністю (actions/фільтри), а не 20 графіків; продуктивність: менше marks на view, extract-агрегати, фільтри-контексти.

## Power BI чи Tableau (коротко для курсу)

- Спільне ядро навичок: зірка з DWH, розуміння гранулярності, обчислення поверх агрегатів (DAX-контексти ↔ LOD/табличні обчислення).
- Power BI: тісна інтеграція з MS-стеком, дешевший вхід, повноцінна семантична модель (DAX) — сильний вибір за замовчуванням для організацій на Microsoft.
- Tableau: сильніший у швидкому візуальному дослідженні (drag&drop exploration) і гнучкості візуалізацій; ліцензії дорожчі.
- Для навчання: обидва читають ті самі вітрини PostgreSQL/MySQL/ClickHouse — тримай еталонні SQL-запити поруч зі звітом для перевірки чисел.
