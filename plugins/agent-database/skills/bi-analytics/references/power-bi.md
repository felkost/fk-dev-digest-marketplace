# Power BI: від джерела до дашборда

Конвеєр Power BI: **Power Query (M) → модель даних (VertiPaq) → DAX-міри → візуалізації**. Джерела: Еренмюллер-Єнсен, Бахши, Розема/Влотман (DAX), Равів (Power Query), DA-100.

## 1. Power Query (мова M) — підготовка даних

- Кожен запит — послідовність **кроків** (Applied Steps), кожен крок — вираз M; редактор пише M за тебе, але читати згенерований код треба вміти (Advanced Editor).
- Типові операції: типізація стовпців (**перший обов'язковий крок** — інакше «текстові числа» ламають агрегати), фільтрація, розгортання вкладених записів, unpivot «широких» таблиць (місяці в стовпцях → рядки!), merge (≈ JOIN) і append (≈ UNION ALL), параметризовані запити.
- **Query folding (згортання запитів):** для реляційних джерел (PostgreSQL/MySQL) Power Query намагається транслювати кроки в **SQL на боці джерела**. Кроки, що ламають folding (наприклад, довільні M-функції) змушують тягти всі дані в пам'ять. Правило: кроки, що folding підтримують — на початок; перевірка — View Native Query. Найкраще — давати вже готові view/таблиці з БД.
- Табличні дані заходять через конектори; для ClickHouse — офіційний конектор або ODBC.

## 2. Модель даних — зірка обов'язкова

- Power BI **розрахований на зірку**: таблиці фактів + виміри, зв'язки «один-до-багатьох» від виміру до факту, фільтри течуть від «одного» до «багатьох». Один широкий flat-файл або клубок «багато-до-багатьох» — головні причини повільних звітів і неправильних чисел (Еренмюллер-Єнсен: розділи model theory ↔ Power BI практика).
- **Import (VertiPaq)** — колонкове стиснення в пам'яті; найшвидший варіант; обсяг обмежений пам'яттю. **DirectQuery** — запити йдуть у джерело (добре для великих/живих даних: ClickHouse, Redshift); повільніші візуалізації, обмеження DAX/M. **Composite/агрегації** — гібрид: детальні дані DirectQuery + агрегати Import.
- Зменшуй кардинальність: прибирай непотрібні стовпці (особливо унікальні ідентифікатори з фактів), дати без часу, числа замість рядків — VertiPaq стискає по стовпцях, кардинальність = розмір і швидкість.
- `dim_date`: окрема таблиця-календар, позначена як Date table — без неї time intelligence не працює коректно.

## 3. DAX — міри й контексти

- **Обчислюваний стовпець** рахується при оновленні й займає пам'ять; **міра (measure)** — на льоту в момент запиту. Правило: усе, що агрегується — міра; стовпець — лише коли значення потрібне як вісь/фільтр.
- Два контексти: **контекст рядка** (row context — в обчислюваних стовпцях/ітераторах `SUMX`) і **контекст фільтра** (filter context — від візуалізації/зрізів). Розуміння, який контекст діє — 90% розуміння DAX.
- **`CALCULATE`** — центральна функція: обчислює вираз у **зміненому контексті фільтра**:
```dax
Sales Amount   = SUM ( fact_sales[amount] )
Sales UA       = CALCULATE ( [Sales Amount], dim_store[country] = "UA" )
Sales LY       = CALCULATE ( [Sales Amount], SAMEPERIODLASTYEAR ( dim_date[date] ) )
Sales YTD      = TOTALYTD ( [Sales Amount], dim_date[date] )
Share of Total = DIVIDE ( [Sales Amount], CALCULATE ( [Sales Amount], ALL ( dim_product ) ) )
```
- Типові граблі: `DIVIDE` замість `/` (безпечне ділення на нуль — те саме правило, що NULLIF у SQL); міра всередині ітератора без розуміння context transition; фільтр таблицею (`FILTER(all_table, ...)`) там, де досить фільтра стовпця — вбиває швидкість; time intelligence без Date table.
- **Зняття фільтрів — три різні функції, три різні питання:** `REMOVEFILTERS(…)` (синонім `ALL` у ролі модифікатора `CALCULATE`, але читається однозначно як намір) — «від загального підсумку»; `REMOVEFILTERS(dim_product)` — «частка товару в поточному місяці»; `ALLSELECTED()` — «від того, що видно на екрані» (поважає зрізи користувача). Питання «частка **від чого?**» має бути сформульоване явно, інакше міра не визначена.
- **`Auto date/time`** (File → Options → Current File → Data Load) — за замовчуванням увімкнено й створює **приховану date-таблицю на кожен стовпець дати** в моделі. Це і роздуває модель, і ламає логіку фільтрів (див. нижче). У навчальних і продуктових моделях — вимикати, календар брати свій.

## 3a. Діагностика «неправильних чисел»: де живе фільтр

Головна евристика: **числа в Power BI діагностуються за топологією фільтрів (звідки прийшов фільтр і на що його накладено), а не за текстом міри.** Найпоширеніший корінь — **вісь часу взята зі стовпця таблиці фактів** замість виміру; він дає одразу три різні на вигляд симптоми:

| Симптом | Механіка | Виправлення |
|---|---|---|
| Time intelligence (`SAMEPERIODLASTYEAR`) повертає **BLANK** або **дублює поточний період** | Зріз із `fact[date]` насправді фільтрує **приховану** `LocalDateTable` (Auto date/time), а `CALCULATE` перевизначає фільтр на **стовпці факту** — два фільтри на різних таблицях перетинаються: зріз «рік+місяць» ∩ торішній місяць = порожньо; зріз «лише місяць» ∩ ті самі місяці всіх років = поточний період. Додатково: стовпець факту має **дірки** в датах (а з типом `timestamp` — ще й час), тому зсув на рік не знаходить відповідних значень | `dim_date` + Mark as date table; time intelligence **тільки** по `dim_date[date]` |
| **Частка = рівно 100%** на будь-якому зрізі | `ALL(fact_sales)` знімає фільтри лише з факту, а фільтр приходить з **іншої** таблиці (прихованої/виміру) → знаменник дорівнює чисельнику. **Сам симптом «рівно 100%» — доказ, що фільтр живе не там, де ви його знімаєте** | знімати фільтр із тієї таблиці, що його **накладає**: `REMOVEFILTERS(dim_date)` / `REMOVEFILTERS()` / `ALLSELECTED()` |
| **Періоди без даних зникають** із таблиці/графіка | вісь бере значення зі **стовпця** (місяця без продажів там фізично немає) **і** Power BI приховує рядки з BLANK-мірою — це дві незалежні причини | вісь із `dim_date` (повний календар) **плюс** міра, що повертає 0 |

```dax
-- 0 замість BLANK, але лише для періодів, що вже минули
Sales (0) =
VAR s = [Total Sales]
RETURN IF ( NOT ISBLANK ( s ), s, IF ( MIN ( dim_date[date] ) <= TODAY (), 0 ) )
```
Наївне `[Total Sales] + 0` теж «показує нулі», але заповнює **всі** дати календаря, включно з майбутніми — звіт отримує довгий хвіст нулів. Опція візуалізації *Show items with no data* допомагає лише тоді, коли міра не BLANK.

**BLANK ≠ 0:** «немає рядка» і «нуль» — різні факти (пряма аналогія `NULL` vs `0` у SQL, §2 каталогу граблів); перетворювати одне в інше можна лише свідомо.

**Паралель для студентів із SQL:** `dim_date` як вісь звіту ≡ `LEFT JOIN dim_date` у SQL — обидва розв'язують ту саму проблему зниклих періодів; а безперервність календаря однаково потрібна і для `SAMEPERIODLASTYEAR`, і для `LAG(…, 12)`. Тому еталонний SQL-запит до джерела — найкращий спосіб довести, що звіт правильний (режим викладача).
- Паралель для студентів SQL: міра ≈ агрегат у `SELECT`, контекст фільтра ≈ `WHERE`+`GROUP BY`, `ALL()` ≈ віконне `OVER ()` без PARTITION — показуй відповідності, вони прискорюють навчання.

## 3b. Python і R усередині Power BI (за Zavarella)

Скрипти Python/R вставляються у **трьох місцях**, і різниця між ними принципова:

| Місце | Як працює | Результат |
|---|---|---|
| **Завантаження** (Get data → Python/R script) | скрипт має повернути `pandas.DataFrame` (Python) або data frame (R) — саме структуру-таблицю, не словник/список | таблиця в моделі |
| **Трансформація** (Power Query → Transform → Run Python/R script) | змінна `dataset` **уже ініціалізована** результатом попередніх кроків | таблиця в моделі |
| **Візуалізація** (Python visual / R visual) | поля перетягуються в Values, скрипт малює графік (matplotlib/ggplot) | **лише зображення** |

**Головне правило:** скрипти в **Power Query персистяться в моделі** (виконуються при оновленні даних, результат доступний іншим візуалізаціям і мірам); скрипти у **візуалах не персистяться** — вони перевиконуються при кожній крос-фільтрації й нічого не додають у модель. Тому: потрібен результат далі в моделі → Power Query; потрібна лише картинка/розвідка → скрипт-візуал.

Обмеження, про які треба попереджати студентів:
- потрібен **локально встановлений** рушій (Python/R) і налаштований шлях у Power BI Desktop; для оновлення в хмарі — personal gateway із тими самими бібліотеками;
- **скрипт-візуал у Power BI Service не має доступу до інтернету** (у Desktop має) — код, що ходить в API, працює локально й «тихо» ламається після публікації; для Power Query цього обмеження немає;
- Python/R підтримуються лише в **Power BI Desktop, Service та Embedded** — не в Report Server, не в мобільних застосунках;
- виконання скриптів вимагає явного дозволу користувача (безпековий ризик).

Коли це доречно в курсі: складне очищення тексту/регулярні вирази, статистика й прогнозування, спеціальні візуалізації, яких немає штатно. **Коли недоречно:** те, що виражається в SQL/Power Query — тоді дешевше й надійніше зробити це в джерелі (див. принцип «джерело правди — БД» у SKILL.md).

## 4. Продуктивність (за Бахши/Еренмюллер-Єнсен)

Перевіряй у такому порядку: (1) модель — зірка, вузькі таблиці, низька кардинальність; (2) кроки Power Query folding-ються в джерело; (3) міри — без зайвих ітераторів по великих таблицях; (4) Performance Analyzer → які візуалізації/запити повільні; (5) агрегації/composite для дуже великих фактів. Import за замовчуванням; DirectQuery — за вимогою живих/надвеликих даних.

## Чек-лист рецензії звіту Power BI

- [ ] Модель — зірка; зв'язки 1:* від вимірів до фактів; bidirectional-фільтри лише обґрунтовано.
- [ ] Типи стовпців задано в перших кроках Power Query; важкі трансформації — у БД/ELT, не в 15 кроках M.
- [ ] Показники — мірами (не обчислюваними стовпцями); є Date table; `DIVIDE` для часток.
- [ ] **`Auto date/time` вимкнено**; вісь часу береться з `dim_date` (позначеної як date table), ніколи зі стовпця факту; ключ дати — день, не `timestamp`.
- [ ] **BLANK vs 0 оброблено свідомо** (порожні періоди показані нулями лише в минулому); «частка» має явну відповідь «від чого» (`REMOVEFILTERS`/`ALLSELECTED`).
- [ ] Числа звірені з SQL-запитом до джерела на кількох зрізах (контракт викладача: еталонний SQL поруч).
- [ ] Для великих джерел обрано свідомо Import/DirectQuery/composite; кардинальність зменшено.
