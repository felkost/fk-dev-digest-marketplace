# DAX: контексти обчислення й CALCULATE (поглиблено)

За М. Розема, Х. Влотман «DAX для профессионалов» (гл. 1.3–1.4). Це «теорія, без якої DAX здається магією»: 90% неправильних чисел у Power BI пояснюються нерозумінням контексту. Базові приклади мір — у `power-bi.md`, техніки моделювання — у `power-bi-modeling.md`.

## Три контексти

| Контекст | Де виникає | Що дає | Обмеження |
|---|---|---|---|
| **Рядка** (row context) | обчислювані стовпці, ітератори (`SUMX`, `FILTER`, …) | доступ до значень **поточного рядка тієї самої таблиці** через пряме посилання `Table[Column]` | **не поширюється зв'язками**; жодні фільтри не діють |
| **Запиту** (query context) | обчислення міри у візуалізації | набір рядків, обраний зрізами/фільтрами/мітками візуалізації | пряме посилання на стовпець **заборонене** (може бути багато рядків) |
| **Фільтра** (filter context) | результат зміни контексту кодом (`CALCULATE`) | будь-який набір фільтрів, який ти сконструював | вимагає абстрактного мислення: які фільтри активні саме зараз |

**Контекст рядка не бачить зв'язків.** Класична демонстрація — обчислювані стовпці:
```dax
TotalTax      = SUM ( fSales[Tax] )            -- у fSales: однакова сума в усіх рядках
TotalShipping = SUM ( fSales[ShippingCosts] )  -- у Date: теж однакова в усіх рядках!
```
Обидва дають той самий підсумок у кожному рядку: у контексті рядка **немає фільтрів**, отже нічого поширювати зв'язками. Щоб «побачити» зв'язок, потрібен перехід до контексту фільтра.

Для роботи в контексті рядка є дві функції-навігатори: **`RELATED`** (з боку «багато» до «один» — вимагає унікальності на іншому кінці) і **`RELATEDTABLE`** (з боку «один» до «багато», повертає таблицю → потрібна агрегація: `COUNTROWS ( RELATEDTABLE ( fSales ) )`).

## Перехід контексту (context transition)

`CALCULATE` **без аргументів фільтра** перетворює контекст рядка на контекст фільтра:
```dax
TotalShipping = CALCULATE ( SUM ( fSales[ShippingCosts] ) )   -- у таблиці Date
```
Тепер поточний рядок `Date` стає фільтром, який **поширюється зв'язком** на `fSales` — і кожна дата отримує свою суму доставки.

**Найважливіший наслідок:** будь-який **виклик міри всередині формули неявно загорнутий у `CALCULATE`**, тобто завжди спричиняє перехід контексту:
```dax
SalesByCustomer = DIVIDE ( [Sales], [Number of Customers] )   -- обидві міри = CALCULATE(...)
```
Саме тому міра всередині `SUMX` поводиться інакше, ніж той самий вираз, написаний напряму — це найчастіше джерело «незрозумілих» результатів у ітераторах.

## `CALCULATE`: чотири кроки виконання

1. **Створити новий контекст фільтра** — копію поточного (з переходом контексту, якщо ми в контексті рядка).
2. **Прибрати наявні фільтри** зі **стовпців, згаданих в аргументах фільтра**.
3. **Застосувати нові фільтри** з аргументів.
4. **Обчислити вираз** у новому контексті.

Три висновки, які треба донести студентам:

**(а) Порядок аргументів фільтра не має значення — кроки 2 і 3 виконуються саме в цьому порядку.**
```dax
Sales373_374 = CALCULATE ( [Sales], Products[ProductID] = 373, Products[ProductID] = 374 )
```
Це **завжди BLANK**: обидва фільтри застосовуються разом (перетин), а товар не може мати два ID одночасно. Інтуїція «другий фільтр перезапише перший» — хибна.

**(б) Стовпці, не згадані в аргументах, зберігають свої фільтри.** Порівняй дві міри в матриці з мітками `Group` + `ProductID`:
```dax
Sales373          = CALCULATE ( [Sales], Products[ProductID] = 373 )
SalesRearWheel525 = CALCULATE ( [Sales], Products[Product] = "REAR WHEEL STEEL #525" )
```
- `Sales373` **прибирає** фільтр із `ProductID` (він згаданий) і ставить свій → у кожному рядку показує продажі товару 373… але фільтр по `Group` лишається, тому в інших групах буде BLANK.
- `SalesRearWheel525` прибирає фільтр лише з `Product`, а `Group` і `ProductID` із контексту запиту **лишаються** → непорожнє значення тільки в рядку самого товару 373.

Одна й та сама «мета» — різні результати, бо згадано різні стовпці. Це і є практичне значення кроку 2.

**(в) `ALL` в аргументі — теж «згадування стовпців».**
```dax
Sales373OrWhat = CALCULATE ( [Sales], Products[ProductID] = 373, ALL ( Products ) )
```
`ALL(Products)` знімає всі фільтри з таблиці, а `ProductID = 373` додає свій → результат: продажі товару 373 **незалежно від контексту запиту**.

**`KEEPFILTERS` пропускає крок 2** для свого аргументу — новий фільтр **додається** до наявного (перетин), а не замінює його:
```dax
SalesLargeUnitAmount KeepFilters = CALCULATE ( [Sales], KEEPFILTERS ( fSales[UnitAmount] > 25 ) )
```

## Алгоритм побудови складної міри (метод із книги)

1. **Вивчи можливі контексти запиту**, у яких мірa працюватиме (які зрізи/мітки будуть у візуалізаціях, включно з рядками підсумків, де фільтрів менше).
2. **Виріши, який контекст фільтра** потрібен для правильного результату.
3. **Визнач спосіб переходу** від (1) до (2) — які фільтри зняти, які додати, які зберегти.

Це принципово інший спосіб мислення, ніж SQL (де ти описуєш вибірку) чи Excel (де посилаєшся на клітинки) — тому студентам із SQL-бекґраундом варто явно проговорити різницю: **у DAX ти не пишеш запит, ти описуєш, як змінити фільтри навколо агрегату**.

## Функції-детектори фільтрів

Для «оборонного» коду й діагностики (усі приймають посилання на стовпець):
- `ISFILTERED(A)` — чи накладено фільтр **прямо** на `A`;
- `ISCROSSFILTERED(A)` — чи є фільтри, що впливають на `A` (зокрема через зв'язки);
- `HASONEFILTER(A)` — чи обрано **одне** значення прямим фільтром;
- `HASONEVALUE(A)` — чи в поточному контексті лишилося одне значення (з будь-якої причини);
- `ISINSCOPE(A)` — чи `A` є поточним рівнем деталізації **у візуалізації** (для ієрархій/матриць; правильний вибір для «підсумок чи рядок деталі»).

Типове застосування — різна логіка для рядків деталі й підсумкових рядків (напр. відсотки, які не можна просто підсумовувати). Пастка: `HASONEVALUE` спрацьовує і тоді, коли значення одне «випадково» (у даних лишився один елемент) — для рівнів ієрархії надійніший `ISINSCOPE`.

## Прямі паралелі із SQL (для навчання)

| DAX | SQL-аналог |
|---|---|
| контекст запиту | `WHERE` + `GROUP BY` поточного зрізу |
| контекст рядка | обчислення в межах рядка (як вираз у `SELECT`) |
| перехід контексту (`CALCULATE`) | корельований підзапит, що бере ключ поточного рядка |
| `CALCULATE(expr, filter)` | той самий агрегат з іншим `WHERE` |
| `ALL(Table)` / `REMOVEFILTERS()` | агрегат по всій таблиці — `OVER ()` без `PARTITION BY` |
| `ALLSELECTED()` | агрегат у межах поточної вибірки звіту |
| `KEEPFILTERS(f)` | додати умову через `AND` замість заміни умови |
| `RELATED` | `JOIN` до виміру за FK |

## Чек-лист рецензії міри

- [ ] Зрозуміло, у якому контексті працює вираз (рядка / запиту / зміненому фільтра).
- [ ] Усі стовпці, згадані в аргументах `CALCULATE`, згадані **свідомо** (їхні фільтри буде знято).
- [ ] Перевірено поведінку в рядках **підсумків** (там фільтрів менше — часті «дивні» числа).
- [ ] Виклик міри всередині ітератора врахований як перехід контексту.
- [ ] `KEEPFILTERS` використано там, де фільтр має **додаватися**, а не замінювати.
- [ ] Для логіки «підсумок vs деталь» узято `ISINSCOPE`, а не `HASONEVALUE` навмання.
- [ ] Числа звірені з еталонним SQL-запитом до джерела.
