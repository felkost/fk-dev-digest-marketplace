# Моделювання даних за стандартами DAMA-DMBOK (гл. 5, 11)

Довідник «як за стандартами»: рівні моделей, схеми представлення, архітектури сховищ. Джерело — DAMA-DMBOK (2-ге вид., гл. 5 «Моделювання й проєктування даних», гл. 11 «Ведення сховищ даних і бізнес-аналітика»); зв'язок вимог із бізнес-аналізом — BABOK v3 (IIBA).

## Три рівні моделі даних (ANSI/SPARC → DMBOK)

| Рівень | Що містить | Кому/навіщо |
|---|---|---|
| **Концептуальна (CDM)** | лише ключові бізнес-сутності та зв'язки між ними, з визначеннями | розмова з бізнесом; узгодження понять |
| **Логічна (LDM)** | + атрибути, ключі (природні), нормалізація/ФЗ — **незалежно від СУБД** | аналітики/архітектори; вимоги до даних |
| **Фізична (PDM)** | типи даних, сурогатні ключі, індекси, партиції, денормалізація — під конкретну СУБД | розробники/DBA; DDL |

Порядок робіт за DMBOK: планування → побудова (CDM→LDM→PDM) → **перевірка й оцінка якості** (нормалізація, іменування, відповідність вимогам) → **супровід** (модель живе разом зі схемою; розсинхрон моделі й БД — типова хвороба). Концептуальне/логічне моделювання — це аналіз вимог (тут працює зв'язка з BABOK: зацікавлені сторони → бізнес-процеси → вимоги до даних), фізичне — проєктування.

## Шість схем представлення даних (DMBOK, табл. 9)

1. **Реляційна** (нотації: IE «пташині лапки», IDEF1X, Баркер, Чен) — точне відображення бізнес-правил, один факт в одному місці; ідеальна для OLTP. CDM/LDM у реляційній нотації можна будувати навіть для NoSQL-цілей.
2. **Багатовимірна** — оптимізація під запити/аналіз; зв'язки = навігаційні шляхи до відповідей на бізнес-питання (деталі — `dimensional-modeling.md`).
3. **Об'єктно-орієнтована (UML)** — класи з атрибутами **й операціями**; для БД сама по собі рідко, але корисна на межі із застосунком.
4. **Фактографічна (fact-based: ORM2/FCO-IM)** — моделювання від елементарних фактів-речень; нішева.
5. **Хронологічні (time-based):**
   - **Data Vault** — гібрид 3НФ і зірки для «сирого» історизованого шару EDW: **хаби** (бізнес-ключі), **лінки** (зв'язки/транзакції між хабами), **сателіти** (атрибути з історією). Плюс: аудит, гнучке дозавантаження джерел; мінус: багато таблиць, звіти будують не з нього, а з вітрин-зірок поверх.
   - **Анкерне моделювання** — якорі/атрибути/зв'язки/вузли; для схем, що часто змінюються структурно (6НФ-стиль).
6. **NoSQL-схеми** — документна, ключ-значення, колонкова, графова; **лише фізичний рівень** — концепт/логіку однаково варто малювати реляційно (див. `db-connectivity-cloud/references/db-families-cap.md` і `nosql-aws-best-practices.md`).

Практичний висновок для курсу: **вибір схеми — від призначення системи**: OLTP → реляційна 3НФ; DWH-звітність → зірка; сирий шар EDW з аудитом → Data Vault; застосунок із гнучкою схемою → документна; й у всіх випадках CDM/LDM спершу.

## Сховище даних: Інмон vs Кімбол (DMBOK, гл. 11)

Спільне в обох: дані надходять із зовнішніх систем-джерел; упорядковуються, щоб підвищити цінність; призначені для аналізу й довіреного доступу.

- **Інмон, «Корпоративна інформаційна фабрика» (CIF):** DW — *предметно-орієнтований, інтегрований, прив'язаний до часу, незмінюваний* набір даних у **нормалізованій** моделі. Компоненти: застосунки-джерела → **staging** (область тимчасового зберігання, користувачам недоступна) → інтеграція/перетворення → **ODS** (операційні дані за 30–90 днів, змінювані; не всім потрібен) → **EDW** (історія за роки, append-only) → **вітрини (data marts)** під конкретних споживачів. Потік: від операцій до аналітики, від рядових користувачів до керівників.
- **Кімбол, багатовимірне DW:** сховище = «копія транзакційних даних, структурована для запитів і аналізу» — тобто **сукупність зіркових вітрин**, інтегрованих через **шину согласованих вимірів** (bus matrix). Нормалізація свідомо відкинута.
- Практика сьогодні — гібрид: сирий/історичний шар (нормалізований або Data Vault, або data lake/Bronze) + вимірні вітрини для BI (зірки/Gold). Ключові властивості DW за Інмоном (незмінюваність, привязка до часу) реалізуються в будь-якому варіанті: **у сховище не пишуть UPDATE — додають версії зі штампом часу**.

## Найкращі практики моделювання (звід)

- Іменування: послідовні конвенції (сутності — однина, зрозумілі повні слова; без крипто-абревіатур), визначення для кожної сутності й атрибута — модель без визначень не пройде рецензію.
- Один факт — одне місце (реляційний шар); свідома надлишковість — лише у вимірному/NoSQL шарі з механізмом синхронізації.
- Модель — частина поставки: діаграма + словник даних (metadata) versionуються разом із DDL-міграціями.
- Рецензія моделі — за чек-лістами (нормальні форми для OLTP; чек-ліст зірки — `dimensional-modeling.md`; аномалії — `analyze-task-conditions/references/relational-modeling.md`).
