# Вимірне моделювання (зірка/сніжинка) з DDL для PostgreSQL / MySQL / ClickHouse

За Кімболом (через DAMA-DMBOK, гл. 5 і 11) та «Data Modeling with Microsoft Power BI» (Еренмюллер-Єнсен). Мета вимірної моделі — **оптимізувати читання й аналіз**, на відміну від нормалізованої OLTP-моделі, оптимізованої під швидкий безпомилковий запис.

## Складники

- **Таблиця фактів** — числові показники бізнес-процесу (сума, кількість). Займає ~90% обсягу сховища, росте безперервно. Рядок = одна подія/вимірювання на визначеному рівні деталізації.
- **Таблиці вимірів** — описові атрибути («по чому» аналізуємо): дата, продукт, клієнт, магазин. Сильно денормалізовані, ~10% обсягу, є «точками входу» запитів (`WHERE`/`GROUP BY` по атрибутах виміру).
- **Grain (гранулярність)** — рівень деталізації рядка фактів; фіксується **першим** («один рядок = один товар у чеку», «= студент-курс-дата»). Всі факти й виміри мають відповідати заявленому grain; змішування рівнів (чек і позиція чека в одній таблиці) — класична помилка.
- **Зірка** — факт у центрі + пласкі виміри навколо. **Сніжинка** — виміри нормалізовані в ієрархії (продукт → категорія → відділ). Для BI-інструментів (Power BI, Tableau) **зірка — стандарт за замовчуванням**: менше JOIN-ів, зрозуміліша користувачу; сніжинка виправдана хіба великими вимірами, що дублюють багато тексту.
- **Согласовані (conformed) виміри** — спільні виміри для кількох фактів (Дата, Продукт у Продажах і Запасах) з єдиними значеннями. **Матриця шини** (bus matrix): рядки — бізнес-процеси (факти), стовпці — виміри; хрестики показують, які виміри якими процесами використовуються — це план інтеграції вітрин у сховище.
- **Согласовані факти** — стандартизовані визначення показників: «приріст клієнтів» має означати одне й те саме в усіх звітах; розбіжність визначень між відділами — виявляй на етапі аналізу.

## Три типи таблиць фактів (і як рахувати воронки)

За Кімболом факти бувають трьох типів — плутанина між ними породжує більшість «неправильних чисел» у звітах:

| Тип | Grain | Коли | Поведінка рядка |
|---|---|---|---|
| **Transaction** | одна подія (продаж, запис, платіж) | 90% випадків; максимальна деталізація | append-only, ніколи не оновлюється |
| **Periodic snapshot** | стан на кінець періоду (залишок на складі на кінець дня, баланс рахунку) | коли важливий *рівень*, а не подія | дописується щоперіоду; факти **напівадитивні** (по часу не підсумовуються — залишки не додають, беруть останній/середній) |
| **Accumulating snapshot** | один екземпляр процесу з кількома етапами (заявка → оплата → відвантаження) | воронки, lifecycle, час між етапами | рядок **оновлюється** при проходженні кожного етапу |

### Воронки: два способи порахувати конверсію

**Спосіб 1 — join двох transaction-фактів** через вироджений вимір (`enrollment_id`, `order_id`):
```sql
SELECT c.category,
       ROUND(100.0 * COUNT(DISTINCT p.enrollment_id)
                   / COUNT(DISTINCT e.enrollment_id), 1) AS conversion_pct
FROM fact_enrollment e
LEFT JOIN fact_payment p ON p.enrollment_id = e.enrollment_id   -- LEFT, не INNER!
JOIN dim_course c ON c.course_sk = e.course_sk
GROUP BY c.category;
```
Дві обов'язкові пересторони: **`LEFT JOIN`** (INNER губить саме тих, хто не конвертувався — тобто знаменник воронки), і **`COUNT(DISTINCT …)`** проти fan-out (кілька платежів за один запис — розстрочка — інакше конверсія вийде >100%; це §6a каталогу граблів на матеріалі DWH).

**Спосіб 2 — accumulating snapshot** (канонічний патерн Кімбола для воронок):
```sql
CREATE TABLE fact_enrollment_funnel (      -- grain: 1 рядок = 1 запис на курс
  enrollment_id          BIGINT PRIMARY KEY,
  student_sk             BIGINT NOT NULL,
  course_sk              BIGINT NOT NULL,
  enrolled_date_key      INT    NOT NULL,
  first_payment_date_key INT,              -- NULL, доки не оплачено
  is_paid                BOOLEAN NOT NULL DEFAULT FALSE,
  days_to_payment        INT,
  total_paid             NUMERIC(14,2) NOT NULL DEFAULT 0
);
```
Конверсія стає `AVG(is_paid)` — без join фактів; BI-інструменти (Power BI/Tableau) працюють із цим набагато краще, ніж із fact-to-fact join (там це прямий антипатерн моделі). Ціна — рядок треба **оновлювати**: у PostgreSQL/MySQL це нормальний `UPDATE`/`MERGE`, у **ClickHouse погано** (мутації дорогі) → там або версії рядків + `argMax`, або повний перерахунок вітрини за вікно.

### Гроші у факті vs прайс у вимірі

Правило, що рятує історичну коректність звітів: **фактичне грошове значення транзакції «вморожується» у факт** (`amount` у `fact_payment`), а **прайс — SCD2-атрибут виміру** (`dim_course.price`). Тому:
- виручку рахують **тільки** `SUM(fact.amount)`, і зміна прайсу заднім числом не переписує історію;
- «скільки коштував курс у березні» відповідає SCD2-вимір;
- поширена помилка студентів — «виручка = поточна ціна × кількість записів»: після будь-якої зміни прайсу всі минулі звіти стають хибними.

## Ключі й SCD

- Виміри отримують **сурогатний ключ** (ціле), а природний/бізнес-ключ зберігається атрибутом. Причини: незалежність від змін у джерелі, компактні join-и, підтримка версій SCD2 (один бізнес-ключ → кілька рядків версій).
- **SCD (повільно змінювані виміри):** тип 1 — перезапис (історія не потрібна); тип 2 — нова версія-рядок із `valid_from`/`valid_to`/`is_current` (стандарт для звітності в минуле); тип 3 — окремий стовпець «попереднє значення» (обмежена історія). Тип фіксуй по кожному атрибуту виміру. Реалізація SCD2 у конвеєрі — див. `build-data-projects/references/etl-pipeline-blueprint.md` (MERGE/AUTO CDC).
  - **Тип фіксується по атрибуту, а не по виміру** — і це не стилістика. Якщо версіонувати вимір по швидкозмінному атрибуту (рейтинг, лічильник, останній вхід), кожна його зміна породжує новий рядок, і вимір починає рости зі швидкістю таблиці фактів — «rapidly changing monster dimension». Ліки: SCD2 лише по повільних атрибутах, швидкий оновлювати на місці (тип 1) у поточній версії, а якщо він таки потрібен історично — виносити в **міні-вимір** бендами (`rating_band`), не в основний.
  - **Спроєктувати SCD2 — це половина роботи; друга половина — коректно на нього заджойнитися.** Предикат as-of join («значення виміру на момент події») має чотири тихі точки відмови — `is_current = TRUE` замість темпорального предиката, `BETWEEN` замість напіввідкритого інтервалу, `NULL` у відкритій межі, `INNER JOIN` при `NULL` у FK факту. Розібрано з замірами в `analyze-sql-examples/references/fundamentals-and-gotchas.md` §6c. Конвенція `valid_to NOT NULL DEFAULT '9999-12-31'` (як у DDL нижче) робить третю пастку неможливою **конструктивно** — це аргумент на користь сентинела проти `NULL` як відкритої межі.
  - **Перш ніж проєктувати SCD2 — спитай, звідки візьметься історія.** Якщо джерело перезаписує рядок (`UPDATE` в операційній таблиці), історії в ньому **фізично немає** і побудувати її заднім числом неможливо: нічний снапшот дає гранулярність доба й лише вперед, інкремент по `updated_at` губить проміжні зміни й не бачить `DELETE`, повну історію дає лише log-based CDC (Debezium/DMS). Наслідок, який треба записати у вітрину, а не замовчати: **історія починається з дати ввімкнення захоплення**, а старіші факти джойняться на найранішу відому версію.
- Факт посилається на виміри FK-стовпцями; **дата** майже завжди окремий вимір `dim_date` (календар з атрибутами: день тижня, місяць, квартал, свято) — генерується заздалегідь (рецепт календаря — `analyze-sql-examples/references/dialect-recipes.md`).

## Еталонний приклад: продажі (зірка)

Grain: **один рядок = одна позиція чека**.

**PostgreSQL:**
```sql
CREATE TABLE dim_product (
  product_sk   BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  product_id   TEXT NOT NULL,            -- бізнес-ключ джерела
  name         TEXT NOT NULL,
  category     TEXT NOT NULL,
  valid_from   DATE NOT NULL,
  valid_to     DATE NOT NULL DEFAULT '9999-12-31',   -- SCD2
  is_current   BOOLEAN NOT NULL DEFAULT TRUE
);
CREATE TABLE fact_sales (
  date_key     INT  NOT NULL REFERENCES dim_date(date_key),
  product_sk   BIGINT NOT NULL REFERENCES dim_product(product_sk),
  store_sk     BIGINT NOT NULL REFERENCES dim_store(store_sk),
  receipt_id   TEXT NOT NULL,
  qty          NUMERIC(12,3) NOT NULL,
  amount       NUMERIC(14,2) NOT NULL
) PARTITION BY RANGE (date_key);          -- партиції за часом
```

**MySQL** — те саме з діалектними замінами: `BIGINT AUTO_INCREMENT PRIMARY KEY`, тип `DECIMAL` замість `NUMERIC` (синоніми), `TEXT`→`VARCHAR(n)` для індексованих полів, `PARTITION BY RANGE (date_key)` із явними межами секцій; FK підтримуються (InnoDB), `BOOLEAN` = `TINYINT(1)`.

**ClickHouse** — колонкова аналітична СУБД, модель інша:
```sql
CREATE TABLE fact_sales (
  date_key   Date,
  product_sk UInt64,
  store_sk   UInt64,
  receipt_id String,
  qty        Decimal(12,3),
  amount     Decimal(14,2)
) ENGINE = MergeTree
PARTITION BY toYYYYMM(date_key)
ORDER BY (date_key, store_sk, product_sk);   -- сортувальний ключ = «індекс»
```
Ключові відмінності ClickHouse (завжди коментуй): **немає FOREIGN KEY/UNIQUE-обмежень** (цілісність забезпечує ETL), PRIMARY KEY = префікс ORDER BY (розріджений індекс, не унікальність); виміри або живуть звичайними таблицями і join-яться, або підключаються **словниками** (`dictGet`) — денормалізація в широкий факт тут теж нормальна практика; оновлення рядків дорогі (`ALTER ... UPDATE` — мутації), тому SCD2 краще вести append-only (версії рядків + `argMax`/ReplacingMergeTree за `valid_from`).

## Чек-лист рецензії вимірної моделі

- [ ] Grain заявлено явно; усі факти адитивні на цьому grain (неадитивні — %, залишки — позначені: snapshot-факт, semi-additive).
- [ ] Виміри денормалізовані (зірка), сурогатні ключі; природні ключі збережені атрибутами.
- [ ] SCD-тип зафіксовано для змінюваних атрибутів; для SCD2 є `valid_from/valid_to/is_current` і факт посилається на **версію** (product_sk), а не бізнес-ключ.
- [ ] `dim_date` існує і заповнений наперед; дати у фактах — ключами на нього.
- [ ] Conformed-виміри спільні між фактами (матриця шини), а не скопійовані з різними значеннями.
- [ ] Факт не містить текстових описів (їх місце у вимірах); вимір не містить метрик, що підсумовуються.
- [ ] Для ClickHouse: ORDER BY відповідає патернам запитів; партиції за часом; цілісність — на боці ETL.
