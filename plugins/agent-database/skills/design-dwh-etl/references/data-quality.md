# Якість даних: вимірювання, правила, перевірки в конвеєрі

За DAMA-DMBOK (2-ге вид., гл. 13 «Якість даних») з практичною реалізацією в SQL. Використовуй, коли треба спроєктувати перевірки даних у ETL/ELT, пояснити студентам «чому цифри в звіті неправильні», побудувати скоркард якості або розібрати причини проблем із даними. SQL-перевірки нижче прогнано в SQLite 3.44 (узагальнені форми) і подані для **PostgreSQL / MySQL / ClickHouse**.

## 1. Що таке «дані високої якості»

**Дані високої якості = придатні для використання за призначенням** (fitness for purpose). Звідси два наслідки, які треба проговорювати зі студентами:
- Якість **залежить від контексту й споживача**: той самий набір може бути відмінним для маркетингової розсилки й непридатним для фінансової звітності.
- Очікування щодо якості часто **не сформульовані**: споживач не вміє їх висловити, а власник системи не підозрює, що до даних є спеціальні вимоги. Тому вимоги до якості — предмет постійного діалогу, а не разового ТЗ.
- DMBOK радить не вживати «якість даних» без контексту: кажи **«дані високої/низької якості»**, «роботи із забезпечення якості», «засоби контролю якості».

**Критично важливі дані** — фокус зусиль. Кандидати: дані для нормативно-правової відповідності, бухобліку й фінансової звітності, бізнес-правил, поточної операційної діяльності, стратегічного планування. **Основні (master) дані — критичні за визначенням.** Перевіряти все з однаковою суворістю — марна трата ресурсів.

## 2. Шість вимірів якості (DAMA UK) — базовий словник

| Вимір | Питання | Типова метрика |
|---|---|---|
| **Повнота** (completeness) | Скільки даних із потенційно можливих є? | % заповнених обов'язкових полів |
| **Унікальність** (uniqueness) | Чи один реальний об'єкт = один запис? | % записів, що мають дублікат |
| **Актуальність** (timeliness) | Чи відображають дані стан на потрібний момент? | лаг завантаження; % записів, свіжіших за N год |
| **Годність** (validity) | Чи відповідає значення формату/типу/діапазону/довіднику? | % значень, що проходять правило |
| **Точність/відповідність** (accuracy) | Чи відповідає значення реальному об'єкту? | % звірених із верифікованим джерелом |
| **Узгодженість** (consistency) | Чи немає суперечностей між представленнями того самого? | % розбіжностей між системами |

Історичний контекст (за потреби згадати на лекції): Ван і Стронг (1996) — 15 параметрів у 4 групах (внутрішня якість, контекстуальна, репрезентативність, доступність); Редман — виміри для **моделі даних**, значень і представлення; Інгліш — неодмінні (inherent) vs утилітарні (pragmatic) характеристики. Стандарт **ISO 8000** (частини 100/110/120/130/140/150, і 8000-6x про управління) визначає «переносимі дані, що задовольняють вимоги» й дозволяє автоматично підтверджувати відповідність.

**Головне правило метрики:** вимір має бути **вимірним** (у дискретному діапазоні) і **значущим для бізнесу**. «Актуальність» без критерію «свіжіші за 24 год» — порожнє слово.

## 3. Бізнес-правила якості даних → SQL-перевірки

Типи правил за DMBOK і їхня реалізація. Узагальнені запити перевірено в SQLite; діалектні відмінності позначено.

**Обов'язковість / повнота:**
```sql
SELECT COUNT(*) AS violations FROM customers
WHERE email IS NULL OR TRIM(email) = '';        -- порожній рядок ≠ NULL, ловимо обидва
```

**Формат (validity):**
```sql
-- PostgreSQL: оператор регулярного виразу
SELECT COUNT(*) FROM customers WHERE email !~ '^[^@\s]+@[^@\s]+\.[A-Za-z]{2,}$';
-- MySQL 8.0+: REGEXP_LIKE
SELECT COUNT(*) FROM customers WHERE NOT REGEXP_LIKE(email, '^[^@[:space:]]+@[^@[:space:]]+\\.[A-Za-z]{2,}$');
-- ClickHouse: match()
SELECT count() FROM customers WHERE NOT match(email, '^[^@ ]+@[^@ ]+\\.[A-Za-z]{2,}$');
-- Портативний спрощений варіант (працює всюди, зокрема SQLite):
SELECT COUNT(*) FROM customers WHERE email NOT LIKE '%_@_%.__%';
```

**Множина допустимих значень (довідник):**
```sql
SELECT c.customer_id, c.country_code
FROM customers c
LEFT JOIN country_ref r ON r.code = c.country_code
WHERE c.country_code IS NOT NULL AND r.code IS NULL;   -- анти-join = «немає в довіднику»
```
Ця ж перевірка ловить **розбіжності регістру** (`'ua'` vs `'UA'`) — типовий продукт ручного вводу.

**Діапазон / розумність (reasonability):**
```sql
SELECT COUNT(*) FROM customers WHERE created_at > CURRENT_DATE;      -- дати з майбутнього
SELECT COUNT(*) FROM orders    WHERE amount < 0 OR amount > 1000000; -- викиди сум
```

**Унікальність:**
```sql
SELECT email, COUNT(*) AS dup_cnt
FROM customers WHERE email IS NOT NULL
GROUP BY email HAVING COUNT(*) > 1;
```

**Посилальна цілісність (сироти)** — критично там, де FK не забезпечується СУБД (ClickHouse, Redshift, партиційований MySQL, Cassandra/DynamoDB):
```sql
SELECT o.order_id, o.customer_id
FROM orders o
LEFT JOIN customers c ON c.customer_id = o.customer_id
WHERE c.customer_id IS NULL;
```

**Логічна відповідність (крос-польова):**
```sql
-- один поштовий індекс не може належати двом містам
SELECT postal_code, COUNT(DISTINCT city) AS cities
FROM customers WHERE city IS NOT NULL
GROUP BY postal_code HAVING COUNT(DISTINCT city) > 1;
-- інші приклади: end_date >= start_date; paid_at >= created_at; sum(позицій) = total чека
```

**Контрольна звірка (accuracy)** — порівняння з верифікованим джерелом:
```sql
SELECT d.order_id, d.amount AS dwh_amount, s.amount AS source_amount
FROM dwh.fact_orders d JOIN staging.orders s USING (order_id)
WHERE d.amount <> s.amount;     -- обов'язковий крок регресії після зміни ETL
```

**Статистичні правила** (за DMBOK — окремий вид): допустима кількість записів у файлі, допустиме середнє по транзакціях, частка значень за замовчуванням. Це вже міст до §6 (SPC).

## 4. Профілювання даних (data profiling)

Профілювання — **перший крок**, розвідка перед правилами: що в даних насправді є. Мінімальний профіль стовпця одним запитом:
```sql
SELECT COUNT(*)                                  AS rows_total,
       COUNT(email)                              AS filled,
       ROUND(100.0*COUNT(email)/COUNT(*), 1)     AS fill_pct,
       COUNT(DISTINCT email)                     AS distinct_cnt,
       MIN(LENGTH(email))                        AS min_len,
       MAX(LENGTH(email))                        AS max_len
FROM customers;

-- частотний розподіл: ловить сміття, аномальний регістр, «значення за замовчуванням»
SELECT country_code, COUNT(*) AS cnt
FROM customers GROUP BY country_code ORDER BY cnt DESC;
```
На що дивитися: лічильник NULL (чи допустимі?), min/max значень і довжин (від'ємні суми, «хвости» в текстах), частотний розподіл (домінування дефолту `'N/A'`, `'1900-01-01'` — ознака поганого інтерфейсу вводу), відповідність типу/формату. Далі — **крос-стовпцевий** аналіз (перекриття, приховані залежності) і **крос-табличний** (перетини множин, кандидати у FK).

Діалектні зручності: PostgreSQL — розширення `pg_stats`/`ANALYZE` для швидких оцінок; MySQL — `information_schema` + `PROCEDURE ANALYSE()` (застаріло); **ClickHouse** — `countIf()`/`uniqExact()`/`quantile()` роблять профілювання дуже дешевим на великих обсягах:
```sql
SELECT count() AS rows_total,
       countIf(email IS NULL) AS nulls,
       uniqExact(email)       AS distinct_cnt,
       quantile(0.99)(amount) AS p99
FROM customers;
```

**Профілювання ≠ розв'язання проблеми.** Воно виявляє симптоми; далі потрібні аналіз бізнес-процесу, походження даних (lineage) і кореневих причин.

## 5. Метрики, пороги, скоркард

Базова формула по кожному правилу `r`:
```
Придатні(r)   = (Перевірено − Відбраковано) / Перевірено × 100%
Непридатні(r) = Відбраковано / Перевірено × 100%
```
Кожне правило мусить мати **цільовий/пороговий показник** (напр. «не менше 98% чинних email»), інакше вимір не має сенсу. Приклад скоркарда одним запитом (перевірено в SQLite; у ClickHouse зручніше через `countIf`):
```sql
WITH t AS (SELECT COUNT(*) AS n FROM customers)
SELECT 'completeness: email' AS rule_name,
       (SELECT COUNT(*) FROM customers WHERE email IS NULL) AS failed,
       (SELECT n FROM t) AS total
UNION ALL
SELECT 'validity: country_code',
       (SELECT COUNT(*) FROM customers c
        LEFT JOIN country_ref r ON r.code = c.country_code
        WHERE c.country_code IS NOT NULL AND r.code IS NULL), (SELECT n FROM t)
UNION ALL
SELECT 'uniqueness: email',
       (SELECT COUNT(*) FROM (SELECT email FROM customers WHERE email IS NOT NULL
                              GROUP BY email HAVING COUNT(*) > 1) d), (SELECT n FROM t);
```
Результат такого запиту складають у таблицю історії (`dq_run(run_ts, rule_name, total, failed, threshold, status)`) — це дає **тренд якості**, а не разовий знімок, і живить дашборд у Power BI/Tableau (див. скіл `bi-analytics`).

Дві категорії метрик за призначенням: **інформування споживачів** (по критичних елементах даних) і **управління ризиком** (по місцях, де проблеми вже траплялися). Правило DMBOK: усунувши проблему в одній функції обробки, **постав під моніторинг усі аналогічні** — рецидиви типові.

## 6. Життєвий цикл: PDCA + статистичний контроль

Якість даних будується як управління якістю виробництва: дані — продукт процесу, а кожен етап (збір → передача → інтеграція → агрегація) несе свої ризики.

**Цикл Шухарта — Демінга (PDCA):** Plan (виявити критичні дані, правила, корені проблем) → Do (усунути кореневі причини, налаштувати вимірювання) → Check (моніторинг проти порогів) → Act (оперативне усунення поточних проблем; вихід за межі — новий цикл). Цикл перезапускається, коли: показники вийшли за допуски; з'явився новий набір даних; змінилися вимоги; змінилися бізнес-правила.

**SPC / контрольні карти:** якщо процес стабільний, показник (напр. % записів зі значеннями за замовчуванням) коливається довкола середнього в межах контрольних меж (UCL/LCL). Вихід за межі — **особлива причина** (зовнішня подія), яку треба розслідувати; коливання всередині — **загальні причини**, властиві самому процесу. Практичний переклад для ETL: не алерти «щось не так», а **порогові алерти на статистично обґрунтованих межах**; спершу накопич ряд вимірювань, потім став пороги.

**Економіка:** зробити дані правильними одразу дешевше, ніж виправляти; вбудувати контроль якості в процеси з самого початку — на порядок дешевше, ніж «прикрутити» потім. Визначення критеріїв якості **до** проєктування нової системи — ознака зрілості організації.

## 7. Каталог причин проблем (для розбору зі студентами)

**На вводі:** погано спроєктовані форми (немає валідації → оператор економить час і лишає дефолти); задовгі невпорядковані списки; **перепризначення полів** («використаємо старе поле під нову потребу» — плутанина назавжди); низька підготовка операторів; оплата за швидкість, а не за точність; зміни бізнес-правил, не відображені в інтерфейсі.

**На обробці:** хибні уявлення про джерело (експерт звільнився, документації немає); **застарілі бізнес-правила**; **зміна структури джерела без попередження** — найгірший випадок, коли зміна проходить непоміченою й виявляється лише після накопичення браку (звідси правило: схема джерела має перевірятися на кожному запуску).

**Проєктні рішення систем:** вимкнена посилальна цілісність «заради продуктивності» (→ дублі, розбіжні звіти, дефолти замість відсутніх значень); скасовані перевірки унікальності (→ спотворені суми й середні); помилки в мэппінгу/формулах; **некоректна модель даних** (гіпотези, що не відповідають дійсності); хронологічні неузгодженості (різні формати дати/часу без єдиного словника); дублювання через слабке управління основними даними («одне джерело — багато локальних копій» і «багато джерел — один екземпляр» без правил пріоритету).

**Рецидиви від поспішних «фіксів»:** ручні `UPDATE` просто в проді замість виправлення правил — неперевірений код, незворотні зміни (значення переписуються поверх), частина даних (історія) лишається зіпсутою. Правило: виправлення даних — така сама зміна, як зміна коду (тестування, журнал, зворотність).

## 8. Превентивні vs корективні заходи

**Превентивні (пріоритет):** контроль на вході (правила відбраковки); підготовка й мотивація персоналу за точність, а не швидкість; «міжмережевий екран даних» — шар правил перед завантаженням у сховище з алертом при падінні нижче порога; перевірка якості даних **зовнішніх постачальників** (структури, визначення, походження); формалізований контроль змін (жодних прямих правок в обхід процесу).

**Корективні:** три способи виправлення —
1. **Автоматичне** (правила + довідники, без людини) — лише в середовищі з опрацьованими стандартами й відомою структурою помилок (класика: стандартизація адрес);
2. **Напівавтоматичне** (авто-виправлення + ручне підтвердження за порогом упевненості) — типове для MDM;
3. **Ручне** — коли автоматизація неможлива або дані надто чутливі; обов'язкові окремий інтерфейс і журнал правок, без прямого запису в прод.

**Аналіз кореневих причин** — обов'язковий крок перед виправленням: «п'ять чому», діаграма Ісікави («риб'ячий скелет»), аналіз Парето (80/20), аналіз процесу. Приклад із DMBOK: якість файлів падає в січні/квітні/липні/жовтні → файли приходять 30-го, а не 25-го → бухгалтерія в ці місяці закриває квартал і робить вивантаження поспіхом. **Корінь — конфлікт пріоритетів і графік, а не «погані дані»**; рішення — зсунути графік передачі, а не чистити файли щокварталу.

## 9. Як вбудувати в конвеєр (практика для курсу)

- **Перевірки — окремий крок ETL** (патерн ETL-VP із `etl-elt-patterns.md`), а не «десь у трансформації»: браковані рядки йдуть у **карантинну таблицю** з причиною (`dq_quarantine(rule_name, payload, failed_at)`), не зникають мовчки.
- **Контракти на дані (data contracts):** перевіряти схему джерела на кожному запуску (набір і типи стовпців) — ловить «тиху» зміну структури.
- **Обмеження в СУБД — перша лінія оборони, але діалектно різна:**
  - **PostgreSQL** — `NOT NULL`, `CHECK`, `UNIQUE`, FK, `DOMAIN`, `EXCLUDE`; забезпечуються завжди.
  - **MySQL** — `CHECK` **реально працює лише з 8.0.16**; у старіших версіях синтаксис приймався й **тихо ігнорувався** (класична пастка); FK лише в InnoDB і **не в партиційованих таблицях** (див. `analyze-sql-examples/references/mysql.md`).
  - **ClickHouse** — `CONSTRAINT … CHECK` перевіряється на вставці, але **FK/UNIQUE немає** → унікальність і посилальна цілісність забезпечуються ETL і регулярними DQ-запитами (`ReplacingMergeTree` дедуплікує *зрештою*, це не гарантія унікальності в запиті).
- **Тести як код:** у dbt-стилі — `unique`, `not_null`, `accepted_values`, `relationships`; без dbt ті самі перевірки живуть SQL-скриптами в репозиторії й ганяються після кожного завантаження.
- **Порядок:** профілювання (разово, на старті) → правила на критичні дані → пороги й скоркард → алерти → аналіз кореневих причин → превентивні зміни. Не навпаки.

## Чек-лист рецензії якості даних

- [ ] Визначено **критично важливі** дані — перевірки сфокусовані на них, а не «на всьому».
- [ ] Кожне правило прив'язане до **виміру** (повнота/унікальність/годність/…) і має **поріг**.
- [ ] Правила виражені **виконуваним SQL** і ганяються автоматично після завантаження.
- [ ] Браковані рядки потрапляють у **карантин** із причиною; нічого не зникає мовчки.
- [ ] Результати вимірювань **зберігаються історично** (тренд), а не лише «зелено/червоно» зараз.
- [ ] Схема джерела перевіряється на кожному запуску (контракт на дані).
- [ ] Для кожної повторюваної проблеми зроблено **аналіз кореневої причини**, а не лише чистку.
- [ ] Ураховано, які обмеження СУБД **не** забезпечує (ClickHouse/Redshift/партиційований MySQL) — там перевірки обов'язкові в ETL.
- [ ] Виправлення даних відбувається через процес (тест, журнал, зворотність), а не ручним `UPDATE` у проді.
