# Процеси ETL / ELT: патерни, затримка, CDC (за DAMA-DMBOK гл. 8, «Building ETL Pipelines with Python», M. Palmer «Understanding ETL»)

Проєктний рівень: як обрати архітектуру переміщення даних. Практична реалізація (код, blueprint, medallion) — `build-data-projects`.

## ETL vs ELT

- **ETL (Extract → Transform → Load):** перетворення до завантаження, зазвичай через **staging** (проміжне сховище). Staging дає журнал кроків і перезапуск з проміжної точки, але коштує диска й часу — для мінімальної затримки його пропускають (перетворення в пам'яті).
- **ELT (Extract → Load → Transform):** сирі дані вантажаться в цільову систему як є, перетворення виконує **сама цільова система** (SQL у DWH). Обирається, коли ціль потужніша за проміжні сервери (Redshift/BigQuery/ClickHouse/Snowflake, data lake). Бонус: у цілі лишається копія сирих даних для інших процесів (це і є Bronze-шар). Сучасні хмарні DWH зробили ELT типовим вибором: T = SQL/dbt.
- **Мэппінг** — специфікація «джерело → ціль»: звідки береться кожен атрибут, правила відбору рядків, правила заміни в цілі, формули перетворення. Мэппінг-документ (хоч таблицею в md) — обов'язковий артефакт проєктування, а не «код і є документація».
- Типові перетворення (T): зміна формату/кодування, зміна структури (денормалізація/нормалізація), **семантична конверсія** (коди 0/1/2 → «невідомо/чол/жін»), дедуплікація, переупорядкування.

## Спектр затримки (latency): від пакета до потоку

| Режим | Суть | Коли |
|---|---|---|
| **Пакетний (batch)** | за розкладом; «снапшот» (повний стан) або «дельта» (лише зміни) | звітність, DWH; найпростіше й найдешевше |
| **Мікропакети (micro-batch)** | той самий batch кожні кілька хвилин | зменшити затримку без зміни архітектури |
| **CDC (Change Data Capture)** | передаються лише зміни; на основі даних (timestamps/прапорці/тригерні копії) або **журналу транзакцій** (log-based: Debezium, AWS DMS — найменше навантажує джерело) | синхронізація систем, живлення DWH без повних вивантажень |
| **Near-real-time / event-driven** | дрібні порції за подіями, асинхронно (черга/шина) | інтеграція застосунків; затримка секунди-хвилини |
| **Синхронний real-time** | ціль підтверджує запис у тій самій транзакції (2PC) | коли розсинхрон неприпустимий; дорого, зв'язує системи |
| **Потокова обробка (streaming)** | безперервний потік подій (Kafka/Kinesis), обробка на льоту | телеметрія, кліки, антифрод; найскладніше експлуатувати |

Правила проєктування: (1) затримку диктує **бізнес-потреба споживача**, а не мода — для добової звітності CDC/стрімінг надлишкові; (2) порядок завантаження має шануватися: спершу виміри/довідники, потім факти, що на них посилаються; (3) пакетні вікна прив'язуй до завершення бізнес-циклів (закриття дня/місяця), інакше в DWH поїдуть неповні набори.

## Реплікація й архівування (суміжні механізми)

- **Реплікація** — ведення точних копій набору даних (читається журнал змін, а не таблиці; мінімальний вплив на джерело). Добре: розвантаження читань, географічна близькість. Погано: якщо копії можна змінювати незалежно — конфлікти й втрати оновлень; реплікація ≠ інтеграція (структури мають бути ідентичні).
- **Архівування** — перенесення неактивних даних у дешевше сховище засобами ETL; стеж за сумісністю форматів архіву при зміні технологій, інакше архів стане нечитним.

## Дизайн-патерни конвеєрів (Pandey, Kishore «Building ETL Pipelines with Python»)

- **Basic ETL** — лінійний E→T→L; для простих одноджерельних задач.
- **ETL-P (parallel)** — незалежні гілки джерел обробляються паралельно, збірка в кінці.
- **ETL-VP (validate + parallel)** — до T додається явний крок валідації даних (схеми/діапазони/повнота) з відбраковкою в quarantine-зону.
- **ELT two-phase** — фаза 1: E+L сирого в ціль; фаза 2: T усередині цілі (SQL/dbt). 
- **Повне vs інкрементальне завантаження:** full load — простий, ідемпотентний, але не масштабується; incremental — за водяним знаком (`updated_at`/id) або CDC; **обов'язково ідемпотентний** (патерн «видали вікно → встав заново» — див. blueprint) і з обробкою пізніх даних.
- Інструменти Python-екосистеми за рівнями: бібліотеки трансформацій (pandas; Dask — коли не влазить у пам'ять), легкі ETL-фреймворки (pETL, Bonobo), оркестратори (Luigi, **Airflow** — стандарт де-факто; хмарні: Glue/Step Functions, MWAA). Вибір: скрипт+cron → поки конвеєрів мало; оркестратор — коли з'являються залежності, ретраї, моніторинг.

## Оцінка джерела й призначення (чек-лісти Palmer)

Перед вибором інструмента — по кожному джерелу відповісти (це каркас аналізу вимог, пор. `analyze-task-conditions`):
**Джерело:** з ким працюємо (стейкхолдери, їхні OKR)? як використовуватимуться дані («проблема за проблемою», список так/ні-вимог)? дані **обмежені чи безперервні** (bounded/unbounded — первинніше за «batch vs streaming»)? мінімальна частота оновлення джерела (API оновлюється раз на добу → частіше тягнути безглуздо)? обсяг? формат (JSON API / FTP / реляційна БД; чи є схема)? якість (скільки трансформацій/збагачення)? → як зберігатимемо.
**Призначення:** те саме, але з пріоритетом стейкхолдера (BI/ML/застосунок). Призначень зазвичай значно менше, ніж джерел.

- Частоти: **batch → micro-batch → streaming** — це шкала затримки, не різні світи; Spark Structured Streaming технічно micro-batch із латентністю ~100 мс. Правило **«right-time», а не «real-time»**: кожен крок зменшення затримки збільшує вартість/складність непропорційно; перехід «доба → година» зазвичай закриває потребу.
- Класи рішень інджесту: **декларативні** (Fivetran/Airbyte/Stitch; нативні для платформи: Auto Loader/`CREATE STREAMING TABLE`) — дешево будувати/підтримувати, дорого перемикатися (lock-in), обмежена розширюваність; **імперативні** (свій код: lambda + API + інкремент) — максимально гнучко, але дорого (потрібна інженерна команда); **гібрид** — типова відповідь: конектор для 90 % джерел + свій код для екзотики (пор. dlt-бібліотеку).
- Стрімінгові цеглинки: транспорт (Kafka / Redpanda / Pub/Sub / Kinesis), обробка (Flink / Spark Structured Streaming / Kafka Streams), вікна (fixed / sliding / sessions; розрізняй event time vs processing time).

## Каталог трансформацій і патернів оновлення (Palmer)

Трансформації — вправа на розпізнавання патернів: **enrichment** (доджойнити довідник: код статусу → назва), **joining**, **filtering** (якомога раніше), **structuring** (JSON → таблиця), **conversion** (типи, timestamps), **aggregation** (агрегуй «якнайправіше» — ближче до аналізу, зберігаючи дрібну гранулярність), **anonymization** (хеш PII), **splitting** (email → prefix+domain), **deduplication**.
Оновлення цілі: **overwrite** (просто, ідемпотентно, не масштабується) → **insert/append** (незалежні нові дані) → **upsert/MERGE** (CDC, дедуплікація; див. `lakehouse-delta-glue.md` про `APPLY CHANGES INTO`) → **delete** (**soft** — прапорець deleted, зберігає історію; **hard** — фізично, вимога GDPR). Наскрізні вимоги: **ідемпотентність** (повторний запуск без нових даних = той самий результат) та **інкрементальність**.

## Оркестрація (Palmer, гл. 3)

- **Оркестратор ≠ планувальник**: cron уміє лише розклад; оркестратор — залежності (DAG: вузли=задачі, ребра=залежності), тригери за подіями, ретраї, моніторинг, backfill. **Оркестратор ≠ трансформатор**: «диригент не грає на тромбоні» — залежності мапуй на зовнішні сервіси (dbt, Databricks, Fivetran), оркестратор їх запускає й стежить. dbt/DLT/SQLMesh — самі «SQL-оркестратори» (DAG моделей усередині сховища) → потрібна наскрізна lineage між рівнями, інакше помилку «аналітик чи інженер?» шукатимеш вічно.
- Вибір: Airflow (стандарт; керовані: MWAA/Cloud Composer/ADF) vs нові (Dagster, Prefect) vs платформні (Databricks Workflows — максимальна видимість, ціна — lock-in). Критерії: масштабованість (що буде за 10× DAG-ів?), повторне використання коду, нативні конектори, підтримка/зрілість, спостережуваність.
- Патерни: **backfill вбудований у DAG** (правило: «якщо дані зникнуть завтра — чи відтворю?»; параметризований start_date), **event-driven** запуски (по завершенню інджесту, не за «дивним cron-ом»; менше вартість — рідше запуски порожніх даних), **умовна логіка** (if-else на випадок типових збоїв), **конкурентність/dynamic tasks** (fan-out тисяч дрібних задач), **retry+fallback** (розумні ліміти, без нескінченних перезапусків), **параметризоване виконання** (дата як параметр → backfill = цикл по датах), **декомпозиція** (DAG-и як мікросервіси: автономні, паралельні, збій ізольовано), **lineage до рівня стовпця**.

## Моніторинг, помилки, стосунки (Palmer, гл. 4)

- Метрики даних: **freshness** (лаг до джерела, latency, частота), **volume** (рядки/розмір), **quality** (унікальність, повнота/NULL, валідність діапазонів). Методи: verbose-логування («data lumberjack»), lineage, **anomaly detection** (статистичний прогноз ловить «тихі» баги постачальника), **data diffs** (що зміна коду зробила з даними — Datafold/SQLMesh), **assertions** (Great Expectations; явні бізнес-інваріанти) — пор. `data-quality.md`.
- Обробка збоїв: умовна логіка, ретраї, декомпозиція → **graceful degradation** (ламається шматок, не все; фінзвітність не мусить залежати від product-логів), алерти в останню чергу (реактивні; бережи від **alert fatigue**). Відновлення: staged-дані (time travel) + **backfill**.
- Incident-метрики для KPI: N (кількість інцидентів), TTD (час виявлення), TTR (час відновлення), **data downtime = N × (TTD + TTR)** → вартість.
- Організаційне: **SLA між командами** (що таке «якісні вхідні дані», хто й за скільки лагодить) → **data contracts** (перевірка метаданих/схеми перед запуском — інженерне втілення SLA; популяризовано dbt); постмортеми; документація; юніт-тести коду (≠ assertions на дані); CI/CD; усе — **як код** у git (навіть YAML-конфіги GUI-інструментів).

## Ефективність і масштабування (Palmer, гл. 5)

- Ресурси: кластери (вузли; job-specific vs all-purpose), **spot vs on-demand** (spot — дешево для перезапускних batch; on-demand — критичні), пули прогрітих кластерів, **автоскейлінг** (вертикальний = потужніші вузли, горизонтальний = більше вузлів), **serverless** (BigQuery/Databricks SQL — платиш за використання, не тюниш).
- Технічні прийоми: інкрементальна обробка замість truncate+reload, колонкові формати+партиціювання (див. `lakehouse-delta-glue.md` §5), матеріалізація (таблиця vs view vs MV — свідомо).
- Люди: **демократизація** (self-service патерни: аналітик здає SQL+YAML → джоб генерується сам; інакше беклог команди даних = вузьке місце) і **DevEx** (короткі цикли зворотного зв'язку, низьке когнітивне навантаження — локальна розробка DAG-ів має бути легкою).

## Чек-ліст проєктування конвеєра

- [ ] Визначено споживача й допустиму затримку; режим (batch/CDC/streaming) не «сильніший», ніж треба.
- [ ] ETL чи ELT обрано свідомо (де відбувається T і чому); staging/Bronze передбачено, якщо потрібні аудит і перезапуск.
- [ ] Мэппінг задокументовано (джерело→ціль→правила) до написання коду.
- [ ] Інкремент ідемпотентний; є водяний знак/CDC; пізні й повторні дані не дублюються.
- [ ] Валідація даних — окремий крок (ETL-VP): схема, обов'язковість, діапазони; браковані рядки — в карантин, не «тихо викинуті».
- [ ] Порядок: довідники/виміри → факти; завантаження прив'язане до бізнес-циклу.
- [ ] Логування, алерти, ретраї, backfill-процедура — спроєктовані, не «потім».
