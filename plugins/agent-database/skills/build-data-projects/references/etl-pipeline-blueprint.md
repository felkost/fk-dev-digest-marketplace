# Канонічний ETL/аналітичний конвеєр (blueprint)

Виведено з розбору 4 реальних репозиторіїв (див. `example-projects.md`) — усі влаштовані за одним патерном. Це еталон, який можна пояснювати студенту й адаптувати.

## Схема потоку
```
REST API  ──requests──▶  Python (pandas: очистка/трансформація)
                              │
                              ▼  psycopg2 / SQLAlchemy
                        PostgreSQL (таблиця-вітрина, індекси)
                              │
                              ▼
                 BI-дашборд (Metabase / Yandex DataLens / Google Sheets)

Оркестрація: cron (щодня, напр. 07:00) · Логування + ротація логів · Алерти (email/Telegram)
```

## Модульна структура (як у прикладах)
- `api_client.py` — витяг даних з API (пагінація, ретраї, ліміти).
- `data_processor.py` — трансформація/нормалізація (`pandas`).
- `db.py` — з'єднання (singleton/пул), вставка, upsert; секрети з env.
- `main.py` — оркестрація кроків; `get_historical_data.py` (backfill) + `get_daily_data.py` (щоденне).
- `config.py` / `.env` — конфіг і секрети (не в git).
- Логування з ротацією; сповіщення про успіх/помилку.

## Дві траєкторії завантаження
- **Історичний backfill** — одноразово залити весь період (напр. із 2022).
- **Щоденний інкремент** — за розкладом вантажити лише новий день/період.

## Ідемпотентне інкрементне оновлення (патерн «вікно»)
Щоб перезапуск за період не дублював дані й не давав «плаваючих» цифр — видали вікно й встав його заново, параметризувавши межі:
```sql
-- Псевдо-процедура: перерахунок одного вікна дат
DELETE FROM sales_mart WHERE sale_date >= :start AND sale_date < :end;
INSERT INTO sales_mart (channel, leads, sales, conv_rate)
SELECT channel,
       COUNT(*)                                                   AS leads,
       SUM(CASE WHEN status = 'Купил' THEN 1 ELSE 0 END)          AS sales,
       SUM(CASE WHEN status = 'Купил' THEN 1 ELSE 0 END)*1.0/COUNT(*) AS conv_rate
FROM marketing_leads
WHERE date >= :start AND date < :end
GROUP BY channel;
```
Щоб перерахувати конкретний місяць — міняєш дві межі й запускаєш; інші періоди не чіпаються. Використовуй **напівінтервал** `>= :start AND < :end` (див. gotcha про `BETWEEN` + timestamp).

**Портативність меж вікна (обери під діалект, не змішуй):**
- PostgreSQL/PL/pgSQL: `date_trunc('month', now()) - interval '2 month'` … `date_trunc('month', now()) + interval '1 month'`.
- T-SQL: змінні `@start/@end` через `DATEADD`/`EOMONTH`/`DATEFROMPARTS`.
- MySQL: `DATE_SUB`, `LAST_DAY`, `CURDATE()`.
- (Приклад-першоджерело змішував `@var` T-SQL з `sysdate/add_months/last_day/trunc` Oracle — так робити не можна.)

### Наскільки широке вікно перераховувати (формула lookback)

Патерн вище розвʼязує «як» перерахувати; лишається «скільки». Дописати рядок за вчора й піти — найпоширеніша помилка інкрементних вітрин, бо **минуле змінюється** щонайменше двома шляхами: дані приходять із запізненням і сутність живе далі (замовлення, створене вчора, сьогодні скасували).

```
lookback = макс. затримка джерела
         + макс. тривалість життєвого циклу сутності
         + ширина бізнес-вікна метрики
```

Третій доданок забувають — а він і є найдорожчим. Для **ковзної N-денної** метрики одна змінена подія входить у **N** різних вихідних рядків, тобто радіус ураження в N разів ширший за денну вітрину **при тому самому потоці змін**:

| Вітрина | Бізнес-вікно | Скільки рядків псує одна зміна | lookback |
|---|---|---|---|
| денна виручка | 1 день | 1 | ≈ затримка + цикл + 1 день |
| частка скасувань за 7 днів | 7 днів | **7** | ≈ затримка + цикл + 7 днів |

Механіка «які саме дні перераховувати»: узяти множину зачеплених бізнес-днів зі змінених рядків (Delta **Change Data Feed**, `updated_at`-водяний знак або журнал CDC), розширити її на ширину бізнес-вікна й прогнати патерн «видали вікно → встав заново» **лише по цих днях**. Це і є професійна відповідь — не «інкремент» і не «повний перерахунок», а **прицільний перерахунок зачеплених партицій**.

**Дві суміжні пастки, які випливають із того самого кореня:**

- **Метрики-відношення не адитивні.** Середній чек, конверсію, частку скасувань **не можна** зберігати готовим числом і потім усереднювати по тижнях/містах — вийде «середнє від середніх». Зберігай **чисельник і знаменник** (`sales` + `leads`, `revenue` + `orders_cnt`), ділення роби на читанні або мірою в BI. Тільки так вітрина коректно згортається на вищі рівні. (Саме тому в прикладі вище поруч із `conv_rate` лежать і `leads`, і `sales`.)
- **«Незрілі» останні періоди.** Якщо метрика привʼязана до дати створення, останній день-два ще не дозріли: частина сутностей не дійшла до термінального стану, тому свіжа частка скасувань **систематично занижена**, а конверсія — завищена. Ліки — прапорець у вітрині (`is_final_day`), який BI приглушує; без нього аналітики щоранку бачать «покращення», якого немає.

## Аналітичні рецепти вітрин

**Когортний аналіз / retention одним SQL (PostgreSQL):**
```sql
WITH first_seen AS (
  SELECT user_id, date_trunc('week', min(event_at)) AS cohort_week
  FROM events GROUP BY user_id
),
activity AS (
  SELECT e.user_id, fs.cohort_week,
         (date_trunc('week', e.event_at) - fs.cohort_week) AS week_offset
  FROM events e JOIN first_seen fs USING (user_id)
)
SELECT cohort_week,
       COUNT(DISTINCT user_id) FILTER (WHERE week_offset = interval '0 week') AS w0,
       COUNT(DISTINCT user_id) FILTER (WHERE week_offset = interval '1 week') AS w1,
       COUNT(DISTINCT user_id) FILTER (WHERE week_offset = interval '2 week') AS w2
FROM activity GROUP BY cohort_week ORDER BY cohort_week;
```
Кроки: `date_trunc` знаходить тиждень першої події (когорта) → зсув події в тижнях → `GROUP BY + COUNT(...) FILTER (...)` → трикутна матриця ретеншну. Перед показом зафіксуй два визначення: що вважається «поверненням» і від чого рахується когорта.

## Продуктивність завантаження
- **Масову вставку роби через `COPY` (PostgreSQL), а не цикл `INSERT`** — мільйон рядків: хвилини проти секунд. У psycopg 3 — `cursor.copy(...)`; для CSV з аналітики/міграцій — найшвидший шлях. У ClickHouse — `client.insert(...)` батчами; у MongoDB — `insert_many`.
- Індекси створюй **після** масового backfill, а не до (кожен індекс уповільнює вставку).

## Надійність
- **Логування** кожного кроку + ротація (напр. зберігати 21 день).
- **Алерти** на помилку/успіх (email SMTP або Telegram-бот).
- **Індекси** під запити дашборду (складені за `client_id`, `product_id`, `purchase_datetime` тощо).
- **Оркестрація:** cron для простих випадків; для складних DAG — Airflow; трансформації у сховищі — dbt.

## Ризики / обмеження патерну
- cron не має ретраїв/залежностей між задачами — на складніших конвеєрах бери Airflow/Prefect.
- Один невеликий сервер (1 CPU/2GB у прикладах) — вузьке місце при зростанні; розглянь керовані сервіси (див. скіл `db-connectivity-cloud`).
- API можуть змінюватися/лімітувати — закладай ретраї, backoff, версіювання схеми.

## Medallion-архітектура шарів (Bronze → Silver → Gold)

Загальноприйнята назва для пошарової організації даних у сховищі/лейкхаусі (термінологія Databricks, але патерн універсальний і відтворюється навіть у одному PostgreSQL схемами `raw`/`staging`/`mart`):
- **Bronze** — сирі дані як прийшли (JSON/CSV з API, лог подій) + історія; нічого не губимо, можна перечитати.
- **Silver** — очищені/типізовані/збагачені таблиці (дедуплікація, приведення типів, довідники).
- **Gold** — бізнес-агрегати під конкретні дашборди/звіти (вітрини).
Правило: трансформації течуть лише «вниз за течією»; будь-який шар можна перебудувати з попереднього. Наш blueprint вище — та сама ідея: API → сира відповідь → pandas-очистка (Silver) → вітрина в PostgreSQL (Gold).

## Імперативний vs декларативний ETL

- **Імперативний:** ти описуєш *як* — кроки, стан, обробку помилок (Python-скрипти, збережені процедури, курсори, cron). Повний контроль, але код багатослівний і крихкий; інкрементальність (що вже оброблено, «нові файли», ретраї) пишеш вручну.
- **Декларативний:** ти описуєш *що* (SQL-визначення результату), а платформа сама планує виконання, інкрементальний refresh і моніторинг. Приклади: **матеріалізовані viewи** (PostgreSQL: `CREATE MATERIALIZED VIEW` + `REFRESH`, у Databricks/BigQuery — з автоматичним інкрементальним оновленням), **dbt** (трансформації = набір SELECT-моделей із залежностями), streaming tables (Databricks Lakeflow: `CREATE STREAMING TABLE … FROM STREAM read_files(...)` — інжест нових файлів/Kafka без ручного трекінгу оброблених).
- Навчальний місток: класичний приклад різниці — «порахувати суму» курсором у циклі (T-SQL `WHILE @@FETCH_STATUS`) проти `SELECT SUM(...)`. Декларативний підхід — це той самий стрибок, але на рівні цілого конвеєра.
- Тренд індустрії: SQL стає мовою всього ETL (аналітик доводить конвеєр до продуктиву без переписування на PySpark), Python лишається для ML/складної логіки. Для курсу: спершу навчи «ручний» blueprint (розуміння механіки), потім покажи dbt/materialized views як «те саме, але декларативно».

## CDC і повільно змінювані виміри (SCD)

- **CDC (Change Data Capture)** — перенесення *змін* (insert/update/delete) із джерела в сховище замість повного перевантаження. Джерела: журнал транзакцій БД (Debezium, AWS DMS), тригери, або колонка `updated_at` + інкрементальне вікно. Ускладнення, які треба проговорювати: події поза порядком (потрібна колонка-секвенсор), часткові оновлення, еволюція схеми.
- **SCD Type 1** — оновлення на місці, історії немає (перезаписали адресу клієнта).
- **SCD Type 2** — історія зберігається версіями рядків: нова версія при кожній зміні + колонки `valid_from`/`valid_to` (або `is_current`); дозволяє звіт «як виглядав клієнт на дату X». Класична ручна реалізація — MERGE із «закриттям» старої версії та вставкою нової; сучасні платформи мають готові механізми (Databricks `AUTO CDC … STORED AS SCD TYPE 2`, dbt snapshots).
- Для викладання: SCD2 — найкращий приклад, чому «UPDATE у вітрині» — антипатерн для звітності, що дивиться в минуле.

## Стек дата-інженерії на AWS (за «Ultimate AWS Data Engineering» / «AWS in Action»)

Типовий розклад ролей, який варто знати студенту (аналог для GCP/Azure — див. `db-connectivity-cloud`):
- **S3** — озеро даних (сирі файли, Bronze); **Glue** — керований Spark-ETL + каталог метаданих; **Athena** — SQL по файлах у S3 без сервера;
- **Kinesis** — потокова інжестія подій (аналог Kafka); **Lambda** — легкі трансформації за подією;
- **Redshift** — колонкове MPP-сховище (Gold-шар, важка аналітика); **RDS/Aurora** — операційні БД-джерела; **DynamoDB** — NoSQL-джерело;
- Оркестрація — Step Functions / MWAA (керований Airflow); візуалізація — QuickSight (або Tableau/Power BI поверх Redshift/Athena).
- Наскрізні теми курсу, які книга кладе в основу: реплікація (син/асинхронна, консистентність), партиціювання vs шардинг, відмовостійкість, балансування навантаження — усе це та сама теорія, що в `db-families-cap.md` і `sql-fundamentals.md` (партиціювання), лише в хмарному виконанні.
