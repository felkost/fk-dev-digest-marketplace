# SQL vs Python: коли що використовувати

Часте питання джунів. Короткий орієнтир: **SQL — для роботи з даними у сховищі; Python — для всього, що SQL робить незручно або не вміє.** Далі — деталі.

## Коли SQL
Усе, що формулюється як **трансформація таблиці**: агрегації, фільтрація, джойни, віконні функції, побудова вітрин/звітів. SQL зробить це швидше й читабельніше, бо обчислення виконуються **поруч із даними**, без перевезення в пам'ять застосунку.
```sql
SELECT user_id,
       COUNT(order_id)  AS orders_count,
       SUM(revenue)     AS total_revenue,
       MIN(created_at)  AS first_order_date
FROM orders
GROUP BY user_id;
```
Те саме на `pandas` буде довше й повільніше на великих обсягах (тягнеш дані з БД у пам'ять, потім рахуєш).

## Коли Python
Коли дані **вже дістали** і треба щось, чого SQL не вміє зручно: статистика й моделювання (ML), візуалізація, автоматизація, робота з API та нетабличними форматами, або **ітеративна логіка**, де результат рядка залежить від попереднього.
```python
import pandas as pd
from scipy import stats
df = pd.read_csv("orders.csv")
a = df[df["group"] == "control"]["revenue"]
b = df[df["group"] == "test"]["revenue"]
t_stat, p_value = stats.ttest_ind(a, b)
print(f"p-value: {p_value:.4f}")
```
A/B-тест технічно можливий і в SQL, але ти витратиш час на те, для чого вже є `scipy`.

## Коли обидва (типовий реальний випадок)
Більшість задач: **SQL дістає й агрегує** дані зі сховища → **Python доробляє** те, що SQL не вміє (статистика/ML/візуалізація/API). Межа проходить по «що зручніше рахувати де».

## Типові помилки джунів
- **Тягнути все в `pandas`**, бо звичніше: отримують код, який робить те саме, що один SQL-запит у три рядки, і падає на пам'яті на обсягах у кілька мільйонів рядків.
- **Навпаки — крутити статистику/ML суто в SQL** там, де потрібен Python.

Уміння обрати інструмент, коли умова цього прямо не диктує, — одна з ознак, що відрізняє мідла від джуна. Це варто відпрацьовувати на задачах із відкритим вибором інструмента.
