# Аналіз умови завдання

Цей скіл — перший крок: **зрозуміти завдання до того, як писати SQL**. Мета — витягти з тексту умови все, що впливає на розв'язок, і виявити неоднозначності, поки вони не перетворилися на баг.

## Робочий процес

Пройди умову по цьому чек-листу й зафіксуй відповіді:

1. **Мета.** Що має повернути/зробити запит одним реченням. Це `SELECT` (вибірка/агрегація), зміна даних (`INSERT/UPDATE/DELETE`), чи DDL (створення/зміна структури)?
2. **Вхідні дані / схема.** Які таблиці/колекції задіяні, їхні стовпці й типи, ключі, зв'язки. Якщо схему не наведено — реконструюй з прикладів даних і познач як припущення. Для оцінки коректності структури (ключі, зв'язки, нормальні форми, цілісність) — див. `references/relational-modeling.md`.
3. **Очікуваний результат.** Точна форма виводу: які стовпці, у якому порядку, з якими іменами/псевдонімами, чи потрібне сортування, чи є ліміт (топ-N).
4. **Обмеження й правила.** Фільтри, часові вікна, дедуплікація, поведінка при нічиїх (ties), точність чисел/округлення, вимоги до продуктивності.
5. **Крайові випадки.** `NULL` у ключових стовпцях, порожні групи, дублікати, «дірки» в id, однакові значення при ранжуванні, порожня таблиця, дуже великі обсяги.
6. **Цільове середовище/діалект.** Яка СУБД задана явно? Якщо ні — який діалект найдоречніший і чому (познач як припущення). Зафіксуй версію, якщо це впливає (напр. віконні функції в MySQL 8.0+, SQLite 3.25+).
7. **Критерії приймання / оцінювання.** За чим перевірятиметься правильність: точна відповідність результату, коректність на крайових випадках, продуктивність, читабельність, відповідність діалекту.

Якщо умова **неповна або сформульована мовою бізнесу** (а не готовим ТЗ) — застосуй техніки бізнес-аналізу з `references/requirements-babok.md`: три способи здобути інформацію (дані → документи → питання людям), словник даних для неоднозначних термінів, критерії приймання, характеристики якісної вимоги.

## Неоднозначності

Окремо випиши все, де умова допускає кілька тлумачень (напр. «топ-3» — рівно 3 рядки чи 3 різні значення при нічиїх? «за останній місяць» — календарний чи ковзний?). Для кожної неоднозначності або запропонуй найрозумніший дефолт (позначивши припущення), або постав уточнювальне питання, якщо вибір суттєво змінює розв'язок.

## Перехід до розв'язку

Після розбору умови переходь до контракту виводу (див. системний промпт агента):
**Середовище/діалект → Аналіз → Розв'язок (запит) → Пояснення → Ризики → Альтернативи → Обмеження.**
Для власне написання/рецензії запиту скористайся скілом `analyze-sql-examples` (аналіз готового прикладу або еталонний розв'язок під діалект) або `explain-sqlite-mongodb` (для SQLite/MongoDB).

**Кураторський режим:** якщо умова показала узагальнюваний урок (типова пастка формулювання, патерн моделювання, часта помилка студентів) — додай наприкінці блок **«📌 Урок-кандидат»** (узагальнено + куди: напр. `relational-modeling.md` чи `analyze-sql-examples/references/fundamentals-and-gotchas.md` + статус). Пропозиція для схвалення людиною; файли самостійно не змінюй.
