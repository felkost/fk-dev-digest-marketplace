# Аудит якості даних для EDA

Виявляти дефекти й ризики до feature engineering. Спочатку вимірювати проблему, потім вирішувати, чи виправляти, позначати, виключати або залишати значення.

## Допоміжні скрипти

Leakage-safe утиліти в [scripts/](scripts/) (навчати лише на train/fold):

- `profile_schema.py` — семантична схема, mixed-type/sentinel-детекція, `n_inf` окремо від `n_missing`, mojibake, константні/домінантні значення-заглушки (крок 2);

- `consistency.py` — неузгоджені написання (`near_duplicate_categories`), конфлікт атрибутів під одним ключем (`key_attribute_conflicts`), дублікати + докази для вибору стратегії злиття (`duplicate_report`), пастки дат/часових поясів (`datetime_consistency`), логічні обмеження для похідних колонок (`logical_constraints`), `proxy_label_diagnostics` (мітка від недосконалого детектора: PPV, роздування базової частоти, частка хибних позитивів у мітках) (кроки 3, 7);
- `missingness.py` — патерни пропусків за колонкою/часом/target/split, `lowrank_impute` (ітеративна PCA-імпутація) і `imputation_quality_probe` (маскування відомих значень — валідація будь-якого імпутера) (крок 4);
- `text_profile.py` — для тексту: довжини, NaN vs порожній рядок, дублікати нормалізованого тексту, частоти токенів, OOV-покриття pretrained-словника, all-OOV документи, перетин словників між сплітами (див. [references/text-nlp.md](references/text-nlp.md));

- `sampling_design.py` — скільки насправді є **незалежних** спостережень: `intraclass_correlation` (ICC + design effect для середніх і часток, з F-гардом проти хибних тривог), `weight_design_effect` (Kish, нерівні ваги відбору), `effective_n_for_association` (deff для зв'язку двох колонок = добуток ICC, **не** deff середнього), `overlapping_label_deff` / `label_concurrency` (мітка на forward-вікні: uniqueness = 1/h як **вага**, deff = `1+(h−1)(2h−1)/(3h)` як **інфляція дисперсії** — різні величини), `design_effect_report` (крок 1, наслідки — кроки 5, 7, 8; див. [references/sampling-design.md](references/sampling-design.md));
- `outliers.py` — domain bounds + IQR/MAD/robust-z + опційно IsolationForest/LOF (крок 6);
- `split_designer.py` — побудова stratified/group/chronological/spatial split + `purge_and_embargo` (прибрати train-рядки, чия мітка на forward-вікні дістає до test-блоку) (крок 8);
- `leakage_checks.py` — дублікати між split, group-disjointness, availability-vs-cutoff, `composite_target_probe` (чи не є ціль лінійною комбінацією власних ознак) (кроки 7–8);
- `image_profile.py` — для зображень: `folder_census` (зображень на клас на спліт + відсутні папки класів), цілісність, властивості, per-channel норм-константи (train-only; для pretrained-енкодера — константи його передтренування, не train), near-duplicate (див. [references/computer-vision.md](references/computer-vision.md)).

## 1. Зафіксувати контекст

Спочатку прочитати застосовні правила користувача: `applicable("audit-eda-data-quality/<метод>")` з `insights.py` скіла `$plan-eda-dataset` — доменні рішення, зафіксовані раніше (напр. «пропуск тиску = датчик вимкнено, не імпутувати»). Застосовне правило переказати перед дією. Нове виправлення від користувача записати через `append_insight(rule, scope, why=...)`.

Встановити одиницю спостереження, ключі, джерела, часову семантику, target, горизонт прогнозу, групи залежних записів і передбачуваний split. Якщо контекст неповний, не підміняти його припущеннями без маркування.

Зберегти immutable raw layer і контрольні характеристики: row count, column count, хеш/версію, джерела та діапазон часу.

### Скільки з цих рядків незалежні

Два питання, які треба поставити явно, а не залишити припущенням: **у що вкладений рядок** (користувач, пристрій, школа, домогосподарство, сесія, майданчик) і **чи всі рядки мали однакову ймовірність потрапити у вибірку** (якщо ні — де ваги). Якщо обидві відповіді «нічого» і «так», design effect = 1 — але це треба записати як перевірений факт.

Інакше `sampling_design.py` дає `n_eff = n / deff`. Ціна ігнорування вимірювана: при 60 кластерах × 25 рядків і ICC = 0.30 номінальний 95% CI покриває істину у **48.8%** випадків, а 1500 рядків несуть інформацію ~183 незалежних. Це успадковують усі подальші числа — CI, bootstrap, p-value, PSI-гард на ≥50 рядків у бакеті (він рахує рядки, не незалежні спостереження).

Правило, що обмежує паніку: кластеризація роздуває контрасти **між** кластерами і не чіпає контрасти **всередині** кластера (виміряно: 0.039/0.038/0.049 проти 0.038/0.184/0.359 при ICC 0/0.10/0.30). Для зв'язку двох колонок працює добуток ICC — якщо кластеризована лише одна колонка, корекція не потрібна взагалі.

Group-aware split цього **не** лікує: він проти leakage, а не проти інференсу (виміряно: 0.164 на випадковій половині проти 0.247 на кластерній). Потрібні обидва контролі.

Окремий випадок, який створюємо ми самі: **мітка на forward-вікні** (дохідність за наступні 20 днів, відтік за 90 днів, повторна госпіталізація за 30 днів, будь-яка rolling-агрегація). Тоді рядки `t` і `t+1` ділять `h−1` з `h` складників. Uniqueness = `1/h` — правильна **вага** спостереження; інфляція дисперсії — `1+(h−1)(2h−1)/(3h)` (→ `2h/3`), і це **різні числа**: підстановка `h` замість формули занижує alpha до 0.011 при номінальних 0.05 (виміряно), тобто тихо викидає справжні ознаки. Без корекції при h=50 три чверті **незалежних** пар колонок читаються як значущі.

Для non-probability вибірок (convenience, opt-in панелі, snowball) ваги незастосовні — жодна схема зважування не відновлює те, що ніколи не було відібране. Чесний вихід — опис досягнутої вибірки + явне твердження, про яку популяцію вона може і не може говорити.

## 2. Побудувати семантичну схему

Для кожної колонки визначити не лише storage dtype, а роль:

- continuous, count, ordinal, nominal, binary;
- datetime, duration, cyclic time;
- identifier, group key, source key;
- sampling/survey weight або інша метадата вибірки (наприклад, census `fnlwgt`) — використовувати для зважування/діагностики, не як predictor;
- free text, list, set або multi-label categorical;
- image/audio/path/embedding/array;
- target, post-outcome field або potential proxy;
- sensitive/protected або operationally unavailable feature.

Перевіряти mixed values, числа-коди категорій, одиниці вимірювання, locale, sentinel values, whitespace/case variants, невідомі категорії й неоднозначні спискові рядки. Прочитати [references/mixed-categorical.md](references/mixed-categorical.md) для multi-label і змішаних категорій.

Окремо перевірити те, що не потрапляє в `missing_rate`:

- `inf` — `pd.isna()` його не рахує, тож колонка проходить як «0% пропусків», а потім mean/std/квантилі стають NaN. Дивитися колонку `n_inf`. Типове джерело — ділення на нуль у похідній ознаці, тобто лагодити треба формулу, а не рядки;
- значення-заглушки замість пропуску (`email@company.com`, `1900-01-01`, `0` від upstream-системи): `constant_value` і `dominant_value_share`. Дата-сентинел усередині правдоподібного діапазону ніколи не спрацює на перевірці меж — концентрація значення є єдиним сигналом;
- `encoding_artifacts` (mojibake `cafÃ©`, `�`) — перекодувати з джерела правильним кодеком, а не підміняти символи: на цьому етапі інформація ще відновлювана, далі — ні.

## 3. Перевірити цілісність

- Перевірити primary key, дублікати рядків, near-duplicates та конфліктні labels.
- Перевірити дублікати між train/validation/test, між entities, часом, файлами або похідними зображеннями.
- Перевірити joins: many-to-many expansion, втрачені ключі, orphan rows, зміщення row count.
- Перевірити діапазони, одиниці, монотонність часу, impossible combinations і взаємні обмеження колонок.
- Для файлів перевірити читаність, формат, розмір, channels, aspect ratio, duration, corrupted/blank/near-identical content.

Прочитати [references/consistency-validity.md](references/consistency-validity.md) для дефектів, де кожне окреме значення коректне:

- **Неузгоджені написання** одного елемента (`Apples`/`apples`/`appels`) роздувають cardinality й розщеплюють підрахунки. `near_duplicate_categories`: нормалізація регістру/пробілів детермінована, а fuzzy-збіг — це **черга на перегляд, не автозаміна** (`apples`/`appels` = 0.833, але `xl`/`xxl` = 0.800 — поріг схожості їх не розділяє, розділяє довжина). Мапу нормалізації навчати на train.
- **Конфлікт атрибутів під одним ключем** (`key_attribute_conflicts`): розрізняти помилку вводу, slowly changing dimension (справжній ключ — `(id, valid_from)`; згортання в один рядок вносить майбутнє в минулі рядки) і перевикористаний ідентифікатор.
- **Дублікати**: `duplicate_report` показує, які колонки реально конфліктують, бо це визначає допустиму стратегію — залишити один рядок можна лише за нуля конфліктів; інакше потрібне правило «найсвіжіший» або злиття в сутність. Дублікати не є дефектом автоматично: повторні вимірювання однієї сутності лікуються **group-aware split**, а не видаленням.
- **Дати/типи** (`datetime_consistency`): рядки-псевдодати, epoch-цілі, неоднозначність день/місяць (усі дні ≤ 12 — формат треба знати, а не вгадувати), змішані tz-aware/naive. Для крос-регіональних даних нормалізувати в UTC **і зберегти локальний офсет окремою ознакою** — локальна година доби поведінково не те саме, що UTC-година.

Для зображень (класифікація, детекція/сегментація, медичні/супутникові) читати [references/computer-vision.md](references/computer-vision.md): цілісність, розподіли властивостей, аналіз міток/боксів/масок, shortcut/confound, image-специфічний leakage (near-duplicate + group за пацієнтом/сценою/відео) і per-channel норм-константи лише на train.

Для тексту читати [references/text-nlp.md](references/text-nlp.md): NaN і порожній рядок — різні дефекти (не конвертувати мовчки), дублікати нормалізованого тексту, OOV-покриття словника ембедингів по сплітах/класах, all-OOV документи (mean-pooling дає NaN — політика явна, не «магічна константа»), узгодженість регістру токенізатора і словника.

Для геоданих (точки, полігони, растрові тайли, траєкторії) читати [references/geospatial.md](references/geospatial.md): узгодженість CRS і порядку осей (відстані/площі лише в проєктованій системі координат), валідність геометрій (`make_valid` явно, з логом), координатні сентинели (Null Island, центроїди геокодера), просторова автокореляція (Moran's I, семіваріограма) як механізм leakage → block/buffered spatial split, H3/geohash-агрегати лише на train.

**Рядки, яких немає взагалі.** Крок 4 аудитує пропущені **комірки**; окремий і менш помітний дефект — пропущені **рядки**: вибіркова рамка мовчки виключає частину популяції, яку модель зустріне в бою. Жодна колонка не хибна, жодне обмеження не порушене, і профілювання таблиці цього не покаже — доказ лежить поза файлом. Канонічний випадок — **виживання**: таблиця «наші клієнти», зібрана сьогодні з поточної бази, містить лише тих, хто не пішов, а саме відтік мала б пояснювати модель. Обсяг не рятує: п'ять років історії — це п'ять років тих, хто вижив; великі дані лікують дисперсію, але не зміщення відбору. Перевірки: реконструювати рамку **на момент рішення** (point-in-time snapshot сутності, що тоді існувала, з міткою, доданою пізніше), а не на момент вивантаження; спитати, що мало статися, щоб рядок сюди потрапив (сутність — вижити, сесія — завершитись, товар — бути **показаним**, щоб отримати клік); звірити з зовнішнім лічильником, де він є. Найгостріший варіант — **логи поведінки**: рекомендер записує лише те, що сам показав, тобто покази MNAR із **попередньою моделлю** як механізмом пропуску, і «користувачам не сподобалось» плутається з «користувачам не показували». Зафіксувати правило побудови рамки в dataset contract поруч з одиницею спостереження, а «кого в цій таблиці немає і чому» — в dataset card.

Не видаляти підозрілі записи автоматично. Додати issue flag і provenance.

## 4. Проаналізувати пропуски

Виміряти missing rate загалом, за колонкою, рядком, source, time, target, class і subgroup. Відрізняти справжній NA від `0`, порожнього рядка, `unknown`, sentinel code або «не застосовується».

Дослідити патерни спільних пропусків і зв'язок missingness з target та доступними ознаками. MCAR/MAR/MNAR подавати як гіпотези, а не як автоматично доведені класи.

**Спершу спитати, чи пропуск не є частиною дизайну.** Пропуск не завжди дефект: існують *planned missing designs*, де його створюють навмисно — multiform (кожен респондент отримує підмножину блоків), accelerated longitudinal (вік як вісь часу, кожна когорта покриває свій відрізок), two-method measurement (дешевий вимір усім, дорогий — випадковій підвибірці, яка калібрує дешевий). Там пропуск **MCAR за побудовою**, бо його призначив рандомізатор, і «полагодити» його видаленням рядків/колонок означає викинути дизайн. Діагностична ознака — блокова, регулярна, надто акуратна структура пропусків (рівні частки, чисті прямокутники в missingness-матриці), а не розсіяні поодинокі дірки. Спитати замовника, чи є кодбук дизайну; якщо так — зафіксувати механізм у dataset contract і не трактувати ці клітини як дефект. Практичний наслідок: висока частка пропусків сама по собі не є вироком — вирок дає **механізм**, а не відсоток.

**«Ми використали FIML/імпутацію» — не є описом даних.** Метод працює під припущенням (MAR), а не безумовно, і рядок у звіті його не підтверджує. Мінімум, що має бути задокументовано: частка пропусків по кожній колонці й патернах, які змінні пов'язані з пропуском (саме вони й роблять MAR правдоподібним), які auxiliary-змінні додано заради цього, і що робили з рядками, де бракує target. Пропуск як MNAR не лікується жодним із цих методів. І про масштаб: 5% і 60% пропусків — різні задачі, навіть якщо назва методу однакова.

Вибирати delete/impute/indicator/domain reconstruction лише після оцінювання механізму, обсягу й наслідків. Глобальна медіана/мода — базовий варіант; групова імпутація (статистика по домен-релевантних групах, як-от медіана ціни за класом), low-rank (`lowrank_impute`, використовує кореляції між колонками) і модельна імпутація точніші, але дорожчі й ризикованіші. Якість імпутера не постулювати, а міряти: `imputation_quality_probe` маскує частину відомих значень і оцінює відновлення (validation-підхід до імпутації; ним же обирати ранг M). Missing-індикатор часто корисний сам по собі. Fit imputation (включно з груповими статистиками) виконувати тільки на train/fold і зберігати як артефакт для inference.

## 5. Обчислити властивості розподілів

**Kurtosis міряє хвости, а не «гостроту піка».** Поширене пояснення («що вищий пік, то більший ексцес») відправляє агента дивитися не туди. Ексцес — стандартизований четвертий момент, а `z⁴` арифметично визначається тим, що далеко від центру: заміряно на нормальному, `E[z⁴] = 2.9914`, з якого область `|z| < 1` дає **3.75%** статистики, вміщаючи при цьому **68.26%** рядків (`|z| < 0.5` → 0.15% статистики при 38.29% рядків). Пік для цього числа практично невидимий. Хибне уявлення живуче, бо на звичних розподілах пік і ексцес справді ростуть разом (Spearman +0.94 за точними значеннями для uniform/normal/Laplace/t(5)/t(8)/t(20)) — але порядок **інвертується**: при дисперсії 1 щільність у моді Лапласа **0.7071** проти **0.4901** у t(5), тоді як ексцес — **+3.000** проти **+6.000** (точні аналітичні значення, не симуляція). Читати високий ексцес як «рідкісні екстремальні значення відносно нормального» — тобто твердження про ризик викидів, валідність інтервалів і оцінюваність середнього, — а не як «гострий центр». Окремо: **вибірковий ексцес поганий саме там, де потрібен** — 400 000 значень із t(5) дали 5.337 замість справжніх 6.000, бо четвертий момент t(5) ледве існує; тому велике значення — привід побудувати хвіст (`tail_index`, `family_router`), а не порівнювати число між колонками.

Для числових ознак подавати `n`, missing, unique, min/max, median, mean, standard deviation, MAD/IQR, quantiles, skewness і kurtosis з урахуванням sample size. Центр і розсіяння звітувати узгодженою парою (mean↔std, median↔MAD, midrange↔піврозмах), не змішувати. Використовувати histogram/density, box/violin, ECDF, Q-Q або log-scale за потреби.

Статистику, на яку спирається рішення, супроводжувати bootstrap-CI (`bootstrap_ci`), особливо на малих вибірках; дві групи порівнювати через CI різниці (`bootstrap_diff_ci`), а не через перетин двох окремих інтервалів; коли теоретичний null сумнівний (мале n, важкі хвости) — permutation-тест (`permutation_test_groups`). Мультимодальність заявляти лише якщо вона стійка до зміни числа бінів (`binwidth_sensitivity`): форма гістограми — функція кількості бінів, надійного теоретичного правила вибору немає.

**«n > 30 вистачає для ЦГТ» — це твердження про симетрію, а не про `n`.** Керує **асиметрія**: перекос вибіркового середнього спадає як `skew(x)/√n` (перевірено на логнормальному з асиметрією 6.185: заміряно 1.2551/0.6355/0.3092 при n=25/100/400 проти `6.185/√n` = 1.2370/0.6185/0.3092). Тому фіксований поріг працювати не може. Заміряно покриття 95% t-інтервалу для **середнього** (20 000 повторів, номінал 0.950): симетричні розподіли дають 0.945–0.952 уже при n=10, а при n=50 експоненційний дає 0.934, логнормальний(0,1) — **0.900**, Парето(α=3) — 0.882, логнормальний(0,2) — **0.695** (і навіть при n=2000 лише 0.874). Виручка на користувача, тривалість сесії, розмір заявки живуть саме в цій половині таблиці.

**Промах односторонній — і це головне.** При n=50 (номінал 0.025 з кожного боку): інтервал цілком **нижче** істини у 0.0587 (експоненційний), **0.0943** (логнормальний(0,1)), 0.1150 (Парето), **0.3054** (логнормальний(0,2)) випадків, а вище — 0.0099/0.0028/0.0023/0.0001. Правоасиметрична вибірка зазвичай не містить рідкісних великих значень, які й тримають середнє, тож і `x̄`, і `s` виходять заниженими, а весь інтервал сідає **під** істину — на логнормальному(0,1) у ~34 рази частіше, ніж над нею. Будь-яке «середнє на користувача» з малої вибірки занижене конструктивно. Правило `n > 25·skew²` знати варто, але воно не замінює вимірювання: точно влучило на експоненційному (100 проти 100), близько на логнормальному(0,1) (956 проти 750), але в 6× надто консервативне на Парето і у 8× на Пуассоні(0.1). Плюс у дискретного лічильника покриття **немонотонне за n** (0.623→0.949→0.870→0.932), тож «n, з якого стало добре» там беззмістовне — читати криву, не клітину.

**Bootstrap тут не рятує — і це виправляє попередню рекомендацію самого проєкту.** Заміряно (3000 повторів, B=2000) t-інтервал проти percentile та BCa: на **малих** вибірках percentile **гірший** за t, зокрема на цілком нормальній популяції (0.890 проти 0.951 при n=10); BCa, створений саме для корекції зсуву й асиметрії, відіграє близько відсоткового пункту (0.899→0.911 на логнормальному при n=50) і теж не доходить до номіналу. Механізм той самий: bootstrap перевибирає **наявні** рядки, а мала правоасиметрична вибірка просто не містить великих значень. **Жодна формула інтервалу не відновить масу, якої не було в вибірці.** Порядок дій: більше рядків; або змінити оцінювану величину на ту, яку вибірка витримує (медіана, усічене чи геометричне середнє); або оцінювати середнє через модель сімейства (`mean_variance_scaling`, див. `references/distribution-families.md`).

Для категорій подавати cardinality, frequency, rare/unseen levels, entropy, Gini та concentration. Для target — class counts/rates і їх зміни за time/source/group.

Не використовувати один normality p-value як рішення. На великих вибірках тести виявляють практично неважливі відхилення; на малих мають низьку потужність. Поєднувати графік, effect size, domain meaning і robust statistics.

Визначати не лише форму, а **механізм** — [references/distribution-families.md](references/distribution-families.md):

- `mean_variance_scaling` оцінює `b` у `Var ≈ mean^b` по реальних групах (сегмент, період, майданчик): `b≈0` — адитивний шум, `b≈1` — лічильні дані (sqrt), `b≈2` — мультиплікативний процес (**log**), стабілізувальна трансформація загалом `x^(1−b/2)`. Це параметр степеня сімейства Tweedie. Логарифм виправданий тим, що процес мультиплікативний (логнормальний замкнений щодо множення так само, як нормальний щодо додавання), а не тим, що гістограма скошена. Не бінувати змінну за її ж значенням — це створює залежність із нічого.
- **Пастка нулів.** Між `b=1` і `b=2` лежить compound Poisson-gamma: маса в нулі плюс неперервна додатна частина (витрати на клієнта, страхові виплати, дохід на користувача). Заміряно: `b=2.02` при **30.4% нулів** — сама форма каже «log», а голий логарифм дає `-inf` у 3804 з 12500 рядків, і `pd.isna()` цього не рахує як пропуск. Тому функція звітує `zero_fraction` і за наявності недодатних значень не рекомендує голий log, а вказує на `log1p`, дводольну (hurdle) модель або Tweedie GLM. Нулі тут зазвичай і є цікавою половиною даних: «чи була транзакція взагалі» і «скільки за умови, що була» — два різні питання.
- Для лічильних колонок `count_dispersion`: читати **VMR**, не p-value. `VMR>1` (overdispersion) — зазвичай **невраховані коваріати**, а не хибне сімейство: пуассонівська інтенсивність, яка сама варіює, і є gamma-Poisson сумішшю з негативним біноміальним маргіналом; спершу шукати змінну, що пояснює варіацію, і перевірити дублікати. `VMR<1` — обмежений процес (біноміальний має `VMR = 1−p` конструктивно). Надлишок нулів проти `exp(−mean)` — окреме питання **структурних нулів**, тобто визначення даних, а не моделі.
- Коли сигнатура даних не вкладається в жодне зі стандартних сімейств — `family_router.route(series, by=...)`. Він міряє сигнатуру (цілочисельність, межі, частка нулів, VMR, `b`, хвостовий індекс Хілла з дискримінатором Pareto/логнормальний/log-важкий) і повертає ранжованих кандидатів **із тригером, що їх підняв, і посиланням**. На добропорядних даних і на сімействах, які вже є на чарті, він мовчить — маршрутизатор, що завжди має відповідь, є генератором випадкових назв.
- **Дисципліна походу за посиланням** (деталі — [references/distribution-families.md](references/distribution-families.md)): ходити лише за тим посиланням, яке підняв **вимір**, а не назва, що звучить доречно; спершу зіставити параметризацію джерела з бібліотечною (scipy бере `scale`, підручники дають rate — це різні речі); перевіряти на відкладених даних проти простішого базового сімейства (Q-Q/PP + власне правило скорингу), бо більше параметрів завжди краще підганяються in-sample; вимагати механізму — сімейство, що підганяється, але описує процес, у який ніхто в домені не вірить, є кривою, а не поясненням; фіксувати відхилених кандидатів, не лише переможця. Кандидат, який не пройшов перевірку, потрапляє у звіт як «розглянутий і відхилений» — це теж результат.
- Перед агрегацією (rolling sum, тижневі підсумки, агрегати на користувача) — `rate_homogeneity`. Сумувати арифметично можна завжди, але сімейство зберігається лише за сталого параметра: згортка позначена умовно (`C_p` для біноміального/Pascal, `C_α` для гамми). За дрейфу ставки підсумок виглядає нормально, а всі CI й контрольні межі на ньому вже хибні. Тест відкалібрований, тому ~1 із 20 однорідних рядів позначається помилково — читати `rate_min`/`rate_max` поруч із p-value.

### p-value відповідає на питання про `n` не менше, ніж про дані

Наскрізне застереження цього кроку, і воно ріже в **обидва** боки. Жоден вердикт тут не керується значущістю, бо p-value змішує величину ефекту з розміром вибірки.

- **Великі `n` — значуще все.** Критичне значення двовибіркового KS спадає як `1.36·√(2/N)`. Заміряно на зсуві 0.005 sd (надто малому, щоб змінити бодай один прогноз): p-value = 0.70 / 0.65 / 0.39 / 0.19 / **0.0068** при N = 10²…10⁶. Дані не змінювались — змінювалось лише `n`. На продакшн-обсягах кожна колонка «значуще» дрейфує, і поріг за p-value стає сигналізацією, що не змовкає. При цьому тест **не зламаний**: під істинною нульовою гіпотезою частка відхилень лишалась ~0.05 для всіх `n` від 50 до 20 000 — він чесно відповідає на питання «чи є хоч якась різниця», просто це не те питання. Питати треба **наскільки** (`distribution_shift` → PSI, Wasserstein), а поріг калібрувати за історичними епізодами з відомою втратою якості, а не за 0.05.
- **Малі `n` — не значуще ніщо, і «post-hoc power» тут не рятує.** Дзеркальна помилка — читати незначущий результат як доказ відсутності ефекту. Заміряно на 3000 симульованих досліджень: рангова кореляція між p і спостережуваною потужністю дорівнює рівно **−1.000000**, потужність на межі p=0.05 становить ≈0.50, а максимум серед **усіх** незначущих досліджень — 0.5027. Тобто «результат незначущий» і «потужність низька» — це одне речення двічі; обчислення не може дізнатись нічого, чого не сказало p. Потужність рахувати **до** дослідження, щоб вирішити, чи здатне воно виявити цікавий ефект; після нульового результату це театр.
- Практичний наслідок для EDA: незначущий приріст від набору ознак **не є** доказом його марності — це твердження про вибірку, а не про ознаку. Звітувати величину ефекту з інтервалом (`bootstrap_diff_ci`, `paired_feature_significance`). Структурно та сама пастка, що вже задокументована для викидів: `|z|>3` не може спрацювати при n≤10, бо `max|z| = √(n−1)` — порожній результат означає замало даних, а не відсутність викидів.
- **Зсув розподілу міряти `distribution_shift`** (train vs test — чи репрезентативний split; train vs live — covariate shift; період vs період — базова лінія дрейфу). Читати `psi_ratio`, а не сирий PSI: нульове очікування PSI дорівнює `(B−1)·(1/n_ref + 1/n_cur)` (перевірено з точністю 5% на 14 конфігураціях), тож на **ідентичних** розподілах із 10 бінами PSI сягає 0.64 при n=50 і 0.20 при n=100 — звичайна пам'ятка «>0.25 = сильний дрейф» оголосила б сильний дрейф між вибіркою та нею ж. Пороги за сирим PSI застосовні лише коли `psi_reliable` = True (≥ ~50 рядків на бін). Біни беруться з **reference**-квантилів, тому міра асиметрична — подавати train як `reference`.

Прочитати [references/distributions-outliers.md](references/distributions-outliers.md).

## 6. Дослідити викиди й аномалії

Розрізняти:

- data error;
- рідкісний, але валідний випадок;
- окремий режим/сегмент;
- multivariate anomaly;
- future signal або leakage artifact.

Використовувати domain bounds, IQR/MAD/robust z-score, ECDF tails і, за потреби, Isolation Forest/LOF або reconstruction error як допоміжні сигнали. Не визначати «частку викидів» через значення поза 1-м і 99-м процентилями: така процедура конструктивно позначає приблизно 2% даних.

Відрізняти **невалідні** значення від викидів: викид екстремальний, але можливий, а невалідне значення неможливе за визначенням (вік 170, −2 години в застосунку, доставка раніше замовлення) — тут немає розподільного питання. Кодувати визначення як виконувані правила (`logical_constraints`) і запускати їх **після кожного кроку створення похідних ознак**, а не лише на сирих даних: `time_in_app = finish_hour − start_hour` дає −22 для сесії 23:00→01:00. Лагодити функцію (віднімати timestamps, а не години доби), а не видаляти рядки — видалення зміщує вибірку в бік сесій, що не перетинають північ, тобто саме «чищення» вносить selection effect.

**«Видалити за 1.5·IQR, перебудувати boxplot, викидів більше немає» — це цикл без нерухомої точки.** Межа означена **відносно того, що зараз у вибірці**, тож видалення зсуває квартилі й породжує новий набір «викидів». Заміряно на 100 000 чистих значень, застосовуючи правило повторно до зупинки: нормальний — 0.73%/0.04%/0.00%, 4 проходи, разом 0.77%; експоненційний — 4.69%/1.77%/0.71%/…, 12 проходів, разом 7.70%, sd 0.9934→0.6351; логнормальний — 7.87%/3.39%/1.60%/…, 12 проходів, разом **14.40%**, sd 2.1433→**0.6866**. Тобто на бездоганно чистих логнормальних даних цикл видаляє кожен сьомий рядок і **68% розмаху змінної** — а «викидами» був її власний правий хвіст, тобто саме та частина, від якої залежить рішення для виручки, тривалості чи розміру заявки. Навіть на нормальних даних перший прохід видаляє 0.73% рядків, які не є помилками. Цикл справді зупиняється — але тільки тому, що обрізав розподіл до такого, чиї хвости вміщаються в його ж межі: «на boxplot немає точок» — твердження про boxplot, а не про дані.

Порівняти результати з і без flagged records. Winsorization, clipping, log/power transform або removal дозволяти лише з поясненням і split-safe fit.

## 7. Перевірити target і дисбаланс

- Перевірити визначення label, момент його появи, censoring, delayed labels, conflict rate і noise.
- Перевірити, чи дисбаланс не є артефактом дублікатів: порахувати duplicate rate окремо в кожному класі й порівняти співвідношення класів до і після дедуплікації.
- Аналізувати class balance не лише загалом, а за source, time, entity, subgroup і split.
- Не ранжувати підгрупи за сирою часткою target: малі групи опиняються на обох краях рейтингу чисто випадково (дисперсія частки ~ 1/n). Використовувати `group_rate_funnel` + `rate_funnel`-графік; заявляти відхилення лише для груп поза біноміальним коридором.
- Пам'ятати, що рідкісний клас у частки відсотка (fraud, скринінг, дефекти) — властивість домену, а не помилка даних; «дисбаланс» у регресії — це важкі хвости/аномалії, а не задача ресемплінгу.
- Якщо мітку створив недосконалий детектор/правило/рев'ю — порахувати, з чого вона складається (`proxy_label_diagnostics`). При sens=spec=0.90 і справжній поширеності 0.02 PPV = **0.155**, тобто ~85% позитивних міток хибні, а частка позитивних у даних читається як 0.116 замість 0.02 (**роздування 5.8×**). Наслідок: рішення про дисбаланс, узяте з власної частоти мітки, успадковує це роздування. При рідкісному класі важіль — **специфічність, не чутливість** (0.80/0.999 дає PPV 0.890 при поширеності 0.01). Невизначеність поширеності домінує над невизначеністю заявленої точності на порядок.
- Перевірити мінімальну кількість positive/rare examples у кожному fold.
- **Перевірити, чи ціль не обчислена з самих ознак** (`composite_target_probe`). Індексні та скорингові цілі часто *визначені* як функція колонок, що приїхали в тій самій таблиці: Happiness Score у World Happiness Report дорівнює рівно `Economy + Family + Health + Freedom + Trust + Generosity + Dystopia Residual`; «ризик-скор» — зважена сума бюро-колонок поруч; `total` — сума своїх частин. Корелювати таку ціль із її ж компонентами і подавати це як знахідку — циклічно: відновлюється означення, а не зв'язок у світі. **Однофакторний скрин цього не бачить**: `suspicious_single_features` дивиться на одну колонку і потребує бінарної цілі. Заміряно на структурі WHR (n=158, 7 компонент): окремі кореляції +0.36…+0.45, у реальному датасеті heat map показує 0.72–0.78 — жодна не переходить поріг leakage. Тотожність існує лише **спільно**: OLS на всіх 7 дає `R² = 1.000000000000` і відносний залишок **1.07e-15**, тоді як чесна модель зі справжнім `R² = 0.99` дає **0.0992** — п'ятнадцять порядків різниці, тобто це не питання підбору порога. **Головне тут — запобіжник**: при `n ≤ p+1` МНК відтворює **будь-яку** ціль точно (заміряно на чистому шумі: мінімальний відносний залишок 0.0000 при `n/(p+1)=1.0`, 0.0008–0.0720 при 1.2–1.3, 0.246–0.428 при 2.0), тому проба відмовляється працювати нижче `n ≥ 2(p+1)`. Спрацювання — це **питання до власника даних** («як рахувалася ця колонка?»), а не дозвіл автоматично викидати ознаки.
- Не робити висновок про корисність ознаки лише з Pearson correlation з target; слабка лінійна кореляція не виключає nonlinear або interaction signal.
- Не балансувати дані на цьому етапі. Сформувати діагноз і передати рішення `$engineer-select-eda-features`.

## 8. Спроєктувати split і перевірити leakage

Прочитати [references/splits-leakage.md](references/splits-leakage.md). Обрати split за механізмом появи нових даних:

- stratified random — лише для приблизно IID classification;
- group-aware — для повторних entities, users, patients, devices, sites;
- chronological/rolling — для forecasting і point-in-time prediction;
- spatial/source holdout — для географічної або source generalization; для геоданих із виміряною автокореляцією — block/buffered spatial CV ([references/geospatial.md](references/geospatial.md));
- nested CV — якщо одночасно відбираються ознаки/гіперпараметри й оцінюється generalization.

Holdout не є автоматично «реалістичнішим» за CV. Реалістичність визначає відповідність deployment scenario, а стабільність — протокол і розмір даних.

Якщо target визначений на forward-вікні довжини `h` — **ніколи не перемішувати** фолди. Виміряно на дизайні, де істинна передбачуваність рівно нульова: shuffled 5-fold вигадує кореляцію **+0.151/+0.203/+0.205** при h=5/20/50, тоді як суміжні (contiguous) фолди дають −0.026/−0.048/−0.116. Головна шкода перемішування — не межа фолду, а те, що train-рядки опиняються *між* test-рядками; тому суміжність робить майже всю роботу, а `purge_and_embargo` прибирає залишковий крайовий витік (у цьому дизайні понад суміжність він нічого вимірюваного не додав, але коштує кілька рядків і лишається правильним запобіжником). Окремо: contiguous k-fold при великому `h` має **від'ємне** зміщення, тож одне число CV на таких мітках ненадійне в обидва боки — подавати розкид, а не крапку.

Group-aware split і `n_eff` — різні контролі проти різних загроз: перший захищає оцінку generalization error від leakage, другий захищає кожне твердження, зроблене під час самого EDA. Кластерний split сам собою не відновлює номінальний рівень значущості (виміряно: 0.164 → 0.247), тому при `deff > 2` розмір ефективної вибірки треба явно нести у звіт разом із розміром split.

## 9. Візуалізувати знахідки

Ключові знахідки підкріплювати графіками (спільні хелпери — `eda_plots.py` скіла `plan-eda-dataset`, сценарії — його `references/visualization.md`): гістограми/ECDF по target/source/time (`hist_by_group`, log-вісь для важких хвостів), box/violin по групах (`box_by_group`), матриця пропусків (`missingness_matrix`), бар-чарти класів за сегментами, funnel-графік часток по підгрупах (`rate_funnel`). Будувати по train, зберігати як артефакти, підписувати межі інтерпретації. Автозвіт ydata-profiling дозволяється як стартовий чекліст першого проходу, але він не замінює аудит: не знає одиницю спостереження, split, availability і leakage.

## 10. Сформувати звіт

Для кожної знахідки вказати evidence, severity, affected scope, likely cause, recommended action, validation check і status.

Окремо фіксувати **походження дефекту**, бо воно визначає, де його лагодити: джерело/збір → виправити, позначити або виключити рядки; **наша власна трансформація** → виправити функцію; **структура пайплайну** (дубль-вставка на ретраї, неправильний кодек на імпорті, join, що розмножує рядки) → виправити ETL-крок. Два останні класи **відтворюються**: очищений датасет, пайплайн якого досі пише дефект, чистий рівно один раз, а наступне завантаження мовчки повертає проблему — часто вже після виходу моделі в продакшн. Такі знахідки оформляти як задачі на пайплайн, а не як комірки ноутбука; у звіті вказувати, чи виправлення належить цьому датасету, чи вище за течією.

Розділити:

- факти, прямо виміряні в даних;
- інтерпретації;
- гіпотези;
- інформацію, якої бракує.

Завершити blockers для подальшого EDA, non-blocking warnings і питання, які передаються `$discover-eda-structure`.
