# README для початківця — EDA-скіли

Ця інструкція — «на пальцях»: як узяти свій файл із даними й підготувати з нього
**якісну таблицю для моделі**. Без жаргону. Повна версія — у [README.md](README.md).

---

## Що це і навіщо
У тебе є таблиця (Excel/CSV) із «сирими» даними. Ці 4 скіли — **помічник, що
перетворює сиру таблицю на чисту, готову таблицю**, на якій *потім* можна навчати
модель. Сам він модель **не навчає** — він готує дані й перевіряє, щоб усе було
чесно (без помилок і «підглядання відповіді»).

Що ти отримаєш наприкінці: **готову таблицю** + **графіки** кожного етапу +
**опис**, що з даними робили й чому + чесний вердикт `ready` / `not_ready`.

## Що приготувати (3 обов'язкові + 2, що дуже допомагають)
1. **Дані** — найчастіше файл **CSV** або Excel. Рядок = один об'єкт (клієнт,
   машина…), колонки = його ознаки. Краще давати **сирі** дані, а не вже
   почищені: саме сміття найкраще розповідає, що не так із твоїм процесом.
2. **Що прогнозуєш** — яка колонка є «відповіддю» (напр. `churn` = пішов клієнт).
   Це називають *target / ціль*.
3. **Місце** — просто чат із Claude (можна прикріпити CSV або вставити кілька
   рядків). Код запускати **не обов'язково**.

**Ці дві — не обов'язкові, але дають найбільший приріст якості:**

<!-- markdownlint-disable-next-line MD029 -->
4. **Словник даних** — що означає кожна колонка, в яких одиницях. Це найцінніше,
   що ти можеш додати: без нього сенс колонки доводиться вгадувати з назви, і
   `status_2` буде вгадано неправильно.
<!-- markdownlint-disable-next-line MD029 -->
5. **Коли поле стає відомим** — які колонки вже є в момент прогнозу, а які
   заповнюються пізніше. Це **єдиний надійний захист від «підглядання
   відповіді»**, і з самих даних це не видно.

Ще корисно сказати: як відбирались рядки (чи це всі клієнти, чи лише активні),
і які проблеми ти вже підозрюєш («до березня сенсор писав нулі замість пропусків»).

> **Проси діагноз, а не метод.** «Перевір, чи є в даних стійкі групи» краще, ніж
> «зроби кластеризацію на 5 кластерів»: у першому випадку скіл перевірить, чи
> групи взагалі існують, у другому — змушений видати 5 груп, навіть якщо їх немає.
> Те саме з «прибери викиди» проти «розберись, що це за викиди».

## Про дані: CSV чи що?
Так, **CSV — найпростіше**. Три способи дати дані:
- прикріпити сам файл `data.csv`;
- або вставити в чат перші 10–20 рядків;
- або **просто описати словами**: «~150 тис. рядків, колонки: вік, активність,
  сума транзакцій, churn (0/1)». Реальний файл давати не обов'язково, щоб почати.

## Найлегший старт — один промпт
Якщо не знаєш, з чого почати, скопіюй це (заміни у `[...]`):

```
Я новачок. У мене є датасет у CSV про [тема, напр. клієнтів банку].
Хочу підготувати його для моделі, що прогнозує [ЩО, напр. відтік — колонка churn].
Проведи мене через увесь EDA крок за кроком, простими словами,
пояснюй кожен метод (навіщо він і що дає) і показуй графіки на кожному етапі.
Питай мене, якщо чогось бракує.
```

Далі просто відповідай на уточнення. Оркестратор (`plan-eda-dataset`) поведе тебе етапами.

## Або крок за кроком (5 промптів)
Пиши по черзі, один за одним:

**1. План** (прикріпи CSV або опиши колонки)
```
Ось мій датасет [файл або опис колонок]. Ціль — прогноз [ЩО].
Склади план EDA і почни підготовку датасета.
```
**2. Якість і leakage**
```
Перевір якість даних і leakage перед моделюванням.
Покажи графіки: розподіли, пропуски, баланс класів.
```
**3. Структура**
```
Знайди зв'язки між ознаками, надлишковість і кластери.
Покажи кореляційну карту й графік вибору кількості кластерів.
```
**4. Ознаки й відбір**
```
Зроби нові ознаки, відбери найкорисніші (mRMR) і, якщо треба, збалансуй класи.
Покажи важливість ознак із шумовим baseline. Валідуй фінальний датасет.
```
**5. Готовність**
```
Перевір готовність датасета (readiness gate) і покажи, що ще лишилось.
```

## Графіки: що ти побачиш і як їх читати

Скіли малюють графіки на кожному етапі — це не прикраса, а **докази**. Головні:

| Графік | Що показує | Як читати |
|---|---|---|
| Гістограма розподілу | як розкидані значення однієї колонки | два «горби» = можливо, дві різні групи в даних |
| Box plot («ящик») | медіана, основна маса, викиди | крапки за «вусами» — кандидати у викиди, **не** автоматично помилки |
| Матриця пропусків | де саме порожні клітинки | вертикальні смуги = проблемна колонка; горизонтальні = биті рядки |
| Кореляційна карта | які ознаки дублюють одна одну | яскраві квадрати-блоки = надлишковість (кандидати на прибирання) |
| Графік вибору k | скільки кластерів брати | «лікоть» — лише **підказка**; дивитись разом із силуетом |
| Важливість ознак | які ознаки корисні | стовпчики **нижче червоної лінії** (шум) — сигналу не доведено |
| Порівняння варіантів | чи допомогла нова обробка | якщо «вуса» помилок перетинаються — різниці не доведено |

Три правила, які скіл дотримує сам:
1. Графіки будуються **лише по навчальній частині** (дивитись на тестову = підглядати).
2. Кожен графік **зберігається у файл** — щоб потім показати у звіті.
3. Під кожним є підпис, що з нього **можна** і **не можна** висновувати. Напр.:
   гарна картинка з двома «острівцями» точок — це *гіпотеза* про кластери, а не доказ.

Промпт, якщо хочеш більше візуалізації:
```
Покажи графіки для кожного етапу EDA й поясни простими словами,
що на них видно і які висновки з них робити НЕ можна.
```

## Якщо в тебе часовий ряд (є колонка з датою/часом)
Скажи про це — і скіл застосує правильні кроки для часу (щоб оцінка була чесною):
- поставить дані на **регулярну сітку часу** (заповнення лише з минулого);
- перевірить **стаціонарність** (тести ADF/KPSS — як діагностику, не як єдиний вердикт);
- зробить **лаги та ковзні середні без зазирання в майбутнє**;
- поділить дані **хронологічно** (тест — строго пізніший час), для кількох
  об'єктів (машин/користувачів) — з групуванням.

Промпт:
```
Мої дані — часовий ряд: є колонка часу [назва], крок [година/день],
[одна серія / кілька об'єктів у колонці ...]. Хочу прогнозувати [ЩО] на [горизонт].
Підготуй датасет чесно: вирівняй час, перевір стаціонарність, зроби лаги/ковзні
без витоку майбутнього і поділи хронологічно.
```

### «Різниця з учора» викидає більше, ніж треба

Моделям часто потрібен **стабільний** ряд — такий, що не «пливе» вгору-вниз з
роками. Класичний спосіб цього досягти: замість самого значення взяти
**різницю «сьогодні мінус учора»**. Спрацьовує, але має неочевидну ціну: ти
залишаєш «наскільки змінилось» і **втрачаєш «а де ми взагалі перебуваємо»**.
Для ціни акції, залишку на складі чи температури рівень зазвичай важить не
менше за зміну.

Дробове диференціювання — це **регулятор між «не чіпати» (0) і «повна різниця»
(1)**, який можна поставити посередині. Заміряно на випадковому блуканні:

| Що робимо | Ряд став стабільним? | Скільки зв'язку з початковим рівнем лишилось |
|---|---|---|
| нічого (0) | ні | 100% |
| **регулятор на 0.30** | **так** | **89%** |
| повна різниця (1) | так | **1%** |

Тобто звичайна різниця викидає майже весь рівень заради стабільності, якої
вистачило б і при 0.30. Скіл сам підбирає це число й показує всю криву — тобі
не треба нічого рахувати, лише знати, що таке питання існує.

Дві речі, про які варто знати:
- **Дірки в часі це ламають.** Кожне значення рахується з кількох попередніх
  поспіль, тож один пропуск гасить цілий шматок далі — у тесті ряд із дірою
  кожні 10 рядків дав **нуль** придатних значень. Тому спершу регулярна сітка
  без пропусків (скіл робить це першим кроком — і скаже, якщо дірки заважають).
- **Підібране число — не «скільки в ряду пам'яті».** Це просто найменше
  втручання, яке проходить перевірку на стабільність. Не читай його як
  характеристику самого явища.

## Якщо в тебе зображення (комп'ютерний зір)
Дані — тека з картинками. Мітки — залежно від задачі:
- **класифікація**: тека на клас, або CSV `шлях → клас`;
- **детекція/сегментація**: файл із рамками/масками (COCO, YOLO);
- **медичні/супутникові**: часто ще й id пацієнта / сцени / ділянки.

Що скіл перевірить (простими словами):
- чи всі картинки читаються, їх розміри й формати;
- баланс класів (чи не занадто мало якогось);
- **дублікати між train і test** (та сама картинка в обох — це «підглядання відповіді»);
- чи одна й та сама **сцена / пацієнт / відео** не потрапила і в train, і в test;
- чи немає «підказки» (фон, вотермарк, лінійка, тег лікарні), за якою модель шахраює;
- нормалізацію (середні кольори каналів) рахує **лише на train**.

Акцент під три типи задач:
- **класифікація** → баланс класів, дублікати, помилкові мітки;
- **детекція/сегментація** → скільки об'єктів, їх розміри, крихітні об'єкти, чи координати в межах картинки;
- **медичні/супутникові** → не змішувати пацієнта/ділянку між train/test і перевірити, чи не відрізняється «стиль» знімків (апарат/освітлення).

Промпт:
```
Мої дані — зображення для [класифікації / детекції / сегментації],
приблизно [скільки] картинок, мітки у форматі [тека-на-клас / CSV / COCO / YOLO].
[Є / немає] id пацієнта/сцени/джерела.
Проведи EDA для зображень: цілісність і розміри, баланс класів, дублікати між
train/test, витік за групою (пацієнт/сцена), можливі «підказки» (фон/вотермарк),
і порахуй нормалізацію лише на train. Поділи чесно й перевір готовність.
```

## Якщо скіл зробив не так: навчи його раз і назавжди

Буває, що ти знаєш про свої дані те, чого з них не видно. Наприклад: «порожній
тиск — це не пропуск, це датчик був вимкнений, не заповнюй його середнім».

Просто скажи це в чаті — і **попроси запам'ятати**:
```
Запиши в insights.md: у наших даних пропуск тиску означає, що датчик вимкнено —
не заповнювати його середнім, а робити окрему ознаку «датчик вимкнено».
```
Скіл збереже правило у файл `insights.md` поруч із даними — і **в наступних
сесіях застосує його сам**, без твого нагадування. Перед тим як застосувати, він
скаже: «застосовую ваше правило від [дата]…» — щоб ти міг заперечити, якщо
правило вже застаріло.

Важливо знати три речі:
- **Пояснюй «чому»** — правило без причини скіл не зможе правильно перенести на
  іншу ситуацію.
- **Файл звичайний** — це текстовий `insights.md`, можеш відкрити й виправити руками.
- **Правила безпеки скасувати не можна.** Якщо попросиш щось, що ламає чесність
  оцінки (напр. «використай тестові дані при підготовці»), скіл запише прохання,
  але **не виконає** — і запропонує безпечний варіант. Це не впертість: саме ця
  помилка робить модель, яка «чудова» на папері й провальна в житті.

## Пастки, від яких скіл тебе вбереже

Це реальні помилки, на яких «горять» навіть досвідчені — просто щоб ти знав, що
скіл їх перевіряє:

- **«Найважливіша ознака» без сенсу.** Буває, що на першому місці опиняється
  колонка на кшталт «номер рядка» чи технічної ваги вибірки. Це не відкриття —
  це збій. Якщо топова ознака не має життєвого пояснення, скіл шукає причину.
- **Шум виглядає корисним.** Якщо додати в дані випадкові числа, погано
  налаштована модель може поставити їх вище за реальні ознаки — та ще й «точність»
  зросте. Скіл підсаджує такі «канарки» навмисне: усе, що нижче за них, —
  без доведеного сигналу.
- **Дисбаланс може бути несправжнім.** Якщо в даних багато дублікатів, класи
  здаються перекошенішими, ніж є. Скіл рахує дублікати **до** рішення про балансування.
- **Одна крива не вирішує.** «Лікоть» на графіку кластерів — підказка, а не
  відповідь; скіл дивиться кілька сигналів одразу.
- **Занадто хороший результат = підозра.** Якщо одна ознака «вгадує» майже все —
  найімовірніше, відповідь випадково потрапила в дані.
- **Колонки-двійники обманюють двічі.** Якщо дві колонки кажуть майже те саме,
  вони «ділять» між собою заслугу — і жодна не виглядає важливою, хоча разом
  вирішальні. А ще буває підступніше: колонка `x` дорівнює сумі `w + z`, і на
  кореляційній карті **нічого підозрілого не видно** — таке ловить окрема
  перевірка (VIF), яку скіл робить завжди, а не «за потреби».
- **Найгірші «покращаться» самі.** Візьми 10% найгірших магазинів і зміряй ще
  раз — вони покращаться, навіть якщо нічого не робити. Це перевірено на даних,
  де між двома вимірами **не сталося взагалі нічого**: найгірші 10% «виросли»
  на +7 балів, найкращі 10% так само «впали». Тому «ми втрутились — і стало
  краще» саме по собі не доводить нічого; скіл рахує, скільки з цього руху було
  неминучим, і показує лише **залишок**.
- **«Не корелює» ≠ «не пов'язано».** Зв'язок може бути дугою (мало — погано,
  багато — теж погано, добре посередині), або протилежним у двох групах — і
  кореляція покаже нуль. Найпідступніший випадок: у **кожній** групі зв'язок
  від'ємний, а в об'єднаних даних — додатний. Тож нуль у таблиці кореляцій — це
  привід подивитись на графік, а не викинути колонку.
- **Відфільтровані дані ховають зв'язки.** Якщо в таблиці лише схвалені кредити,
  лише ті, хто лишився, лише завершені сесії — усі зв'язки в ній занижені
  **конструктивно**. Класика: «оцінка на вступі не впливає на успішність» —
  порахували лише на тих, кого зарахували. Це арифметика, а не висновок.
- **На великих даних «значуще» все.** Слово «значущий» залежить від кількості
  рядків не менше, ніж від самих даних: на мільйоні рядків значущою стає різниця,
  яка не змінює нічого. Тому скіл дивиться на **величину** ефекту, а не на
  «значущість». І навпаки: на маленьких даних «не значущо» не означає «немає
  ефекту» — означає «цих даних забракло, щоб побачити».
- **Кого в таблиці немає.** Таблиця «наші клієнти», зібрана сьогодні, містить
  лише тих, хто **не** пішов — а саме ті, хто пішов, і цікавлять модель відтоку.
  Більше даних цього не лікує. Питання «хто сюди не потрапив і чому» часто
  важливіше за всі колонки разом.
- **Стрибок у часі може бути зміною лінійки.** Якщо дані різко змінилися з
  певної дати — найімовірніше, змінили форму опитника, версію застосунку або
  сенсор, а не світ. Скіл спершу перевіряє саме це.
- **Тисяча рядків — не завжди тисяча спостережень.** Якщо на одного клієнта
  припадає 25 рядків, ці 25 схожі між собою й разом важать значно менше, ніж
  25 різних клієнтів. Заміряно: 1500 таких рядків несуть інформацію приблизно
  183 незалежних, і «надійний на 95%» інтервал виявляється правильним менш ніж
  у половині випадків. Тому скіл питає, **у що вкладений рядок** (клієнт,
  пристрій, магазин, сесія), і рахує «чесний» розмір вибірки.
- **Чим більше колонок перебираєш, тим легше знайти пусте.** Якщо перевірити
  1000 випадкових колонок, найкраща з них виглядатиме переконливо навіть коли
  жодного зв'язку немає насправді. Заміряно: серед 1000 кандидатів із 5
  справжніх ознак у топ-5 потрапляє в середньому лише півтори справжні, а
  на першому місці стоїть чистий шум у 43% випадків. Скіл перевіряє відібрані
  ознаки на **інших рядках**: справжня ознака підтверджується, випадкова —
  розсипається.
- **«Спробував десять способів, один спрацював» — це не результат.** Кожен
  вибір під час аналізу (як прибрати викиди, чи логарифмувати, яку підгрупу
  взяти) — окрема розвилка. Заміряно на даних, де зв'язку **немає взагалі**:
  18 цілком розумних комбінацій дають 30% шансу «знайти» неіснуючий зв'язок
  замість 5%. І найнеприємніше — більше даних від цього **не рятує**. Тому скіл
  записує, який шлях обрано, і вважає знахідку гіпотезою, доки вона не
  підтвердиться на даних, які її не формували.
- **«Точний на 90%» детектор при рідкісній події помиляється частіше, ніж
  вгадує.** Якщо подія трапляється у 2% випадків, то навіть детектор, який
  ловить 90% справжніх і лише 10% разів кричить дарма, дає позначку, де
  **85% позначених — не ті**. Причина проста: хибні спрацювання беруться з
  величезної маси «нормальних», а справжніх мало. Побічний ефект: у даних
  здається, що подія трапляється в 11.6% випадків замість 2% — майже вшестеро
  частіше. Тому скіл питає, звідки взялася позначка, і рахує це явно.
- **Якщо ціль описує «наступні N днів», не можна перемішувати дані при
  перевірці.** Коли ціль — «дохід за наступні 20 днів» або «пішов протягом
  90 днів», сусідні рядки описують той самий шматок майбутнього. Якщо
  перемішати рядки й частину віддати на перевірку, модель фактично побачить
  відповідь. Заміряно на даних, де передбачити було **неможливо в принципі**:
  перемішана перевірка показала помітне «вміння», а перевірка без
  перемішування — нуль. Тобто весь результат був фікцією.
- **Ціль теж буває не тим, чим здається.** Колонка `шахрайство` майже завжди
  означає «шахрайство, яке помітили й підтвердили», `відтік` — чиєсь правило
  про 90 днів. Модель сумлінно вивчить саме це означення, включно з тим, чого
  ти не мав на увазі. Скіл питає, як саме заповнювалася ціль.
- **«Середнє ± похибка» на малій вибірці бреше, і завжди в один бік.** Якщо
  дані «з довгим хвостом» (виручка на клієнта, тривалість сесії, розмір
  заявки — де більшість значень маленькі, а зрідка трапляються дуже великі),
  то маленька вибірка зазвичай **не містить** тих рідкісних великих значень,
  на яких і тримається середнє. Тому й середнє, і його «похибка» виходять
  заниженими. Заміряно: інтервал, який має накривати правду у 95% випадків,
  на 50 рядках накриває її у 90%, а промахи майже всі в один бік — інтервал
  опиняється **нижче** правди у 34 рази частіше, ніж вище. Порада «бери 30
  рядків, і цього досить» стосується симетричних даних; для «хвостатих» вона
  просто неправильна. Що робить скіл: пропонує або більше даних, або **медіану**
  замість середнього (її мала вибірка витримує), і чесно каже, коли числу
  не варто вірити.
- **Іноді ціль порахована з тих самих колонок, з якими ти її порівнюєш.**
  У відомому наборі «World Happiness Report» підсумковий бал щастя — це рівно
  сума колонок, що лежать поруч (економіка + сім'я + здоров'я + свобода +
  довіра + щедрість + залишок). Побудувати кореляцію балу з економікою і
  сказати «економіка впливає на щастя» — це не відкриття, а перевідкриття
  формули, за якою бал склали. Найпідступніше те, що **жодна окрема колонка не
  виглядає підозріло**: кореляції там 0.72–0.78, цілком «нормальні» числа.
  Видно лише коли взяти всі колонки разом — тоді ціль відновлюється точно, до
  останнього знака. Скіл це перевіряє і питає: «як саме рахувалася ця колонка?»
- **«Видалив викиди за правилом — і на графіку їх більше немає» нічого не
  доводить.** Правило 1.5·IQR означене *відносно того, що зараз у вибірці*.
  Прибрав крайні точки — межі зсунулися — з'явилися нові «викиди». Заміряно на
  ідеально чистих даних без жодної помилки: для «хвостатої» змінної цикл
  зупинився аж на 12-му проході, викинувши **14% рядків** і **68% розмаху**
  змінної. Викидами виявився її власний правий хвіст — тобто саме великі
  замовлення чи довгі сесії, заради яких усе й робилося. Порожній boxplot —
  твердження про boxplot, а не про дані.
- **«Ця група відрізняється!» — коли груп багато, хтось відрізнятиметься
  завжди.** Порівнюючи 4 групи, ти робиш 6 порівнянь, а 8 груп — це вже 28.
  Заміряно на даних, де **всі групи однакові за побудовою**: хоча б одна пара
  виглядає «значуще різною» у 20% випадків при 4 групах і у **51%** при 8.
  Класична порада «спочатку зроби загальний тест, і лише якщо він спрацював —
  порівнюй пари» звучить розумно, але заміряно: щойно хоч одна група справді
  відрізняється, загальний тест спрацьовує **завжди** і більше нічого не
  фільтрує — частота хибних знахідок повертається до тих самих 12–44%. Скіл
  використовує поправку, яка працює незалежно від загального тесту.

## Терміни простими словами
- **Target / ціль** — колонка, яку прогнозуємо.
- **Split** — поділ даних на «навчальні» і «перевірочні», щоб чесно оцінити.
- **Leakage (витік)** — коли модель випадково «підглядає відповідь». Головна
  помилка; скіл її ловить.
- **Балансування** — коли одного класу мало (напр. 5%); скіл радить *не* робити
  примусово 50/50, а м'якше (ваги класів + помірний ресемплінг). Часто найкраще
  рішення — **нічого не робити**, лише правильно міряти.
- **Асиметрія (skew)** — наскільки дані «перекошені». У симетричних даних
  середнє посередині; у «хвостатих» більшість значень маленькі, а рідкісні
  великі тягнуть середнє вгору. Саме асиметрія, а не кількість рядків,
  вирішує, чи можна вірити середньому на малій вибірці.
- **Поправка на множинні порівняння** — коли перевіряють багато пар/ознак
  одразу, частина «знахідок» буде випадковою; поправка піднімає планку так,
  щоб їх кількість лишалася під контролем.
- **Кластери** — групи схожих об'єктів, які знаходять *без* підказки. Їх може
  вийти більше, ніж «типів» у житті (напр. кілька стилів написання однієї цифри) —
  це нормально.
- **Важливість ознак** — наскільки модель спирається на колонку. Це **не** означає,
  що колонка «спричиняє» результат.
- **Лаг** — значення ознаки «N кроків тому» (для часових рядів).
- **Стаціонарність** — чи не «пливуть» середнє/розкид із часом (тренд/сезонність).
- **Дробове диференціювання** — регулятор між «лишити ряд як є» і «взяти різницю
  з попереднім днем». Дає стабільність, потрібну моделі, не викидаючи при цьому
  майже весь рівень (див. розділ про часові ряди).
- **Probe** — маленька проста модель, щоб *вивчити дані*, а не для фінального прогнозу.
- **Регресія до середнього** — схильність «рекордних» значень при повторному
  вимірі бути ближчими до середини. Не ефект втручання, а властивість вимірювання.
- **p-value / «значущість»** — наскільки результат схожий на випадковість. Сильно
  залежить від кількості рядків, тому сам по собі нічого не каже про **розмір**
  ефекту й про те, чи він важливий на практиці.
- **Пропуски за задумом** — іноді порожні клітинки створені навмисно (різним
  людям дали різні блоки анкети). Тоді це не дефект, і «лагодити» його не треба.
- **insights.md** — файл, куди скіл записує твої доменні правила, щоб застосовувати
  їх у наступних сесіях без нагадувань.

import pandas as pd
df = pd.read_csv("твій_файл.csv")
