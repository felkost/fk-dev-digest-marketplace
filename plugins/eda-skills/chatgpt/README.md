# EDA-скіли в Custom GPT (ChatGPT)

Той самий leakage-safe EDA-конвеєр (план → аудит → структура → ознаки → readiness
gate), що й у Claude, запакований для **Custom GPT** у ChatGPT. Інші способи
розгортання — у кореневому [README.md](../README.md).

Пакет — це два файли, що йдуть у пару:

- **[gpt_instructions.md](gpt_instructions.md)** — вставляється в поле «Instructions»
  Custom GPT (стиснута інструкція: bootstrap, обов'язкові інваріанти, 5 етапів,
  маршрутизація за модальністю, обмеження версії ChatGPT).
- **`dist/eda_skills_knowledge.zip`** — увесь методологічний зміст (усі 4 `SKILL.md`,
  `references/*.md`, `scripts/*.py`, ноутбук, `requirements.txt`), який GPT
  розпаковує собі через Code Interpreter і читає на вимогу. Будується скриптом
  **[build_gpt_package.ps1](build_gpt_package.ps1)**.

Чому саме так: поле Instructions має практичний ліміт **8000 UTF-8 байтів**
(перевірте живий ліміт у редакторі GPT на момент встановлення), а весь
методологічний текст важить ~530 КБ. Тому Instructions несуть лише **процедуру**
(що робити і в якому порядку), а Knowledge-архів — **зміст** (як саме, з
обґрунтуванням), який GPT читає вибірково через Code Interpreter.

> **Ліміт рахується в БАЙТАХ, не в символах** — і це не дрібниця для української
> інструкції: кирилиця займає 2 байти на літеру, тож 5563 символи = 8498 байтів
> і ChatGPT відхилить вставку. `build_gpt_package.ps1` міряє саме байти
> (`UTF8.GetByteCount`). Станом на зараз файл займає **7968 з 8000 байтів** —
> запасу ~32 байти, тож будь-яке додавання потребує зустрічного скорочення.

## Встановлення (перший раз)

1. **Зібрати пакет** (шлях залежить від того, де ви зараз):
   ```
   # якщо ви в корені eda_skills\:
   powershell -ExecutionPolicy Bypass -File chatgpt\build_gpt_package.ps1

   # якщо ви вже всередині eda_skills\chatgpt\:
   powershell -ExecutionPolicy Bypass -File build_gpt_package.ps1
   ```
   Скрипт очищує `__pycache__`, збирає `dist/eda_skills_knowledge.zip` і друкує
   довжину `gpt_instructions.md` **у UTF-8 байтах** (і довідково в символах)
   проти ліміту 8000 — переконайтеся, що бачите `OK`. Якщо `Warning` —
   інструкцію треба скоротити **до** вставки в ChatGPT, інакше він її відхилить.
   Скрипт сам знаходить кореневу теку `eda_skills\` відносно себе
   (`$PSScriptRoot`), тож працює правильно з обох місць — головне передати
   `-File` правильний шлях *до самого скрипта*.

2. **ChatGPT → Explore GPTs → Create → вкладка Configure** (не Create-чат-режим).

3. **Name / Description** — довільні, напр. «EDA Dataset Prep».

4. **Instructions** — відкрити [gpt_instructions.md](gpt_instructions.md),
   скопіювати весь текст, вставити в поле Instructions.

5. **Capabilities** — увімкнути **Code Interpreter & Data Analysis** (обов'язково:
   без нього GPT не зможе розпакувати архів і виконати скрипти). Web
   browsing/DALL·E не потрібні.

6. **Knowledge** — завантажити `dist/eda_skills_knowledge.zip`.

7. **Save → Create.**

8. **Перевірка:** у новому чаті написати «Розпакуй knowledge і перелічи файли в
   audit-eda-data-quality/scripts». GPT має відповісти списком `.py`-файлів. Якщо
   каже, що не бачить архів — переконатися, що Code Interpreter увімкнено і файл
   справді завантажений у Knowledge (а не лише текстовий індекс).

## Як формулювати завдання і що прикладати

GPT почне добре навіть із неповного запиту, але результат тим кращий, чим
чіткіше сформульовано задачу. Повний чекліст — усередині пакета,
`plan-eda-dataset/references/intake.md` (GPT читає його сам, коли запит
нечіткий); нижче — стисла версія для людини.

**Мінімум, щоб почати** — чотири речення: (1) для чого модель (рішення, не
«передбачити X»), (2) що таке один рядок (одиниця спостереження — клієнт /
клієнт-місяць / транзакція / сесія), (3) ціль (колонка / треба сконструювати /
немає), (4) чи є час і прогноз у майбутнє.

> Шаблон: «Маю [дані] по [сутність] за [період]. Один рядок = [одиниця
> спостереження]. Хочу підготувати датасет, щоб модель [рішення]. Ціль —
> [target / сконструювати / немає]. [Є/немає] час, прогноз на [горизонт].
> Обмеження: [приватність / розмір / інтерпретованість].»

**Стартові фрази за метою** (проси **діагноз, а не метод** — «знайди сегменти»,
не «зроби KMeans k=5»):

| Мета | Фраза |
|---|---|
| Повний конвеєр | «Склади план EDA й підготуй датасет для [рішення]» |
| Лише якість/leakage | «Перевір якість і leakage цього датасета перед моделюванням» |
| Split | «Спроєктуй leakage-safe split: рядок = [X], є [group/time] ключі» |
| Зв'язки/надлишковість | «Підбери міру зв'язку між [A] і [B]» / «знайди надлишкові ознаки» |
| Сегменти | «Перевір, чи є стійкі сегменти» |
| Часові ознаки | «Побудуй лаги/rolling без зазирання в майбутнє» |
| Ознаки + відбір | «Зроби feature engineering і відбір, поясни кожне рішення» |
| Дисбаланс | «Класи 98/2 — діагностуй, чи це справжній дисбаланс» |
| Train/test | «Порівняй розподіли train і test — чи репрезентативний split» |
| Вердикт | «Датасет готовий до навчання? Дай readiness-вердикт з обґрунтуванням» |

**Що прикладати:** дані (краще сирі, не вже почищений експорт) або
репрезентативний семпл 5–20k рядків із поясненням, як відбирався; **словник
даних** (найцінніше — без нього семантику колонок доводиться вгадувати); момент
доступності кожного поля (єдиний надійний захист від target leakage); як
відбирались рядки і кого відкинуто; відомі проблеми й доменні правила
(«пропуск тиску = датчик вимкнено»).

## Оновлення (коли скіли зміняться тут, у Claude)

Два випадки:

### A. Змінився лише вміст `references/*.md` або `scripts/*.py`

Найчастіший випадок (нова стат-функція, уточнення reference-файлу тощо).
Instructions **не чіпати**.

1. Зібрати пакет (див. точну команду залежно від поточної теки в розділі
   «Встановлення» вище).
2. У ChatGPT → редагувати GPT → Knowledge → видалити старий `eda_skills_knowledge.zip`
   → завантажити новий → Save.

### B. Змінився сам `SKILL.md` (новий етап, новий інваріант, нова гілка модальності)

Тоді потрібно синхронізувати й **gpt_instructions.md**. Таблиця відповідності
(розділ `SKILL.md` → розділ `gpt_instructions.md`, щоб знати, що саме правити):

| Змінили в `SKILL.md` | Правити в `gpt_instructions.md` |
|---|---|
| `plan-eda-dataset` → «Межі роботи» / «Обов'язкові правила» | «Обов'язкові інваріанти» |
| `plan-eda-dataset` → «Початковий контракт» / `references/intake.md` | «Старт сесії» (+ блок «Як формулювати завдання» у цьому README) |
| `plan-eda-dataset` → «Оркестрація чотирьох скілів» (порядок/склад етапів) | «П'ять етапів» |
| `plan-eda-dataset` → «Маршрутизація за модальністю та цільовою моделлю» | «Маршрутизація за модальністю» |
| `plan-eda-dataset` → «Уточнення користувача (`insights.md`)» | «Обмеження цієї версії» (абзац про persistence) |
| `plan-eda-dataset` → «Розпаралелювання субагентами» | «Обмеження цієї версії» (абзац про відсутність субагентів) — **не** переносити fan-out-логіку в ChatGPT-інструкції, там немає паралельних субагентів |
| Новий скрипт/референс у будь-якому «Допоміжні скрипти» | Зазвичай **нічого правити** — GPT сам читає `SKILL.md` розпакованого архіву й бачить оновлений список |
| Новий 5-й скіл / нова модальність поза існуючими гілками | Додати рядок і в «П'ять етапів»/«Маршрутизація», і перевірити довжину після редагування |

Після правки `gpt_instructions.md`:

1. Перезапустити `build_gpt_package.ps1` — переконатися, що звіт про довжину каже `OK`.
2. У ChatGPT → редагувати GPT → вставити оновлений текст в Instructions.
3. Замінити Knowledge-архів, як у випадку A.

## Відоме обмеження: `insights.md` не персистентний тут

У Claude/Codex `insights.md` живе на диску проєкту й переживає сесії. У Custom
GPT пісочниця Code Interpreter **скидається між розмовами** — файл, який GPT
запише під час чату, зникає, коли ви відкриваєте новий чат.

Практичний обхід (уже прописаний у `gpt_instructions.md`): якщо в розмові
з'явилося варте збереження доменне правило — попросити GPT показати поточний
вміст `insights.md`, зберегти його собі, і **перезавантажити як окремий файл
Knowledge** перед наступною серйозною сесією (або просто повторити правило на
початку нового чату — для 1–2 правил цього достатньо). Якщо потрібна справжня
персистентна пам'ять між сесіями — це природно працює в Claude Code/Codex
(локальна файлова система), а не в Custom GPT.

## Відоме обмеження: немає субагентів

Розділ «Розпаралелювання субагентами» в `plan-eda-dataset/SKILL.md` — можливість
Claude Code (паралельні read-only гілки на одному етапі). Custom GPT такого
примітиву не має; `gpt_instructions.md` явно каже GPT ігнорувати цю логіку й
виконувати все послідовно. Це не втрата функціональності відносно послідовного
режиму — лише відсутність прискорення на великих багатогілкових задачах.
