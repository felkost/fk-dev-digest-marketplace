# Наскрізне консультування з побудови й тюнінгу моделі

Оркестратор: веде повний цикл і **делегує кожне рішення профільному скілу**.
Вузьке питання («який поріг?») сюди не йде — воно тригерить профільний скіл
напряму.

**Відповіді українською; ідентифікатори й код — англійською.**

## Крок 0 — анкета підпису задачі (один раз, на старті)

1. Задача: класифікація / регресія / кластеризація / RL / LLM-пайплайн?
2. n об'єктів, d ознак; типи ознак (числові/категорії/текст)?
3. Ціль: баланс класів або розподіл значень (нулі? хвости? цензурування?)
4. Ціна помилок: що гірше — FP чи FN? У чому вимірюється?
5. Обмеження: інтерпретованість? латентність? регуляторика?
6. Що вже є: базлайн? train/test спліт? групи/час у даних?

Відповіді решта скілів отримують у спадок — не перепитувати.

**Якщо на п.1 немає впевненої відповіді** (або задача описана як бізнес-запит,
ціль не означена, незрозуміло, що таке рядок) — спершу `ml-task-framing`, і
лише потім решта. Тюнінг задачі, поставленої неправильно, не рятує.

## Крок 1 — маршрут за задачею

| Задача | Послідовність |
|---|---|
| **Тип задачі неясний** | `ml-task-framing` → повернутись сюди з визначеним типом |
| **Класифікація** | метрика → спліт → базлайн → сімейство → діагноз → ручки → пошук → поріг → звіт (повний цикл нижче) |
| **Регресія** | `ml-distribution-choice` (розподіл цілі → втрата/метрика) → базлайн → діагноз → сімейні → пошук → звіт |
| **Кластеризація** | `ml-dimensionality-features` (чи знижувати) → `ml-clustering-k` → перевірка стабільності → звіт |
| **RL** | `rl-hyperparameters` цілком |
| **LLM-пайплайн** | `llm-parameter-choice` цілком |

## Повний цикл для supervised (посилання = делегування)

0. **Постановка** (якщо ще не зроблена). `ml-task-framing`: тип задачі, одиниця
   спостереження, ціль з вікном і горизонтом, контракт входу point-in-time.
1. **Метрика.** `ml-metric-choice`: баланс + ціна помилок → scoring для всього
   далі. Для регресії замість цього — `ml-distribution-choice`: розподіл цілі →
   втрата (deviance) і метрика.
2. **Спліт до всього.** `ml-validation-design`: схема зі структури даних (групи →
   GroupKFold, час → TimeSeriesSplit із gap, і те й те → комбінація руками),
   вибір K, чи потрібна вкладена CV. Test запечатати.
3. **Базлайн за 5 хвилин.** Константа/мажоритарний + одна проста модель
   (логістична або дерево depth=3) на обраній метриці. Все подальше міряється
   від нього.
3a. **Сімейство моделі.** `ml-model-selection`: сім обмежень (пояснюваність,
   пам'ять, n і d, типи ознак, лінійність, час навчання, час прогнозу) →
   короткий список 2–3 сімейств із різним зміщенням, порівняних на однаковій CV.
4. **Діагноз.** `ml-overfitting-diagnosis`: розрив train/val за порогами
   15/20/30%, форма кривої навчання, канарки за підозри на витік.
5. **Ознаки за потреби.** Забагато/колінеарні/підозрілі важливості →
   `ml-dimensionality-features` (+ VIF-драбина в `ml-linear-regularization`).
6. **Сімейні ручки.** Дерева/ансамблі → `ml-tree-ensemble-params`; лінійні →
   `ml-linear-regularization`; нейромережі → `nn-training-params`. Звідти —
   стартові значення й сітки.
7. **Пошук.** `ml-search-strategy`: метод за бюджетом, усе в Pipeline,
   лог-шкали, чесне звітування (test або вкладена CV, правило 1 SE).
8. **Поріг.** Лише для класифікації з рішенням так/ні:
   `ml-decision-threshold` — критерій за асиметрією витрат, поріг як плато.
9. **Звіт.**

## Крок 2 — контракт фінального звіту

- Задача і підпис (з анкети), метрика і ЧОМУ вона.
- Базлайн проти фінальної моделі на запечатаному test (одне число кожній).
- Конфігурація для відтворення: модель + параметри + seed + Pipeline + версії
  бібліотек.
- Поріг (якщо є) з плато і матриця плутанини в абсолютних числах.
- Діагностика: розрив train/test, результат канарок.
- Межі: на яких даних НЕ перевірено; що б змінило висновки.
- Позначки тверджень: факт (пораховано) / інтерпретація / припущення.

## Крок 2a — журнал рішень і перевірка на стійкість (S3E07)

Уздовж циклу ви приймаєте десятки рішень, кожне з яких могло бути іншим:
викидати чи лишити викиди, трансформувати чи ні, який імп'ютер, де різати поріг,
які ознаки. Це **дослідницькі ступені свободи**. Вони не є зловживанням — це
робота експерта; зловживанням їх робить **невидимість**.

**Журнал рішень** (аналог лабораторного зошита) — обов'язковий додаток до звіту:

| Рішення | План був | Зробили | Чому змінили | Вплив на метрику |
|---|---|---|---|---|
| викиди | лишити всі | видалено 3 | явні помилки датчика (−999) | PR-AUC +0.004 |

Правило одне: **читач має право не погодитись із рішенням — але не має права
про нього не знати.** Хто хоче — переграє інакше.

**Перевірка на стійкість (multiverse-lite).** Для 2–4 рішень, які реально
могли піти інакше, прогнати варіанти на тій самій CV і показати розкид
висновку — не лише найкращого числа:

- висновок стійкий у всіх гілках → так і сказати, це сильний аргумент;
- висновок тримається на одній гілці → це і є головна межа роботи;
- **не всі гілки рівноцінні.** Варіант, що суперечить домену (лишити −999 як
  значення), у порівняння не включають — інакше «стійкість» рахується проти
  свідомо хибних альтернатив.

Гілки перебирають **до** погляду на test; інакше це вибір рішення за
результатом, тобто те саме, від чого захищає вкладена CV (`ml-search-strategy`).

## Наскрізний тестовий прогін пака

Датасет курсу: `adult_data_witn_att_names01.csv` (тека курсу; **`sep=';'`,
категорії з провідним пробілом** — реальна пастка читання; ціль `salary`
уже 0/1). Маршрут: метрика (дисбаланс ~24% → PR-AUC/MCC) → базлайн →
діагноз → канарки (очікуваний урок: `fnlwgt` у топі RF — метадана перепису
без механізму) → RF/HGB ручки → пошук → поріг → звіт.

## Режими відмови оркестратора

- **Стрибок у пошук без метрики** — тюнінг невідомо чого; кроки 1–3 обов'язкові.
- **Переробляти рішення профільного скіла** — оркестратор маршрутизує, не
  перевирішує.
- **Анкета замість роботи** — якщо користувач уже дав відповіді в питанні, не
  перепитувати; заповнювати з контексту.
- **Пропущений базлайн** — без нього «покращення» нема від чого міряти.
- **Звіт без версій і seed** — невідтворюваний.
