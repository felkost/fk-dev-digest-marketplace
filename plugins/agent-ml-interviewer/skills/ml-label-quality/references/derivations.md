# Виведення: стеля 1−p, асиметричний шум, каппа, зміщення проксі

## 1. Стеля метрики при симетричному шумі

Нехай `y*` — істинна мітка, `y` — записана, і кожна мітка незалежно перевернута
з імовірністю `p`. Модель `f` ідеальна щодо конструкту: `f(x) = y*`. Тоді

```
P(f(x) = y) = P(y = y*) = 1 − p
```

Тобто **точність ідеальної моделі, виміряна проти зашумлених міток, дорівнює
рівно `1 − p`** — це стеля, якої не подолає жодна модель і жоден тюнінг.

Жива перевірка (n = 20 000):

| p | виміряно | теорія `1−p` |
|---|---|---|
| 0.05 | 0.9500 | 0.9500 |
| 0.15 | 0.8500 | 0.8500 |
| 0.30 | 0.7000 | 0.7000 |

Збіг до 4 знаків.

**Наслідок для інтерпретації:** якщо ви знаєте `p` з аудиту, то нормувати
досягнуту точність треба на стелю. Точність 0.78 при `p = 0.20` — це 0.78/0.80
= 97.5% доступного, а не «є куди рости на 22 пункти».

### Чому шум у train шкодить менше

У навчанні незалежний шум входить у функцію втрат як **додатковий незміщений**
доданок: оптимум усе ще досягається на `y*`, лише з більшою дисперсією оцінки.
При достатньому `n` модель відновлює межу. У тесті ж шум входить **у сам
критерій оцінки**, і жодне збільшення `n` не допомагає — росте лише точність,
з якою ви оцінюєте... зміщене число.

Виміряно (шум 20%): train зашумлений → 0.9941 (з 0.9985); test зашумлений →
0.7987. Різниця на два порядки за шкодою.

**Практичний висновок про бюджет:** при обмеженому ресурсі на переразмітку
чистити треба **тестові** мітки. Це протилежно інтуїції «даємо моделі чисті
дані для навчання».

## 2. Асиметричний шум: чому страждає саме precision

Нехай позитиви пропускаються з імовірністю `α` (подія сталась, але записана як
негатив), а хибних позитивів у мітках немає. Ідеальна щодо конструкту модель
позначає позитивом усі істинні позитиви. Тоді серед її позитивних передбачень
частка `α` вважається помилкою — хоча помиляється мітка:

```
precision_виміряна = (1 − α) · precision_істинна
recall_виміряна ≈ recall_істинна        (чисельник і знаменник масштабуються разом)
```

Жива перевірка (α = 0.30):

| | проти шумних | проти істинних |
|---|---|---|
| базова частка | 0.3501 | 0.5001 |
| precision | 0.6982 | 1.0000 |
| recall | 0.6130 | 0.6145 |

`0.6982 ≈ (1 − 0.30) · 1.0000` — формула підтверджена. Recall зсунувся на
0.0015, тобто практично не зачеплений.

**Діагностична ознака:** аномально низька precision при нормальному recall і
підозріло низькій базовій частці — це підпис пропущених позитивів у мітках, а
не поганої моделі. Рухати поріг тут марно: він переміщує компроміс, а не
виправляє знаменник.

Зворотний випадок (хибні позитиви в мітках, `β`) — дзеркальний: занижується
recall, precision майже не чіпається.

## 3. Каппа: чому вона падає на рідкісних класах

```
κ = (p_o − p_e) / (1 − p_e)
```

`p_o` — спостережена частка згоди, `p_e` — очікувана випадкова згода:
`p_e = Σ_k P(A=k)·P(B=k)`.

Для збалансованих класів (50/50) двоє незалежних анотаторів випадково збігнуться
у `0.5² + 0.5² = 0.50` випадків, тож `p_e = 0.5`. При `p_o = 0.80`:

```
κ = (0.80 − 0.50)/(1 − 0.50) = 0.60      (виміряно 0.5999)
```

Для рідкісного класу (5/95) випадкова згода вже `0.05² + 0.95² = 0.9050`.
Той самий `p_o = 0.80` дає **від'ємний або мізерний** κ, бо анотатори
збігаються **рідше**, ніж дали б монетки з тими самими частками. Виміряно на
фікстурі: `p_o = 0.800`, **κ = 0.2449**.

**Практичне правило:** що рідкісніший клас, то вищий `p_e`, то жорсткіша каппа.
Тому «85% згоди» на задачі з базовою частотою 2% може означати надійність,
близьку до нуля — і саме тому відсоток згоди на рідкісних класах не звітують.

Орієнтири (конвенція, не закон, як і VIF 5/10):

| κ | Читання |
|---|---|
| < 0.20 | згоди практично немає; інструкція анотування не працює |
| 0.20–0.40 | слабка; мітка непридатна як основна істина |
| 0.40–0.60 | помірна; можна вчити, але стеля метрики вже помітна |
| 0.60–0.80 | суттєва; робочий діапазон |
| > 0.80 | висока; або задача проста, або анотатори не незалежні |

Останній рядок не жарт: **дуже висока κ — привід перевірити**, чи не бачили
анотатори роботи одне одного і чи не розмічали обидва за виходом однієї моделі.

### Fleiss для трьох і більше

`statsmodels.stats.inter_rater.fleiss_kappa` приймає таблицю «об'єкт × скільки
анотаторів дали кожну категорію»; будується `aggregate_raters`. Виміряно:
випадкові мітки → κ = 0.0133 (≈0, як і має бути); три анотатори з точністю 85%
кожен → κ = 0.4397.

**κ вимірює надійність, не валідність.** Три анотатори з однаковою хибною
інструкцією дадуть високу κ на систематично неправильній мітці. Валідність —
Крок 6 SKILL.md (зіставлення з незалежним вимірюванням).

## 4. Зміщення проксі, залежне від групи

Модель: істинна потреба `T` однакова в групах; проксі `Y = T − δ·G + ε`, де
`G ∈ {0,1}` — група, `δ > 0` — систематичне заниження. Модель, навчена на `Y`,
відтворює `Y`, а не `T`. Відбір топ-q за передбаченим `Y` дає:

```
P(відібрано | G=1) < P(відібрано | G=0)      навіть при однаковому T
E[T | відібрано, G=1] > E[T | відібрано, G=0]
```

Друга нерівність — суть проблеми: **відібрані з ущемленої групи «кращі» за
конструктом**, бо мусили перекрити зсув `δ`.

Живий прогін (n = 20 000, δ = 0.8, `T` розподілена однаково в групах):

| Величина | Число |
|---|---|
| частка `G=1` у популяції | 0.501 |
| частка `G=1` серед відібраних топ-10% | **0.161** |
| `E[T | відібрано, G=0]` | 1.327 |
| `E[T | відібрано, G=1]` | **2.054** |

Група представлена втричі менше, а її відібрані мають на 0.73 σ вищу істинну
потребу. **Жодна метрика щодо `Y` цього не показує** — щодо проксі модель
бездоганна. Виявляється лише зіставленням із незалежним вимірюванням `T`
(Крок 6) або доменним аргументом.

Це формальна структура Obermeyer et al. (2019): проксі «витрати на лікування»
замість конструкту «тяжкість хвороби», де `δ` виникає з нерівного доступу до
медицини.

**Чому це не лікується додаванням `G` як ознаки.** Модель із `G` краще
передбачить `Y`, тобто ще точніше відтворить зміщений проксі. Виправляти треба
мітку (перевизначити ціль ближче до конструкту), а не набір ознак.

## 5. Закон Гудгарта у формі, придатній для ML

«Коли міра стає ціллю, вона перестає бути хорошою мірою». Формально: зв'язок
`corr(Y, T)` оцінений на **історичних** даних, де ніхто не оптимізував `Y`.
Після впровадження моделі поведінка агентів зсувається у напрямі `Y`, і саме
та частина `Y`, що не пов'язана з `T`, найдешевша для накрутки.

Практичний наслідок: зв'язок проксі з конструктом треба **перевіряти повторно
після впровадження**, а не вважати сталою характеристикою мітки. Це та сама
логіка, що й дрейф означення в `ml-validation-design`, але причина інша:
не зміна системи обліку, а реакція на саму модель.
