# API: згода анотаторів і шум міток — перевірено 2026-07-18

Середовище: **Python 3.12.10 · scikit-learn 1.9.0 · statsmodels 0.14.6**.
Усе нижче — живі виклики, не документація.

## 1. Два анотатори: `cohen_kappa_score` (sklearn)

```python
from sklearn.metrics import cohen_kappa_score
cohen_kappa_score(ann1, ann2)                       # номінальні категорії
cohen_kappa_score(ann1, ann2, weights="linear")     # порядкові шкали
cohen_kappa_score(ann1, ann2, weights="quadratic")  # штраф за квадратом відстані
```

**`weights` тут — НЕ ваги спостережень**, а схема штрафу за відстанню між
категоріями. Для порядкових міток (1–5 зірок) `weights=None` втрачає інформацію:
розбіжність «1 проти 5» рахується так само, як «4 проти 5».

### `replace_undefined_by` — доступний у 1.9 (перевірено)

```python
cohen_kappa_score([0, 0], [0, 0], replace_undefined_by=0.0)   # → 0.0
```

Каппа невизначена, коли в обох наборах присутня лише одна спільна категорія
(`1 − p_e = 0`). У 1.9 параметр **є**, повертає задане значення й видає
`UndefinedMetricWarning` замість `nan`. Ручний `np.isnan`-фолбек більше не
потрібен (те саме, що зафіксовано в `ml-metric-choice`).

**Не плутати з κ як метрикою моделі.** `cohen_kappa_score(y_true, y_pred)` в
`ml-metric-choice` — це узгодження **моделі з міткою**. Тут — узгодження
**анотатора з анотатором**, тобто верхня межа першого. Функція одна, зміст
різний, і в звіті вони мають називатись по-різному.

## 2. Три і більше анотаторів: `fleiss_kappa` (statsmodels)

```python
from statsmodels.stats.inter_rater import fleiss_kappa, aggregate_raters
table, categories = aggregate_raters(labels)   # labels: (n_обʼєктів, n_анотаторів)
fleiss_kappa(table)
```

- `aggregate_raters` перетворює матрицю «об'єкт × анотатор» на «об'єкт ×
  скільки голосів за кожну категорію» — саме цього чекає `fleiss_kappa`.
- Перевірено: випадкові мітки (3 анотатори, 300 об'єктів) → **κ = 0.0133**;
  три анотатори з індивідуальною точністю 85% → **κ = 0.4397**.
- Fleiss не вимагає, щоб об'єкти оцінювали **ті самі** анотатори — лише щоб
  їх була однакова кількість. Для змінної кількості потрібен Krippendorff.

## 3. Чого в середовищі НЕМАЄ (перевірено імпортом)

| Пакет | Статус | Наслідок |
|---|---|---|
| `krippendorff` | **немає** | α Кріппендорфа недоступна; для рівної кількості анотаторів Fleiss — коректна заміна |
| `cleanlab` | **немає** | автоматичний пошук підозрілих міток (confident learning) недоступний; заміна — ручна процедура нижче |

### Заміна cleanlab у три рядки

Ідея confident learning відтворюється без пакета: рядки, де впевнене
передбачення **зі спліту, у якому рядок не навчався**, суперечить мітці, — це
кандидати на хибну мітку.

```python
from sklearn.model_selection import cross_val_predict
proba = cross_val_predict(est, X, y, cv=5, method="predict_proba")
suspect = (proba.max(1) > 0.95) & (proba.argmax(1) != y)   # ранжувати за max
```

**Обов'язково out-of-fold** — інакше модель просто відтворить власні навчальні
мітки й підозрілих не знайдеться. Це та сама вимога, що й до канарок у
`ml-overfitting-diagnosis`. Отриманий список — не «помилки», а **черга на
ручний аудит**: автоматичне перекидання міток за передбаченням моделі
перетворює мітку на вихід моделі (Крок 2 SKILL.md, мітка з попередньої моделі).

## 4. Ваги класів проти шуму міток — не плутати

`class_weight="balanced"` компенсує **дисбаланс**, а не шум. При асиметричному
шумі (пропущені позитиви) базова частка вже зміщена вниз — виміряно 0.3501
замість істинної 0.5001 — тож `balanced` рахуватиме ваги з **неправильної**
частки й закріпить помилку. Спершу оцінка `p` з аудиту, потім рішення про ваги
(`ml-metric-choice`, `ml-sampling-design`).

## 5. Дрібниця, що економить годину

`sklearn.metrics.confusion_matrix(y_ann1, y_ann2)` між двома анотаторами читається
так само, як між моделлю і міткою, і одразу показує **напрям** розбіжності:
позаддіагональна асиметрія означає, що один анотатор систематично щедріший на
позитивний клас. Це діагностика інструкції анотування, а не даних, — і
виправляється до навчання моделі, а не після.
