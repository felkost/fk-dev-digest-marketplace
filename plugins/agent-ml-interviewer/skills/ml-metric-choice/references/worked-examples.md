## Провал accuracy: діагностика хвороби

Класифікатор «хворий / здоровий», 1075 пацієнтів.

```
                  передбачено:
                  хворий    здоровий
істина: хворий      20 (TP)    5 (FN)
        здоровий    50 (FP) 1000 (TN)
```

```
Accuracy  = (20 + 1000) / 1075 = 1020/1075 = 0.94884   →  94.88%
Precision = 20 / (20 + 50)     = 20/70     = 0.28571   →  28.57%
Recall    = 20 / (20 + 5)      = 20/25     = 0.80000   →  80.00%
F1        = 2·0.2857·0.8 / (0.2857 + 0.8)  = 0.42105   →  42.11%
```

Тепер вироджений класифікатор, який **завжди** каже «здоровий»:

```
TP = 0,  FP = 0,  TN = 1000 + 50 = 1050,  FN = 20 + 5 = 25
Accuracy = 1050/1075 = 0.97674   →  97.67%
Recall   = 0/25 = 0
```

**97.67% > 94.88%.** Модель, що не робить нічого, виграє за accuracy у моделі, яка
знаходить 80% хворих. F1 при цьому падає з 42.11% до 0.

Що подавати замість: `balanced_accuracy` = (0.8 + 1000/1050)/2 = (0.8 + 0.9524)/2 =
**0.8762** для першої моделі проти **0.5** для виродженої. Різниця видима.

## Cohen's kappa: кредитний комітет

Два експерти незалежно оцінили 50 заявок «схвалити / відхилити». Збіглися у 35 з 50.

```
p₀ = 35/50 = 0.70          спостережувана згода
pₑ = 0.50                  очікувана за незалежності (з маргінальних часток)

κ = (p₀ − pₑ) / (1 − pₑ) = (0.70 − 0.50) / (1 − 0.50) = 0.20 / 0.50 = 0.40
```

Читання: 70% згоди звучить пристойно, але половину з них пояснює випадковий збіг.
Згода **понад випадкову** — лише 0.40.

Орієнтири інтерпретації κ (Landis & Koch): <0.20 слабка, 0.21–0.40 посередня,
0.41–0.60 помірна, 0.61–0.80 істотна, >0.80 майже повна. 0.40 — на межі
посередньої та помірної.

## Спам-фільтр: чому MCC стійкий до порогу

Лекція 2 показує таблицю метрик того самого класифікатора при трьох порогах.
Ключове спостереження: зі зростанням порогу accuracy, precision, recall і F1
змінюються — а **MCC лишається практично сталим**.

Причина у формулі: MCC використовує всі чотири комірки й нормується на добуток
усіх чотирьох маргінальних сум.

```
MCC = (TP·TN − FP·FN) / √((TP+FP)(TP+FN)(TN+FP)(TN+FN))
```

Precision і recall дивляться лише на стовпець і рядок відповідно, тому реагують на
зсув порогу асиметрично й у протилежні боки. MCC «бачить» усю таблицю.

**Практичний висновок:** якщо треба порівняти дві моделі, а поріг ще не зафіксовано —
беріть MCC або AUC. Якщо порівнювати за F1 на різних порогах, порівнюватимете
пороги, а не моделі.

## Вибір β: два сценарії

### Псувний товар на складі → β = ½

Модель прогнозує, чи буде попит на партію. FP («буде попит», а попиту немає) →
товар псується, прямий збиток. FN («не буде», а попит є) → недоотриманий прибуток,
менша втрата.

**FP дорожчий ⟹ важливіша precision ⟹ β < 1 ⟹ β = ½.**

Наслідок для порогу: поріг піднімається, модель обережніша в позитивних
передбаченнях.

### Контроль в аеропорту → β = 2

FP (хибна тривога) → додаткова перевірка, кілька хвилин затримки. FN (пропущена
загроза) → катастрофа.

**FN дорожчий ⟹ важливіший recall ⟹ β > 1 ⟹ β = 2.**

Наслідок: поріг опускається. Лекція 2 простежує ланцюжок явно —
**FN↓ ⟹ FP↑ ⟹ Precision↓**, і це прийнятна ціна.

### Перевірено на даних

На фікстурі `evals/golden/scores_imbalanced.csv` (7.3% позитивних):

| β | поріг | Recall | Precision |
|---|---|---|---|
| 0.5 | 0.3428 | 0.6420 | 0.8968 |
| 1.0 | 0.2901 | 0.6648 | 0.8603 |
| 2.0 | 0.1545 | 0.7443 | 0.5928 |
| 4.0 | 0.0630 | 0.8523 | 0.2664 |

Монотонність саме така, як передбачає теорія: більша β → нижчий поріг → вищий
recall ціною precision.

## PR проти ROC: спам на мільйоні листів

~10⁶ листів, серед них 100 справжніх спамів.

| | TP | FN | FP | TN | FPR | Precision |
|---|---|---|---|---|---|---|
| Алгоритм 1 | 90 | 10 | 10 | 999 890 | 0.00001 | **0.900** |
| Алгоритм 2 | 90 | 10 | 1 910 | 997 990 | 0.00191 | **0.045** |

Алгоритм 2 робить у **191 раз більше** хибних спрацювань. Для користувача це
різниця між «10 нормальних листів у спамі» і «1910 нормальних листів у спамі».

- **ROC каже, що вони майже однакові**: FPR 0.00001 проти 0.00191 — на графіку
  обидві точки притиснуті до нуля.
- **PR каже, що це прірва**: precision 0.900 проти 0.045.

Причина арифметична: FPR ділить на всі негативні (999 900), precision — лише на
передбачені позитивні (2 000). Величезний TN розбавляє FPR до непомітності.

Джерело: Saito T., Rehmsmeier M. (2015), *PLoS One*.

## Гармонійне проти арифметичного

Чому F1 = гармонійне середнє, а не арифметичне:

| P | R | арифметичне | гармонійне (F1) | коментар |
|---|---|---|---|---|
| 0.9 | 0.1 | 0.500 | **0.180** | перекіс покарано |
| 0.7 | 0.3 | 0.500 | 0.420 | |
| 0.6 | 0.4 | 0.500 | 0.480 | |
| 0.5 | 0.5 | 0.500 | 0.500 | збігаються лише при рівності |
| 1.0 | 0.0 | 0.500 | **0.000** | вироджений випадок |

Арифметичне середнє дає 0.5 усім рядкам — воно не розрізняє збалансований
класифікатор і той, що просто передбачає «позитивний» рідко й точно.

Фізична аналогія Лекції 2: середня швидкість на двох ділянках однакової довжини —
`2/(1/v₁ + 1/v₂)`, а не `(v₁+v₂)/2`. Якщо половину шляху їхати 100 км/год, а
половину 1 км/год, середня — 1.98 км/год, а не 50.5.
