# Аудит «RAG у проді»: розбір типового туторіал-каркасу

Дата перевірки: **2026-07-18**. Живі виміри зроблено в середовищі пака
(Python 3.12.10, Windows). Позначки: **[вимір]** — мій прогін, **[текст]** —
видно з коду джерела без запуску, **[джерело]** — чужий факт із посиланням.

Розбираний матеріал: Rizvi, *From Zero to LLMOps Hero: Your 101 Guide to Running
LLMs in Production*, Analytics Vidhya, 11.11.2025
(`analyticsvidhya.com/blog/2025/11/llms-in-production/`). Каркас: LangChain +
OpenAI + FAISS + FastAPI + Streamlit + ngrok + Docker, зібраний у Colab.

**Навіщо це в паку.** Стаття не дала жодного факту, придатного до цитування, —
її цінність у тому, що вона є **еталонним зразком поширеної хибної відповіді**
на питання «як вивести LLM у прод». Кожен її дефект — тиха відмова: помилки
немає, система просто не працює або не відтворюється. Це той самий клас, що
`None` в object-колонці і незважений scorer у `cross_validate`.

## 1. Незапінені версії: «відтворюваність», що ламає сама себе

Стаття генерує `requirements.txt` рядком
`fastapi\nuvicorn\nstreamlit\nlangchain\nopenai\nfaiss-cpu\npyngrok\nrequests\n`
— **жодного піна** — і поруч заявляє «Reproducibility → the same environment
everywhere».

**[вимір]** Чистий venv, `pip install langchain` → **1.3.14**. Усі шість
імпортів каркасу статті падають:

| Імпорт статті | Результат на langchain 1.3.14 |
|---|---|
| `from langchain.chains import RetrievalQA` | `ModuleNotFoundError: No module named 'langchain.chains'` |
| `from langchain.vectorstores import FAISS` | `ModuleNotFoundError` |
| `from langchain.llms import OpenAI` | `ModuleNotFoundError` |
| `from langchain.text_splitter import RecursiveCharacterTextSplitter` | `ModuleNotFoundError` |
| `from langchain.docstore.document import Document` | `ModuleNotFoundError` |
| `from langchain.embeddings import OpenAIEmbeddings` | `AttributeError: module has no attribute` |

**[джерело]** LangChain 1.0 виніс легасі-ланцюги в окремий пакет
`langchain-classic` (`from langchain_classic.chains import ...`); заміна
`RetrievalQA` — `create_retrieval_chain`
(docs.langchain.com/oss/python/migrate/langchain-v1).

**Наслідок, який варто проговорити вголос:** LangChain 1.0 вийшов **до** дати
публікації, тобто код був непрацездатним **у день виходу статті** — і саме
через відсутність пінів, тобто через ту саму рису, яку стаття рекламує як
відтворюваність. Незапінений `requirements.txt` у Dockerfile дає образ, зібраний
сьогодні ≠ образ, зібраний учора; Docker відтворює **шари**, не версії пакетів.

Що робити: пінити точні версії (`pip freeze` / lock-файл), а дату перевірки
тримати у файлі — рівно та сама дисципліна, що `references/api-2026.md` у
кожному скілі цього пака.

## 2. Два сервіси в одному `CMD`: контейнер «здоровий» при мертвому бекенді

Dockerfile статті:
`CMD bash -c "uvicorn <модуль>:app --port 8000 & streamlit run streamlit_app.py --server.port 8501"`

**[вимір]** Семантика оболонки, що це спричиняє (Docker-демон у середовищі не
запущений, тож міряно саме `bash`, а не статус контейнера):

```
bash -c "python crash.py & python -c 'print(...)'"   # crash.py робить sys.exit(3)
→ foreground alive
→ exit code усього CMD = 0
```

Код виходу складеної команди береться з **переднього** процесу; падіння
фонового не потрапляє нікуди. У контейнері це означає: PID 1 живий, `docker ps`
показує `Up`, healthcheck на порт UI зелений — а API мертвий. Плюс PID 1 тут —
`bash`, який не пересилає сигнали дітям, тож `docker stop` вироджується в
SIGKILL по таймауту.

Що робити: **один процес на контейнер** (два образи, два сервіси); якщо процесів
конче два — супервізор (`supervisord`, `tini` як init) і healthcheck на **кожен**
порт окремо, а не на один.

## 3. Секрет, запечений у шар образу

**[текст]** Ноутбучна версія записує ключ у код згенерованого файла:
`os.environ["OPENAI_API_KEY"] = "''' + os.environ["OPENAI_API_KEY"] + '''"`.
Docker-версія робить `COPY . .` по всьому робочому каталогу.

Наслідок: ключ потрапляє в **шар образу**. Видалення файла наступним шаром його
не прибирає — шари незмінні, значення дістається з історії образу. Якщо образ
десь публікується, ключ опублікований разом із ним.

Додатково `ENV OPENAI_API_KEY=${OPENAI_API_KEY}` у Dockerfile не робить того, що
здається: у контексті збірки змінна береться з `ARG`, а не з оточення хоста; без
оголошеного `ARG` це порожнє значення. Працює все лише завдяки `-e` при
`docker run`, тобто рядок `ENV` — декорація, що створює хибну впевненість.

Що робити: ключі — тільки рантайм-змінні або секрет-сховище; `.dockerignore` на
все, що може містити секрети; ніколи не писати ключ у файл, який лежить у
контексті збірки.

## 4. Код Docker-версії не міг працювати в принципі

**[текст]** Три незалежні розриви всередині самої статті:

1. Streamlit звертається до `GET http://localhost:8000/ask`, а FastAPI-додаток
   статті визначає **лише** `@app.get("/")`. Ендпоінта `/ask` немає в жодному
   сніпеті. Кожен запит → 404 → «Something went wrong with the backend».
2. `uvicorn` запускається на модулі
   `from_zero_to_llmops_hero_your_101_guide_to_running_llms_in_production` —
   слаг заголовка статті. Файла з такою назвою не створює жоден сніпет
   (створюються `app.py` і `streamlit_app.py`).
3. Через дефект №2 бекенд падає одразу, а через дефект №1 (пункт 2 цього файла)
   це нікому не видно.

Тобто розділ «Containerization» демонструє контейнер, який **не обслуговує
жодного запиту**, і подає це як перехід від «playing with LLMs» до «running LLMs
like a product». Урок для агента: скріншот працюючої демки і працюючий
контейнер — різні твердження; перевіркою є запит до API, а не `docker ps`.

## 5. Публічний тунель як «прод»

**[текст]** `ngrok.connect(8501)` дає ефемерний URL без автентифікації, який
стаття пропонує «поділитися з друзями». Позаду цього URL — ваш платний ключ
OpenAI без ліміту запитів, без квот на користувача, без логів. Це не деплой, а
відкритий рахунок; вартість — єдина метрика, що при цьому справді зросте.

Що робити: автентифікація на вході (навіть найпростіший токен), ліміт запитів на
клієнта, стеля витрат на боці провайдера, і облік вартості на запит як окрема
величина.

## 6. Розділи «моніторинг» і «оцінка»: назви без означень

Стаття називає `latency, retrieval accuracy, hallucination rate, version drift`
і переносить їх у «наступну статтю». Жодне з чотирьох не має означення, способу
виміру чи порога дії. Це рівно та помилка, проти якої побудований увесь пак:
**метрика без процедури виміру — не метрика**.

Мінімум, який робить із цього переліку роботу:

1. **Golden set** — питання з еталонними відповідями і з позначкою, який саме
   документ/чанк є правильним джерелом. Без нього наступні пункти неможливі.
2. **Ретрив міряти окремо від генерації.** Частка питань, де потрібний чанк
   узагалі потрапив у топ-`k` (recall@k) — це стеля всієї системи: якщо чанк не
   дійшов, жоден параметр LLM не врятує, а симптом виглядатиме як «галюцинує»
   і лікуватиметься промптом — не там.
3. **Тільки після цього** крутити `chunk_size`, `chunk_overlap`, `k`, модель
   ембедингів. У статті це `800/100/дефолтний k` без жодного слова про те,
   звідки числа; це **гіперпараметри**, і до них застосовна вся дисципліна
   пака — чесний eval-набір до оптимізації (`ml-validation-design`), вибір
   метрики під ціну помилки (`ml-metric-choice`), і те саме застереження про
   переоснащення на десяток прикладів, що в Кроці 4 цього скіла.
4. **Контамінація golden set** — питання, згенеровані тією ж моделлю з тих
   самих чанків, вимірюють переказ, а не пошук; процедура аудиту —
   `ml-validation-design/references/schemas.md`.

## 7. Дрібніше, але варте згадки

- `allow_dangerous_deserialization=True` при `FAISS.load_local` подано без
  жодного коментаря. Прапорець вмикає `pickle`-десеріалізацію: індекс із
  чужих рук = виконання чужого коду. У статті індекс свій, тож ризику немає —
  але й попередження немає, і читач переносить рядок у контекст, де індекс уже
  не свій.
- `from langchain.llms import OpenAI` — це **completions**-ендпоінт
  (лінія `gpt-3.5-turbo-instruct`), а не chat. Для RAG у 2026 це вибір проти
  власних інтересів.
- `temperature=0` подано як налаштування за замовчуванням і без прив'язки до
  моделі. Головний факт цього скіла: на актуальних моделях Claude параметр
  **видалено** (HTTP 400), на reasoning-моделях OpenAI він теж не приймається.
  Порада «постав temperature=0» має строк придатності й моделе-залежність,
  яких стаття не позначає.
- `python:3.10-slim` як база у листопаді 2025 — свідомо старіша гілка без
  причини; на дату цього аудиту тим паче.

## Що з цього джерела НЕ взято

Опис пайплайну (ingestion → chunking → embeddings → retriever → generation) і
перелік сервісів AWS — коректні, але це загальновідомий каркас, який модель
знає без пака; додавати його означало б роздувати скіл нульовою дельтою. За
задокументованим правилом пака дельту дає лише те, чого базлайн не знає або
знає застаріло, — тут це версійний обрив LangChain (пункт 1) і два тихі
механізми відмови (пункти 2–3).
