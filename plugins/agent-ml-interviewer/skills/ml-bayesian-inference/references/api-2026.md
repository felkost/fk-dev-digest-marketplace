# API: PyMC 6.1 / ArviZ 1.2

**Перевірено 2026-07-18 живими викликами** (`pm.sample`, `az.summary`,
`az.compare`, `az.loo`, `az.rhat`, `az.ess` — усе прогнано реально, не з
документації). Обидві бібліотеки встановлені щойно для цього скіла — до цього
в середовищі не було жодного PyMC/ArviZ. Книги пака писані під **PyMC2/PyMC3**
(2018), тож майже кожен виклик нижче — уже інший API, не косметична різниця.

## Головна пастка середовища: `cores=1`

```python
import pymc as pm
with pm.Model() as m:
    p = pm.Beta("p", alpha=1, beta=1)
    obs = pm.Binomial("obs", n=50, p=p, observed=33)
    idata = pm.sample(1000, tune=1000, chains=4, cores=1, random_seed=0,
                      progressbar=False)
```

**Виміряно:** з дефолтними `cores` (спроба паралелити ланцюги окремими
процесами) прогін підвис і не завершився за 3 хвилини в цьому shell-середовищі
(типова причина — обмеження на спавн підпроцесів у пісочниці; на звичайному
Windows це теж часто повільніше й крихкіше, ніж на Linux). З `cores=1`
(послідовні ланцюги в тому самому процесі) — **0.5 секунди** на ту саму
модель. Якщо код застряг на `pm.sample()` без видимої помилки — це перше, що
перевірити.

## `idata.groups` — властивість, не метод

```python
idata.groups   # ('/', '/posterior', '/sample_stats', '/observed_data', ...)
```

У старіших матеріалах трапляється `idata.groups()` (виклик) — у 6.1 це
**властивість**, виклик з дужками кидає `TypeError: 'tuple' object is not
callable`.

## `az.summary`: ETI89 за замовчуванням, не HDI94

**Найважливіша знахідка цього скіла.** Дефолтний виклик:

```python
az.summary(idata, var_names=["p"])
#     mean     sd eti89_lb eti89_ub  ess_bulk  ess_tail r_hat mcse_mean mcse_sd
# p  0.652  0.066     0.54     0.75      1722      2429  1.00    0.0016  0.0011
```

повертає колонки `eti89_lb`/`eti89_ub` — **89% equal-tailed interval**
(конвенція МакЕлріта, *Statistical Rethinking*), а не звичний із книг та
старіших версій ArviZ **94% HDI**. Щоб отримати класичний HDI:

```python
az.summary(idata, var_names=["p"], ci_kind="hdi", ci_prob=0.94)
#     mean     sd hdi94_lb hdi94_ub  ess_bulk  ess_tail r_hat mcse_mean mcse_sd
# p  0.652  0.066     0.53     0.77      1722      2429  1.00    0.0016  0.0011
```

**Параметр називається `ci_prob`/`ci_kind`, НЕ `hdi_prob`.** Виклик
`az.summary(idata, hdi_prob=0.94)` кидає
`TypeError: unexpected keyword argument 'hdi_prob'` — цей kwarg існував у
старіших ArviZ і в жодному коді 2026 року використовуватись не повинен.

## `az.hdi`: `prob=`, не `hdi_prob=`

```python
az.hdi(idata, var_names=["p"], prob=0.94)
```

Повертає `DataTree` з координатою `ci_bound` (`lower`/`upper`), не плаский
масив, як у старих версіях.

## Діагностика напряму (без `az.summary`)

```python
az.rhat(idata, var_names=["p"])["p"]     # скаляр/масив r_hat
az.ess(idata, var_names=["p"])["p"]      # ESS (bulk за замовчуванням)
int(idata.sample_stats["diverging"].values.sum())   # кількість дивергенцій
```

Пороги: `r_hat > 1.01` — не збіглося; ESS менший за кілька сотень при тисячах
семплів — сильна автокореляція, довіряти з обережністю.

## Posterior predictive check

```python
with model:
    pm.sample_posterior_predictive(idata, extend_inferencedata=True)
```

`extend_inferencedata=True` дописує групу `/posterior_predictive` прямо в
існуючий `idata` (перевірено: `idata.groups` після цього містить її). Без
цього прапорця повертається окремий об'єкт, і доведеться зводити вручну.

## `az.compare` / `az.loo`: потрібен `log_likelihood`, і спосіб його додати змінився

```python
idata = pm.sample(1000, tune=1000, chains=4, cores=1, random_seed=0,
                  progressbar=False)
pm.compute_log_likelihood(idata)          # ОКРЕМИЙ виклик, не kwarg до sample
cmp = az.compare({"model_a": idata_a, "model_b": idata_b})
loo = az.loo(idata_a, pointwise=True)
```

**Живо спіймана деприкація:** `pm.sample(..., idata_kwargs={"log_likelihood":
True})` кидає `FutureWarning: Passing log_likelihood via idata_kwargs is
deprecated ... Call pm.compute_log_likelihood(idata) instead`. Працює й досі
(поки що не помилка), але новий код писати одразу через
`pm.compute_log_likelihood(idata)` **після** `pm.sample()`, у тому самому
`with model:`-контексті (перевірено: додає групу `/log_likelihood` до
`idata.groups`, яку до виклику там не було).

Незалежно від способу — і старого, і нового — **отримати `log_likelihood`
заднім числом на вже завершеному `idata` без доступу до контексту моделі не
можна**: плануйте порівняння моделей до семплювання, не після.

`az.compare` сортує за **`elpd`, де БІЛЬШЕ — краще** (log predictive density) —
це протилежний знак до класичної формули з книг `WAIC = −2·lppd + 2·p_WAIC`
(де МЕНШЕ — краще). Читаючи вивід `az.compare`/`az.loo` після книги, легко
переплутати напрям порівняння — перевіряйте колонку `rank` (0 = найкраща), а
не сирий знак числа.

**Пастка Pareto-k, перевірена наживо:** LOO вимагає правдоподібності **на
рівні окремого спостереження**. Модель із агрегованим `Binomial(n=50, k=33)`
(одне «спостереження») дає Pareto-k «дуже погано» (1 зі 100% у категорії
`(1, Inf) bad`) — LOO вироджується, бо «прибрати одне спостереження» тут
означає прибрати єдине. Та сама модель через 50 окремих `Bernoulli`
(50 спостережень) дає **всі 50 Pareto-k «добре»** і коректний `elpd_loo`.
Якщо LOO/WAIC поводяться дивно на, здавалося б, правильній моделі — перше, що
перевірити: чи правдоподібність не агрегована там, де мала б бути
поспостережною.

## Ієрархічна модель: центрована проти нецентрованої (перевірено)

```python
# центрована — схильна до дивергенцій при малих групах
theta = pm.Normal("theta", mu=mu, sigma=tau, shape=J)

# нецентрована — лік
theta_offset = pm.Normal("theta_offset", 0, 1, shape=J)
theta = pm.Deterministic("theta", mu + theta_offset * tau)
```

На 8 групах по 5 спостережень (`target_accept=0.8`, інакше дефолт):
центрована дала **2 дивергенції**, нецентрована — **0**. Моделі математично
еквівалентні, різна лише геометрія постеріора, яку бачить семплер.
