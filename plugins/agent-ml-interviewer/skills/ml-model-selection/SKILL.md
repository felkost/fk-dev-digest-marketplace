# Обґрунтований вибір сімейства моделі

Вибір моделі — це **рішення за обмеженнями**, а не змагання на лідерборді.
Спершу відсіюємо за вимогами задачі, і лише те, що лишилось, порівнюємо чесно.

**Відповіді українською; ідентифікатори й код — англійською.**

## Не відповідає на

- **Який тип задачі взагалі** → `ml-task-framing` (сюди приходять із типом).
- **Ручки обраного сімейства** → `ml-tree-ensemble-params`, `ml-linear-regularization`,
  `ml-clustering-k`, `rl-hyperparameters`, `nn-training-params` (мережі).
- **Як порівнювати кандидатів** → `ml-validation-design` (одна CV на всіх).
- **Сам байєсів workflow** (пріор, PyMC, MCMC-діагностика) → `ml-bayesian-inference`
  (тут — лише порівняння вже навчених моделей через WAIC/LOO, Крок 4a).

## Крок 0 — базлайн, який не є моделлю

До будь-якого сімейства: **константа/мажоритарний клас** і одна проста модель
(логістична або дерево `max_depth=3`). Без цього «покращення» нема від чого
міряти, а частина задач закривається правилом і моделі не потребує.

## Крок 1 — сім питань-обмежень

Відповіді звужують коло **до** експериментів:

1. **Пояснюваність.** Треба захищати кожен прогноз перед нетехнічною людиною
   чи регулятором? Тоді лінійна, дерево, kNN. Ансамблі й мережі — «чорні
   скриньки» (SHAP пояснює post hoc, але це не те саме, що прозора модель).
2. **Дані в пам'ять.** Влазять? Якщо ні — інкрементальні алгоритми
   (`SGDClassifier`, `partial_fit`, HistGradientBoosting) замість тих, що
   вимагають усього набору.
3. **n і d.** Мережі й бустинг тягнуть мільйони ознак; SVM з ядром — ні
   (O(n²)–O(n³) за пам'яттю й часом). Мало n і багато d → лінійні з регуляризацією.
4. **Типи ознак.** Тільки числові? Змішані? Багато категорій високої
   кардинальності? Дерева терплять категорії, лінійні й kNN вимагають кодування,
   а one-hot на високій кардинальності роздуває d.
5. **Лінійність межі.** Якщо класи розділяються лінійно — логістична або лінійний
   SVM дадуть майже той самий результат за мізерну ціну. Інакше — ядра, дерева,
   мережі.
6. **Час навчання.** Мережі — довго; лінійні й дерева — швидко; RF паралелиться
   по ядрах.
7. **Час прогнозу.** Лінійні, SVM і мережі прогнозують швидко; kNN платить на
   інференсі (шукає сусідів), глибокі/рекурентні мережі — теж.

Питання 1, 6 і 7 — **вимоги продукту**, а не смак: вони часто вбивають
найточніший варіант ще до першого fit.

## Крок 2 — сімейства й що вони вимагають

| Сімейство | Коли доречне | Підготовка даних | Ціна |
|---|---|---|---|
| Лінійна / логістична | базлайн завжди; лінійна межа; мале n; потрібна інтерпретація | **скейлінг обов'язковий** при регуляризації; кодування категорій | не бачить взаємодій, поки не задасте їх руками |
| Дерево рішень | прозорість; змішані типи; нелінійність | нічого (масштаб байдужий) | нестабільне, легко перенавчається |
| Random Forest | табличні дані за замовчуванням; шум; мало тюнінгу | нічого | пам'ять; «чорна скринька»; повільніший інференс |
| Градієнтний бустинг (HGB/XGB) | **дефолт для табличних даних**, коли ціль — точність | нічого; HGB тримає NaN і категорії нативно | більше ручок; чутливіший до перенавчання |
| kNN | мале n; локальна структура; прозорість «схожі випадки» | **скейлінг обов'язковий** | прокляття розмірності; дорогий інференс |
| SVM (лінійний) | висока d, мале n (тексти) | **скейлінг обов'язковий** | не дає ймовірностей напряму |
| SVM (RBF) | нелінійність при середньому n | **скейлінг обов'язковий** | O(n²)+ — не для великих n |
| Наївний Байєс | текст, лічильники, дуже швидкий базлайн | дискретизація/лічильники | припущення незалежності майже завжди хибне |
| Нейромережа | багато даних; сигнал/зображення/текст/послідовності; ознаки треба вчити | скейлінг; багато даних | час, тюнінг, непрозорість, потреба в GPU |

**Правило за замовчуванням для табличних даних:** лінійна як базлайн →
градієнтний бустинг як робочий кінь. Мережі на табличних даних середнього
розміру рідко перемагають бустинг і завжди коштують дорожче.

## Крок 3 — сигнатура даних → короткий список

- **n < 1000, табличні** → лінійні/регуляризовані, дерево, kNN. Бустинг перенавчиться,
  мережа не має на чому вчитись.
- **n великі, табличні, змішані типи** → HistGradientBoosting; RF як стійкий
  другий варіант.
- **Текст** → лінійна або наївний Байєс на TF-IDF як базлайн; трансформер, якщо
  базлайн упирається і є бюджет.
- **Зображення/аудіо/послідовності** → мережі (згорткові/рекурентні/трансформер);
  класичні алгоритми лише на витягнутих ознаках.
- **d ≫ n** → лінійні з L1/ElasticNet (вбудований відбір) — див.
  `ml-linear-regularization`.
- **Потрібні калібровані ймовірності** → лінійна; або будь-що + `CalibratedClassifierCV`.

## Крок 4 — як порівнювати кандидатів чесно

1. **2–3 сімейства з РІЗНИМ зміщенням** (напр. лінійне + деревне), не п'ять
   варіацій одного.
2. **Однакова схема CV** для всіх — інакше порівнюються схеми, а не моделі
   (`ml-validation-design`).
3. **Однакова метрика**, обрана заздалегідь (`ml-metric-choice`).
4. Порівнювати **із розкидом**: різниця, менша за одне стандартне відхилення по
   фолдах, не є різницею. За інших рівних — простіша модель (правило 1 SE).
5. Тюнінг — **після** відбору сімейства, не паралельно: інакше порівнюються
   бюджети пошуку.

Різні за природою сильні моделі іноді варто **об'єднати** (2–3 моделі, не сотні) —
але тільки якщо приріст переживає перевірку з п.4.

## Крок 4a — байєсова альтернатива: WAIC/LOO замість CV

Якщо кандидати вже навчені байєсово через PyMC (`ml-bayesian-inference`), для
порівняння не обов'язковий CV-розкид із Кроку 4 — є прямий аналог:

- **Бритва Оккама, кількісно.** Точність у межах вибірки (in-sample) завжди
  оптимістична — модель з рівно стількома параметрами, скільки точок даних,
  проходить крізь усі точки й не узагальнює нічого. Потрібна точність **поза**
  вибіркою, а держати test — розкіш, якої часто немає при малих n. WAIC/LOO
  оцінюють позавибіркову точність **без** відкладеного набору.
- **`az.compare({"a": idata_a, "b": idata_b})`** — ранжує моделі за `elpd`
  (**більше — краще**; протилежний знак до класичної формули з підручників
  `WAIC = −2·lppd + 2·p_WAIC`, де менше краще). Дивитись на колонку `rank`
  (0 = найкраща), не на сирий знак числа.
- **`az.loo(idata, pointwise=True)`** — надійніший практичний варіант (PSIS-LOO);
  дає діагностику Pareto-k на кожне спостереження.
- **Пастка, перевірена наживо:** LOO вимагає правдоподібності **на рівні
  окремого спостереження**. Агрегована `Binomial(n=50, k=33)` як одне
  «спостереження» дає Pareto-k «дуже погано» (0.9+, вироджено); ті самі дані як
  50 окремих `Bernoulli` — усі Pareto-k «добре». Дивна поведінка LOO на,
  здавалося б, правильній моделі — перше, що перевірити.
- Передумова: `idata` з `log_likelihood` (`pm.compute_log_likelihood(idata)`,
  одразу після семплювання — деталі й приклади в `ml-bayesian-inference`).

## Режими відмови

- **Вибір за test.** Модель, обрана за запечатаним набором, робить його
  валідацією; фінальне число стає оптимістичним.
- **kNN або SVM без скейлінгу** — обидва працюють на відстанях; ознака в гривнях
  затопить ознаку в роках.
- **Мережа на 500 рядках табличних даних** — параметрів більше, ніж спостережень.
- **Порівняння нетюнованої моделі А з тюнованою Б** — міряється праця, не сімейство.
- **Ігнорування вимог продукту** (латентність, пояснюваність) до вибору — потім
  доводиться викидати найкращу модель.
- **Ансамбль заради ансамблю** — складність, яку ніхто не супроводжуватиме.

## Що повідомити

1. Короткий список сімейств **і чим відсіяні решта** (за яким із семи обмежень).
2. Базлайн і його число — точка відліку.
3. Таблиця кандидатів: метрика ± розкид по фолдах на однаковій CV.
4. Обране сімейство з причиною, включно з непродуктивними причинами
   (пояснюваність, латентність), якщо вони переважили точність.
5. Що саме передається в сімейний скіл для тюнінгу.
